# オンボーディングワークフロー

> 新メンバーを「伴走」で主力選手に育てる -- 知識習得・実践・メンタリングの一気通貫フロー

## ワークフロー全体像

```
Phase 0               Phase 1               Phase 2                Phase 3              Phase 4
オンボーディング準備   基礎知識習得          実践トレーニング       メンタリング・        自立・卒業判定
                                                                    フィードバック
  |                     |                     |                      |                    |
  v                     v                     v                      v                    v
+--------------+   +----------------+   +------------------+   +-----------------+   +-----------------+
| ドキュメント  |-->| 技術・ビジネス  |-->| プロジェクト運用  |-->| 1on1 準備        |-->| 独立作業         |
|  蒸留        |   |  ・文化・運用   |   |  実践            |   |  振り返り        |   |  品質検証        |
| ウェルカム   |   | クイズ・ナビ   |   | 意思決定演習     |   |  フィードバック  |   |  卒業判定        |
|  コミュニケ  |   | 意思決定原則   |   | コミュニケーション|   |  成長計画        |   |  ナレッジ還元    |
|  ーション設計|   |  学習          |   |  実践            |   |                  |   |                  |
+--------------+   +----------------+   +------------------+   +-----------------+   +-----------------+
```

### 反復サイクル

Phase 2 と Phase 3 は反復して成長を加速させる。Phase 3 のフィードバックに基づき Phase 2 のトレーニングを調整する。

```
Phase 0 --> Phase 1 --> +-> Phase 2 --> Phase 3 -+ --> Phase 4
                        |                         |
                        +--- 追加トレーニング <---+
```

### 品質ゲート

| ゲート | 条件 | 不合格時のアクション |
|:---|:---|:---|
| Phase 0 → 1 | Culture/Biz/Tech/Ops の 4 カテゴリ蒸留済み、週別学習パス策定済み、メンター確定・1on1 スケジュール確保、ウェルカムメール送信済み | Phase 0 を継続 |
| Phase 1 → 2 | 基礎クイズ: Culture 10 問・Biz 10 問・Tech 15 問・Ops 10 問（計 45 問）で加重平均 70% 以上、既存 ADR 3 本の読解サマリーを提出済み | 弱点カテゴリを特定し、該当教材を再学習。再テストは 1 週間後 |
| Phase 2 → 3 | 実践課題 3 つ以上完了（依存関係マップ・練習 ADR・プレゼン骨格）、練習 ADR が 7 セクション構造を満たす、プレゼン骨格が WHY-WHAT-HOW-SO WHAT 構造 | Phase 2 で追加課題を実施。メンターがペアワークで支援 |
| Phase 3 → 4 | 1on1 を 2 回以上実施（KPT 記録あり）、応用クイズ: 各カテゴリ 20 問以上（計 80 問）で加重平均 80% 以上、練習 ADR の review で Major = 0、SMART 目標が設定済み | Phase 2-3 を反復。弱点領域に追加トレーニングを設計 |
| Phase 4 | 独立タスク 1 件以上を完了（review で Critical/Major = 0）、ナレッジ共有セッション 1 回実施、最終進捗レポート生成済み | Phase 3 に差し戻し。メンターがペア作業を増やす |

### タイムボックス目安

| Phase | 新卒（IT 未経験） | 中途エンジニア（即戦力） | 中途マネージャー |
|:---|:---|:---|:---|
| Phase 0 | 2-3 日（入社前） | 1-2 日（入社前） | 1 日（入社前） |
| Phase 1 | 3-5 週間 | 1-2 週間 | 1 週間 |
| Phase 2 | 2-3 週間 | 1-2 週間 | 1 週間 |
| Phase 3 | 2-4 週間（Phase 2 と反復） | 1-2 週間 | 1 週間 |
| Phase 4 | 1-2 週間 | 1 週間 | 3 日 |
| **合計** | **10-17 週間** | **5-9 週間** | **3-4 週間** |

| **onboarding** | 新人の背景に基づく学習パス設計、ロードマップ生成 | 「学習ロードマップを設計して」「オンボーディングを準備して」 |

### 成果物

## Phase 1: 基礎知識習得

### 概要

蒸留されたドキュメントに基づき、技術・ビジネス・文化・運用の 4 カテゴリを体系的に学習する。onboarding の学習ナビゲーション・クイズ機能で理解度を確認しながら進める。意思決定の基礎原則もこの段階で学ぶ。

**学習目標**:
- **L1**: Culture/Biz/Tech/Ops の 4 カテゴリの基礎知識を習得する
- **L2**: State Dimensions を理解し、ペルソナ定義に 5 次元（Belief/Value/Stance/Emotion/Communication）を適用できる

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **onboarding** | 学習ナビゲーション、理解度クイズ（4 択・○×・記述式）、スコアレポート | 「次に何を学べばいい？」「クイズを出して」「進捗を確認して」 |
| **distill** | 難解なドキュメントの要約・再構造化、知識サマリーの作成 | 「この技術文書を要約して」「ポイントを 3 つにまとめて」 |
| **decision-framework** | 意思決定の基礎原則の学習（ADR の読み方、トレードオフの考え方） | 「ADR の読み方を教えて」「意思決定の原則を学びたい」 |

### 学習順序

```
Week 1-2: Culture（5 本）--> 会社の「なぜ」を理解する
  |
  v
Week 2-3: Biz（3 本）----> 事業の「何を」を理解する
  |
  v
Week 3-4: Tech（7 本）---> 技術の「どうやって」を理解する
  |                         + 既存 ADR を読み、判断の型を学ぶ
  v
Week 4-5: Ops（6 本）----> 業務の「日常」を理解する
```

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 学習進捗レポート（プログレスバー付き） | `.md` | `30_proposal/onboarding/` |
| カテゴリ別スコアレポート | `.md` | `30_proposal/onboarding/` |
| 弱点分析・復習計画 | `.md` | `30_proposal/onboarding/` |
| 学習メモ（新人が記述） | `.md` | `project/20_notes/onboarding/` |

### チェックリスト（Phase 1 -> Phase 2 ゲート）

- [ ] Culture/Biz/Tech/Ops の 4 カテゴリを学習完了
- [ ] 全カテゴリの基礎クイズ正答率 70% 以上
- [ ] 用語集のドメイン用語を理解している（記述式クイズで確認）
- [ ] 既存 ADR を 3 本以上読み、判断の構造を理解している
- [ ] 学習メモを project/20_notes/ に記録している
- [ ] メンターとの初回 1on1 を完了している

### 次工程への引き渡し

```
「project-ops スキルでチームのフロー構造を学びたい」
「comm-craft スキルでステークホルダーへの報告方法を学びたい」
```

## Phase 2: 実践トレーニング

### 概要

基礎知識を実践に移す段階。プロジェクト運用（チームフロー・依存関係・DORA メトリクス）、意思決定演習（ADR 記述・トレードオフ分析）、コミュニケーション実践（聴衆分析・メッセージ構造化）を通じて「手を動かして学ぶ」。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **project-ops** | チームフロー構造の学習、依存関係分析、DORA メトリクスの読み方 | 「チームの依存関係を分析して」「DORA メトリクスを学びたい」 |
| **comm-craft** | ステークホルダーコミュニケーションの実践、聴衆分析、交渉テクニック | 「プレゼンの構成を考えて」「技術的な説明を準備して」 |
| **decision-framework** | ADR の記述演習、トレードオフ分析の実践、DACI チャート作成 | 「ADR を書く練習をしたい」「トレードオフを分析して」 |
| **onboarding** | 実践課題のシナリオ提供、応用クイズ | 「実践的なシナリオで学びたい」「応用クイズを出して」 |

### 実践トレーニングメニュー

| 課題 | 使用スキル | 完了基準 |
|:---|:---|:---|
| チームの依存関係マップを作成 | project-ops | 3 種の依存（Arch/Exp/Act）を識別 |
| DORA メトリクスの現状を評価 | project-ops | 4 メトリクスの測定値を記録 |
| 練習用 ADR を 1 本記述 | decision-framework | 7 セクション構造を網羅 |
| 技術選定のトレードオフ分析 | decision-framework | 3 候補以上、重み付きスコアリング |
| 技術提案のプレゼン骨格を作成 | comm-craft | WHY-WHAT-HOW-SO WHAT 構造 |
| 聴衆分析シートの作成 | comm-craft | 3 カテゴリの聴衆を分析 |

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 依存関係分析レポート（練習） | `.md` | `30_proposal/onboarding/` |

- [ ] コミュニケーションの WHY-WHAT-HOW-SO WHAT 構造を実践している

## Phase 3: メンタリング・フィードバック

### 概要

メンターとの 1on1、振り返り（KPT）、フィードバック交換を通じて成長を加速する。Phase 2 の実践結果に対する品質評価を行い、弱点を特定して追加トレーニングを設計する。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **onboarding** | 1on1 準備（KPT・SMART 目標）、進捗レポート生成、フォローアップ管理 | 「1on1 の準備を手伝って」「進捗レポートを生成して」 |
| **comm-craft** | フィードバックの伝え方の学習、Elastic Leadership 診断 | 「フィードバックの伝え方を練習したい」 |
| **review** | 実践課題（ADR・プレゼン骨格・コード）の品質レビュー | 「練習 ADR をレビューして」「課題の品質を確認して」 |

### メンタリングサイクル

```
1on1 準備（onboarding: KPT + SMART 目標設定）
  |
  v
フィードバック交換（comm-craft: 理由先行・協働的言語）
  |
  v
実践課題のレビュー（review: Critical/Major/Minor 分類）
  |
  v
成長計画の更新（onboarding: 弱点分析 --> 追加トレーニング設計）
  |
  v
Phase 2 へ戻り追加トレーニング（必要に応じて）
```

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 1on1 準備シート（KPT + SMART） | `.md` | `30_proposal/onboarding/` |
| 実践課題レビューレポート | `.md` | `30_proposal/onboarding/reviews/` |
| 成長計画・フォローアップ | `.md` | `30_proposal/onboarding/` |
| 進捗レポート（プログレスバー付き） | `.md` | `30_proposal/onboarding/` |

### チェックリスト（Phase 3 -> Phase 4 ゲート）

- [ ] メンターとの 1on1 を 2 回以上実施している
- [ ] KPT 振り返りで Keep/Problem/Try を整理している
- [ ] SMART 目標が設定され、進捗が追跡されている
- [ ] 実践課題の review で Major 以上の指摘が解消されている
- [ ] 応用クイズの正答率が 80% 以上
- [ ] メンターが Phase 4 への移行を承認している

### 次工程への引き渡し

```
「review スキルで独立作業の品質を検証して」
「onboarding スキルで卒業基準を確認して」
```

## Phase 4: 自立・卒業判定

### 概要

独立して業務を遂行できることを検証する最終段階。実際のタスクに取り組み、成果物の品質を review と data-validation で検証する。卒業基準を達成したらオンボーディング完了。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **review** | 独立作業の成果物（コード・ドキュメント・ADR）の品質レビュー | 「成果物をレビューして」「品質を確認して」 |
| **data-validation** | 成果物の数値整合性・スコアリング検証 | 「数値を検証して」「データの正確性を確認して」 |
| **onboarding** | 卒業基準の確認、最終進捗レポート生成、ナレッジ還元支援 | 「卒業基準を確認して」「最終レポートを生成して」 |

### 卒業基準

| 基準 | 評価方法 | 合格ライン |
|:---|:---|:---|
| 知識理解度 | 全カテゴリ応用クイズ | 正答率 80% 以上 |
| 成果物品質 | review スキルによるレビュー | Critical/Major = 0 件 |
| 意思決定力 | ADR の品質（Why の十分性） | decision-framework の基準を達成 |
| コミュニケーション力 | プレゼン骨格・聴衆分析の品質 | comm-craft のチェックリスト通過 |
| 独立作業能力 | メンター支援なしでタスクを完遂 | 1 タスク以上を独立で完了 |
| チーム貢献 | チームへのナレッジ還元 | 学んだことを 1 つ以上共有 |

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 独立作業の成果物 | 各種 | プロジェクト内 |
| 最終レビューレポート | `.md` | `30_proposal/onboarding/reviews/` |
| 卒業レポート（最終進捗サマリー） | `.md` | `project/60_snapshots/` |
| ナレッジ還元資料（学びの共有） | `.md` | `project/20_notes/` |

### チェックリスト（Phase 4 完了条件）

- [ ] 卒業基準の全項目を達成している
- [ ] 独立作業の成果物が review で Approved（Critical/Major = 0）
- [ ] 最終進捗レポートが生成されている
- [ ] メンターが卒業を承認している
- [ ] チームへのナレッジ還元（学びの共有）を実施した
- [ ] オンボーディング期間の振り返り（KPT）を完了した

## トラブルシューティング

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| Phase 0 で蒸留資料が膨大 | チームドキュメントが整理されていない | distill で優先カテゴリ（Culture -> Biz）から段階的に蒸留 |
| Phase 1 で技術文書が難しすぎる | 前提知識が不足 | onboarding で難易度を下げた基礎クイズから開始。視覚的学習手法を併用 |
| Phase 1 でクイズの点数が低い | 完璧主義・自己否定 | 「学習の途中経過。弱点が見えたことが成果」とフレーミング |
| Phase 1 で ADR が理解できない | 意思決定の経験不足 | decision-framework で ADR の読み方から段階的に学ぶ |
| Phase 2 で実践課題が進まない | 課題の粒度が大きすぎる | 30 分単位の小タスクに分割。ペアワークを導入 |
| Phase 2 で交渉演習が苦手 | コミュニケーション不安 | comm-craft の NG/OK パターンをロールプレイで練習 |
| Phase 3 で 1on1 が形骸化 | 準備不足 | onboarding の KPT テンプレートを事前に記入 |
| Phase 3 で成長が停滞 | モチベーション低下 | 進捗レポートで小さな成長を可視化する |
| Phase 3-2 の差し戻しが繰り返される | 業務とのギャップ | 実プロジェクトタスクをトレーニング課題に変更 |
| Phase 4 で独立作業の品質が低い | Phase 2-3 のトレーニングが不十分 | 弱点を特定し Phase 2-3 を追加反復。メンターのペア作業を増やす |
| メンタル面の不安が強い | 孤立感・自己否定 | 共感 + コアバリュー「絆」で寄り添い、人間のメンターへの相談を推奨 |

## クイックスタート: 典型的なオンボーディングの進め方

### パターン A: 新卒エンジニア（IT 未経験）-- 基本パス 5 週間

```
1. 「オンボーディング資料を蒸留して」           --> distill
2. 「基本パスで学習ロードマップを設計して」     --> onboarding
3. 「Culture カテゴリから学習を始めよう」       --> onboarding
4. 「基礎クイズを出して」                       --> onboarding
5. 「ADR の読み方を教えて」                     --> decision-framework
6. 「チームの依存関係を一緒に分析してみよう」   --> project-ops
7. 「1on1 の準備を手伝って」                    --> onboarding
8. 「練習 ADR をレビューして」                  --> review
9. 「卒業基準を確認して」                       --> onboarding
```

### パターン B: 中途 BD（BD 経験あり）-- Biz/Culture 優先

```
1. 「Biz と Culture を優先した学習パスを設計して」 --> onboarding
2. 「事業内容とサービス概要を蒸留して」            --> distill
3. 「顧客対応のシナリオクイズを出して」            --> onboarding
4. 「プレゼン構成を考えて」                        --> comm-craft
5. 「技術選定の判断材料を整理したい」              --> decision-framework
6. 「成果物をレビューして」                        --> review
```

### パターン C: 中途マネージャー -- 急ぎパス 1 週間

```
1. 「チーム文化と運用フローを蒸留して」            --> distill
2. 「急ぎパスでロードマップを設計して」            --> onboarding
3. 「チームの DORA メトリクスを把握したい」       --> project-ops
4. 「チームコミュニケーションを改善したい」        --> comm-craft
5. 「DACI チャートを作って」                      --> decision-framework
6. 「ボトルネックを特定して」                      --> project-ops
```

### パターン D: 中途エンジニア（即戦力）-- Tech/Ops 優先

```
1. 「急ぎパスで Tech/Ops 優先の学習パスを設計して」 --> onboarding
2. 「技術スタックの概要を蒸留して」                 --> distill
3. 「応用クイズで理解度を確認して」                 --> onboarding
4. 「ADR を書く練習をしたい」                       --> decision-framework
5. 「チームのフロー構造を分析して」                 --> project-ops
6. 「独立作業の成果物をレビューして」               --> review
```
