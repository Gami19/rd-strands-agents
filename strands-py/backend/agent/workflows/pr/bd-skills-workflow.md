# BD ワークフロー

> bd-skills スキルを中心に、案件発掘からクロージング・フォローまでを一気通貫で支援する。

## ワークフロー全体像

```
Phase 0           Phase 1              Phase 2             Phase 3
案件発掘・引合い  案件アセスメント     課題分析・競合分析  バリュープロポジション
                  + ヒアリング準備                         + 提案設計
  │                 │                    │                   │
  ▼                 ▼                    ▼                   ▼
┌─────────┐   ┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│ RFP受領 │──▶│ BANT確認      │──▶│ 課題4層分解   │──▶│ 技術×価値     │
│ 引合い  │   │ Go/No-Go判定  │   │ ペルソナ整理  │   │ マッピング    │
│ 相談    │   │ ヒアリング準備│   │ 競合4力分析   │   │ ROI試算       │
└─────────┘   └───────────────┘   │ 差別化戦略    │   │ ストーリー設計│
                                  └───────────────┘   └───────────────┘
                                                              │
                ┌─────────────────────────────────────────────┘
                ▼
          Phase 4                  Phase 5
          提案書作成 + DRM         クロージング + フォロー
            │                       │
            ▼                       ▼
      ┌───────────────┐       ┌───────────────┐
      │ 提案書生成    │       │ 受注後立上げ  │
      │ 構成図作成    │       │ 効果測定      │
      │ コピー磨き上げ│       │ 内製化支援    │
      │ DRM レビュー  │       │ 横展開・深掘り│
      │ 品質検証      │       │ 失注分析      │
      └───────────────┘       └───────────────┘
```

---

## 反復サイクル

Phase 4（提案書作成 + DRM）で差し戻しが発生した場合は Phase 3 に戻り修正する。
review で Critical/Major が残る場合も Phase 3 or 4 に差し戻す。

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → ┌→ Phase 4 → Phase 5
                                          │     │
                                          │     ├─ DRM Go     → Phase 5
                                          │     ├─ DRM 修正Go → 修正後 Phase 5
                                          │     └─ DRM 差し戻し
                                          │           │
                                          └───────────┘
```

- **Phase 3 → 4**: バリュープロポジションと提案設計が固まったら提案書作成へ
- **Phase 4 → 3**: DRM 差し戻しや review Critical 検出時は課題分析・提案設計からやり直し
- **Phase 4 → 5**: DRM Go / 修正 Go 後にクロージングへ

## 品質ゲート（Phase 間の通過条件）

| ゲート | 条件 | 不合格時のアクション |
|:---|:---|:---|
| Phase 0 → 1 | 顧客資料が project/10_inbox/ に取り込み済み、案件基本情報が記録済み | Phase 0 に戻り資料収集を継続 |
| Phase 1 → 2 | BANT 4 要素確認済み、Go/No-Go 判定済み、ヒアリング質問リスト準備済み | Phase 1 で情報不足を解消 |
| Phase 2 → 3 | 課題 4 層分解完了、競合分析完了、差別化ポイント 3 つ以内に絞り込み済み | Phase 2 に戻り深掘り |
| Phase 3 → 4 | ROI 定量化（2 項目以上）、3 キーメッセージ策定済み、MoSCoW 分類済み | Phase 3 に戻り設計を補完 |
| Phase 4 → 5 | 全章完成、review Critical/Major = 0、数値検証済み、DRM Go/修正 Go | Phase 4 修正 or Phase 3 差し戻し |
| Phase 5（完了） | 効果測定 KPI 策定済み、フォロー計画策定済み | — |

## ADR 記録タイミング

| Phase | ADR 記録タイミング | 記録内容の例 |
|:---|:---|:---|
| Phase 1 | Go/No-Go 判定時 | 案件 Go/No-Go の根拠、リスク受容判断 |
| Phase 2 | 技術方針決定時 | アーキテクチャ選定理由、競合差別化戦略 |
| Phase 3 | ROI 前提条件確定時 | ROI モデルのパラメータ、スコープ除外理由 |
| Phase 4 | DRM 判定時 | DRM Go/No-Go 結果、修正要件 |
| Phase 5 | 効果測定設計時 | KPI 選定根拠、測定手法の選択理由 |

ファイル命名規則: `project/50_decision_log/ADR-NNNN-<summary>.md`

## Phase 0: 案件発掘・引合い対応

### 概要

RFP の受領、引合い対応、口頭相談への初動を行う。顧客資料が PDF/DOCX/PPTX で提供される場合は、Markdown に変換してプロジェクトの `project/10_inbox/` に蓄積する。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **pdf-convert** | 顧客資料の Markdown 変換 | 「RFP を project/10_inbox に取り込んで」「PDF を変換して」 |
| **bd-skills** (Step 1) | 案件プロファイル初期作成 | 「案件情報を整理して」 |

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 顧客資料（Markdown 化済み） | `.md` | `project/10_inbox/` |
| 案件プロファイル（初期版） | `.md` | プロジェクトルート or `project/20_notes/` |

### チェックリスト（Phase 0 → Phase 1 ゲート）

- [ ] 顧客資料がすべて `project/10_inbox/` に取り込まれた
- [ ] 案件の基本情報（顧客名・業種・相談内容）が記録された
- [ ] 提案期限・RFP 期限を把握した

## Phase 1: 案件アセスメント + ヒアリング準備

### 概要

案件の Go/No-Go を判断し、顧客ヒアリングの準備を行う。BANT 確認で案件の質を評価し、ADS 質問テンプレートでヒアリング項目を準備する。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **bd-skills** (Step 1-2) | BANT 確認、Go/No-Go 判定、ヒアリング準備 | 「案件をアセスメントして」「Go/No-Go 判断をして」「ヒアリングを準備して」 |
| **distill** | 顧客資料の蒸留・構造化 | 「project/10_inbox の資料を project/20_notes に蒸留して」 |
| **drm** (Phase 1) | Go/No-Go 会議の台本生成 | 「Go/No-Go 会議を準備して」 |

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 案件アセスメントシート（BANT + Go/No-Go） | `.md` | `project/20_notes/` |
| 蒸留済み顧客要件 | `.md` | `project/20_notes/` |
| ヒアリング質問リスト | `.md` | `project/20_notes/` |
| DRM Phase 1 台本（Go/No-Go） | `.md` | `project/60_snapshots/` |

### チェックリスト（Phase 1 → Phase 2 ゲート）

- [ ] BANT の 4 要素を確認した
- [ ] Go/No-Go 判定を実施し、根拠を記録した
- [ ] 顧客資料を `project/20_notes/` に蒸留した
- [ ] ヒアリング質問リストを準備した
- [ ] 提案のマイルストーンを設定した

### 概要

顧客の課題を4層に分解し、競合状況を把握して差別化戦略を策定する。marketing-context スキルでペルソナ・競合の基盤情報を構築し、bd-skills スキルで BD 観点の分析を行う。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **bd-skills** (Step 2-3) | 課題4層分解、ペルソナ整理、4力分析、差別化戦略 | 「課題を構造化して」「競合分析して」「差別化ポイントを整理して」 |
| **marketing-context** | ペルソナ定義、競合分析の基盤構築 | 「マーケティングの前提を整理して」「ペルソナを定義して」 |
| **data-arch** | 技術アーキテクチャ選定の根拠 | 「データアーキテクチャを選定して」「DWH を設計して」 |
| **databricks** | Databricks 固有の技術優位性の整理 | 「Databricks の設計をして」 |

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 顧客課題構造化シート（4層分解） | `.md` | `project/20_notes/` |
| ペルソナ4象限マップ | `.md` | `project/20_notes/` |
| 競合分析・差別化マップ | `.md` | `project/20_notes/` |
| ポジショニングマップ | `.md` | `project/20_notes/` |
| 技術選定根拠（該当時） | `.md` | `project/50_decision_log/` |

### チェックリスト（Phase 2 → Phase 3 ゲート）

- [ ] 顧客課題を4層（表層→構造→根本→理想）で分解した
- [ ] ペルソナ4象限で関係者を整理した
- [ ] 競合を直接・間接・代替手段の 3 分類で特定した

| スキル | 役割 | トリガー例 |

| **delta-lake** | パフォーマンス・コスト最適化の根拠 | 「Delta Lake のテーブルを設計して」 |
| **unity-catalog** | ガバナンス設計の根拠 | 「Unity Catalog を設計して」 |

### 成果物

### チェックリスト（Phase 3 → Phase 4 ゲート）

## Phase 4: 提案書作成 + DRM

### 概要

提案書のドラフト作成、構成図生成、コピーライティング、品質検証（DRM + review + data-validation）を行う。このフェーズが最も多くのスキルを連携させる。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **bd-skills** (Step 6-7) | 提案書の章構成、エグゼクティブサマリー | 「提案書を作って」「エグゼクティブサマリーを作って」 |
| **diagram** | As-Is/To-Be 構成図、データフロー図、ロードマップ図 | 「構成図を描いて」「Azure 構成図を作成して」 |
| **marketing-copy** | 提案書のコピーライティング（リード文、CTA） | 「コピーを改善して」「エグゼクティブサマリーを磨いて」 |
| **review** | 提案書の 4 軸レビュー（正確性・一貫性・説得力・実現性） | 「提案書をレビューして」 |
| **data-validation** | 見積金額・ROI 数値の整合性検証 | 「見積の数値を検証して」「ROI の整合性を確認して」 |
| **drm** (Phase 2) | 提案書提出前レビュー会議の台本生成 | 「DRM アジェンダを作って」「提案レビュー会議を準備して」 |

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 提案書ドラフト（全章） | `.md` | `proposal/` |
| エグゼクティブサマリー（A4 1枚） | `.md` | `proposal/` |
| 構成図（As-Is/To-Be、データフロー等） | `.drawio` / `.drawio.svg` | `proposal/` |
| DRM Phase 2 台本 | `.md` | `project/60_snapshots/` |
| レビューレポート | `.md` | `proposal/` |
| 数値検証レポート | `.md` | `proposal/` |

### サブフロー: 品質ゲート

```
提案書ドラフト完成
  │
  ├─▶ review（4軸レビュー）──▶ Critical/Major = 0？
  │                                 ├─ Yes ──▶ 次へ
  │                                 └─ No  ──▶ 修正して再レビュー
  │
  ├─▶ data-validation（数値検証）──▶ 数値整合？
  │                                     ├─ Yes ──▶ 次へ
  │                                     └─ No  ──▶ 修正して再検証
  │
  └─▶ drm Phase 2（提案レビュー会議）
          │
          ├─ Go          ──▶ Phase 5 へ
          ├─ 修正 Go     ──▶ 修正後 Phase 5 へ
          ├─ 差し戻し    ──▶ Phase 4 再実施
          └─ No-Go       ──▶ 案件中止
```

### チェックリスト（Phase 4 → Phase 5 ゲート）

- [ ] 提案書の全章が作成済み
- [ ] 構成図が最低 2 つ生成されている
- [ ] エグゼクティブサマリーが A4 1枚に収まっている
- [ ] review の Critical/Major 指摘が 0 件
- [ ] 数値の整合性が検証済み
- [ ] DRM Phase 2 で「Go」または「修正Go（修正済み）」
- [ ] ADR（設計判断記録）が `project/50_decision_log/` に記録済み

## ディレクトリ構造テンプレート

BD 案件ごとに以下のディレクトリ構造を作成する。

> **Note**: プラットフォーム標準では知識関連ディレクトリは `project/` プレフィックスを使用する（`project/10_inbox/`、`project/20_notes/`、`project/50_decision_log/` 等）。BD 案件を `projects/<案件名>/` 配下で管理する場合も、各サブディレクトリの役割は同じ。

```
projects/<案件名>/
├── inbox/            ← Phase 0: 顧客資料（Markdown 化済み）
├── notes/            ← Phase 1-2: 蒸留済み要件、課題分析、競合分析
├── proposal/         ← Phase 3-4: バリュープロポジション、提案書、構成図
├── snapshots/        ← Phase 1,4: DRM 台本
│   ├── DRM1-<案件名>.md    ← Go/No-Go
│   └── DRM2-<案件名>.md    ← 提案書提出前レビュー
├── decision_log/     ← Phase 1-5: 設計判断、DRM 判定結果
│   ├── ADR-0001-*.md
│   └── ...
└── references/       ← 外部参照資料
```

## クイックスタート: 典型的な BD 案件の進め方

### パターン A: RFP ベースのフル提案

```
1. 「この RFP を project/10_inbox に取り込んで」   → pdf-convert
2. 「project/10_inbox の資料を project/20_notes に蒸留して」 → distill
3. 「案件をアセスメントして」                     → bd-skills Step 1
4. 「顧客の課題を構造化して」                     → bd-skills Step 2
5. 「競合分析して差別化ポイントを整理して」       → bd-skills Step 3
6. 「バリュープロポジションを作って ROI を試算して」 → bd-skills Step 4
7. 「提案ストーリーを設計して」                   → bd-skills Step 5
8. 「提案書を作って」                             → bd-skills Step 6
9. 「構成図を描いて」                             → diagram
10. 「提案書をレビューして」                      → review + data-validation
11. 「DRM アジェンダを作って」                    → drm
12. 「効果測定計画を作って」                      → bd-skills Step 8
```

### パターン B: 口頭相談からの案件化

```
1. 「案件情報を整理して Go/No-Go 判断をして」    → bd-skills Step 1
2. 「ヒアリングの質問を準備して」                 → bd-skills Step 2
3. （ヒアリング実施）
4. 「ヒアリング結果をもとに課題を構造化して」     → bd-skills Step 2
5. 以降はパターン A の Step 5 以降と同じ
```

### パターン C: コスト最適化案件（短期）

```
1. 「既存環境のコスト最適化を提案して」           → bd-skills Step 1 + 4
2. 「ROI を試算して」                             → bd-skills Step 4b
3. 「改善提案書を作って」                         → bd-skills Step 6
4. 「レビューして」                               → review
```

## トラブルシューティング

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| Phase 0 で PDF 変換品質が低い | 画像メインの PDF | OCR 有効化を指示、または手動テキスト入力 |
| Phase 1 で BANT が不明確 | 顧客との接点が薄い | ヒアリング質問リストを事前に準備し、ADS テンプレートで深掘り |
| Phase 1 で Go/No-Go が決まらない | 情報不足 | BANT の不足要素を特定し、追加ヒアリング計画を策定 |
| Phase 2 で課題が表層にとどまる | 4 層分解の深掘り不足 | 「なぜ？」を 5 回繰り返し、構造→根本→理想まで到達 |
| Phase 2 で競合情報が取得できない | 市場情報が乏しい | marketing-context の間接競合・代替手段分析を活用 |
| Phase 3 で ROI の定量化が困難 | ベースライン数値がない | 業界平均値やケーススタディから推計、前提条件を ADR に記録 |
| Phase 3 で 3 キーメッセージが決まらない | 差別化ポイントが多すぎる | ペルソナ 4 象限の最優先ターゲットに絞って選定 |
| Phase 4 で DRM が差し戻しを繰り返す | 根本的な提案設計の問題 | Phase 2 に戻り、課題構造と差別化を再分析 |
| Phase 4 で構成図の品質が低い | diagram への情報提供不足 | project/20_notes の技術選定根拠を diagram に明示的に入力 |
| Phase 4 で review Critical が大量 | 章間の整合性不足 | エグゼクティブサマリーから逆算して各章を修正 |
| Phase 5 で効果測定計画が抽象的 | KPI が定義されていない | Phase 3 の ROI 試算から逆算し、測定可能 KPI を設定 |
| Agent Teams でコンフリクト | 同一ファイルの並行編集 | Phase 4 の DRM1/DRM2 は並行可能だが、提案書本体は逐次作業 |
