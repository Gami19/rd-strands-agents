# PDF → Markdown 変換ワークフロー

> テキストにならなければ、知識にならない — すべてのプロジェクトは変換から始まる

## ワークフロー全体像

```
Phase 0              Phase 1            Phase 2           Phase 3
事前準備・評価        変換実行           品質検証          引き渡し・記録
  │                    │                  │                 │
  ▼                    ▼                  ▼                 ▼
┌──────────┐     ┌──────────────┐   ┌──────────────┐  ┌──────────────┐
│ ソース特定│────▶│ Docling 変換 │──▶│ 構造検証     │─▶│ inbox 配置   │
│ 特性評価  │     │ OCR 判定    │   │ テーブル確認 │  │ ADR 記録     │
│ 戦略決定  │     │ バッチ処理  │   │ エンコーディング│  │ distill 連携 │
└──────────┘     └──────────────┘   └──────────────┘  └──────────────┘
```

---

## 反復サイクル

Phase 1（変換）と Phase 2（検証）は品質が基準を満たすまで反復する。

```
Phase 0 → ┌→ Phase 1 → Phase 2 ─┐ → Phase 3
           │                      │
           └── 品質不足で再変換 ◀──┘
```

- **Phase 1 → 2**: 変換結果の構造・文字・テーブルを検証
- **Phase 2 → 1**: 検証で問題があれば設定を変更して再変換
- **Phase 3**: 品質基準クリア後に inbox/ に配置

---

## 品質ゲート（Phase 間の通過条件）

| ゲート | 条件 | 不合格時のアクション |
|:---|:---|:---|
| Phase 0 → 1 | ソースファイル特定済み、変換戦略決定済み、前提ツールインストール済み | Phase 0 に戻りツールセットアップ |
| Phase 1 → 2 | Markdown ファイルが出力された（0 バイトでない） | Phase 1 に戻りエラー原因を調査 |
| Phase 2 → 3 | 見出し構造が原本と一致、テーブル崩れなし、文字化けなし | Phase 1 に戻り OCR / エンコーディング設定を変更して再変換 |
| Phase 3（完了） | inbox/ に配置済み、変換メモが記録済み | — |

---

## ADR 記録タイミング

| Phase | ADR 記録タイミング | 記録内容の例 |
|:---|:---|:---|
| Phase 0 | 変換戦略の決定時 | OCR 使用判断の根拠、バッチ vs 個別の選択理由 |
| Phase 2 | 品質トレードオフ発生時 | テーブル構造の手動修正判断、画像除外判断 |

ファイル命名規則: `decision_log/ADR-NNNN-<summary>.md`

## Phase 0: 事前準備・評価

### 概要

変換対象のソースファイルを特定し、ファイルの特性（テキスト PDF / スキャン PDF / DOCX / PPTX）を評価して変換戦略を決定する。

### 前提条件チェック

| ツール | 確認コマンド | インストール |
|:---|:---|:---|
| Python 3.10+ | `python --version` | [python.org](https://python.org) |
| docling >= 2.70.0 | `pip show docling` | `pip install docling` or `uv pip install docling` |

### ソースファイル評価

| 特性 | 判定方法 | 変換戦略 |
|:---|:---|:---|
| テキスト PDF（テキスト選択可能） | テキストをコピー&ペーストできる | 標準変換（OCR 不要） |
| スキャン PDF（画像のみ） | テキスト選択不可 | OCR 有効化が必要 |
| DOCX | 拡張子 `.docx` | 標準変換 |
| PPTX | 拡張子 `.pptx` | 標準変換（スライドノート含む） |
| 100 ページ超の大容量 | ページ数確認 | チャンク分割変換 |

### チェックリスト（Phase 0 → Phase 1 ゲート）

- [ ] 変換対象ファイルを特定した
- [ ] ファイルの特性（テキスト/スキャン/DOCX/PPTX）を評価した
- [ ] Python 3.10+ がインストールされている
- [ ] docling がインストールされている
- [ ] 変換戦略を決定した（OCR 有無、バッチ有無）

---

## Phase 1: 変換実行

### 概要

pdf-convert スキルを使って Docling でソースファイルを Markdown に変換する。ファイルの特性に応じた変換モードを選択する。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **pdf-convert** | PDF/DOCX/PPTX → Markdown 変換 | 「PDF を変換して」「inbox に取り込んで」「RFP を Markdown にして」 |

### 変換モード

#### 単一ファイル変換

```
「この PDF を Markdown に変換して inbox に配置して」 → pdf-convert
```

#### バッチ変換（ディレクトリ内の全ファイル）

```
「このフォルダの PDF をすべて変換して inbox に入れて」 → pdf-convert
```

#### スキャン PDF（OCR 有効）

```
「この PDF は画像なので OCR で変換して」 → pdf-convert（OCR モード）
```

#### 大容量ファイル（100 ページ超）

50 ページ単位でチャンク分割を指示:

```
「この PDF は 200 ページあるので 50 ページずつに分けて変換して」 → pdf-convert
```

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 変換済み Markdown ファイル | `.md` | `inbox/` |

### チェックリスト（Phase 1 → Phase 2 ゲート）

- [ ] Markdown ファイルが出力された
- [ ] ファイルサイズが 0 でない
- [ ] エラーメッセージが出ていない

---

## Phase 2: 品質検証

### 概要

変換結果の品質を検証する。原本と比較して、見出し構造・テーブル・文字コード・画像参照の正確性を確認する。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **review**（補助） | 変換品質のレビュー | 「変換結果を確認して」 |

### 検証項目

| 検証項目 | 確認方法 | 合格基準 |
|:---|:---|:---|
| 見出し構造 | 原本の目次と Markdown の見出しを比較 | 全見出しが `#` / `##` / `###` で再現 |
| テーブル構造 | 原本のテーブルと Markdown テーブルを比較 | 列数・行数が一致、セル内容が正確 |
| 文字化け | Markdown ファイルを目視確認 | 文字化け・豆腐（□）なし |
| 画像参照 | 画像ファイルの存在確認 | 画像パスが有効、または画像除外を ADR に記録 |
| エンコーディング | ファイルのエンコーディング確認 | UTF-8（BOM なし） |
| ページ完全性 | 原本のページ数と変換後の内容量を比較 | 全ページの内容が含まれている |

### チェックリスト（Phase 2 → Phase 3 ゲート）

- [ ] 見出し構造が原本と一致する
- [ ] テーブルの列数・行数が正確
- [ ] 文字化け・豆腐文字がない
- [ ] 画像参照が有効（または除外を記録）
- [ ] エンコーディングが UTF-8

> **品質不合格の場合**: Phase 1 に戻り、OCR 設定の変更やエンコーディング指定を行って再変換する

---

## Phase 3: 引き渡し・記録

### 概要

検証済みの Markdown ファイルをプロジェクトの `inbox/` に正式配置し、変換に関する判断を記録する。後続の distill ワークフローへの引き渡し準備を行う。

### 使用スキル

| スキル | 役割 | トリガー例 |
|:---|:---|:---|
| **distill**（後続） | inbox の資料を notes に蒸留 | 「inbox の資料を notes に蒸留して」 |

### 成果物

| 成果物 | 形式 | 配置先 |
|:---|:---|:---|
| 変換済み Markdown（最終版） | `.md` | `inbox/` |
| 変換判断メモ（該当時） | `.md` | `decision_log/` |

### チェックリスト（Phase 3 完了条件）

- [ ] Markdown ファイルが `inbox/` に配置された
- [ ] ファイル名が命名規則に従っている
- [ ] OCR 判断や品質トレードオフがあった場合は ADR に記録した
- [ ] 後続の distill 作業が可能な状態になっている

## トラブルシューティング

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| `python --version` でエラー | Python 未インストール | [python.org](https://python.org) からインストール |
| `pip install docling` でエラー | pip のバージョンが古い | `python -m pip install --upgrade pip` で更新 |
| 変換結果が空ファイル | スキャン PDF でテキスト層がない | `--ocr` フラグで OCR 変換を実行 |
| テーブル構造が崩れる | 複雑な結合セルを含むテーブル | 結合セルが多い場合は手動修正。列ごとの分割変換も検討 |
| 日本語が文字化けする | PDF のフォント埋め込みなし | OCR モードで再変換。エンコーディングを UTF-8 に指定 |
| 数式が正しく変換されない | LaTeX 記法に非対応 | 数式部分を手動で LaTeX マークアップに修正 |
| 変換に時間がかかりすぎる | 大量ページ（100 ページ超） | 50 ページ単位でチャンク分割して並行実行 |
| 画像パスが参照切れ | 画像ファイルの出力先が異なる | 画像ファイルを `inbox/images/` に配置し、パスを相対参照に修正 |
| PPTX のスライドノートが欠落 | デフォルトでノート非抽出の場合 | `--include-notes` オプションを確認、または手動補完 |
| 変換後の Markdown が 1 ファイルに長すぎる | 全ページが単一ファイルに結合 | 章単位でファイル分割する後処理を実施 |
| DOCX の差し込み印刷フィールドが残る | Word のフィールドコードが変換される | 変換後にフィールドコード文字列を検索・削除 |
| macOS で docling がインストールできない | Homebrew Python との競合 | `python3 -m pip install docling` で明示的に Python3 を指定 |

## スキル発動マトリクス（全フェーズ × 全スキル）

| スキル | Ph.0 | Ph.1 | Ph.2 | Ph.3 |
|:---|:---:|:---:|:---:|:---:|
| **pdf-convert** | - | **主** | - | - |
| **review** | - | - | **補** | - |
| **distill** | - | - | - | **後続** |

**凡例**: **主** = フェーズの主要スキル、**補** = 補助的に使用、**後続** = 次ワークフローへの引き渡し、- = 不使用

## クイックスタートパターン

### パターン A: 単一 PDF の標準変換

```
1. 「この PDF を Markdown に変換して inbox に入れて」    → pdf-convert
2. （変換結果を目視確認）
3. 「inbox の資料を notes に蒸留して」                   → distill
```

### パターン B: ディレクトリ一括変換

```
1. 「このフォルダの PDF をすべて変換して inbox に入れて」 → pdf-convert（バッチモード）
2. 「変換結果を確認して」                                → review
3. 「inbox の資料を notes に蒸留して」                   → distill
```

### パターン C: スキャン PDF の OCR 変換

```
1. 「この PDF は画像なので OCR で変換して」              → pdf-convert（OCR モード）
2. （文字化けがないか重点的に確認）
3. 問題があれば「OCR 設定を変えて再変換して」            → pdf-convert
```

### パターン D: DOCX / PPTX の変換

```
1. 「この Word ファイルを Markdown に変換して」           → pdf-convert
2. （テーブル・箇条書きの構造を確認）
3. 「inbox の資料を notes に蒸留して」                   → distill
```
