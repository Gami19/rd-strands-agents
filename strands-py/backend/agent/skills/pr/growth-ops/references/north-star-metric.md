# North Star Metric（NSM）設計ガイド

> プロダクトのコアバリューを表す単一の指標を定義し、チーム全体の方向性を統一する。
> SKILL.md の Step 2 で使用する。

---

## 1. North Star Metric（NSM）とは

### 定義

North Star Metric は、プロダクトが顧客に提供する**コアバリューを表す単一の指標**である。Sean Ellis が提唱し、Amplitude がフレームワーク化した。

**NSM の役割**:
- プロダクト戦略とチームの意思決定を統一する
- 売上やユーザー数といった結果指標ではなく、**顧客への価値提供**を測る
- 全チームが「何を増やすべきか」について共通認識を持てる

**NSM ≠ 売上指標**: 売上は顧客がプロダクトから価値を得た**結果**にすぎない。NSM は価値提供そのものを測る先行指標であり、NSM が成長すれば売上は後から付いてくる。

### ビジネスゲーム分類

Amplitude のフレームワークでは、ビジネスを 3 つの「ゲーム」に分類し、それぞれに適した NSM のパターンがある。

| ゲームタイプ | 説明 | 顧客が求める価値 | NSM の傾向 |
|:---|:---|:---|:---|
| **Attention** | ユーザーの時間・注意を獲得する | エンターテインメント、情報 | エンゲージメント系（滞在時間、DAU） |
| **Transaction** | 取引・購買を促進する | 便利な購買体験、品揃え | 取引量系（GMV、購買回数） |
| **Productivity** | 業務効率を向上させる | 時間短縮、品質向上 | 生産性系（タスク完了数、処理時間） |

### 各ゲームタイプの NSM 例

| ゲーム | 企業例 | NSM |
|:---|:---|:---|
| Attention | Spotify | Weekly Active Listeners |
| Attention | Netflix | 月間視聴時間中央値 |
| Attention | Medium | 月間総閲覧時間 |
| Transaction | Airbnb | 予約泊数 |
| Transaction | Amazon | 月間購買回数 |
| Transaction | Shopify | GMV（流通取引総額） |
| Productivity | Slack | 週間メッセージ送信組織数 |
| Productivity | Notion | 週間アクティブチーム数 |
| Productivity | Figma | 週間コラボレーションセッション数 |

## 2. NSM 選定の 7 基準

NSM 候補を以下の 7 基準で評価する。すべて「Yes」であることが理想。

| # | 基準 | 説明 | 検証方法 |
|:---|:---|:---|:---|
| 1 | **顧客価値を反映** | この指標が増えると、顧客が受ける価値も増えるか | 顧客インタビューで確認 |
| 2 | **戦略と整合** | 自社のプロダクトビジョン・戦略方針と一致するか | 経営チームとの合意 |
| 3 | **先行指標** | 売上・チャーンより前に変動するか（遅行指標は NG） | 時系列相関分析 |
| 4 | **影響可能** | チームのアクションで変化させられるか | 施策 → 指標の因果関係 |
| 5 | **測定可能** | 正確にかつ頻度高く計測できるか | データパイプラインの確認 |
| 6 | **理解可能** | 全チーム・経営陣が直感的に理解できるか | 説明テスト（30秒で説明可能か） |
| 7 | **操作困難** | ゲーミング（指標のハック）が困難か | レッドチーム思考で検証 |

### 評価テンプレート

```markdown
## NSM 候補評価

| 基準 | 候補A: [名前] | 候補B: [名前] | 候補C: [名前] |
|:---|:---:|:---:|:---:|
| 顧客価値を反映 | ○/△/× | ○/△/× | ○/△/× |
| 戦略と整合 | ○/△/× | ○/△/× | ○/△/× |
| 先行指標 | ○/△/× | ○/△/× | ○/△/× |
| 影響可能 | ○/△/× | ○/△/× | ○/△/× |
| 測定可能 | ○/△/× | ○/△/× | ○/△/× |
| 理解可能 | ○/△/× | ○/△/× | ○/△/× |
| 操作困難 | ○/△/× | ○/△/× | ○/△/× |
| **合計 ○ 数** | _ | _ | _ |

### 判断
- ○ が 6-7: 強い候補 → 採用
- ○ が 4-5: 改善余地あり → 定義の調整を検討
- ○ が 3 以下: 不適 → 別の候補を検討
```

### よくある失敗パターン

| 失敗 | 問題 | 改善 |
|:---|:---|:---|
| 売上を NSM にする | 遅行指標、顧客価値を反映しない | 売上の先行指標（利用頻度等）を選ぶ |
| 登録ユーザー数を NSM にする | 蓄積型で増加し続け、改善の感度が低い | アクティブユーザー数に変更 |
| 複数の NSM を設定 | 方向性が分散し、優先順位が曖昧に | 1 つに絞る。他は Input Metrics に |
| バニティメトリクス（PV 等） | 顧客価値との相関が薄い | 価値体験に直結する指標に変更 |

## 3. Input Metrics コンステレーション

### 概要

NSM を 1 つ定めたら、それを**分解した 3-5 の入力指標（Input Metrics）**を定義する。各チームが自分の担当 Input Metric を改善することで、全体の NSM が成長する。

```
                      ┌─ Input 1: 新規獲得数 ← Growth Team
                      │
NSM ─── f(inputs) ────┼─ Input 2: 活性化率 ← Product Team
                      │
                      ├─ Input 3: リテンション率 ← CS Team
                      │
                      └─ Input 4: 機能利用深度 ← Engineering Team
```

### 分解テンプレート

```markdown
## NSM コンステレーション

### North Star Metric
**[NSM 名]**: [定義]
- 計測頻度: [日次/週次/月次]
- データソース: [プロダクトログ/分析DB]

### Input Metrics

| # | Input Metric | 定義 | オーナー | 計測頻度 | 現在値 | 目標値 |
|:---|:---|:---|:---|:---|:---|:---|
| 1 | [指標名] | [計算式・定義] | [チーム名] | [頻度] | [値] | [値] |
| 2 | [指標名] | [計算式・定義] | [チーム名] | [頻度] | [値] | [値] |
| 3 | [指標名] | [計算式・定義] | [チーム名] | [頻度] | [値] | [値] |
| 4 | [指標名] | [計算式・定義] | [チーム名] | [頻度] | [値] | [値] |

### 関係式
NSM ≈ Input1 × Input2 × Input3 × Input4
```

### 具体例: Spotify

```
NSM: Weekly Active Listeners（週間アクティブリスナー数）

Input 1: 新規リスナー獲得数        ← Growth / Marketing
Input 2: 休眠リスナー復帰率        ← Re-engagement Team
Input 3: 週次リテンション率        ← Product / Personalization
Input 4: プレイリスト追加率        ← Content / Recommendation

WAL ≈ (既存リスナー × リテンション率) + 新規獲得 + (休眠 × 復帰率)
```

### 具体例: BtoB SaaS（データプラットフォーム）

```
NSM: Weekly Query Users（週間クエリ実行ユーザー数）

Input 1: 新規ワークスペース作成数  ← Sales / Onboarding
Input 2: オンボーディング完了率    ← CS / Product
Input 3: 週次利用リテンション率    ← Product / Engineering
Input 4: チーム展開率（1→N）      ← CS / Growth

WQU ≈ (既存ユーザー × リテンション) + (新規WS × 完了率 × 平均ユーザー数)
```

## 5. NSM → OKR → story-map 接続パターン

### 接続フロー

```
NSM: Weekly Active Teams
  │
  ├─→ Objective: 顧客のチーム利用率を倍増させる
  │     │
  │     ├─ KR1: 新規チーム作成数を月 200 → 400 に（Input 1 対応）
  │     │     └─ Feature: チーム作成ウィザードの簡略化
  │     │     └─ Feature: テンプレートギャラリーの追加
  │     │
  │     ├─ KR2: オンボーディング完了率を 40% → 60% に（Input 2 対応）
  │     │     └─ Feature: インタラクティブチュートリアル
  │     │     └─ Feature: プログレスバーの追加
  │     │
  │     └─ KR3: 週次リテンションを 65% → 80% に（Input 3 対応）
  │           └─ Feature: 週次サマリーメール
  │           └─ Feature: 未利用機能のレコメンド
  │
  └─→ story-map の Activity/Feature に展開
```

### 変換テンプレート

```markdown
## NSM → OKR 変換シート

### NSM
- 指標名: [Weekly Active Teams]
- 現在値: [1,200]
- 目標値: [2,400]（6 ヶ月後）

### OKR

**Objective**: [NSM の成長を自然言語で表現]

| Key Result | Input Metric | 現在値 | 目標値 | 担当チーム |
|:---|:---|:---|:---|:---|
| KR1: [具体的な成果指標] | [Input 1] | [値] | [値] | [チーム] |
| KR2: [具体的な成果指標] | [Input 2] | [値] | [値] | [チーム] |
| KR3: [具体的な成果指標] | [Input 3] | [値] | [値] | [チーム] |

### story-map 展開

| KR | Activity | Feature | User Story |
|:---|:---|:---|:---|
| KR1 | [ユーザー行動] | [機能名] | As a [ユーザー], I want to [行動] so that [価値] |
| KR2 | [ユーザー行動] | [機能名] | As a [ユーザー], I want to [行動] so that [価値] |
| KR3 | [ユーザー行動] | [機能名] | As a [ユーザー], I want to [行動] so that [価値] |
```

### story-map スキルとの連携

NSM → OKR で KR を導出したら、`story-map` スキルを使って Feature → User Story → Task に分解する。

| ステップ | 使用スキル | 入力 | 出力 |
|:---|:---|:---|:---|
| NSM 定義 | growth-ops | ビジネスモデル分析 | NSM + Input Metrics |
| OKR 設定 | growth-ops | NSM + 目標値 | OKR シート |
| Feature 分解 | story-map | KR → Activity → Feature | ストーリーマップ |
| 施策実行 | growth-ops | 実験設計・分析 | 実験レポート |

## 7. NSM 導入のベストプラクティス

### 導入ステップ

| # | ステップ | 所要期間 | 成果物 |
|:---|:---|:---|:---|
| 1 | ビジネスゲームの分類 | 1 日 | ゲームタイプの合意 |
| 2 | NSM 候補の洗い出し（3-5 個） | 1 週間 | 候補リストと評価シート |
| 3 | 7 基準での評価・スコアリング | 2-3 日 | 評価マトリクス |
| 4 | データ計測可能性の検証 | 1 週間 | データパイプライン確認結果 |
| 5 | 経営チーム・全チームでの合意 | 1 日 | NSM 決定ドキュメント |
| 6 | Input Metrics の分解 | 3-5 日 | コンステレーションマップ |
| 7 | ダッシュボード構築 | 1-2 週間 | NSM ダッシュボード |
| 8 | 全社への展開・教育 | 1 週間 | キックオフ資料 |

### NSM 変更のタイミング

NSM は頻繁に変えるべきではないが、以下の条件では変更を検討する:

| 条件 | 理由 | 例 |
|:---|:---|:---|
| ビジネスモデルの転換 | ゲームタイプ自体が変わる | 広告モデル → サブスクモデル |
| PMF 達成前後 | フェーズにより重要指標が変わる | サインアップ数 → アクティブユーザー数 |
| 市場環境の変化 | 競争軸が変わった | 成長速度 → 収益性 |
| NSM が天井に達した | 改善余地がなくなった | DAU が市場上限に到達 |

**注意**: NSM を変更する場合、過去データとの連続性を保つため、旧 NSM も一定期間は並行計測すること。

### 組織浸透のコツ

| やるべきこと | やってはいけないこと |
|:---|:---|
| 全社ミーティングで NSM の意味を繰り返し説明 | 数値だけをダッシュボードに貼って放置 |
| 各チームの OKR と NSM の接続を明示 | 「NSM を上げろ」と目標だけ伝える |
| NSM の動きを週次で全社共有 | 月次レポートでのみ言及 |
| 成功事例を「NSM への貢献」で語る | 個別施策の成果を NSM と無関係に報告 |
| Input Metrics のオーナーを明確にする | 全チームに全指標の責任を持たせる |

## 8. NSM のアンチパターン集

### パターン 1: バニティメトリクスの罠

```
× 累計登録ユーザー数 500,000
→ 増え続ける一方の指標は改善の感度が低い

○ 週間アクティブユーザー数 12,000
→ 施策の効果がリアルタイムに反映される
```

### パターン 2: 複合指標の罠

```
× エンゲージメントスコア = (DAU × セッション時間 × 機能利用数) / 3
→ 何が変動したかわからない、チームが影響を与えにくい

○ 週間アクティブチーム数
→ シンプルで全員が理解でき、直接改善アクションを取れる
```

### パターン 3: 売上直結の罠

```
× MRR（月次経常収益）
→ 遅行指標。MRR が下がった時にはすでに手遅れ

○ 週間有料機能利用率
→ MRR の先行指標。低下を早期検知してチャーン防止
```

### パターン 4: ゲーミング可能な指標の罠

```
× ページビュー数
→ 自動リフレッシュ、リダイレクトで容易に操作可能

○ 記事完読率
→ 実際のユーザー価値体験を反映し、操作が困難
```

### パターン 5: チーム分断の罠

```
× チームごとに異なる NSM を設定
→ 全社の方向性が分散し、リソース配分が非効率に

○ 1 つの NSM + チーム別 Input Metrics
→ 全社で方向性を統一しつつ、各チームの責任範囲を明確化
```

```markdown

- [ ] 7 基準の評価で ○ が 6 個以上か
- [ ] Input Metrics が 3-5 個に分解されているか
- [ ] 各 Input Metric にオーナーチームが割り当てられているか
- [ ] ダッシュボードで日次/週次に可視化されているか
- [ ] アラート閾値が設定されているか
- [ ] 全チームが NSM の定義と意味を理解しているか
- [ ] OKR の Key Results が Input Metrics と整合しているか
- [ ] 四半期に 1 回の妥当性レビューが予定されているか
```
