# Skill: Growth Ops（グロースオペレーション）

> **計測なき改善は推測にすぎない — データで全てのファネルを最適化せよ**

## strands-py / Strands Agent での利用

`marketing-context.md`・ファネル／広告データ・レポートは **`file_read` / `file_write`** とプロジェクト workspace（`inbox` / `notes` / `proposal` 等）を正とする。references 内の Python 例はエージェント自動実行前提ではない。詳細は [references/strands-py-runtime.md](references/strands-py-runtime.md) を参照。

## Step 1: marketing-context 確認とファネル定義

プロジェクト内の `marketing-context.md` を読み込み、グロースオペレーションの前提を確認する。

### 1a. marketing-context.md の読み込み

以下の情報を確認する:

| 項目 | 確認内容 | グロース施策への影響 |
|:---|:---|:---|
| **カスタマージャーニー** | ファネルの 5 段階定義 | 分析対象の明確化 |
| **KPI・目標** | NORTH STAR メトリクス | 最適化の方向性 |
| **ペルソナ** | ターゲット像と課題 | 施策の訴求軸 |
| **競合分析** | 競合の強み・弱み | 差別化ポイント |

### 1b. marketing-context.md が存在しない場合

最低限のコンテキストを直接ヒアリングする:

1. **プロダクト**: 何を提供しているか（SaaS / EC / アプリ）
2. **ビジネスモデル**: 収益構造（サブスクリプション / 従量課金 / 売り切り）
3. **現在のファネル**: 各段階の概算数値
4. **最大の課題**: 今最も改善したいポイント

### 1c. ファネルの定義

プロダクトに応じたファネルモデルを選定する:

| モデル | 段階 | 最適なプロダクト |
|:---|:---|:---|
| **AARRR**（パイレートメトリクス） | Acquisition → Activation → Retention → Revenue → Referral | SaaS、アプリ |
| **AIDA** | Attention → Interest → Desire → Action | EC、LP |
| **TOFU-MOFU-BOFU** | 認知 → 検討 → 購買 | BtoB リードジェネレーション |

詳細なテンプレートは [funnel-analysis-templates.md](references/funnel-analysis-templates.md) を参照。

**チェックリスト**:
### 品質ガードレール

**CRITICAL**: 実験結果は統計的有意性を担保した上で判断すること
**必須**: ベースライン指標を施策実行前に必ず設定すること

- サンプルサイズ不足のまま A/B テスト結果を確定してはならない
- 相関関係と因果関係を混同して施策判断をしないこと

品質基準:
- A/B テスト期間 最大 4 週間
- 最小サンプルサイズ ≥1000

- [ ] marketing-context.md の有無を確認した
- [ ] カスタマージャーニーと KPI を把握した
- [ ] プロダクトに適したファネルモデルを選定した
- [ ] 現在の最大の課題を特定した

## Step 2: ファネル分析・ボトルネック特定

ファネルの各段階を定量的に分析し、最大のボトルネックを特定する。

### 2a. ファネルデータの収集と可視化

各段階の母数・転換数・転換率を整理し、ファネル図として可視化する。

### 2b. ボトルネックの特定

各段階の転換率を業界ベンチマークと比較し、ギャップが最大の段階を特定する。
ベンチマーク詳細は [funnel-analysis-templates.md](references/funnel-analysis-templates.md) を参照。

### 2c. コホート分析

月別コホートのリテンション推移を追跡し、リテンションカーブが平坦になるポイント（安定化ポイント）を特定する。コホートテンプレートは [funnel-analysis-templates.md](references/funnel-analysis-templates.md) を参照。

### 2d. セグメント別分析

流入チャネル別・デバイス別・ペルソナ別・プラン別で転換率を比較し、高パフォーマンスセグメントと低パフォーマンスセグメントを特定する。

**チェックリスト**:
- [ ] ファネルの全段階の数値を収集した
- [ ] 各段階の転換率をベンチマークと比較した
- [ ] 最大のボトルネック段階を特定した
- [ ] コホート分析でリテンション推移を把握した
- [ ] セグメント別の転換率差異を確認した

## Step 3: CRO（コンバージョン率最適化）設計

ボトルネック段階に対する具体的な CRO 施策を設計する。
詳細なプレイブックは [cro-playbook.md](references/cro-playbook.md) を参照。

### 3a. LP（ランディングページ）最適化

| 最適化領域 | チェック項目 | 改善手法 |
|:---|:---|:---|
| **バリュープロポジション** | 5 秒で価値が伝わるか | 見出しの具体化、ベネフィット訴求 |
| **CTA** | 明確で視認性が高いか | ボタンコピーの改善、配置の最適化 |
| **社会的証明** | 信頼シグナルが十分か | ロゴ、テスティモニアル、事例数値 |
| **ファーストビュー** | スクロールなしで要点が伝わるか | ヒーローセクションの再設計 |
| **反論処理** | 不安要素を解消しているか | FAQ、保証、比較コンテンツ |

### 3b. フォーム最適化

| 原則 | 説明 | 目安 |
|:---|:---|:---|
| **フィールド数最小化** | 各フィールドは転換率を下げるコスト | 3 フィールド以下が理想 |
| **段階的コミットメント** | 簡単な項目から始める | メール → 名前 → 詳細 |
| **リアルタイムバリデーション** | 入力中のフィードバック | エラーはフィールド横に表示 |
| **モバイル最適化** | タップターゲット 44px 以上 | 単一カラムレイアウト |

### 3c. 価格ページ最適化

| 要素 | ベストプラクティス |
|:---|:---|
| **プラン数** | 3 プラン表示（推奨プランにバッジ） |
| **アンカリング** | 高額プランを先に表示して相対感を調整 |
| **年額表示** | 月額換算と年間節約額を併記 |
| **CTA 文言** | 「無料で始める」>「購入する」>「送信」 |
| **FAQ** | 価格に関するよくある反論を処理 |

### 3d. CRO 改善提案の優先度付け

| 優先度 | 基準 | 例 |
|:---|:---|:---|
| **即効性（Quick Win）** | 低工数・高インパクト | CTA 文言変更、社会的証明追加 |
| **高インパクト** | 高工数・高インパクト | LP 全面リデザイン、フォーム再構築 |
| **テスト候補** | 効果が不確実 | ヒーロー画像変更、価格表示形式 |

**チェックリスト**:
- [ ] ボトルネック段階に対応する CRO 施策を設計した
- [ ] LP のバリュープロポジション・CTA・社会的証明を評価した
- [ ] フォームのフィールド数と最適化余地を確認した
- [ ] 価格ページの表示構成を評価した
- [ ] 施策に優先度を付けた（Quick Win / 高インパクト / テスト候補）

## Step 4: グロース実験設計（ICE/RICE スコアリング）

CRO 施策を含むグロース実験を体系的に設計し、優先順位を付ける。
詳細なフレームワークは [experiment-frameworks.md](references/experiment-frameworks.md) を参照。

### 4a. 仮説テンプレート

すべての実験は以下の形式で仮説を記述する:

```
[観察/データ] に基づき、
[変更内容] を実施すると、
[対象ユーザー] の [指標] が [予測される変化] すると考える。
成功基準: [具体的な数値目標]
```

### 4b. ICE スコアリング

| 実験 | Impact (1-10) | Confidence (1-10) | Ease (1-10) | ICE スコア |
|:---|:---|:---|:---|:---|
| CTA 文言変更 | 6 | 8 | 9 | 7.7 |
| LP ヒーロー刷新 | 8 | 5 | 3 | 5.3 |
| フォーム簡素化 | 7 | 7 | 6 | 6.7 |
| 価格ページ再設計 | 9 | 4 | 2 | 5.0 |

**ICE = (Impact + Confidence + Ease) / 3**

### 4c. RICE スコアリング（より詳細な優先度付け）

| 実験 | Reach | Impact (0.25-3) | Confidence (%) | Effort (人週) | RICE |
|:---|:---|:---|:---|:---|:---|
| CTA 文言変更 | 10,000 | 1 (中) | 80% | 0.5 | 16,000 |
| LP ヒーロー刷新 | 10,000 | 2 (高) | 50% | 3 | 3,333 |
| オンボーディング改善 | 5,000 | 3 (最大) | 60% | 4 | 2,250 |

**RICE = (Reach x Impact x Confidence) / Effort**

### 4d. 実験バックログの管理

| ステータス | 説明 |
|:---|:---|
| **バックログ** | 優先度付け済み、未着手 |
| **設計中** | 仮説・計測方法を策定中 |
| **実行中** | A/B テスト配信中 |
| **分析中** | データ収集完了、結果分析中 |
| **完了** | 結論が出て、実装または棄却 |

**チェックリスト**:
- [ ] すべての実験に仮説テンプレートを記述した
- [ ] ICE または RICE スコアで優先順位を付けた
- [ ] 上位 3 つの実験に必要なサンプルサイズを算出した
- [ ] 実験バックログをステータス管理できる状態にした
- [ ] 各実験の成功指標・ガードレール指標を定義した

## Step 5: リテンション施策設計（メール・オンボーディング）

ユーザーの定着率を高めるリテンション施策を設計する。
詳細な施策集は [retention-strategies.md](references/retention-strategies.md) を参照。

### 5a. オンボーディング最適化

| 原則 | 説明 | 実践 |
|:---|:---|:---|
| **Time-to-Value** | 初期価値体験までの時間を最短化 | サインアップ後 30 秒以内に核心体験 |
| **1 セッション 1 ゴール** | 初回セッションは 1 つの成功体験に集中 | 高度な機能は後回し |
| **体験重視** | 説明より実際にやらせる | チュートリアルよりインタラクティブ |
| **進捗の可視化** | 完了度を示しモチベーションを維持 | チェックリスト（3-7 項目） |

### 5b. Aha Moment の定義

リテンション最適化の鍵は「Aha Moment」（核心的価値体験）の特定:

| プロダクトタイプ | Aha Moment の例 | アクティベーション指標 |
|:---|:---|:---|
| プロジェクト管理ツール | 最初のプロジェクト作成 + メンバー招待 | 7 日以内に 2 名以上招待 |
| 分析ツール | トラッキング設置 + 初回レポート閲覧 | 3 日以内にレポート閲覧 |
| EC サイト | 初回購入完了 | 30 日以内に初回購入 |
| デザインツール | 初回デザイン作成 + 共有 | 7 日以内にデザイン共有 |

### 5c. メールによるリテンション施策

| メールタイプ | タイミング | 目的 | KPI |
|:---|:---|:---|:---|
| **ウェルカム** | サインアップ直後 | 次のアクションを促す | 開封率、CTA クリック率 |
| **未完了リマインダー** | 24h / 72h 後 | オンボーディング完了を促す | 完了率 |
| **価値再認識** | 7 日後 | 成功事例を共有 | 再訪率 |
| **機能紹介** | 14 日後 | 未利用機能の発見 | 機能利用率 |
| **再エンゲージメント** | 30 日間未利用時 | プロダクトへの復帰 | 復帰率 |

### 5d. チャーン防止

| リスクシグナル | 検知条件 | 介入施策 |
|:---|:---|:---|
| ログイン頻度低下 | 50% 以上の減少が 2 週間 | 「お困りですか？」メール |
| 主要機能の未使用 | 7 日間コア機能未使用 | 機能チュートリアルの再案内 |
| サポート件数急増後沈黙 | チケット後 2 週間無反応 | CS からのプロアクティブ連絡 |
| 請求ページの閲覧増加 | 3 日以内に複数回閲覧 | 価値再認識メール + ダウングレード提案 |

**チェックリスト**:
- [ ] オンボーディングフローの改善点を特定した
- [ ] Aha Moment を定義し、アクティベーション指標を設定した
- [ ] ライフサイクルメールの設計が完了した
- [ ] チャーンリスクシグナルと介入施策を定義した
- [ ] リテンション指標の計測方法を確認した

## Step 6: クリエイティブ大量制作オペレーション（Figma/Canva テンプレート化）

広告バナー・SNS 画像などのクリエイティブを効率的に大量制作する仕組みを構築する。
詳細なガイドは [creative-production-guide.md](references/creative-production-guide.md) を参照。

### 6a. テンプレート設計原則

| 原則 | 説明 | 実践 |
|:---|:---|:---|
| **変数化** | 差し替え要素をすべて変数にする | テキスト、画像、カラーを変数定義 |
| **サイズ展開** | 1 デザインから複数サイズを生成 | マスターコンポーネントからの派生 |
| **ブランド一貫性** | ブランドガイドを組み込む | カラー・フォント・ロゴの統一管理 |
| **承認フロー** | レビュー工程を組み込む | Figma コメント or スプレッドシート管理 |

### 6b. Figma を使った大量制作ワークフロー

1. **マスターコンポーネント作成**: 可変要素をプロパティで管理
2. **変数設定**: テキスト・カラー・画像の変数を定義
3. **バリアント生成**: コンポーネントプロパティで訴求軸×サイズの組み合わせ
4. **一括書き出し**: プラグイン（例: Export Kit）で全バリアントを書き出し

### 6c. Canva を使った大量制作ワークフロー

1. **ブランドキット登録**: カラー・フォント・ロゴを登録
2. **テンプレート作成**: 差し替え箇所を明確にしたマスターデザイン
3. **一括作成**: Canva Bulk Create 機能で CSV からバリエーション自動生成
4. **サイズ展開**: Magic Resize で複数サイズへ自動変換

### 6d. クリエイティブマトリクス

制作するバリエーションを体系的に管理する:

| 訴求軸 | コピー | ビジュアル | サイズ | プラットフォーム |
|:---|:---|:---|:---|:---|
| 課題解決 | 「作業時間を75%削減」 | Before/After | 1200x628 | Meta Feed |
| 課題解決 | 「作業時間を75%削減」 | Before/After | 1080x1080 | Meta/Instagram |
| 社会的証明 | 「1万社が導入」 | ロゴ集合 | 1200x628 | Meta Feed |
| 社会的証明 | 「1万社が導入」 | ロゴ集合 | 1200x300 | LinkedIn |

**チェックリスト**:
- [ ] テンプレート設計原則に基づいてマスターデザインを定義した
- [ ] 訴求軸×サイズ×プラットフォームのマトリクスを作成した
- [ ] Figma または Canva のワークフローを選定した
- [ ] 一括書き出し・一括生成の手順を確認した
- [ ] ブランドガイドラインとの整合性を確認した

## Step 7: 広告プラットフォーム分析レポート（Meta/Google/LinkedIn）

各広告プラットフォームのパフォーマンスを横断的に分析し、予算配分の最適化を提案する。

### 7a. プラットフォーム別 KPI 一覧

| KPI | Meta Ads | Google Ads | LinkedIn Ads |
|:---|:---|:---|:---|
| **リーチ指標** | インプレッション、リーチ | インプレッション、検索IS | インプレッション |
| **エンゲージメント** | CTR、いいね・シェア | CTR、品質スコア | CTR、エンゲージメント率 |
| **コンバージョン** | CVR、CPA、ROAS | CVR、CPA、ROAS | CVR、CPA、CPL |
| **効率性** | CPC、CPM | CPC、CPM | CPC、CPM |
| **品質** | 関連度スコア | 広告の有効性 | — |

### 7b. クロスプラットフォーム分析

各プラットフォームの広告費・インプレッション・クリック・CTR・コンバージョン・CVR・CPA・ROAS を横並びで比較する。統一的なアトリビューションウィンドウ（例: 7 日クリック）を設定し、プラットフォーム間の比較を可能にする。

### 7c. 予算配分の最適化

各プラットフォームの CPA と ROAS に基づき、予算の再配分を提案する:

| 判断基準 | アクション |
|:---|:---|
| CPA が目標の 50% 以下 | **予算増額**: スケーリング余地あり |
| CPA が目標の 50-100% | **現状維持**: 最適化を継続 |
| CPA が目標の 100-150% | **要改善**: クリエイティブ・ターゲティング見直し |
| CPA が目標の 150% 超 | **予算削減 or 停止**: ROI が合わない |

### 7d. レポートの構成

月次分析レポートは以下の構成で作成する:

1. **エグゼクティブサマリー**: 主要 KPI の前月比・目標比
2. **プラットフォーム別詳細**: 各プラットフォームのパフォーマンス
3. **キャンペーン別分析**: トップ/ワーストキャンペーン
4. **クリエイティブ分析**: 勝ちパターン・負けパターン
5. **予算配分提案**: 次月の最適配分案
6. **アクションアイテム**: 具体的な改善施策リスト

**チェックリスト**:
- [ ] 全プラットフォームの KPI データを収集した
- [ ] クロスプラットフォームの比較分析を実施した
- [ ] CPA/ROAS に基づく予算配分を提案した
- [ ] キャンペーン別・クリエイティブ別の分析を完了した
- [ ] 次月のアクションアイテムを明確にした

## Step 8: ダッシュボード設計と改善サイクル

グロースオペレーション全体を継続的に改善するためのダッシュボードと運用サイクルを設計する。

### 8a. ダッシュボードの構成

| セクション | 含める指標 | 更新頻度 |
|:---|:---|:---|
| **NORTH STAR** | 最重要指標 1 つ + トレンド | リアルタイム |
| **ファネル概況** | 各段階の転換率 + 前週比 | 週次 |
| **広告パフォーマンス** | CPA、ROAS、予算消化率 | 日次 |
| **リテンション** | コホート別リテンション率 | 週次 |
| **実験ステータス** | 実行中の実験と進捗 | 週次 |

### 8b. 推奨ダッシュボードツール

| ツール | 特徴 | 最適なケース |
|:---|:---|:---|
| **Google Looker Studio** | 無料、Google エコシステムとの連携 | GA4 + Google Ads 中心 |
| **Mixpanel** | プロダクト分析特化、ファネル・コホート | SaaS のプロダクト指標 |
| **Amplitude** | コホート分析に強い | リテンション重視 |
| **Notion + API 連携** | カスタマイズ自由度が高い | チーム共有・実験管理 |

### 8c. 週次グロースミーティング

| 時間 | アジェンダ | 成果物 |
|:---|:---|:---|
| 5 分 | NORTH STAR の確認 | 方向性の共有 |
| 10 分 | ファネル数値レビュー | ボトルネックの特定 |
| 10 分 | 実験結果の共有 | 勝ち/負けの判定 |
| 5 分 | 次週の実験計画 | 実験バックログの更新 |

### 8d. 改善サイクル（PDCA）

```
Plan（計画）: ファネル分析 → ボトルネック特定 → 実験設計
  ↓
Do（実行）: A/B テスト実施 → クリエイティブ制作 → 施策展開
  ↓
Check（検証）: 結果分析 → 統計的有意性の確認 → 学習の記録
  ↓
Act（改善）: 勝ちパターンの展開 → 負けパターンの棄却 → 次の仮説設計
  ↓
（次の Plan へ）
```

### 8e. 改善サイクルの記録

各サイクルで以下を記録する: サイクル番号、期間、NORTH STAR の現状値と目標値、ファネルボトルネック、実施した実験と結果、主な学習、次サイクルの優先施策（ICE スコア付き）。

### 8f. Growth Ops Slop 防止チェック（Distributional Convergence 対策）

LLM がグロース施策を設計する際、distributional convergence により「どの案件でも同じフレームワーク・同じ施策・同じダッシュボード」に収束する。プロダクト固有の文脈を反映した施策になっているかを検証する。

| # | Growth Ops Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| GO-1 | **ファネル分析の収束** | 毎回同じAARRRフレームワーク、同じボトルネック特定方法 | プロダクトタイプに応じたファネルモデルを選定（Step 1c）。セグメント別・コホート別で分析角度を変える |
| GO-2 | **CRO プレイブックの単調さ** | 毎回「CTA改善＋社会的証明追加＋フォーム簡素化」の3点セット | ボトルネックの根本原因（価値提案の弱さ？信頼不足？摩擦？）を診断してから施策を選定（Step 3d） |
| GO-3 | **実験仮説のテンプレ化** | 仮説が全て同じ構文で、予測される変化が毎回「15-20%向上」 | 仮説の効果量はベースラインと業界ベンチマークから導出。「なぜこの変化量を予測するか」の根拠を明記 |
| GO-4 | **リテンション施策の画一化** | 全案件でウェルカム→リマインダー→価値再認識→再エンゲージの同じシーケンス | プロダクトの Aha Moment（Step 5b）から逆算してシーケンスを設計。メール以外の介入手段（in-app、CS、コミュニティ）も検討 |
| GO-5 | **ICE/RICE スコアの形骸化** | スコアが主観的で毎回同じような数値配分 | Impact は推定 CV 数で定量化。Confidence はデータの有無で段階分け。Effort は人時で見積もる |
| GO-6 | **クリエイティブマトリクスのコピペ** | 訴求軸×サイズの組み合わせが全案件で同じ | ブランドの視覚的アイデンティティとプラットフォーム特性から組み合わせを設計。front-design / ui-design の AI slop 対策を参照 |
| GO-7 | **ダッシュボード設計の収束** | 毎回 Looker Studio + 同じ5セクション | チームの意思決定プロセスから逆算してダッシュボードを設計。「このダッシュボードで何を意思決定するか」を各セクションに明記 |
| GO-8 | **週次ミーティングのテンプレ化** | 毎回同じ30分アジェンダ（NORTH STAR→ファネル→実験→計画） | チームの成熟度に応じてアジェンダを進化させる。初期は学習重視、成熟期は実験速度重視 |

> **Why**: グロースオペレーションは「計測→仮説→実験→学習」のサイクルを回すことが本質だが、
> distributional convergence により毎回同じフレームワーク・同じ施策に収束すると、
> 「計測なき改善は推測にすぎない」というこのスキルの前提自体が形骸化する。
> プロダクト固有のデータと文脈を反映しない施策は、汎用的なコンサルレポートと変わらない。
>
> **核心原則**: **フレームワークは思考の起点であり、結論ではない** — AARRR も ICE/RICE も 4 象限分析も、
> 分析の出発点として使い、そこからプロダクト固有の文脈に踏み込むこと。
> 「このプロダクトだからこそ」の一言が入らない施策は slop である。

**チェックリスト**:
- [ ] ダッシュボードに必要な指標を定義した
- [ ] ダッシュボードツールを選定した
- [ ] 週次ミーティングのアジェンダを設計した
- [ ] PDCA サイクルの運用ルールを定めた
- [ ] 改善サイクルの記録テンプレートを準備した
- [ ] ファネルモデルがプロダクトタイプに応じて選定されている（GO-1）
- [ ] CRO 施策がボトルネックの根本原因診断に基づいている（GO-2）
- [ ] 実験仮説の効果量がベースラインとベンチマークから導出されている（GO-3）
- [ ] リテンション施策が Aha Moment から逆算して設計されている（GO-4）
- [ ] ICE/RICE スコアの各要素が定量的に算出されている（GO-5）
- [ ] クリエイティブマトリクスがブランド・プラットフォーム固有の設計になっている（GO-6）
- [ ] ダッシュボードの各セクションに「何を意思決定するか」が明記されている（GO-7）
- [ ] 週次ミーティングのアジェンダがチームの成熟度に応じて調整されている（GO-8）
- [ ] GO-1〜GO-8 に 2 つ以上該当しない

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| ファネルデータが不完全で分析できない | トラッキングが未実装 or 不十分 | GA4 + GTM で最低限のイベント計測を先に構築する。完璧を求めず、まず主要ファネルの 3 段階から計測を開始する |
| ボトルネックが複数あり優先順位が付けられない | インパクトの定量化が不足 | Step 4 の RICE スコアリングを使い、Reach（影響人数）と Effort（工数）で機械的に優先度を決める |
| A/B テストのサンプルサイズが不足する | トラフィックが少ない | テスト対象を絞る（全ページではなく最もトラフィックの多い 1 ページ）。ベイズ推定を使い、少サンプルでも意思決定する |
| CRO 施策を実装したが CVR が改善しない | 変更が軽微すぎる or ボトルネックの誤認 | 大胆な変更（ヒーロー全面刷新等）をテストする。Step 2 に戻りファネル分析を再実行してボトルネックを再特定する |
| リテンション施策の効果が出ない | Aha Moment の定義が間違っている | コホート分析で「継続ユーザーと離脱ユーザーの行動差」を再分析し、Aha Moment を再定義する |
| クリエイティブの量産体制が整わない | テンプレート設計が不十分 | 変数化の粒度を見直す。まず 1 つの訴求軸×3 サイズで成功パターンを確立してから横展開する |
| 広告の CPA が目標を大幅に超過している | ターゲティングが広すぎる or クリエイティブの疲弊 | オーディエンスを絞り込み、2-3 週間ごとにクリエイティブを更新する。類似オーディエンスの精度を見直す |
| ダッシュボードが複雑すぎて誰も見ない | 指標が多すぎる | NORTH STAR + ファネル 3 段階の転換率に絞る。「このデータで何を意思決定するか」を基準にフィルタする |
| チーム内でグロース施策の合意が取れない | データに基づく議論ができていない | Step 8c の週次ミーティング形式を導入し、「数値 → 仮説 → 実験 → 結果」のサイクルで意思決定する |
| 実験結果が統計的に有意にならない | サンプルサイズ不足 or 効果が小さい | 実験期間を延長するか、より大きな変更をテストする。MDE（最小検出可能効果）を事前に設定しておく |
| プラットフォーム間で KPI の定義が異なる | 各プラットフォームの計測仕様の違い | Step 7a の KPI 一覧を基準に統一定義を作成する。アトリビューションウィンドウを揃える（例: 7 日クリック） |
| 月次レポートの作成に時間がかかりすぎる | 手動でのデータ収集と集計 | Looker Studio で自動化ダッシュボードを構築し、レポートは分析と提案に集中する |
| グロース施策が全案件で同じ内容になる | AI slop（distributional convergence）— LLM が汎用的なプレイブックに収束 | Step 8f の GO-1〜GO-8 チェックを実行。プロダクト固有の Aha Moment・ファネル構造・チーム成熟度から施策を導出する |
| 実験の効果量予測が毎回「15-20%向上」になる | GO-3（実験仮説のテンプレ化）— ベースラインを参照せず根拠なく予測している | 現在のベースライン値と業界ベンチマークを明示し、変化量の根拠を記述する。過去の実験結果がある場合はそのリフト率を参考にする |
| ダッシュボードを作ったが誰も意思決定に使わない | GO-7（ダッシュボード設計の収束）— 汎用的な5セクション構成で、チーム固有の意思決定フローと乖離 | 各セクションに「このデータを見て何を判断するか」「閾値を超えたら誰がどう動くか」を明記し、チームの意思決定プロセスから逆算して再設計する |

## アンチパターン検出

このスキルの出力品質を検証し、AI Decay（品質劣化）を防止するためのアンチパターン検出リスト。各項目は品質劣化パターンの検出条件を定義し、不合格時は対応する対策（修正ガイドライン）に従って改善する。

- [ ] ファネル分析がプロダクト固有の Aha Moment から逆算されているか（汎用 AARRR テンプレの丸写しでないか）
- [ ] 実験仮説の効果量予測にベースライン値と業界ベンチマークの根拠があるか（「15-20%向上」の根拠なし予測を排除）
- [ ] KPI の定義がプラットフォーム間で統一されているか（アトリビューションウィンドウの差異を考慮）
- [ ] ダッシュボードの各セクションに「このデータで何を判断するか」「閾値超過時の行動」が明記されているか
- [ ] リテンション施策がチーム成熟度に応じたレベルで設計されているか（過度に高度な施策の排除）
- [ ] A/B テストのサンプルサイズが統計的に有意な結果を得られる設計になっているか
