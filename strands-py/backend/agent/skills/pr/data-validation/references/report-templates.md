## Phase 2a: テーブル構造検証の指摘例

```markdown
## テーブル検証レポート

### 対象: `inbox/RFP_教育データ基盤.md` — 表3.2「システム要件一覧」

| # | 問題 | 行 | 内容 | 深刻度 |
|:---|:---|:---|:---|:---|
| 1 | 列数不一致 | 行15 | ヘッダー5列に対し4列 → セル結合崩れ | Major |
| 2 | 数値異常 | 行23 | 「1,000」が「1.000」に変換 | Critical |
| 3 | 空セル | 行8-12 | 「結合セル」の値が最初の行のみに記載 | Minor |

### 推奨アクション
1. 行15: 元 PDF を参照し、手動で列を補完
2. 行23: 桁区切りを「1,000」に修正
3. 行8-12: 結合セルの値を全行にコピー
```

## Phase 2c: スキーマ整合性検証の指摘例

```markdown
スキーマ整合性の問題

**エンティティ**: E-003 GradeRecord（成績）
**問題**: `teacher_id` が FK として定義されているが、
        参照先のエンティティ `Teacher` が domain-model.md に未定義

**影響**: テーブル作成時に参照整合性制約エラーが発生
**修正案**: Teacher エンティティを追加定義するか、
          teacher_id を単純な VARCHAR に変更
```

## Phase 2e: 数値整合性検証の指摘例

```markdown
数値不整合を検出

**箇所**: 08_schedule.md vs 04_development.md

| 項目 | 08_schedule.md | 04_development.md | 差分 |
|:---|:---|:---|:---|
| 開発工数 | 2,400人日 | 2,680人日 | -280人日 |
| サブシステム数 | 6 | 8 | -2 |

**影響**: スケジュール章に2サブシステムが欠落 → 納期超過リスク
**修正案**: 08_schedule.md を 04_development.md に合わせて更新
```
