# Skill: Unit Test（単体テスト設計・生成）

> **すべてのテストが平等に作られているわけではない — 価値の高いテストだけがテスト・スイートに含める価値がある**

## Instructions

### 基本原則：良い単体テストの4本柱

すべてのテスト設計判断は、以下の4本柱の**積**で価値が決まる。1つでもゼロなら、テスト全体の価値はゼロになる。

| 柱 | 説明 | 測定基準 |
|:---|:---|:---|
| **退行に対する保護** | バグを検出する能力 | 実行されるコード量、複雑度、ドメイン重要度 |
| **リファクタリングへの耐性** | 偽陽性を出さない能力 | 実装の詳細ではなく**観察可能な振る舞い**を検証しているか |
| **迅速なフィードバック** | テスト実行の速さ | プロセス外依存の少なさ、テスト実行時間 |
| **保守のしやすさ** | テストの読みやすさ・実行の簡単さ | テストのサイズ、プロセス外依存の数 |

CRITICAL: テストを書く前に、そのテストが4本柱のすべてにおいて十分な価値を持つか自問すること。

---

## Step 1: テスト対象の分析

テスト対象のコードを読み、以下の4種類に分類する。

```
コードの複雑度 / ドメイン重要度（高）
  ├── 協力者オブジェクトが少ない → ドメインモデル / アルゴリズム【単体テスト最優先】
  └── 協力者オブジェクトが多い  → 過度に複雑なコード【リファクタリング必要】

コードの複雑度 / ドメイン重要度（低）
  ├── 協力者オブジェクトが少ない → 取るに足らないコード【テスト不要】
  └── 協力者オブジェクトが多い  → コントローラ【統合テスト対象】
```

**判断基準**:
- **ドメインモデル / アルゴリズム**: 複雑なビジネスロジック、純粋な計算 → 単体テストで最大の価値
- **コントローラ**: 依存の調整役、薄いロジック → 統合テストで検証
- **取るに足らないコード**: 単純なgetter/setter、自明なコード → テスト不要
- **過度に複雑なコード**: Humble Object パターンでリファクタリングを推奨

### State Alignment の観点（HumanLM 知見）

ユーザーシミュレーションやペルソナ駆動のテスト設計において、
HumanLM 論文（Stanford/NYU, 2026）の State Alignment アプローチが示唆する観点:

- **テスト対象がユーザー入力を処理する場合**: ユーザーの心理状態（Belief, Stance, Emotion）によって
  入力パターンが変わることを考慮し、テストデータに心理状態の多様性を反映する
- **出力がユーザーに表示される場合**: 同一機能でも Emotion（焦燥 vs 余裕）や Stance（懐疑的 vs 推進的）に
  よって期待される UX が異なることをテストケースの境界値設計に活用する
- **State Alignment スコア**: ユーザーシミュレーション品質の検証に、応答模倣（SFT）ではなく
  状態次元ごとの整合性（Goal r=0.61, Stance r=0.59 が最も予測力が高い）で評価する

## Step 2: テスト手法の選択

3つのテスト手法から、対象に最適なものを選択する。**出力値ベースを最優先で検討**すること。

| 手法 | 説明 | 適用場面 | 4本柱の評価 |
|:---|:---|:---|:---|
| **出力値ベース** | 戻り値を検証 | 純粋関数、副作用なし | 全柱で最高 |
| **状態ベース** | 操作後のオブジェクト状態を検証 | 状態変化を伴うドメインモデル | リファクタリング耐性に注意 |
| **コミュニケーション・ベース** | メソッド呼び出しを検証（モック使用） | 外部システムとの通信 | 保守性に注意 |

**選択の優先順位**:
1. 出力値ベース（最優先）: 関数型アーキテクチャを活用し、副作用を分離
2. 状態ベース: 出力値ベースが不可能な場合
3. コミュニケーション・ベース: 外部システム境界でのみ使用

---

## Step 3: テストの構造設計（AAAパターン）

すべてのテストは **Arrange-Act-Assert** パターンで構造化する。

### 構造ルール

- **Arrange（準備）**: テスト対象と依存の初期状態を設定
- **Act（実行）**: テスト対象のメソッドを**1回だけ**呼び出す
- **Assert（確認）**: 期待する結果を検証

### 厳守事項

- 1つのテストに**Actは1つだけ**（複数のActは統合テストの兆候）
- if文をテスト内に書かない（テストは分岐のない直線的なコードであるべき）
- Arrangeが大きくなる場合はファクトリメソッドに抽出（ただし、コンストラクタでの共有は避ける）
- Assertの数は論理的に1つの振る舞いを検証する範囲に留める

---

## Step 4: テストの命名

テスト名は**振る舞いを自然言語で記述**する。メソッド名やパラメータを機械的に組み合わせたパターンは避ける。

### 命名の指針

- テスト名はプログラマでない人が読んでも理解できるようにする
- テスト対象メソッド名をテスト名に含めない（リファクタリング耐性の低下を防ぐ）
- 「べき（should）」で始めない。事実を平叙文で記述する

### 命名例

```
// 悪い例（実装に結合した命名）
Test_IsStringLong_Input_LessThan5_ReturnsFalse

// 良い例（振る舞いを記述）
test_short_string_is_not_considered_long
delivery_with_past_date_is_invalid
sum_of_two_positive_numbers_returns_correct_result
```

## Step 5: モックの適切な使用

### コマンドとクエリの区別

| 種別 | 定義 | テスト・ダブル | 検証すべきか |
|:---|:---|:---|:---|
| **コマンド** | 副作用を起こす操作（メール送信、DB書込み） | **モック** | 呼び出し回数を検証する |
| **クエリ** | 値を返すだけの操作（データ取得） | **スタブ** | 呼び出し自体は検証しない |

### モックのルール

CRITICAL: 以下のルールを厳守すること。

1. **モックはアプリケーション境界を超える通信のみに使用**する
   - 外部API、メッセージキュー、メール送信等の管理下にない依存
   - データベースは管理下にある依存 → 実物またはインメモリ版を使用
2. **スタブの呼び出しを検証しない**（過剰検証 = 偽陽性の原因）
3. **モックの対象は自プロジェクトが所有する型のみ**にする
   - サードパーティライブラリのインターフェースを直接モックしない
   - アダプタパターンで自プロジェクトのインターフェースに変換してからモック
4. **モックの呼び出し回数を必ず確認**する

## Step 6: テストの品質検証

作成したテストを4本柱で自己レビューする。

### チェックリスト

**退行に対する保護**:
- ビジネスロジックの重要なパスを検証しているか
- 境界値（0, -1, null, 空文字列, 最大値）を検証しているか
- 異常系（例外、エラー状態）を検証しているか
- 十分なコード量が実行されているか

**リファクタリングへの耐性**:
- 実装の詳細ではなく、観察可能な振る舞いを検証しているか
- privateメソッドを直接テストしていないか
- テスト内にドメイン知識（本番コードと同じ計算ロジック）が漏洩していないか
- モックで内部通信を検証していないか（システム間通信のみ）

**迅速なフィードバック**:
- プロセス外依存（DB、ファイルシステム、ネットワーク）を最小化しているか
- テスト実行が高速か

**保守のしやすさ**:
- テストコードが短く、読みやすいか
- Arrangeフェーズが肥大化していないか
- テスト間で不要な依存がないか（テスト実行順序に依存していないか）

### テスト設計 Slop 防止チェック（Distributional Convergence 対策）

LLM が生成するテストは「それらしいが価値のない」汎用パターンに収束しやすい。このようなテストはカバレッジ数値を満たしても退行保護に寄与せず、保守コストだけが増大する。

| # | テスト設計 Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| TS-1 | **実装転写テスト名** | テスト名が `test_function_returns_value`、`test_method_called` のように実装構造を転写している | 振る舞いを自然言語で記述する（例: `test_expired_coupon_cannot_be_applied_to_order`、`test_overdraft_triggers_notification_to_account_holder`） |
| TS-2 | **金太郎飴 Arrange** | 全テストが同じファクトリ/フィクスチャパターンで、テストごとの文脈が見えない | 各テストの Arrange はそのテストの意図を表現する最小限の状態設定にする（例: 期限切れテストなら `coupon = Coupon(expires_at=yesterday)` のように意図が自明な値を使う） |
| TS-3 | **全方位モック症候群** | 内部依存まで含めて全てをモック化し、テストが実装の鏡像になっている | モックはアプリケーション境界（外部 API、メッセージキュー等）のみに限定。内部依存はリアル実装またはインメモリ版を使う |
| TS-4 | **Happy + 1 Error テスト** | 正常系 1 件 + `ValueError` 等の汎用エラー 1 件だけで、ドメイン固有の異常系がない | ドメイン固有の境界条件を網羅する（例: 在庫切れ、同時購入競合、権限不足、有効期限境界、上限金額超過） |
| TS-5 | **プレースホルダーテストデータ** | テストデータが `"test"`, `"example"`, `"sample"`, `123`, `"foo@bar.com"` 等の汎用値ばかり | ドメインに即したリアルなデータを使う（例: 実際の商品名 `"有機栽培コシヒカリ 5kg"`, 学籍番号 `"S2024001"`, 請求金額 `Decimal("148500")` ） |
| TS-6 | **ファイル対ファイル構造写像** | テストファイル構造が実装ファイル構造を 1:1 で鏡像化し、振る舞い単位でグループ化されていない | テストは振る舞い（ユースケース、ドメイン概念）単位でグループ化する（例: `test_order_lifecycle.py`、`test_payment_authorization.py` のように業務シナリオで分割） |

> **核心原則**: **テストはシステムの「仕様書」であり、実装の「写経」ではない** — テスト名とテストデータだけを読んで、非エンジニアがシステムの振る舞いを理解できるか？「別のプロジェクトのテストに差し替えても違和感がないか？」→ 違和感がないならテスト設計 Slop である。

**総合チェック**:
### 品質ガードレール

- テストコードからプロダクション環境へ直接接続してはならない
- テスト間の実行順序依存を許容しないこと

品質基準:
- テスト実行時間 最大 10 秒/テスト
- カバレッジ 閾値: 80%

- [ ] テスト設計 Slop チェック（TS-1〜TS-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## アンチパターン一覧

以下のアンチパターンを検出した場合は修正を提案すること。

| アンチパターン | 問題 | 対処 |
|:---|:---|:---|
| **privateメソッドの直接テスト** | リファクタリング耐性がゼロ | publicインターフェース経由でテスト。テスト困難な場合は設計を見直す |
| **private状態の公開** | テストのためだけにカプセル化を破壊 | 観察可能な振る舞いのみ検証 |
| **テストへのドメイン知識の漏洩** | テストが本番コードのロジックを複製 | 期待値はハードコードする。計算ロジックをテストに書かない |
| **プロダクションコードへのテスト用汚染** | テスト専用の分岐・フラグ | テスト用のインターフェース分離 |
| **具象クラスへのテスト・ダブル適用** | 不安定な継承依存 | インターフェースを導入してからモック |
| **網羅率至上主義** | 価値のないテストの量産 | 4本柱で各テストの価値を判断 |
| **取るに足らないテスト** | getter/setterの検証 | テスト不要と判断して削除 |
| **壊れやすいテスト** | 実装詳細への結合 | 振る舞いベースに書き換え |
| **過剰検証** | スタブの呼び出し検証 | コマンドのみモックで検証 |

## テスト・ピラミッド

テスト戦略は以下のピラミッドに従う。

```
        /  E2E  \          ← 少数: 最も重要なユーザーシナリオのみ
       /  統合テスト \       ← 中程度: コントローラ、外部連携
      / 単体テスト     \    ← 多数: ドメインモデル、アルゴリズム
     ------------------
```

**配分の目安**:
- 単体テスト: ドメインモデル・アルゴリズムに集中。最も多く書く
- 統合テスト: コントローラ層の主要パスを網羅。ハッピーパス + 重要な異常系
- E2E テスト: ビジネス上最も重要なシナリオのみ

## 実践的テストワークフロー

テスト設計から実行までの4ステップ:

```
1. 未テストコードの特定
   → カバレッジレポートの確認、テストファイルが存在しないモジュールの探索

2. テストスキャフォールドの生成
   → 既存テストのパターン・フレームワーク・アサーションスタイルを自動検出し、一貫性を保つ

3. エッジケースの追加
   → 境界値、null/undefined、空配列、並行処理、タイムアウト、権限エラー

4. テスト実行と検証
   → 全テスト green 確認、カバレッジの改善度を測定
```

**原則**: 既存のテストパターン（命名規則、ディレクトリ構造、使用ライブラリ、アサーションスタイル）を自動検出して合わせる。プロジェクトの文化に溶け込むテストを書く。

```

→ Step 2: 失敗パターンの分類
  - 環境依存（2件）: タイムゾーン、OS パス区切り
  - モック不足（2件）: 外部 API レスポンスの変更
  - ロジック変更の未追従（1件）: リファクタリングでインターフェースが変更
→ Step 3: 各テストの修正
  - 環境依存 → `vi.useFakeTimers()` / `path.join()` に変更
  - モック不足 → 最新の API レスポンスでモック更新
  - 未追従 → テストを新インターフェースに合わせて修正
→ Step 4: 全テスト green を確認、カバレッジが低下していないことを検証
```

### Example 6: エッジケース網羅

```
「この認証モジュールのエッジケースを洗い出して、テストを追加して」

→ Step 1: 対象モジュールの振る舞い分析（入力パターン、分岐条件）
→ Step 2: エッジケース洗い出し
  - 空文字列のパスワード
  - 最大長超過のユーザー名（256文字）
  - SQLインジェクション文字列を含む入力
  - 同時ログイン試行（race condition）
  - トークン有効期限ちょうどの境界
  - 無効な JWT フォーマット
→ Step 3: AAA パターンでテスト実装（各エッジケースに 1 テスト）
→ Step 4: カバレッジ 72% → 91% に改善を確認
```

## Troubleshooting

| 問題 | 原因 | 対処 |
|:---|:---|:---|
| テストが頻繁に壊れる | 実装の詳細に結合している | 観察可能な振る舞いのみ検証するようリファクタリング |
| テストが遅い | プロセス外依存が多い | ドメインロジックを分離し、単体テストで検証 |
| Arrangeが肥大化 | テスト対象が過度に複雑 | Humble Object パターンでロジックを分離 |
| モックだらけのテスト | 内部通信までモック化 | モックはシステム間通信のみに限定 |
| 網羅率は高いが品質が低い | 取るに足らないテストが多い | 4本柱で各テストの価値を再評価し、低価値テストを削除 |
| テスト名が意味不明 | 実装詳細を命名に含めている | 振る舞いを自然言語で記述するよう改名 |
| テストデータに機密情報が含まれるリスク | 本番データのテスト環境流用 | テストフィクスチャの匿名化とセキュリティレビュー |
| 日本語テストケースの文字化けで偽の失敗 | テストファイルの文字コード不一致 | UTF-8 統一と多言語テストフィクスチャの整備 |
| 並列テスト実行で共有リソースが競合 | テスト間の暗黙的な状態依存 | テスト分離と排他制御で同時実行の安全性を確保 |

### generate_test_template.py の機能

- **Python 解析**: `ast` モジュールによる正確な関数・メソッドシグネチャ抽出
- **TypeScript 解析**: 正規表現ベースの export 関数・クラスメソッド抽出
- **AAA パターン**: 全テストケースを Arrange-Act-Assert 構造で生成
- **4 本柱コメント**: 各テストに Khorikov の 4 本柱（退行保護・リファクタリング耐性・迅速なフィードバック・保守性）の観点をコメントで付与
- **3 テストケース/関数**: ハッピーパス + 境界値 + エラーケースを関数ごとに生成
- **フレームワーク対応**: pytest（Python）、jest / vitest（TypeScript）

### 使用例

```bash
# Python ファイル → pytest テンプレート（デフォルト）
python scripts/generate_test_template.py src/calculator.py

# TypeScript ファイル → vitest テンプレート
python scripts/generate_test_template.py src/userService.ts --framework vitest

# 出力先を指定
python scripts/generate_test_template.py src/auth.py --output tests/test_auth.py
```

## アンチパターン検出

このスキルの出力品質を検証するためのチェックリスト。

- [ ] テスト名が振る舞いを自然言語で記述しているか
- [ ] テストデータがドメインに即したリアルな値を使用しているか
- [ ] モックがアプリケーション境界のみに限定されているか
- [ ] 異常系テストにドメイン固有の境界条件が含まれているか
- [ ] テストファイルが振る舞い単位でグループ化されているか
- [ ] Arrange セクションがテストの意図を表現する最小限の状態設定になっているか
- [ ] ユーザーシミュレーションを含むテストで、応答模倣（SFT）ではなく State Dimensions（Belief/Stance/Emotion 等）の整合性でテストデータが設計されているか
