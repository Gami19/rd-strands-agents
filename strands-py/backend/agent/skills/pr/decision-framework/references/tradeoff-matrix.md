# Trade-Off Analysis Matrix Template & Guide

> Source: Fundamentals of Software Architecture, 2nd Edition (Mark Richards & Neal Ford, O'Reilly 2025)
> Progressive Disclosure: Step 4（トレードオフ分析）で参照するテンプレートと記入ガイド

---

## 基本原則

> **First Law of Software Architecture**: "Everything in software architecture is a trade-off."
> 利点のないものは存在しないが、欠点のないものも存在しない。
> トレードオフ分析は「最良の選択」ではなく「最も害の少ない選択（least worst）」を見つけるプロセス。

---

## 記入ガイド

### Weight（重み）の設定

| 重み | 意味 | 判断基準 |
|:---:|:---|:---|
| **5** | 最重要 | ビジネスの成否を直接左右する |
| **4** | 重要 | ビジネス目標の達成に大きく影響する |
| **3** | 中程度 | 考慮すべきだが決定的ではない |
| **2** | 低い | あると望ましいが妥協可能 |
| **1** | 最低 | ほぼ影響しないが記録のために含める |

**ルール**:
- 全軸を同一重みにしない（差分がなければ重み付けの意味がない）
- 重み 5 は最大 2 軸まで（全てが最重要では優先順位が付けられない）
- ステークホルダーと合意してから重みを確定する

### Score（評価）の設定

| Score | 意味 | 判断基準 |
|:---:|:---|:---|
| **5** | 優秀 | この軸において最良の選択肢 |
| **4** | 良好 | 十分に要件を満たす |
| **3** | 中程度 | 要件を満たすが改善の余地あり |
| **2** | 不十分 | 要件を部分的にしか満たさない |
| **1** | 不適 | この軸において著しく劣る |

**ルール**:
- 必ず根拠を記録する（「なんとなく 4」は禁止）
- 可能な限り定量データで裏付ける（ベンチマーク結果、コスト見積等）
- 不明な場合はスコアを仮置きし、検証タスクとして記録する

## 評価軸カタログ

判断の種類に応じて、以下から 5-8 個を選定する。

### 技術的評価軸

| 評価軸 | 説明 | 測定方法の例 |
|:---|:---|:---|
| **性能** | レスポンスタイム、スループット、レイテンシ | ベンチマーク、負荷テスト |
| **スケーラビリティ** | 負荷増大への対応力 | 水平/垂直スケーリングの可否 |
| **可用性** | アップタイム、障害耐性 | SLA/SLO の達成見込み |
| **保守性** | 変更の容易さ、コードの可読性 | 変更頻度、修正にかかる時間 |
| **テスト容易性** | テストの書きやすさ、自動化適性 | テストカバレッジの達成見込み |
| **デプロイ容易性** | デプロイ頻度とリスク | デプロイの自動化度 |
| **セキュリティ** | 攻撃対象面、データ保護 | セキュリティ監査結果 |
| **相互運用性** | 他システムとの統合のしやすさ | API 互換性、標準準拠 |
| **可観測性** | 監視・デバッグのしやすさ | ログ・メトリクス・トレースの充実度 |

### 組織的評価軸

| 評価軸 | 説明 | 測定方法の例 |
|:---|:---|:---|
| **学習コスト** | チームが習得にかかる時間 | 習得期間の見積もり |
| **採用可能性** | スキルを持つ人材の市場供給 | 求人市場の状況 |
| **チーム自律性** | チームが独立して運用できるか | 外部依存の有無 |
| **エコシステム成熟度** | ドキュメント・コミュニティの充実度 | GitHub stars、Stack Overflow 質問数 |
| **ベンダーロックイン** | 特定ベンダーへの依存度 | 移行コストの見積もり |

### ビジネス的評価軸

| 評価軸 | 説明 | 測定方法の例 |
|:---|:---|:---|
| **初期コスト** | 導入時の費用 | 見積金額 |
| **運用コスト** | 継続的な費用（TCO） | 月額/年額の見積もり |
| **市場投入速度** | 実装完了までの期間 | スプリント数の見積もり |
| **リスク** | 失敗時の影響度 | リスクマトリクスの結果 |
| **コンプライアンス** | 規制要件への適合 | 規制チェックリスト |

## 記入例（完成形）

```markdown
## Trade-Off Analysis: API Gateway の選定

**日付**: 2026-02-28
**Author**: 田中太郎
**判断対象**: マイクロサービスの API Gateway として Kong / AWS API Gateway / Envoy のいずれを採用するか

### マトリクス

| # | 評価軸 | W | Kong | | AWS API GW | | Envoy | |
|:---:|:---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| | | | S | WS | S | WS | S | WS |
| 1 | 性能 | 4 | 4 | 16 | 3 | 12 | 5 | 20 |
| 2 | 運用コスト | 5 | 3 | 15 | 4 | 20 | 2 | 10 |
| 3 | 拡張性 | 3 | 5 | 15 | 3 | 9 | 4 | 12 |
| 4 | 学習コスト | 4 | 3 | 12 | 5 | 20 | 2 | 8 |
| 5 | ベンダーロックイン | 3 | 4 | 12 | 2 | 6 | 5 | 15 |
| 6 | 可観測性 | 3 | 4 | 12 | 3 | 9 | 5 | 15 |
| | **合計** | | | **82** | | **76** | | **80** |

### スコアリング根拠

| 評価軸 | 選択肢 | Score | 根拠 |
|:---|:---|:---:|:---|
| 性能 | Kong | 4 | Lua プラグインでオーバーヘッドあるが実用範囲 |
| 性能 | AWS API GW | 3 | コールドスタートのレイテンシ問題あり |
| 性能 | Envoy | 5 | C++ 実装、最高のスループット |
| 運用コスト | Kong | 3 | OSS 版は無料だが運用工数が必要 |
| 運用コスト | AWS API GW | 4 | マネージドで運用工数最小 |
| 運用コスト | Envoy | 2 | 運用知識が必要、サポート契約が高額 |
| 学習コスト | Kong | 3 | 中程度の学習曲線 |
| 学習コスト | AWS API GW | 5 | チーム全員が AWS 経験あり |
| 学習コスト | Envoy | 2 | xDS API の習得にチームが時間を要する |

### 感度分析

| シナリオ | 変動 | 結果 | 結論 |
|:---|:---|:---|:---|
| ベースライン | — | Kong:82, AWS:76, Envoy:80 | Kong 推奨 |
| 運用コスト +1 | W: 5→6 | Kong:85, AWS:80, Envoy:82 | 変化なし（堅牢） |
| 運用コスト -1 | W: 5→4 | Kong:79, AWS:72, Envoy:78 | 変化なし（堅牢） |

### 定性的トレードオフ

| 観点 | Kong | AWS API GW | Envoy |
|:---|:---|:---|:---|
| **不可逆性** | 中（移行可能） | 高（AWS 依存） | 低（標準的な Proxy） |
| **段階的適用** | 可能 | 可能 | 可能 |
| **組織的準備度** | Ready | Ready | Not Ready |

### 推奨

**Kong** を推奨する。スコアの差は僅差だが、感度分析で結論が安定しており、拡張性・ベンダー非依存性で優位。Envoy は性能では最良だが、チームの学習コストと組織的準備度を考慮すると現時点では適切でない。AWS API Gateway はベンダーロックインのリスクが中長期的な懸念。
```

## よくある間違いと対策

| 間違い | 原因 | 対策 |
|:---|:---|:---|
| 全評価軸の重みが同じ（全て 3） | 優先順位の合意が取れていない | ステークホルダーと重み付けを議論してから記入 |
| スコアの根拠がない | 「なんとなく」で付けた | 全てのスコアに 1 行以上の根拠を記述する |
| 選択肢が 2 つだけ | 偽りの二項対立 | "Do nothing" やハイブリッド案を含め最低 3 つ |
| 定性的トレードオフの未記入 | スコアで全てが表現できると誤解 | 不可逆性・段階的適用・組織的準備度を必ず補足 |
| 感度分析の省略 | 結果が明確だと思い込む | 僅差（10%以内）の場合は必ず実施。明確な差でも記録として残す |
