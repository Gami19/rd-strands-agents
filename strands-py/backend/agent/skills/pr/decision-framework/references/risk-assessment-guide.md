# Risk Assessment Guide: Risk Storming & Risk Matrix

> Source: Fundamentals of Software Architecture, 2nd Edition (Mark Richards & Neal Ford, O'Reilly 2025)
> Progressive Disclosure: Step 5（リスク評価）で参照する手法ガイドとテンプレート

---

## 基本原則

> **リスク評価の第一原則**: 影響度を先に評価し、確率を後に評価する。確率が不明な場合は高 (3) を仮定する。

リスクは「避ける」ものではなく「管理する」もの。全てのリスクを排除することは不可能であり、コストに見合わない。目的は、リスクを**可視化**し、**優先順位付け**し、**緩和策を策定**すること。

---

## リスクマトリクス

### マトリクス構造

2 次元（影響度 x 発生確率）でリスクを定量化する。

| | 低確率 (1) | 中確率 (2) | 高確率 (3) |
|:---|:---:|:---:|:---:|
| **高影響 (3)** | 3 (中) | 6 (高) | 9 (高) |
| **中影響 (2)** | 2 (低) | 4 (中) | 6 (高) |
| **低影響 (1)** | 1 (低) | 2 (低) | 3 (中) |

### リスクレベルと対応方針

| レベル | スコア | 色 | 対応方針 |
|:---|:---:|:---:|:---|
| **低** | 1-2 | 緑 | 監視のみ。緩和策は不要。定期的にスコアを再評価 |
| **中** | 3-4 | 黄 | 緩和策を策定し、定期的にモニタリング。トリガー条件を設定 |
| **高** | 6-9 | 赤 | 緩和策を即時策定し、判断への影響を再評価。必要に応じて判断を変更 |

### 影響度の評価基準

| 影響度 | スコア | 判断基準 |
|:---|:---:|:---|
| **高** | 3 | システム全体のダウン、データ損失、セキュリティ侵害、大幅なコスト超過、顧客への直接的影響 |
| **中** | 2 | 部分的な機能低下、パフォーマンス劣化、スケジュール遅延、特定ユーザーグループへの影響 |
| **低** | 1 | 軽微な不便、ワークアラウンドで回避可能、内部プロセスへの限定的影響 |

### 発生確率の評価基準

| 確率 | スコア | 判断基準 |
|:---|:---:|:---|
| **高** | 3 | 過去に発生実績あり、技術的に発生しやすい、外部要因で発生する可能性が高い |
| **中** | 2 | 発生の可能性はあるが条件が限定的、類似事例が存在する |
| **低** | 1 | 発生の可能性は低い、十分な予防策が既に存在する、稀な条件でのみ発生 |

> **不明な場合の扱い**: 確率が不明な場合は**高 (3) を仮定する**。楽観的な見積もりは危険。確率を正確に評価できない時点で、そのリスクは十分に理解されていない証拠。

## リスクストーミング

### 概要

リスクストーミングは、チームで協調的にリスクを識別する手法。個人のバイアスを排除し、多様な視点からリスクを洗い出す。

### 事前準備

| 準備項目 | 説明 |
|:---|:---|
| **参加者** | 3-7 名。異なる専門性を持つメンバーを含める |
| **対象資料** | アーキテクチャ図、プロセスフロー、技術選定資料 |
| **付箋（色分け）** | 赤 = 高リスク、黄 = 中リスク、緑 = 低リスク |
| **時間枠** | 30-45 分（3 フェーズの合計） |
| **ファシリテーター** | 議論を導く人（自身はリスク識別に参加しない方がよい） |

### Phase 1: 個別識別（5-10 分）

**目的**: 各参加者が独立にリスクを識別する（他者の影響を排除）

**手順**:
1. 対象資料（アーキテクチャ図等）を壁に貼り出す
2. 各参加者が独立にリスクを付箋に書く
   - 1 枚の付箋に 1 つのリスク
   - 色で重要度を表現（赤/黄/緑）
3. 付箋を対象資料の該当箇所に貼る
4. この間、他の参加者の付箋を見ない・話さない

**記入テンプレート**:
```
[リスクの簡潔な説明]
領域: [影響を受けるコンポーネント/プロセス]
影響: [高/中/低]
```

### Phase 2: コンセンサス形成（10-15 分）

**目的**: 個人差を議論し、チームとしてのリスク認識を統一する

**手順**:
1. 全員の付箋を確認し、類似リスクをグルーピングする
2. リスクレベルの認識が大きく異なる項目を議論する
   - 「Aさんは高リスク、Bさんは低リスクと評価した理由は？」
3. 議論の結果、チームとしてのリスクレベルを確定する
4. リスクマトリクスに全リスクを転記する

**議論のガイドライン**:
- 反対意見を歓迎する（Groupthink の防止）
- 「なぜそう思うか」を掘り下げる
- 合意に至らない場合は高い方のリスクレベルを採用する
- 時間切れの場合は未議論リスクを次回に持ち越す

### Phase 3: 緩和策の策定（10-15 分）

**目的**: 高リスク項目に対する具体的な緩和策を策定する

**手順**:
1. 高リスク（6-9）の項目を優先的に取り上げる
2. 各リスクに対して以下を決定する:

| 決定事項 | 説明 |
|:---|:---|
| **緩和策** | リスクを軽減する具体的なアクション |
| **責任者** | 緩和策を実行する人 |
| **期限** | 緩和策を完了する期限 |
| **残存リスク** | 緩和策実施後のリスクレベル |
| **許容可否** | 残存リスクを組織として許容するか |

3. 中リスク（3-4）の項目にはモニタリング方針を決定する
4. 低リスク（1-2）の項目は記録のみで対応不要

## リスクカテゴリカタログ

判断の種類に応じて、以下のカテゴリからリスクを洗い出す。

### 技術リスク

| リスクカテゴリ | 例 |
|:---|:---|
| **性能リスク** | レスポンスタイムの劣化、スループットの不足 |
| **可用性リスク** | 単一障害点、カスケード障害 |
| **セキュリティリスク** | データ漏洩、認証の突破、権限昇格 |
| **データリスク** | データ損失、データ不整合、移行失敗 |
| **統合リスク** | API 互換性の破壊、外部サービスの停止 |
| **スケーラビリティリスク** | 負荷増大時の性能低下、リソース枯渇 |
| **技術的負債リスク** | 保守コストの増大、リファクタリングの困難化 |

### 組織リスク

| リスクカテゴリ | 例 |
|:---|:---|
| **人材リスク** | キーパーソンの離職、スキル不足 |
| **コミュニケーションリスク** | チーム間の認識齟齬、情報の断絶 |
| **知識リスク** | 暗黙知への依存、ドキュメント不足 |
| **変更管理リスク** | 要件変更の頻発、スコープクリープ |

### ビジネスリスク

| リスクカテゴリ | 例 |
|:---|:---|
| **コストリスク** | 予算超過、隠れたコスト |
| **スケジュールリスク** | 遅延、外部依存による停滞 |
| **コンプライアンスリスク** | 規制違反、監査指摘 |
| **ベンダーリスク** | ベンダーロックイン、サービス終了 |
| **市場リスク** | 競合の動向変化、顧客ニーズの変化 |

## ユーザーストーリーリスク分析

リスクをユーザーストーリーの観点から分析する手法。新機能の実装順序をリスクベースで決定する場合に有用。

### 手順

1. 各ユーザーストーリーを列挙する
2. 各ストーリーが影響するアーキテクチャ領域を特定する
3. 各領域のリスクスコアを評価する
4. 高リスクのストーリーを先に実装する（リスクの早期検証）

### テンプレート

```markdown
## User Story Risk Analysis

| ストーリー | 影響領域 | 影響する特性 | リスクスコア | 実装優先度 |
|:---|:---|:---|:---:|:---|
| [US-001: ユーザー登録] | 認証, DB | Security, Performance | 6 | 高（先行実装） |
| [US-002: 商品検索] | API, Cache | Performance, Scalability | 4 | 中 |
| [US-003: 通知設定] | Notification | Availability | 2 | 低 |
```

> **原則**: 高リスクのストーリーを先に実装することで、アーキテクチャ上の問題を早期に発見できる。低リスクのストーリーは安定したアーキテクチャの上で効率的に実装できる。

## リスク評価の品質チェックリスト

リスク評価の完了時に以下を確認する。

- [ ] 全リスクに影響度と確率のスコアが付与されている
- [ ] 影響度を先に評価し、確率を後に評価した
- [ ] 確率不明のリスクは高 (3) を仮定した
- [ ] 高リスク（6-9）の全項目に具体的な緩和策がある
- [ ] 緩和策に責任者と期限が設定されている
- [ ] 残存リスクの許容可否を判断した
- [ ] 中リスク（3-4）のモニタリング方針を決定した
- [ ] リスクマトリクスのヒートマップを作成した
- [ ] リスクの見落としがないか、複数の視点（技術/組織/ビジネス）で確認した
- [ ] リスク評価の結果をステークホルダーと共有した
