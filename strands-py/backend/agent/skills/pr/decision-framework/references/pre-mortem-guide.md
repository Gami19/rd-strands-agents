# プレモーテム・ファシリテーション・ガイド

> Source: Prospective Hindsight — Gary Klein (1989), "Performing a Project Premortem" (HBR, 2007)
> Progressive Disclosure: リスク評価の前段階で使用。risk-assessment-guide.md と組み合わせて運用する。

---

## プレモーテムとは

### 定義

プレモーテム（Pre-Mortem）は、プロジェクト開始**前**に「このプロジェクトは壊滅的に失敗した」と仮定し、
失敗の原因を逆算的に洗い出す手法。Gary Klein が 1989 年に提唱した Prospective Hindsight（前向きの後知恵）に基づく。

> **核心的な問い**: 「1 年後、このプロジェクトは大失敗に終わった。何が起きたのか？」

### 通常のリスク分析との違い

| 観点 | 通常のリスク分析 | プレモーテム |
|:---|:---|:---|
| **出発点** | 「何がリスクか？」（未来の可能性を列挙） | 「失敗は既に起きた。なぜか？」（確定した前提から逆算） |
| **思考モード** | 予防的（可能性を網羅する） | 説明的（既に起きた結果を説明する） |
| **心理的効果** | 楽観バイアスに対抗しにくい | 失敗を既成事実化することで楽観バイアスを無効化 |
| **参加者の姿勢** | 「問題を指摘する人」になりにくい | 「失敗の原因を発見した人」になれる |
| **出力** | リスク一覧（発生確率と影響度のマトリクス） | 失敗シナリオ一覧（具体的なストーリー） |
| **研究結果** | — | Klein の研究で、リスク識別精度が 30% 向上（HBR 2007） |

### 克服する認知バイアス

プレモーテムが特に有効な認知バイアスは 3 つある。

| バイアス | 説明 | プレモーテムの対抗メカニズム |
|:---|:---|:---|
| **計画錯誤（Planning Fallacy）** | 楽観的な見積もりに偏る傾向。Kahneman & Tversky が実証 | 「失敗した前提」から逆算することで、楽観的シナリオを強制的に破壊する |
| **確証バイアス（Confirmation Bias）** | 自説を支持する情報のみ注目する傾向 | 全員が「失敗の理由」を探す側に回るため、否定的情報の収集が正当化される |
| **集団思考（Groupthink）** | 集団の合意を維持するために異論を抑制する傾向。Janis (1972) | 「失敗の理由を見つけるのが仕事」という役割設定により、異論が歓迎される構造を作る |

## 3 段階リスク分類フレームワーク

プレモーテムで洗い出したリスクを 3 カテゴリに分類する。
カテゴリごとに対応方針が明確に異なるため、分類精度がプレモーテムの成否を決める。

### Tigers（猛虎）

**定義**: 確実に発生し、プロジェクトを殺す致命的リスク。

| 属性 | 内容 |
|:---|:---|
| **発生確率** | 高い（過去に類似プロジェクトで発生実績あり、または構造的に避けられない） |
| **影響度** | 致命的（プロジェクトの目的達成が不可能になる） |
| **対応方針** | **ローンチブロッカー** — 緩和策が完了するまでプロジェクトを進めない |
| **ADR 接続** | 必ず ADR を起票し、緩和策を意思決定として記録する |

**判定基準**:
- 過去 3 件以上の類似プロジェクトで発生している
- 緩和策なしでプロジェクト目的の 50% 以上が達成不能になる
- 技術的・構造的に発生を回避する手段が存在しない

**具体例**:

| 例 | 説明 | なぜ Tiger か |
|:---|:---|:---|
| 主要 API の廃止予定 | 依存する外部 API が 6 ヶ月後に廃止される | 代替手段がなければプロダクトが動作しない |
| 規制要件の未充足 | GDPR 対応が未完了のまま EU 展開を計画 | 法的にサービス提供が不可能 |
| キーパーソン 1 名依存 | 特定技術の知見がチーム内 1 名のみ | 離職時にプロジェクト継続不能 |
| データ移行の不可逆性 | レガシーシステムからの移行で rollback 手段がない | 失敗時に業務停止 |

### Paper Tigers（張り子の虎）

**定義**: 恐ろしく見えるが、実際のインパクトは小さいリスク。

| 属性 | 内容 |
|:---|:---|
| **発生確率** | 低〜中（条件が限定的、または既に予防策がある） |
| **影響度** | 小さい（ワークアラウンドが存在する、または影響範囲が限定的） |
| **対応方針** | **監視のみ** — リソースを投入しない。モニタリング指標を設定するだけ |
| **ADR 接続** | ADR 不要。リスク一覧表に記録のみ |

**判定基準**:
- ワークアラウンドが 3 つ以上存在する
- 影響範囲がプロジェクトスコープの 10% 未満
- 発生してから対応しても間に合う（回復時間 < 許容停止時間）

**具体例**:

| 例 | 説明 | なぜ Paper Tiger か |
|:---|:---|:---|
| 特定ライブラリのバージョン非互換 | 依存ライブラリのメジャーバージョンアップ | 代替ライブラリが複数あり、移行コストも小さい |
| UI フレームワークの流行変化 | 採用した UI フレームワークの人気低下 | プロダクトの価値に直接影響しない。段階的移行が可能 |
| 開発環境の不安定さ | CI/CD パイプラインが時々失敗する | リトライで解決。本番環境に影響なし |

> **注意**: Paper Tiger を Tiger と誤認すると、無駄なリソース投入が発生する。
> 逆に Tiger を Paper Tiger と誤認すると、致命的な見落としとなる。
> 分類に迷った場合は Tiger として扱い、緩和策策定後に再分類する。

### Elephants（象）

**定義**: 全員が知っているが、誰も言い出さないリスク。"Elephant in the room" の比喩。

| 属性 | 内容 |
|:---|:---|
| **発生確率** | 高い（既に兆候が現れている場合が多い） |
| **影響度** | 中〜致命的（放置すると Tiger に成長する） |
| **対応方針** | **心理的安全性で引き出す** — プレモーテムの最大の価値はここにある |
| **ADR 接続** | 引き出された後、影響度に応じて ADR 起票を判断する |

**判定基準**:
- 1 対 1 の会話では言及されるが、全体会議では誰も触れない
- 「あれは大丈夫なの？」と曖昧に言及されるが、具体的な議論にならない
- 特定の人物・チーム・組織に関わるため、指摘が政治的にリスキー

**具体例**:

| 例 | 説明 | なぜ Elephant か |
|:---|:---|:---|
| スポンサーのコミットメント不足 | 経営層が口頭では支持するが、予算・人員を出さない | 指摘すると「経営批判」と受け取られるリスク |
| チーム間の技術的負債 | 他チームが管理する API の品質問題 | 他チームへの批判と受け取られるため、自チーム内で回避策を取り続ける |
| スケジュールの非現実性 | 誰もが間に合わないと思っているが、上層部が設定した期限 | 「できない」と言うことがキャリアリスクになる環境 |
| 前提技術の成熟度不足 | 採用した新技術が本番レベルに達していない | 技術選定を主導した人物への配慮で指摘できない |

**引き出すためのファシリテーション技法**:

1. **匿名投票**: 付箋に無記名で書く（Phase B のサイレントライティング）
2. **「私が聞いた話」形式**: 「あるチームメンバーから聞いた懸念として...」と間接話法で導入する
3. **ノーマライゼーション**: ファシリテーターが「よくあるリスクとして、経営のコミットメント不足がありますが、この案件ではどうでしょう？」と一般論として提示する
4. **事前ヒアリング**: プレモーテム前に個別に 1 対 1 で懸念を収集し、匿名化して提示する

## ファシリテーション手順 — 90 分テンプレート

### 事前準備チェックリスト

| 準備項目 | 説明 | 必須 |
|:---|:---|:---:|
| **参加者選定** | 5-8 名。異なる役割・専門性を含める。意思決定者を 1 名以上含む | Yes |
| **プロジェクト概要** | 目的、スコープ、スケジュール、主要マイルストーンの 1 枚サマリー | Yes |
| **アーキテクチャ図** | システム構成図、データフロー図（存在する場合） | 推奨 |
| **付箋 + ペン** | 物理開催の場合。オンラインの場合は Miro/FigJam 等のボード | Yes |
| **タイマー** | 各フェーズの時間管理用 | Yes |
| **リスク一覧テンプレート** | 本ガイドの「プレモーテム結果テンプレート」を印刷 | 推奨 |
| **事前ヒアリング結果** | Elephant 引き出しのため、個別に収集した懸念（匿名化済み） | 推奨 |

### Phase A: セットアップ（10 分）

**目的**: 参加者のマインドセットを「成功を計画する」から「失敗を分析する」に切り替える。

**ファシリテーター台詞（テンプレート）**:

> 「今日はプレモーテムを実施します。ポストモーテム（事後検証）の反対で、
> プロジェクト開始**前**に失敗を分析する手法です。
>
> **前提**: いまから 1 年後、このプロジェクトは**壊滅的に失敗**しました。
> ローンチできなかった、ローンチしたが使われなかった、予算を 3 倍超過した、
> チームが崩壊した ── どんな失敗でも構いません。
>
> 皆さんの仕事は、**なぜ失敗したかを説明する**ことです。
> 正しい答えはありません。思いつく限りの失敗原因を書いてください。
> 突飛に見えるものほど歓迎します。」

**やること**:
1. プロジェクト概要を 3 分で確認する（参加者の認識を揃える）
2. プレモーテムのルールを説明する
3. 「批判ではなく発見」であることを強調する

**ルール掲示（壁/ボードに貼る）**:
- 失敗は**既に起きた**前提で語る（「起きるかもしれない」ではなく「起きた」）
- 他の人のアイデアを否定しない
- 「そんなことは起きない」は禁止
- 小さなリスクも歓迎する

### Phase B: 個人ブレスト — サイレントライティング（20 分）

**目的**: 集団思考を排除し、個人の視点で失敗原因を洗い出す。

**手順**:
1. 各参加者に付箋を 10 枚以上配布する
2. **沈黙のまま**、失敗原因を 1 枚 1 件で書く
3. 最低 5 件の記入を必須とする
4. 各付箋に以下を記入する:

```
失敗原因: [1 行で記述]
影響: [プロジェクトにどんなダメージを与えたか]
兆候: [どの時点で気づけたはずか]
```

**ファシリテーターの行動**:
- 5 分経過: 「最低 5 件書けていない方は、技術面・人材面・スケジュール面・予算面・外部環境のそれぞれで 1 件ずつ考えてみてください」
- 10 分経過: 「残り 10 分です。突飛なものほど歓迎します。『さすがにこれは...』と思うものこそ書いてください」
- 15 分経過: 「残り 5 分。書いた内容を見返して、それぞれの兆候（いつ気づけたか）を記入してください」

**プロンプトカード（詰まった参加者向け）**:

| カテゴリ | プロンプト |
|:---|:---|
| 技術 | 「選択した技術が想定通りに動かなかったとしたら、何が原因か？」 |
| チーム | 「チームが機能不全に陥ったとしたら、最初の兆候は何か？」 |
| ステークホルダー | 「スポンサーが途中で方針を変えたとしたら、どんな状況でそうなるか？」 |
| 外部環境 | 「市場・競合・規制が変化したとしたら、何が一番痛いか？」 |
| スコープ | 「スコープが制御不能になったとしたら、どの要件がトリガーか？」 |
| 品質 | 「本番で重大インシデントが発生したとしたら、何が見落とされていたか？」 |

### Phase C: 共有・分類（30 分）

**目的**: 個人の視点を共有し、Tigers / Paper Tigers / Elephants に分類する。

**手順**:

1. **ラウンドロビン共有**（15 分）:
   - 各参加者が 1 件ずつ順番に発表する（全員が発表するまで繰り返す）
   - 重複は「同意」として統合する（票数を記録）
   - ファシリテーターは発表内容をボードに貼り出す

2. **グルーピング**（5 分）:
   - 類似リスクをクラスタリングする
   - 各クラスタにラベルを付ける

3. **3 カテゴリ分類**（10 分）:
   - 各クラスタを Tigers / Paper Tigers / Elephants に分類する
   - 分類に迷う場合は Tigers として扱う
   - Elephant の候補は、事前ヒアリング結果と照合する

**分類の判断フロー**:

```
リスク項目を確認する
  │
  ├─ 全員が知っているのに誰も言い出さなかった？
  │    └─ Yes → Elephants
  │
  ├─ ワークアラウンドが 3 つ以上存在する？
  │    └─ Yes → Paper Tigers
  │
  ├─ 発生時にプロジェクト目的の 50% 以上が達成不能？
  │    └─ Yes → Tigers
  │
  └─ 上記いずれにも該当しない
       └─ 影響度と確率で risk-assessment-guide.md のマトリクスを適用
```

### Phase D: 緩和策設計（20 分）

**目的**: Tigers に対する具体的な Action Item を策定する。

**対象**: Tigers のみ（Paper Tigers と Elephants は Phase D では扱わない）

**手順**:
1. Tigers を影響度順にソートする
2. 各 Tiger に対して以下を決定する:

| 決定項目 | 説明 | 記入例 |
|:---|:---|:---|
| **緩和策** | リスクを軽減する具体的アクション | 「API 廃止に備え、代替 API の PoC を 2 週間で実施」 |
| **担当者** | 緩和策を実行する責任者 | 「鈴木（バックエンドリード）」 |
| **期限** | 完了期限 | 「2026-04-15」 |
| **トリガー条件** | 緩和策を発動する条件（事前設定型の場合） | 「API 廃止アナウンスが公式に出た時点」 |
| **検証方法** | 緩和策が有効かを確認する方法 | 「代替 API で現行機能の 100% が動作することを確認」 |
| **残存リスク** | 緩和策実施後の残るリスク | 「代替 API のレイテンシが 20% 増加する可能性」 |

3. Elephants については:
   - 引き出せたこと自体を記録する（「表面化した」ステータス）
   - 影響度を再評価し、Tiger レベルなら Phase D で緩和策を追加
   - Tiger レベルでなければ、モニタリング方針のみ設定する

4. Paper Tigers については:
   - モニタリング指標のみ設定する（「月次で再評価」等）
   - リソースは投入しない

### Phase E: まとめ・ADR 記録（10 分）

**目的**: プレモーテムの結果を組織知として記録する。

**手順**:
1. Tigers の緩和策を全員で確認する
2. 全リスクをリスク一覧テーブルに転記する（次章のテンプレート使用）
3. ADR 起票の判断:

| 条件 | ADR 起票 |
|:---|:---|
| Tiger の緩和策がアーキテクチャ・技術選定に影響する | **必須** |
| Tiger の緩和策がスコープ・スケジュールに影響する | **必須** |
| Tiger の緩和策がチーム構成・役割に影響する | **推奨** |
| Paper Tigers / Elephants（Tiger 以外） | **不要**（リスク一覧表に記録のみ） |

4. 次回レビュー日を設定する（推奨: 1 ヶ月後、または主要マイルストーン通過時）
5. 参加者への感謝を伝える（特に Elephant を引き出した勇気に対して）

## よくある失敗パターン — 5 つのアンチパターンと対策

### AP-1: 「批判セッション化」

| 項目 | 内容 |
|:---|:---|
| **症状** | 特定の人物・チーム・意思決定への批判に変質する。建設的なリスク識別ではなく、過去の不満の吐露になる |
| **原因** | ファシリテーターがルールを徹底しない。Phase A のマインドセット切り替えが不十分 |
| **対策** | 「人」ではなく「状況・構造」に焦点を当てるルールを明示する。「〇〇さんがミスした」→「〇〇の知見がチーム内で共有されていない構造」に言い換える |
| **予防** | Phase A で「批判ではなく発見」を 2 回以上繰り返し強調する |

### AP-2: 「声の大きい人支配」

| 項目 | 内容 |
|:---|:---|
| **症状** | Phase C の共有時に、上位者やベテランの意見に全員が同調する。少数意見が埋没する |
| **原因** | Phase B のサイレントライティングが不十分。共有時にラウンドロビンを徹底しない |
| **対策** | Phase B を厳格に沈黙で実施する。Phase C は必ずラウンドロビン（全員 1 件ずつ順番）で進行する |
| **予防** | 匿名投票ツール（Miro の付箋、Google Forms 等）を活用する |

### AP-3: 「分類の形骸化」

| 項目 | 内容 |
|:---|:---|
| **症状** | 全リスクが Tigers に分類される。または全リスクが Paper Tigers に分類される。分類に差が出ない |
| **原因** | 分類基準が曖昧。「念のため全部 Tiger にしておこう」という安全志向、または「大丈夫だろう」という楽観志向 |
| **対策** | 本ガイドの判定基準（発生確率、影響度の閾値）を掲示し、各リスクについて基準に照らして判定する |
| **予防** | 「Tigers は全体の 20-30% が目安。全部 Tiger は分類できていない証拠」と事前に伝える |

### AP-4: 「緩和策の抽象化」

| 項目 | 内容 |
|:---|:---|
| **症状** | 緩和策が「注意する」「モニタリングする」「検討する」等の曖昧な表現にとどまる |
| **原因** | Phase D の時間不足。または具体的なアクションまで落とし込む習慣がない |
| **対策** | 緩和策には必ず「誰が」「何を」「いつまでに」「どうやって確認するか」の 4 要素を含める |
| **予防** | Phase D のテンプレート（担当者・期限・検証方法）を事前に配布し、埋めなければ完了としない |

### AP-5: 「やりっぱなし」

| 項目 | 内容 |
|:---|:---|
| **症状** | プレモーテムの結果が記録されない。記録されても参照されない。緩和策が実行されない |
| **原因** | Phase E が時間切れで省略される。記録の保管場所が不明確。フォローアップの仕組みがない |
| **対策** | Phase E の時間を確保する（10 分は必須）。結果を `project/50_decision_log/` に保管し、ADR と紐付ける |
| **予防** | 次回レビュー日をカレンダーに登録する。Tiger の緩和策を既存のタスク管理ツールに登録する |

## decision-framework との統合

### スキル連携フロー

```
distill / story-map
     ↓ プロジェクト計画の蒸留・整理
[Pre-Mortem 実施]  ← 本ガイド
     ↓ リスク一覧 + 緩和策
decision-framework
     ↓ ADR 起票（Tiger の緩和策 → 意思決定記録）
     ↓ tradeoff-matrix.md（緩和策の選択肢比較）
     ↓ risk-assessment-guide.md（定量的リスク評価）
review
     ↓ ADR レビュー
drm
     ↓ ステークホルダーへの説明
```

### 使い分けガイド

| 状況 | 使うべき手法 | ガイド |
|:---|:---|:---|
| プロジェクト開始前のリスク洗い出し | **プレモーテム**（本ガイド） | pre-mortem-guide.md |
| 特定の技術判断のリスク評価 | **リスクストーミング + リスクマトリクス** | risk-assessment-guide.md |
| 選択肢間の比較・評価 | **トレードオフ分析マトリクス** | tradeoff-matrix.md |
| 意思決定の記録 | **ADR** | adr-template.md / adr-template-library.md |

### project/ ディレクトリへの記録

```
project/
├── 20_notes/
│   └── [project-name]/
│       └── risks.md              ← プレモーテム結果（リスク一覧表）
├── 50_decision_log/
│   ├── ADR-XXX-[tiger-mitigation].md  ← Tiger 緩和策の ADR
│   └── ...
```

## 実施タイミングの判断基準

| タイミング | プレモーテムの要否 | 理由 |
|:---|:---|:---|
| 新規プロジェクトのキックオフ直後 | **必須** | 全ステークホルダーが揃い、計画のバイアスが最も強い時期 |
| 主要マイルストーンの前 | **推奨** | スコープ・技術・チームの状況が変わっている可能性 |
| 外部環境の大きな変化時 | **推奨** | 規制変更、市場変動、主要メンバーの異動等 |
| 小規模な機能追加 | **不要** | 既存のリスク評価手法（risk-assessment-guide.md）で十分 |
| 既に進行中で大きな問題が発生 | **不要** | ポストモーテム（事後検証）が適切 |

## 品質チェックリスト

プレモーテム完了時に以下を確認する。

- [ ] 参加者全員が最低 5 件のリスクを記入した
- [ ] 全リスクが Tigers / Paper Tigers / Elephants のいずれかに分類された
- [ ] Tigers の全項目に担当者・期限・検証方法が設定された
- [ ] Elephant が 1 件以上識別された（0 件の場合は心理的安全性が不足している可能性を検討）
- [ ] 緩和策に「注意する」「検討する」等の曖昧な表現がない
- [ ] 結果がリスク一覧テーブルに記録された
- [ ] Tiger の緩和策から ADR 起票の要否を判断した
- [ ] 次回レビュー日が設定された
