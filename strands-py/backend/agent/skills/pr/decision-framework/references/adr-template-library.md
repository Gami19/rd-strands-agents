# ADR テンプレートライブラリ

> **ドメイン固有の 8 テンプレートで、技術選定を即座に構造化する**

本ライブラリは、頻出する技術選定シナリオごとに**事前定義済みの評価軸・選択肢・リスク**を提供する。
汎用 ADR テンプレート（[adr-template.md](adr-template.md)）と組み合わせて使用する。

## 使い方

1. 判断対象に最も近いテンプレートを選択する
2. コンテキストを自プロジェクトの状況に合わせて加筆・修正する
3. 選択肢テーブルに自社の候補を追加・削除する
4. 評価マトリクスの**重み（W）をビジネス優先度に合わせて調整**する（デフォルト値はあくまで出発点）
5. 各セルにスコア（1-5）を記入し、根拠をコメントで残す
6. 決定・根拠・リスクと軽減策を記入して ADR を完成させる

> **注意**: 評価マトリクスのスコアは空欄（`-`）で提供される。プロジェクト固有の状況に基づいて記入すること。

## 1. クラウドプロバイダー選定 ADR

### コンテキスト

新規システムの構築、またはオンプレミスからのクラウド移行にあたり、主要クラウドプロバイダーを選定する必要がある。選定は数年にわたるインフラ投資に直結し、一度決定すると移行コストが高いため、慎重な評価が求められる。組織のガバナンスポリシー、既存のクラウド契約、チームのスキルセットも考慮に入れる。

### 選択肢

| 選択肢 | 概要 |
|:---|:---|
| AWS | 最大のマーケットシェアと最も豊富なサービスカタログ。成熟したエコシステムとパートナーネットワーク |
| Azure | Microsoft 製品との緊密な統合。エンタープライズ向け ID 管理（Entra ID）が強力。ハイブリッドクラウド対応 |
| GCP | データ分析・ML/AI 領域に強み（BigQuery、Vertex AI）。Kubernetes のネイティブサポート（GKE） |
| マルチクラウド | 複数プロバイダーを併用。ベンダーロックイン回避と可用性向上。ただし運用複雑性が増大 |

## 2. 認証方式選定 ADR

### コンテキスト

アプリケーションまたは API の認証方式を選定する必要がある。対象ユーザー（エンドユーザー / サービス間 / 外部パートナー）、セキュリティ要件、既存の ID 基盤（Active Directory, IdP 等）を考慮する。B2B/B2C の区別、シングルサインオン（SSO）要件、モバイル対応も判断に影響する。

### 選択肢

| 選択肢 | 概要 |
|:---|:---|
| OAuth 2.0 + OIDC | 業界標準の認可フレームワーク。委任認可とユーザー認証（OIDC）を分離。豊富なライブラリとプロバイダー |
| SAML 2.0 | エンタープライズ SSO の事実上の標準。XML ベースで冗長だが、既存 IdP との統合が容易 |
| JWT（自前発行） | 自前でトークンを発行・検証。シンプルだが、鍵管理・トークン失効の設計が必要 |
| API Key | 最もシンプルな認証方式。サービス間通信やパートナー API 向け。ユーザー単位の認証には不向き |
| ハイブリッド（OAuth + API Key） | ユーザー認証に OAuth、サービス間 / パートナー連携に API Key を併用 |

| 自前 JWT 実装のセキュリティ脆弱性 | 高 | 実績のあるライブラリを使用。セキュリティレビューとペネトレーションテストを実施 |

| API Key の管理不備（ハードコード、ローテーション漏れ） | 中 | シークレット管理サービスを使用。定期ローテーションを強制。使用状況を監査 |

## 3. データベース選定 ADR

### コンテキスト

新規システムまたは既存システムのリプレースにあたり、データストアを選定する必要がある。データの特性（構造化 / 半構造化 / 非構造化）、アクセスパターン（OLTP / OLAP / 時系列）、一貫性要件（強い一貫性 / 結果整合性）、データ量の成長予測を考慮する。

### 選択肢

| 選択肢 | 概要 |
|:---|:---|
| RDBMS（PostgreSQL / MySQL） | 強い一貫性と ACID トランザクション。構造化データの複雑なクエリに最適。スキーマ変更は計画的に |
| NoSQL ドキュメント DB（MongoDB / DynamoDB） | 柔軟なスキーマと水平スケーリング。半構造化データ、高書込みスループットに適する。結合クエリは苦手 |
| NewSQL（CockroachDB / TiDB） | RDBMS の ACID 保証と NoSQL のスケーラビリティを両立。分散トランザクション対応。運用が複雑 |
| 時系列 DB（TimescaleDB / InfluxDB） | 時系列データに特化した書込み最適化とクエリ。IoT、メトリクス、ログ収集に最適 |
| ポリグロット永続化 | 用途に応じて複数 DB を併用。最適なデータストアを選べるが、運用・データ整合性の複雑さが増大 |

## 4. フロントエンドフレームワーク選定 ADR

### コンテキスト

Web アプリケーションのフロントエンドフレームワークを選定する必要がある。アプリケーションの規模（SPA / MPA / マイクロフロントエンド）、チームの構成とスキル、パフォーマンス要件（初回表示速度、操作レスポンス）、長期保守性を考慮する。

### 選択肢

| 選択肢 | 概要 |
|:---|:---|
| React | 最大のエコシステム。JSX によるコンポーネント設計。Next.js / Remix でフルスタック対応。学習リソースが豊富 |
| Vue | 段階的導入が容易。テンプレート構文で学習曲線が緩やか。Nuxt.js でフルスタック対応 |
| Angular | フルスタックフレームワーク。TypeScript ファースト。DI・ルーティング・HTTP クライアント等を組込み。エンタープライズ向け |
| Svelte / SvelteKit | コンパイラベースで最小バンドル。ランタイムオーバーヘッドなし。エコシステムは発展途上 |

## 5. API 設計方式選定 ADR

### コンテキスト

サービス間通信またはクライアント-サーバー間通信の API 設計方式を選定する必要がある。通信パターン（同期 / 非同期）、クライアントの種類（SPA / モバイル / サーバー間）、データ量、リアルタイム要件、API のバージョニング戦略を考慮する。

### 選択肢

| 選択肢 | 概要 |
|:---|:---|
| REST（HTTP/JSON） | 最も広く採用されている設計方式。リソース指向。キャッシュが容易。ブラウザとの親和性が高い |
| GraphQL | クライアント主導のクエリ。過剰取得・過少取得を解消。スキーマファーストの型安全な API |
| gRPC（Protocol Buffers） | 高性能バイナリプロトコル。HTTP/2 ストリーミング対応。サービス間通信に最適。ブラウザ直接対応は限定的 |
| Event-Driven（Kafka / SQS） | 非同期メッセージング。疎結合。高スループット。結果整合性が前提。デバッグが複雑 |
| ハイブリッド（REST + gRPC） | 外部 API に REST、内部サービス間に gRPC を併用。各方式の強みを活かす |

## 6. CI/CD パイプライン選定 ADR

### コンテキスト

ソフトウェアのビルド・テスト・デプロイを自動化する CI/CD パイプラインを選定する必要がある。コードホスティング先（GitHub / Azure DevOps / GitLab）、デプロイ先（クラウド / オンプレ / ハイブリッド）、セキュリティ要件（セルフホスト要件、シークレット管理）、チームの規模と成熟度を考慮する。

### 選択肢

| 選択肢 | 概要 |
|:---|:---|
| GitHub Actions | GitHub との深い統合。YAML ベース。豊富な Marketplace。パブリックリポジトリは無料 |
| Azure DevOps (Pipelines) | Azure との統合。ボード・リポジトリ・テスト管理を一元化。エンタープライズ向けガバナンス機能 |
| GitLab CI/CD | GitLab との一体型。Auto DevOps 機能。セルフホスト可能。セキュリティスキャン組込み |
| Jenkins | OSS でフル制御可能。プラグインエコシステムが最大。セルフホスト前提。運用負荷が高い |

## 8. データアーキテクチャ選定 ADR

### コンテキスト

組織のデータ基盤アーキテクチャを選定する必要がある。データの種類と量（構造化 / 半構造化 / 非構造化）、分析ユースケース（BI / ML / リアルタイム分析）、データガバナンス要件、組織のデータ成熟度レベルを考慮する。data-arch スキルで詳細な設計を行う前段として、大方針を決定する。

### 選択肢

| 選択肢 | 概要 |
|:---|:---|
| Data Warehouse（Snowflake / BigQuery / Redshift） | 構造化データの集中管理と高速分析。スキーマオンライトで品質担保。BI ユースケースに最適 |
| Data Lake（S3 + Glue / ADLS + Synapse） | 全データの低コスト蓄積。スキーマオンリードで柔軟性確保。非構造化データ対応。ガバナンスが課題 |
| Lakehouse（Databricks / Delta Lake） | DWH と Data Lake の利点を統合。ACID トランザクション + スキーマ進化。ML と BI を単一基盤で |
| Data Mesh | ドメイン駆動の分散データ所有。データプロダクトとして品質・SLO を定義。組織成熟度が前提条件 |
| 段階的移行（DWH → Lakehouse） | 既存 DWH を維持しつつ Lakehouse を段階的に導入。リスク最小化だが移行期間のコスト増 |
