# Skill: Diagram (作図 & 図解生成)

> **構造を可視化し、複雑さを一枚の図で制圧する**

VS Code + Draw.io Integration 環境向けに、編集可能な `.drawio` ファイルを生成するスキル。

## Instructions

### ワークフロー内の位置

```
distill → [diagram] → 成果物に埋め込み → review
             ↑                               ↓
        20_notes/ のモデル定義          40_decision_log/ に記録
```

### プロジェクトディレクトリ構造

| 番号 | ディレクトリ | 図面との関係 |
|:---|:---|:---|
| **20** | `20_notes/` | ドメインモデル・ビジネスフローの入力ソース |
| **30** | `30_proposal/` | 提案書に埋め込む図面の出力先 |
| **90** | `90_references/` | 参考図面・外部構成図の保管場所 |

## Step 1: 入力解析

ユーザーの指示から以下を特定する。

| 解析項目 | 確認内容 | デフォルト値 |
|:---|:---|:---|
| **図の対象** | 何を図にするか（システム構成、業務フロー等） | — (必須) |
| **出力形式** | `.drawio` or `.drawio.svg` | `.drawio` |
| **出力先** | ファイルパス | `outputs/` またはカレントディレクトリ |
| **クラウドアイコン** | Azure / AWS アイコンが必要か | 不要 |
| **入力ソース** | テキスト指示 / 既存ファイル参照 / 画像 | テキスト指示 |
| **図のサイズ感** | ノード数の見積もり（小: ~10、中: ~30、大: 50+） | 中 |

「図の対象」が不明な場合は必ずユーザーに質問する。

### 画像入力パターン

既存の図面やホワイトボード写真から draw.io を生成する場合:

| 入力方法 | 説明 | 例 |
|:---|:---|:---|
| **スクリーンショット貼り付け** | 既存の構成図画像を Claude にドラッグ&ドロップし、draw.io XML で再現する | 「この構成図の画像を draw.io で再現して」 |
| **ホワイトボード写真** | 写真からノード・エッジの構造を読み取り、整形された draw.io に変換する | 「このホワイトボードの図を draw.io にして」 |
| **画像ファイルパス** | `@path/to/image.png` でファイルを参照し、構造を解析する | 「@docs/arch-diagram.png を draw.io で再現して」 |

**チェックリスト**:
- [ ] 図の対象と種別を特定した
- [ ] 出力形式（.drawio / .drawio.svg）を決定した
- [ ] クラウドアイコンの要否を確認した
- [ ] 入力ソース（テキスト / ファイル / 画像）を確認した
- [ ] 図のサイズ感（ノード数）を見積もった

## Step 2: 図の種別判定とリファレンス読み込み

対象に応じて図の種別を判定し、必要なリファレンスを読み込む。

| 図の種別 | 例 | 読み込むリファレンス |
|:---|:---|:---|
| アーキテクチャ図 | システム構成、Azure/AWS 構成 | `references/cloud-icons.md` + `references/style-guide.md` |
| フローチャート | 業務フロー、ログインフロー | `references/style-guide.md` |
| シーケンス図 | API 呼び出し、処理順序 | `references/mxcell-structure.md` + `references/style-guide.md` |
| ER図 / クラス図 | データモデル、クラス設計 | `references/mxcell-structure.md` + `references/style-guide.md` |
| ネットワーク図 | VNet、サブネット構成 | `references/cloud-icons.md` + `references/style-guide.md` |
| 状態遷移図 | ステートマシン、画面遷移 | `references/mxcell-structure.md` + `references/style-guide.md` |
| C4 モデル図 | システムコンテキスト、コンテナ、コンポーネント | `references/style-guide.md` |
| ArchiMate 図 | エンタープライズアーキテクチャ（ビジネス/アプリ/テクノロジー） | `references/style-guide.md` |
| その他 | データフロー図、配置図等 | `references/mxcell-structure.md` + `references/style-guide.md` |

CRITICAL: Azure / AWS アイコンを使う場合は、XML 生成前に `references/cloud-icons.md` を必ず読み込むこと。

### 図の種別ごとの設計指針

| 種別 | ノード配置 | エッジスタイル | 推奨レイアウト |
|:---|:---|:---|:---|
| アーキテクチャ図 | 階層型（上→下 or 左→右） | 直線 + 矢印 | 3-4 層構造 |
| フローチャート | 上→下（判定は菱形） | 直線 + Yes/No ラベル | 縦長 |
| シーケンス図 | ライフライン横並び | 矢印（実線/破線） | 横長 |
| ER 図 | エンティティ群をグルーピング | リレーション線 + カーディナリティ | グリッド配置 |
| ネットワーク図 | ゾーン別（DMZ/内部/外部） | 接続線 + ポート番号 | ゾーン分割 |
| 状態遷移図 | 円形ノード + 開始/終了 | トリガー/アクション付き矢印 | 左→右 |
| C4 モデル図 | 4 レベル階層（L1→L4） | 境界線で範囲を明示 | レベルごとに分割 |
| ArchiMate 図 | 3 レイヤー（Business/Application/Technology） | 色分け（黄/青/緑） | レイヤー分割 |

**チェックリスト**:
- [ ] 図の種別を判定した
- [ ] 必要なリファレンスを特定した
- [ ] Azure/AWS アイコン使用時に cloud-icons.md を読み込んだ
- [ ] 種別に応じた配置・レイアウト方針を決定した

## Step 3: 構造設計

XML 生成の前に、図の構造を設計する。

### ノード・エッジの洗い出し

1. **ノード一覧**: 図に含めるすべての要素をリスト化（名称、種別、グループ）
2. **エッジ一覧**: ノード間の接続をリスト化（ソース、ターゲット、ラベル、方向）
3. **グルーピング**: 論理的なまとまりをコンテナ（グループノード）で囲む
4. **座標見積もり**: ノード数とグルーピングからレイアウト領域を見積もる

### レイアウト設計ルール

| ルール | 値 | 理由 |
|:---|:---|:---|
| ノード間の最小間隔 | 40px | 視認性確保 |
| グループ内パディング | 20px | コンテナとの余白 |
| ノードの標準サイズ | 120×60px | テキスト可読性 |
| クラウドアイコンサイズ | 50×50px | アイコン視認性 |
| キャンバスマージン | 40px | 全体の余白 |

### C4 モデルの設計ガイド

| レベル | 名称 | 対象 | draw.io での表現 |
|:---|:---|:---|:---|
| L1 | System Context | システムと外部アクターの関係 | 大きな矩形 + 人型アイコン |
| L2 | Container | アプリ、DB、メッセージキュー等 | 矩形 + テクノロジーラベル |
| L3 | Component | コンテナ内のコンポーネント | 矩形 + ステレオタイプ |
| L4 | Code | クラス・関数レベル（通常不要） | UML クラス図 |

L1 → L2 → L3 の順に詳細化し、必要に応じてマルチページ（Step 6）で管理する。

**チェックリスト**:
- [ ] 全ノードをリスト化した
- [ ] 全エッジ（接続）をリスト化した
- [ ] 論理グループを定義した
- [ ] レイアウト方針（方向、サイズ）を決定した

## Step 4: XML 生成

`references/mxcell-structure.md` の構造定義に従い、draw.io 互換の XML を生成する。

### 生成ルール

1. 各ノードの `id` は一意にする（`node-1`, `node-2`, ... 等の連番を推奨）
2. エッジの `source` / `target` は既存ノードの `id` を正確に参照する
3. `value` 属性内の HTML 特殊文字は必ずエスケープする（`&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;`）
4. Azure アイコンは `img/lib/azure2/...` 形式のみ使用する（`mxgraph.azure.*` 形式は使用禁止）
5. スタイルは `references/style-guide.md` の配色・レイアウト指針に従う
6. Write ツールで `.drawio` または `.drawio.svg` ファイルを出力する

```xml
<!-- OK: Azure2 形式 -->
<mxCell style="aspect=fixed;image=img/lib/azure2/networking/Front_Doors.svg;..." />

<!-- NG: 古い形式 (VS Code で青い四角形になる) -->
<mxCell style="shape=mxgraph.azure.front_door;..." />
```

### グループ化のパターン

```xml
<!-- グループコンテナ -->
<mxCell id="group-1" value="VNet: hub-vnet" style="group;rounded=1;fillColor=#dae8fc;strokeColor=#6c8ebf;" vertex="1" parent="1">
  <mxGeometry x="100" y="100" width="400" height="300" as="geometry" />
</mxCell>

<!-- グループ内ノード (parent="group-1") -->
<mxCell id="node-1" value="Firewall" style="..." vertex="1" parent="group-1">
  <mxGeometry x="20" y="40" width="120" height="60" as="geometry" />
</mxCell>
```

**チェックリスト**:
- [ ] 全ノード ID が一意である
- [ ] 全エッジの source/target が実在するノード ID を参照している
- [ ] value 属性の HTML 特殊文字がエスケープされている
- [ ] Azure アイコンが全て `img/lib/azure2/` 形式である
- [ ] グループ内ノードの parent 属性が正しい

## Step 5: レイアウト最適化

生成した XML のレイアウトを最適化する。

### 最適化の観点

| 観点 | チェック内容 | 修正方法 |
|:---|:---|:---|
| **重なり** | ノード同士が重なっていないか | 座標を再計算 |
| **交差** | エッジが不必要に交差していないか | ノード配置を調整 |
| **対称性** | 同じ層のノードが水平/垂直に揃っているか | Y/X 座標を統一 |
| **密度** | 情報が偏っている領域がないか | ノードを分散配置 |
| **フロー方向** | データの流れが自然か（上→下 or 左→右） | 座標を入れ替え |

### ラベル配置ルール

- **ノードラベル**: ノード中央配置（デフォルト）
- **エッジラベル**: エッジの中点付近（`style` に `align=center` を指定）
- **長いラベル**: 改行を挿入（`<br>` タグ使用）し、ノードサイズを拡大

**チェックリスト**:
- [ ] ノードの重なりがないか
- [ ] エッジの不要な交差がないか
- [ ] 同じ階層のノードが揃っているか
- [ ] ラベルが読みやすく配置されているか
- [ ] 全体のバランスが取れているか

## Step 6: 大規模図の分割管理

ノード数が 50 を超える場合、図を分割して管理する。

### 分割戦略

| 戦略 | 適用場面 | 例 |
|:---|:---|:---|
| **概要図 + 詳細図** | システム全体が大きい | L1: システム概要図、L2: サブシステム詳細図 |
| **ゾーン分割** | ネットワーク構成が広範 | DMZ 図、内部ネットワーク図、外部接続図 |
| **レイヤー分割** | アーキテクチャ層が多い | フロントエンド層、バックエンド層、データ層 |
| **時系列分割** | フローが長い | フェーズ 1 フロー、フェーズ 2 フロー |

### draw.io のマルチページ機能

```xml
<mxfile>
  <diagram id="page-1" name="L1: システム概要">
    <mxGraphModel>...</mxGraphModel>
  </diagram>
  <diagram id="page-2" name="L2: バックエンド詳細">
    <mxGraphModel>...</mxGraphModel>
  </diagram>
</mxfile>
```

**チェックリスト**:
- [ ] 50 ノード超の場合、分割戦略を選定したか
- [ ] 概要図と詳細図の間のナビゲーション（リンク or 凡例）があるか
- [ ] 分割した図の間で用語・配色が統一されているか
- [ ] マルチページの場合、ページ名が分かりやすいか

## Step 7: エクスポートと埋め込み

生成した図面を成果物に埋め込む方法。

### 出力形式の選択

| 拡張子 | 説明 | 推奨用途 | Markdown 埋め込み |
|:---|:---|:---|:---|
| `*.drawio` | 標準形式 | 編集・保守用（推奨） | 不可 |
| `*.drawio.svg` | SVG + メタデータ | ドキュメント埋め込み + 編集可 | `![図名](path.drawio.svg)` |
| `*.drawio.png` | PNG エクスポート | プレゼン・メール添付用 | `![図名](path.drawio.png)` |
| 設計判断記録 | Markdown テーブル | 図表種別の選定理由と 3 種以上の代替案比較表 |

### Markdown への埋め込みパターン

```markdown
## システム構成図

![システム構成図](./diagrams/system-architecture.drawio.svg)

> 詳細な編集は [system-architecture.drawio](./diagrams/system-architecture.drawio) を VS Code で開いてください。
```

**チェックリスト**:
- [ ] 用途に応じた出力形式を選択したか
- [ ] Markdown 埋め込み用に .drawio.svg も生成したか（必要な場合）
- [ ] 埋め込みパスが相対パスで正しいか

## Step 8: 検証

生成した XML を以下の観点でセルフチェックする。1つでも NG があれば Step 4 に戻って修正。

### 図面 Slop 防止チェック（Distributional Convergence 対策）

図面生成では「対象システム固有の構造を正確に可視化する」ことが本質であり、LLM がどのシステムでも同じレイアウト・同じ構成要素の「テンプレート図」を生成すると、図面が設計判断の根拠ではなく装飾になってしまう。

| # | 図面 Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| DG-1 | **3層アーキテクチャの常套化** — どのシステムでも「フロントエンド→バックエンド→データベース」の3層構成を生成し、対象システム固有のアーキテクチャスタイル（イベント駆動、マイクロサービス、CQRS 等）を反映しない | 図のレイヤー構造が対象システムの実際のアーキテクチャスタイルを反映しているか確認する。3層で十分なシステムなのか、それとも異なる構造が必要なのかを検証する | 対象システムの実際のアーキテクチャスタイルに基づいてレイアウトを設計する。イベント駆動ならイベントバスを中心に配置し、マイクロサービスならサービスメッシュ構成を反映する |
| DG-2 | **矢印スタイルの画一化** — すべての接続を同じ実線矢印で描き、通信プロトコル・データフロー方向・同期/非同期の区別をスタイルで表現しない | エッジのスタイル（実線/破線/太さ/色）が接続の意味（同期API呼出/非同期メッセージ/データ参照/管理通信）を区別しているか確認する | 接続の種類に応じてエッジスタイルを差別化する。同期 API は実線矢印、非同期メッセージは破線矢印、データフローは太線、管理通信は点線等で視覚的に区別する |
| DG-3 | **色の装飾的使用** — ノードやグループに色を付けるが、色に意味（セキュリティゾーン、デプロイ環境、責任範囲等）がなく、単に見栄えのために使用する | 図の配色に凡例があるか、色が一貫した意味を持っているか確認する。「なぜこの色か」を説明できるか検証する | 色に意味的な規則を持たせ、凡例で明示する。例: 青=内部ネットワーク、赤=DMZ、緑=外部サービス。または: 黄=ビジネス層、青=アプリ層、緑=テクノロジー層（ArchiMate 準拠） |
| DG-4 | **ラベルの汎用化** — ノードラベルが「Service A」「Module B」「Database」等の汎用名で、対象システム固有のサービス名・コンポーネント名・テクノロジー名が記載されていない | ノードラベルだけを読み、どのシステムの図か特定できるか確認する | ノードラベルに対象システム固有の名称を使用する。「Database」ではなく「PostgreSQL 15（学籍マスタDB）」、「API Server」ではなく「教育データ統合 API（FastAPI）」と記述する |
| DG-5 | **レイアウト方向の固定化** — フローチャートもアーキテクチャ図もシーケンス図もすべて左→右の同一方向で配置し、図の種別や情報の流れに応じたレイアウト変更をしない | 図の種別に応じた最適なレイアウト方向（Step 2 の設計指針）が採用されているか確認する | 図の種別と情報の流れに応じてレイアウト方向を選択する。フローチャートは上→下、ネットワーク図はゾーン分割、シーケンス図は横長ライフライン配置等、種別固有の方向を採用する |
| DG-6 | **クラウドアイコンの飾り化** — Azure/AWS アイコンを配置するが、実際のサービス構成（SKU、リージョン、冗長構成）を反映せず、アイコンが「それっぽく見せる」ためだけの装飾になっている | クラウドアイコンが実際に使用するサービス・SKU・構成を正確に表しているか確認する。使用していないサービスのアイコンが含まれていないか検証する | クラウドアイコンは実際のサービス構成を反映する。アイコン付近にSKU・インスタンスサイズ・リージョン等の技術仕様をラベルで付記する。使用しないサービスのアイコンは配置しない |

> **核心原則**: **設計判断の可視化** — 図面はシステムの「設計判断を視覚化した証拠」であり、見栄えのための装飾画ではない。「別のシステムの図に差し替えても違和感がないか？」→ 違和感がないなら図面 Slop である。

**チェックリスト**:
- [ ] XML が well-formed である（タグの開閉が正しい）
- [ ] 全ノードの `id` が一意である
- [ ] 全エッジの `source` / `target` が実在するノード `id` を参照している
- [ ] `value` 属性内の HTML 特殊文字がエスケープされている
- [ ] Azure アイコン使用時、全て `img/lib/azure2/` パスである
- [ ] ファイル拡張子が `.drawio` または `.drawio.svg` である
- [ ] ノード座標が重複せず、適切な間隔で配置されている
- [ ] グループ内ノードの parent 参照が正しい
- [ ] ラベルが図のサイズ内に収まっている
- [ ] VS Code Draw.io Integration で開いて表示崩れがないことを確認（可能な場合）
- [ ] 図面 Slop チェック（DG-1〜DG-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## VS Code 設定 (初回のみ)

1. `.drawio` ファイルを VS Code で開く
2. 左下の「+ その他の図形」をクリック
3. Azure および AWS にチェックを入れて「適用」

## 出力仕様

| 出力 | 形式 | 説明 |
|:---|:---|:---|
| アーキテクチャ図 | draw.io XML / Mermaid | 3 種以上のダイアグラム形式対応 |
| 設計判断記録 | Markdown テーブル | 図表種別の選定理由と代替案比較表 |
| 構成要素カタログ | Markdown 一覧 | システム構成要素の定義書 |

## Examples

### Example 1: 基本的なフローチャート

```
「ログインフローの図を作成して」

→ Step 1: 入力解析 — フローチャート、テキスト指示、.drawio 形式
→ Step 2: フローチャートと判定、style-guide.md を読み込み
→ Step 3: ノード洗い出し（開始→入力→認証→判定→成功/失敗→終了）
→ Step 4: 判定ノード（菱形）+ Yes/No 分岐の XML 生成
→ Step 5: 上→下のフロー配置を最適化
→ Step 8: ID 一意性・エッジ参照・HTML エスケープを検証
→ 出力: login-flow.drawio
```

### Example 2: Azure アーキテクチャ図

```
「Azure ハブ＆スポーク構成図を作成して (VNet, Firewall, Bastion を含む)」

→ Step 1: アーキテクチャ図、Azure アイコン必須、.drawio 形式
→ Step 2: cloud-icons.md + style-guide.md を読み込み
→ Step 3: ハブ VNet（Firewall, Bastion）+ スポーク VNet 2 つをリスト化
→ Step 4: Azure2 形式アイコン + VNet グループコンテナの XML 生成
→ Step 5: ハブ中央、スポーク左右に配置してレイアウト最適化
→ Step 8: 全アイコンが img/lib/azure2/ 形式であることを検証
→ 出力: azure-hub-spoke.drawio
```

### Example 3: ドキュメントからの図面生成

```
「inputs/requirements.md の「システム構成」セクションからアーキテクチャ図を作成して」

→ Step 1: ファイル参照入力、テキストから構造を解析
→ Step 2: アーキテクチャ図と判定、リファレンス読み込み
→ Step 3: 要件書からコンポーネント・接続を抽出しリスト化
→ Step 4: 3 層構造（フロント/バックエンド/データ）の XML 生成
→ Step 8: 要件書の全コンポーネントが図に含まれることを検証
→ 出力: system-architecture.drawio
```

### Example 4: スクリーンショットからの図面再現

```
「この構成図の画像を draw.io で再現して」

→ Step 1: 画像を解析し、ノード（サーバー、DB、LB）とエッジ（通信経路）を特定
→ Step 2: アーキテクチャ図と判定、cloud-icons.md + style-guide.md を読み込み
→ Step 3: 画像内の要素をノード・エッジ一覧にリスト化
→ Step 4: 画像内の配置を再現する座標で XML 生成
→ Step 5: 元画像のレイアウトに合わせて座標を微調整
→ Step 8: ノード数・エッジ数が画像と一致することを検証
→ 出力: reproduced-architecture.drawio
```

### Example 5: ER 図の生成

```
「20_notes/domain-model.md のエンティティ定義から ER 図を draw.io で作成して」

→ Step 1: domain-model.md からエンティティ・属性・リレーションを抽出
→ Step 2: ER 図と判定、mxcell-structure.md + style-guide.md を読み込み
→ Step 3: エンティティごとにテーブル形式ノード（PK/FK/属性リスト）を設計
→ Step 4: テーブルノード + リレーションエッジ（カーディナリティ付き）の XML 生成
→ Step 5: グリッド配置でエンティティを整列
→ Step 8: 全エンティティが図に含まれ、カーディナリティが正確であることを検証
→ 出力: domain-model-er.drawio
```

### Example 6: シーケンス図の生成

```
「ユーザー認証の API 呼び出しシーケンス図を作成して」

→ Step 1: シーケンス図、テキスト指示、.drawio.svg 形式（ドキュメント埋め込み用）
→ Step 2: mxcell-structure.md + style-guide.md を読み込み
→ Step 3: ライフライン（Client, API Gateway, Auth Service, DB）とメッセージを設計
→ Step 4: ライフライン縦線 + メッセージ矢印（実線: 要求 / 破線: 応答）の XML 生成
→ Step 5: 横長レイアウトでライフライン間隔を均等に配置
→ Step 8: メッセージの順序と方向（要求→応答）が正しいことを検証
→ 出力: auth-sequence.drawio.svg
```

### Example 7: 大規模構成図の分割管理

```
「マイクロサービス 15 個の全体構成図を作成して」

→ Step 1: ノード数 50+ と見積もり、分割管理が必要と判断
→ Step 6: 概要図（L1: サービス間通信図）+ 詳細図（L2: 各サービス内部）の 2 層に分割
→ Step 3-4: L1 概要図を先に生成（15 サービス + API Gateway + Message Bus）
→ Step 3-4: L2 詳細図を主要サービス 3 つ分生成（内部コンポーネント + DB 接続）
→ Step 7: マルチページ .drawio で L1/L2 を 1 ファイルに統合
→ Step 8: 全サービスが L1 に含まれ、L2 との命名一貫性を検証
→ 出力: microservices-architecture.drawio（4 ページ構成）
```

## Troubleshooting

| 問題 | 原因 | 対策 |
|:---|:---|:---|
| 図が空白になる | `value` 属性の HTML 特殊文字が未エスケープ | `&`, `<`, `>` をエスケープする |
| アイコンが青い四角になる | Azure アイコンが旧形式 `mxgraph.azure.*` | [cloud-icons.md](references/cloud-icons.md) を参照し `img/lib/azure2/` 形式に修正 |
| エッジが表示されない | `source` / `target` の ID 参照が不正 | ノード ID の一意性と参照整合性を確認 |
| ファイルが開けない | XML が well-formed でない | XML パーサーでバリデーションし、タグの開閉を修正 |
| ノードが重なる | 座標 (`x`, `y`) が未指定または重複 | 各ノードに適切な座標を設定。`references/style-guide.md` 参照 |
| 画像入力で構造が誤認識される | 画像の解像度が低い / 手書きが不明瞭 | ユーザーにノードとエッジのリストを口頭で確認し、テキスト入力と併用する |
| 大規模な図でレイアウトが崩れる | ノード数が多すぎる（50+） | Step 6 の分割戦略で概要図 + 詳細図の 2 層構成にする |
| AWS アイコンのパスが不明 | cloud-icons.md に未掲載のサービス | `img/lib/aws4/` 配下を検索し、正式パスを特定 |
| SVG 形式で文字化けする | 日本語フォントの埋め込み不足 | `fontFamily` に Web セーフフォント（sans-serif）を指定 |
| VS Code で図が重い | ノード数 100+ でレンダリング遅延 | 分割管理（Step 6）でページあたり 50 ノード以下に制限 |
| グループ内ノードがグループ外に表示される | `parent` 属性の設定漏れ | グループ内ノードの `parent` をグループ ID に設定 |

## アンチパターン検出

このスキルの出力品質を検証し、AI Decay（品質劣化）を防止するためのアンチパターン検出リスト。各項目は品質劣化パターンの検出条件を定義し、不合格時は対応する対策（修正ガイドライン）に従って改善する。

- [ ] ノード数が 1 ページあたり 50 以下に制限されているか（超過時は分割管理）
- [ ] グループ内ノードの `parent` 属性が正しく設定されているか
- [ ] AWS/Azure アイコンのパスが cloud-icons.md の正式パスと一致しているか
- [ ] `fontFamily` に Web セーフフォント（sans-serif）が指定されているか（日本語フォント文字化け防止）
- [ ] XML 構造が mxCell の仕様に準拠しているか（style 属性の構文エラーがないか）
- [ ] 概要図と詳細図の 2 層構成が適切に設計されているか（大規模図面の場合）
