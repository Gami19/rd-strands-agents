# Presentation Structures Reference

> テクニカルプレゼンテーションの構成パターン・聴衆別設計・物語弧・Q&A 戦略の実践リファレンス

---

## 1. 構成パターンカタログ

### パターン A: 問題提起型（Problem-Solution-Impact）

最も汎用的なテクニカルプレゼン構成。技術提案・アーキテクチャ刷新に適する。

| スライド群 | 内容 | 枚数目安 | メッセージ類型 |
|:---|:---|:---|:---|
| Opening | 現状の痛み・課題を数値で提示 | 1-2 枚 | Type A（事実共有） |
| Root Cause | 原因分析・技術的負債の構造化 | 2-3 枚 | Type B（示唆） |
| Proposed Solution | 提案するアーキテクチャ・アプローチ | 3-4 枚 | Type C（アクション） |
| Expected Impact | 定量的な改善見込み・ROI | 1-2 枚 | Type D（判断促進） |
| Next Steps | 具体的なアクションプラン | 1 枚 | Type C（アクション） |

**適する場面**: 技術提案、アーキテクチャ刷新、レガシー移行の承認取得

**物語弧**: 痛み → 原因の深掘り → 光明 → 期待効果 → 行動喚起

---

### パターン B: 比較分析型（Criteria-Options-Recommendation）

技術選定・ベンダー比較に最適。聴衆の意思決定を構造的に支援する。

| スライド群 | 内容 | 枚数目安 | メッセージ類型 |
|:---|:---|:---|:---|
| Context | 背景と意思決定が必要な理由 | 1-2 枚 | Type A（事実共有） |
| Evaluation Criteria | 評価軸の定義と重み付け | 1-2 枚 | Type B（示唆） |
| Options Analysis | 各選択肢の定量比較 | 3-5 枚 | Type A（事実共有） |
| Trade-off Summary | トレードオフの明示 | 1 枚 | Type B（示唆） |
| Recommendation | 推奨案と根拠 | 1-2 枚 | Type D（判断促進） |

**適する場面**: フレームワーク選定、クラウドベンダー比較、Build vs Buy 判断

**物語弧**: 問い → 判断基準の合意 → 客観的分析 → トレードオフの可視化 → 推奨

---

### パターン C: デモ駆動型（Concept-Demo-Implication）

PoC 結果の共有や新技術の紹介に適する。「百聞は一見にしかず」を体現する。

| スライド群 | 内容 | 枚数目安 | メッセージ類型 |
|:---|:---|:---|:---|
| Concept | 技術コンセプトの簡潔な説明 | 1-2 枚 | Type A（事実共有） |
| Live Demo / Walkthrough | 実際の動作・コード・結果 | 3-5 枚 | Type C（アクション） |
| Benchmark Data | パフォーマンス・品質の定量データ | 1-2 枚 | Type A（事実共有） |
| Implications | 本番適用時の影響・課題・計画 | 2-3 枚 | Type B（示唆） |
| Ask | 必要なリソース・承認事項 | 1 枚 | Type D（判断促進） |

**適する場面**: PoC 報告、新ツール導入提案、ハッカソン成果発表

**物語弧**: 仮説 → 実証 → 数値 → 含意 → 次の一手

---

### パターン D: 進捗報告型（Status-Risks-Plan）

ステアリングコミッティや定期報告に適する。経営層の意思決定に必要な情報を凝縮する。

| スライド群 | 内容 | 枚数目安 | メッセージ類型 |
|:---|:---|:---|:---|
| Executive Summary | 全体ステータスを1枚で集約 | 1 枚 | Type A（事実共有） |
| Progress | マイルストーン進捗・KPI | 2-3 枚 | Type A（事実共有） |
| Risks & Issues | リスク・課題・ブロッカー | 1-2 枚 | Type B（示唆） |
| Mitigation Plan | リスク緩和策・代替案 | 1-2 枚 | Type C（アクション） |
| Decisions Needed | 経営判断が必要な事項 | 1 枚 | Type D（判断促進） |

**適する場面**: 月次報告、フェーズゲートレビュー、ステアリングコミッティ

**物語弧**: 全体像 → 詳細 → 警告 → 対策 → 判断要請

## 2. 聴衆分析に基づく構成選択

### 選択マトリクス

| 聴衆 | 推奨パターン | 枚数 | キーポイント |
|:---|:---|:---|:---|
| **経営層（CxO）** | D → A | 5-8 枚 | 数字先行。技術詳細は Appendix に |
| **技術リーダー** | B → A | 10-15 枚 | トレードオフと根拠を厚く |
| **開発チーム** | C → A | 15-25 枚 | 動くもの・コード例を中心に |
| **混合聴衆** | A（階層構造） | 8-12 枚 | エグゼクティブサマリー + 段階的詳細化 |
| **外部顧客** | A → B | 10-15 枚 | 課題共感 → 差別化 → 実績 |

### 聴衆の認知スタイル別アプローチ

| 認知スタイル | 特徴 | プレゼン設計の原則 |
|:---|:---|:---|
| **データ駆動型** | 数値と根拠で判断 | ベンチマーク・比較表・ROI 計算を前面に |
| **ビジョン駆動型** | 将来像と可能性で判断 | あるべき姿のビジュアル → ギャップ → ロードマップ |
| **合意形成型** | チームの総意を重視 | 選択肢提示 → 議論誘導 → 合意ポイントの確認 |
| **リスク回避型** | 失敗回避を最優先 | 現状維持リスク → 段階的移行 → ロールバック計画 |

## 3. 物語弧設計（SCR フレームワーク）

### Situation → Complication → Resolution

McKinsey の SCR フレームワークをテクニカルプレゼンに適用する。

```
Situation（状況）
  → 聴衆が「そうだ、それは事実だ」と頷く現状の描写
  → 共有コンテキストの確立（データ・事実ベース）

Complication（複雑化）
  → 「しかし、このままでは...」という緊張の導入
  → 現状のままでは達成できないこと・失うもの

Resolution（解決）
  → 提案する解決策とその根拠
  → 聴衆にとっての具体的なベネフィット
```

### 物語弧のバリエーション

| 弧の形 | 構造 | 適する場面 |
|:---|:---|:---|
| **単純弧** | S → C → R | シンプルな技術提案 |
| **二段弧** | S → C1 → R1 → C2 → R2 | 短期策と長期策を分離提案 |
| **逆転弧** | R(結論先行) → S → C → R(詳細) | 経営層向け（結論を先に知りたい） |
| **探索弧** | S → C → Q(問い) → 分析 → R | 技術選定（一緒に考える形式） |

### スライド間の接続語パターン

プレゼンの流れを滑らかにする接続語:

| 遷移タイプ | 接続パターン | 例 |
|:---|:---|:---|
| 事実→示唆 | 「つまり、これは...を意味します」 | 「SLA 違反が月 3 回。つまり、四半期で解約リスクが...」 |
| 示唆→アクション | 「そこで提案するのが...」 | 「この状況を打開するために...」 |
| アクション→判断 | 「これを実現するために必要なのが...」 | 「ご承認いただきたいのは...」 |
| セクション切替 | 「ここからは視点を変えて...」 | 「技術面を確認したので、次はコスト面から...」 |

## 4. Q&A 対策フレームワーク

### 4a. 事前準備: 想定質問マトリクス

| 聴衆カテゴリ | 典型的な質問テーマ | 準備すべき回答素材 |
|:---|:---|:---|
| **ビジネス** | コスト、期間、ROI、リスク | 見積比較表、TCO 計算、リスクマトリクス |
| **技術** | スケーラビリティ、セキュリティ、パフォーマンス | ベンチマーク結果、アーキテクチャ図、制約一覧 |
| **実装** | 学習コスト、移行手順、既存コードへの影響 | 移行計画、PoC 結果、サンプルコード |

### 4b. 回答テクニック（PREP 法）

```
Point    — 結論を最初に述べる
Reason   — 理由・根拠を示す
Example  — 具体例・データで補強する
Point    — 結論を繰り返す
```

### 4c. 難しい質問への対処パターン

| パターン | 状況 | 対処法 |
|:---|:---|:---|
| **Bridge（橋渡し）** | 質問が本題から逸れる | 「その点は重要です。関連して、今日の提案では...」 |
| **Defer（後回し）** | 今答えられない / 調査が必要 | 「正確にお答えするために、X を確認して Y 日までにご報告します」 |
| **Reframe（再構成）** | 質問の前提が誤っている | 「少し視点を変えると、実はこの問題は...という構造です」 |
| **Acknowledge（受容）** | 正当な批判・懸念 | 「ご指摘の通りです。そのリスクに対しては...を計画しています」 |
| **Redirect（委譲）** | 自分の専門外 | 「その領域は [名前] が詳しいので、補足をお願いします」 |

### 4d. Q&A で避けるべきアンチパターン

| アンチパターン | なぜ危険か | 代替行動 |
|:---|:---|:---|
| 質問を遮る | 信頼を損なう | 最後まで聞き、要約してから回答 |
| 「わかりません」で終わる | 無能に見える | 「調査して X 日までに回答します」と期限を示す |
| 防御的になる | 対立構図を作る | 感謝→受容→対案の順で応答 |
| 長すぎる回答 | 焦点がぼける | 30 秒ルール: 回答は 30 秒以内、詳細は後で個別に |
| 質問者を否定する | 心理的安全性を破壊 | 「興味深い視点ですね」で始める |

## 5. スライド設計の原則

### 1 スライド 1 メッセージの実践

| レベル | 情報密度 | 文字数目安 | 要素数 |
|:---|:---|:---|:---|
| **経営層向け** | 低 | 50 文字以下 | メッセージ + 1 ビジュアル |
| **技術リーダー向け** | 中 | 100 文字以下 | メッセージ + 比較表 or 図 |
| **開発チーム向け** | 高 | 150 文字以下 | メッセージ + コード例 or 詳細図 |

### ビジョンとプラグマティズムの配分

| プレゼン種別 | ビジョン:プラグマティズム | 理由 |
|:---|:---|:---|
| 戦略提案（経営層） | 60:40 | 将来像でインスピレーションを与えつつ、実現可能性で安心感 |
| 技術選定（アーキテクト） | 30:70 | トレードオフと実証データが中心 |
| チーム説得（開発者） | 40:60 | 理想を示しつつ、具体的な実装パスを明確に |
| 顧客提案（外部） | 50:50 | 差別化ビジョン + 実績・体制で裏付け |

## 6. 構成パターンの組み合わせ

### 複合構成の例

**経営層 → 技術リーダーへの二段階プレゼン**:

```
[経営層 8 枚]  パターン D（進捗報告型）
  → Executive Summary → Impact → Decision Ask
  → 技術詳細は「次のセッションで」と予告

[技術リーダー 15 枚]  パターン B（比較分析型）
  → 評価基準の合意 → 3 案の詳細比較 → 推奨案の技術根拠
  → Appendix にベンチマーク全データ
```

**混合聴衆への階層化プレゼン**:

```
[全員 3 枚]    Executive Summary（パターン A の圧縮版）
[経営層 5 枚]  ビジネスインパクト + ROI + リスク
[技術者 7 枚]  アーキテクチャ設計 + トレードオフ + 移行計画
[全員 2 枚]    Next Steps + Decision Ask
```
