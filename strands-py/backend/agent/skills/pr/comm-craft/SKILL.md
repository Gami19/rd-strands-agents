# Skill: Comm Craft（技術コミュニケーション・交渉・リーダーシップ）

> **伝えるのではなく、伝わる設計をせよ — 理由が先、結論は後、実証が最強**

## strands-py / Strands Agent での利用

聴衆分析・メッセージ構造・交渉・プレゼン骨格は **`file_read` / `file_write`** とプロジェクト workspace（`notes` / `proposal` 等）を正とする。ツール名やパスの読み替えは [references/strands-py-runtime.md](references/strands-py-runtime.md) を参照。

## Instructions

### ワークフロー内の位置

```
要件・課題の把握 → [comm-craft] → コミュニケーション成果物
                      ↓
                 ・聴衆分析レポート
                 ・メッセージ構造（スライド骨格 / 交渉スクリプト）
                 ・チーム効果性の改善提案
                      ↓
              slide-gen / kc-slide-gen（PPTX 化）
              drm（DRM 会議への適用）
              review（品質検証）
```

### 入力

| 入力 | 説明 | 例 |
|:---|:---|:---|
| コミュニケーションの目的 | 何を達成したいか | 「アーキテクチャ刷新の承認を得たい」 |
| 聴衆 | 誰に伝えるか | 「経営層」「開発チーム」「他のアーキテクト」 |
| コンテキスト | 背景情報・制約 | 「予算が限られている」「チームが反対している」 |
| 伝えたい内容 | 技術的な判断・提案・変更 | 「マイクロサービスへの移行提案」 |

### 出力

| 出力 | 形式 | 説明 |
|:---|:---|:---|
| 聴衆分析 | Markdown テーブル | 聴衆の関心事・懸念・意思決定基準の整理 |
| メッセージ構造 | Markdown | 聴衆別のメッセージフレーミング |
| 交渉戦略 | Markdown | 交渉シナリオと対応テクニックの選定 |

## Step 1: コミュニケーションコンテキスト分析

聴衆・目的・制約を構造化し、コミュニケーション戦略の基盤を構築する。

### 1a. 聴衆の特定と分類

聴衆を以下の 3 カテゴリに分類する:

| カテゴリ | 典型的な人物 | 関心事 | 意思決定基準 |
|:---|:---|:---|:---|
| **ビジネスステークホルダー** | VP、プロダクトオーナー、経営層 | コスト、時間、ROI、リスク | ビジネスインパクト |
| **技術ステークホルダー** | 他のアーキテクト、テックリード | 技術的正確性、実現可能性、トレードオフ | 技術的根拠 |
| **実装チーム** | 開発者、QA エンジニア | 実装の明確さ、制約の妥当性、自律性 | 実用性と公正さ |

### 1b. 聴衆の懸念マッピング

各聴衆について以下を特定する:

- **表明された懸念**: 聴衆が明示的に述べている問題
- **隠れた懸念**: バズワードや行動から推測される本当の関心事（NT-1: バズワード解読）
- **意思決定の軸**: 何があれば「Yes」と言えるか

### 1c. 目的の明確化

| 要素 | 質問 | 記入 |
|:---|:---|:---|
| **ゴール** | この場で何を達成したいか？ | _____ |
| **成功基準** | 何が起きれば成功か？ | _____ |
| **最低ライン** | 最低限何を持ち帰りたいか？ | _____ |
| **リスク** | 失敗した場合の影響は？ | _____ |

**チェックリスト**:
- [ ] 聴衆を 3 カテゴリ（ビジネス/技術/実装チーム）で分類した
- [ ] 各聴衆の表明された懸念と隠れた懸念を特定した
- [ ] コミュニケーションのゴール・成功基準・最低ラインを定義した
- [ ] バズワードの裏にある本当の関心事を解読した

## Step 2: メッセージの構造化

聴衆に応じてメッセージを最適な形にフレーミングする。

### 2a. 理由先行の原則（Justification First）

すべてのメッセージは以下の順序で構築する:

```
1. WHY  — なぜこの変更・判断が必要か（理由・背景）
2. WHAT — 具体的に何を提案するか（内容）
3. HOW  — どのように実現するか（方法・スケジュール）
4. SO WHAT — 聴衆にとっての意味（インパクト・ベネフィット）
```

> **原則**: 人は反対意見を聞いた瞬間に聞く耳を閉じる。理由を先に伝えることで受容率が上がる。

### 2b. 聴衆別メッセージフレーミング

| 聴衆 | フレーミングの軸 | 使う言語 | 避ける言語 |
|:---|:---|:---|:---|
| **ビジネスステークホルダー** | ROI、リスク軽減、市場投入速度 | 「ビジネスインパクト」「コスト削減」「期間短縮」 | 技術用語、実装の詳細 |
| **技術ステークホルダー** | トレードオフ、アーキテクチャ特性、実証データ | 「可用性」「スケーラビリティ」「結合度」 | 曖昧な表現、根拠のない主張 |
| **実装チーム** | 実装の明確さ、自律性の範囲、具体的ガイダンス | 「検討してみて」「どう思う？」 | 「〜しなさい」「〜する必要がある」 |

### 2c. 抽象的要求の定量化（NT-3）

曖昧な要求を具体的な数値に変換する:

| 曖昧な表現 | 定量化の例 |
|:---|:---|
| 「高可用性」 | 99.9% = 年間 8 時間 46 分のダウンタイム |
| 「高速」 | P95 レイテンシ 200ms 以下 |
| 「スケーラブル」 | 同時接続 10,000 → 100,000 への線形スケール |
| 「セキュア」 | OWASP Top 10 準拠 + SOC 2 Type II 認証 |

### 2d. 4Cs チェック

メッセージが以下を満たしているか検証する:

| C | 質問 |
|:---|:---|
| **Communication（伝達）** | メッセージは聴衆に確実に届く形か？ |
| **Collaboration（協働）** | 一方的な通知ではなく、対話を促進するか？ |
| **Clear（明瞭）** | 曖昧さや誤解の余地はないか？ |
| **Concise（簡潔）** | 偶発的複雑さ（accidental complexity）を排除したか？ |

**チェックリスト**:
- [ ] メッセージを WHY → WHAT → HOW → SO WHAT の順序で構築した
- [ ] 聴衆に応じた言語とフレーミングを選択した
- [ ] 抽象的な要求を具体的数値に変換した
- [ ] 4Cs（Communication, Collaboration, Clear, Concise）を検証した
- [ ] 偶発的複雑さを排除した

## Step 3: 交渉戦略の設計

具体的な交渉シナリオに対して、適切なテクニックを選定・組み合わせる。

### 3a. 交渉テクニック選定マトリクス

| テクニック | ビジネス交渉 | アーキテクト間 | 開発者交渉 | 効果が高い場面 |
|:---|:---|:---|:---|:---|
| **NT-1: バズワード解読** | ★★★ | ★ | ★ | 相手が曖昧な要求を出す時 |
| **NT-2: 事前情報収集** | ★★★ | ★★★ | ★★ | すべての交渉の準備段階 |
| **NT-3: 定量化** | ★★★ | ★★ | ★ | 抽象的な要求を議論する時 |
| **NT-4: 分割統治** | ★★★ | ★★ | ★ | 過大な要求を切り崩す時 |
| **NT-5: コスト・時間（最終手段）** | ★★ | ★ | ★ | 他の手段が尽きた時のみ |
| **NT-6: 実証による決着** | ★ | ★★★ | ★★ | 技術的意見が対立する時 |
| **NT-7: 冷静さの維持** | ★★ | ★★★ | ★★ | 議論が感情的になった時 |
| **NT-8: 自己発見の促進** | ★ | ★★ | ★★★ | 相手が別案を強く主張する時 |
| **NT-9: 命令の質問化** | ★ | ★★ | ★★★ | 指示・依頼を伝える時 |
| **NT-10: 依頼の好意化** | ★ | ★ | ★★★ | 忙しいメンバーへの依頼時 |

### 3b. 交渉シナリオの設計

交渉ごとに以下のフレームワークで戦略を立てる:

```
■ 交渉シナリオ設計シート

1. 相手の立場
   - 表明している主張: _____
   - 推測される本当の関心事: _____
   - 相手にとっての Win: _____

2. 自分の立場
   - 達成したいゴール: _____
   - 譲歩できる範囲: _____
   - 譲歩できない一線: _____

3. 戦略
   - 使用するテクニック（優先順）: NT-_ → NT-_ → NT-_
   - 事前に準備する実証データ: _____
   - 想定される反論と対応: _____

4. 合意形成の出口
   - Win-Win シナリオ: _____
   - 最低限の合意ライン: _____
   - 合意に至らない場合の次のアクション: _____
```

### 3c. 交渉における言語パターン

| NG パターン | OK パターン | 理由 |
|:---|:---|:---|
| 「〜しなさい」「〜する必要がある」 | 「〜を検討しましたか？」 | 制御を相手に戻す |
| 「それはダメだ」 | 「こういう懸念があるのですが、どう思いますか？」 | 協働を維持する |
| 「コストがかかりすぎる」（冒頭で） | 「技術的にはこの理由で X が適切です」 | コスト議論は最後の手段 |
| 「私がシニアだから」 | 「この環境での検証結果を見てください」 | 権威ではなく実証 |
| 「何を（What）」から始める | 「なぜ（Why）」から始める | 理由先行で受容率向上 |

**チェックリスト**:
- [ ] 交渉相手のカテゴリ（ビジネス/技術/実装）を特定した
- [ ] 適切なテクニックを選定マトリクスから選んだ
- [ ] 交渉シナリオ設計シートを記入した
- [ ] 事前に必要な情報・データを洗い出した
- [ ] NG 言語パターンを排除し、OK パターンに変換した
- [ ] Win-Win の出口と最低合意ラインを設定した

## Step 4: プレゼンテーション設計

スライドの物語構造を設計する（PPTX 生成自体は slide-gen / kc-slide-gen に委譲）。

### 4a. プレゼン構成の 3 パターン

| パターン | 構成 | 適する場面 |
|:---|:---|:---|
| **問題解決型** | 現状の課題 → 原因分析 → 提案 → 期待効果 → Next Steps | 技術提案、アーキテクチャ刷新 |
| **比較評価型** | 背景 → 選択肢の提示 → 評価基準 → 比較分析 → 推奨案 | 技術選定、ベンダー比較 |
| **進捗報告型** | エグゼクティブサマリー → 進捗 → 課題・リスク → 計画 → 依頼事項 | 定期報告、ステアリングコミッティ |

### 4b. 聴衆別スライド設計原則

| 聴衆 | スライド枚数 | 情報密度 | 強調すべき要素 |
|:---|:---|:---|:---|
| **経営層** | 5-8 枚 | 低（1 スライド 1 メッセージ） | ビジネスインパクト、ROI、リスク |
| **技術リーダー** | 10-15 枚 | 中（根拠とトレードオフ） | アーキテクチャ図、比較表、実証データ |
| **開発チーム** | 15-25 枚 | 高（詳細な設計と手順） | コード例、設計図、チェックリスト |

### 4c. Pragmatic Visionary のバランス

プレゼンには「ビジョン（将来像）」と「プラグマティズム（現実的制約）」の両方を含める:

```
■ ビジョン要素（聴衆にインスピレーションを与える）
  - 将来のあるべき姿
  - 戦略的な方向性
  - 長期的なベネフィット

■ プラグマティック要素（聴衆に安心感を与える）
  - 予算・時間の制約を反映した計画
  - チームのスキルセットを考慮した段階的アプローチ
  - トレードオフの明示と判断根拠
  - リスクと緩和策
```

**チェックリスト**:
- [ ] プレゼン構成パターンを選択した
- [ ] 聴衆に応じたスライド枚数と情報密度を設定した
- [ ] 各スライドが「1 スライド 1 メッセージ」を満たしている
- [ ] ビジョンとプラグマティズムのバランスが取れている
- [ ] 理由先行（WHY → WHAT → HOW → SO WHAT）の流れになっている

## Step 5: チーム効果性パターンの適用

チームのコミュニケーション課題を診断し、改善策を提案する。

### 5a. Elastic Leadership 診断

5 要因で現在のチーム状態を評価し、アーキテクトの適切な関与度を判定する:

| 要因 | 評価（-20〜+20） | 判定基準 |
|:---|:---|:---|
| **チーム熟知度** | ___ | 相互理解が深い(-20) ↔ 新しいメンバーが多い(+20) |
| **チームサイズ** | ___ | 小規模: 5人以下(-20) ↔ 大規模: 12人以上(+20) |
| **全体的な経験値** | ___ | シニア中心(-20) ↔ ジュニア中心(+20) |
| **プロジェクト複雑度** | ___ | 単純(-20) ↔ 非常に複雑(+20) |
| **プロジェクト期間** | ___ | 短期: 2ヶ月(-20) ↔ 長期: 2年(+20) |
| **合計スコア** | ___ | -100〜-40: 放任 / -40〜+40: バランス / +40〜+100: 積極関与 |

### 5b. チーム警告サインの検出

| 警告サイン | 検出方法 | 対応策 |
|:---|:---|:---|
| **プロセスロス（Brooks の法則）** | マージコンフリクトの頻発、同一コードへの複数人作業 | 並行作業ストリームの特定。並行化できないなら人員追加を拒否 |
| **多元的無知（裸の王様）** | 会議中の表情・ボディランゲージ、全員一致の不自然さ | 懐疑的なメンバーに直接意見を求め、発言を支持する |
| **責任の拡散** | タスクの漏れ、「誰かがやるだろう」の空気、担当不明の問題 | 責任の明示的な割り当て、チームサイズの適正化 |

### 5c. 制約の「部屋」の設計

適切な制約を定義し、チームが効果的に作業できる空間を作る:

| 領域 | 開発者の裁量 | アーキテクト承認が必要 | アーキテクトの責任 |
|:---|:---|:---|:---|
| **特殊用途ライブラリ** | 自由に決定可 | — | — |
| **汎用ライブラリ** | 分析・推薦 | 承認が必要 | — |
| **フレームワーク** | — | — | 完全にアーキテクトの判断 |
| **コンポーネント内部設計** | 自由に決定可 | — | — |
| **コンポーネント間インターフェース** | — | — | アーキテクトが定義 |

### 5d. チェックリスト戦略

チェックリストを導入する際のガイドライン:

- **適切な対象**: 手順の順序が固定されないプロセス、頻繁にステップが抜ける作業
- **不適切な対象**: 依存関係のある手続き的フロー、エラーなく頻繁に実行される単純作業
- **ホーソン効果の活用**: 「チェックリストの遵守を確認する」と伝え、実際はスポットチェックのみ
- **当たり前を恐れない**: 「当たり前のことこそ見落とされる」

**チェックリスト**:
- [ ] Elastic Leadership の 5 要因で関与度を診断した
- [ ] チーム警告サイン（プロセスロス/多元的無知/責任の拡散）を検出した
- [ ] 制約の「部屋」を設計し、裁量と制約のバランスを定義した
- [ ] 必要なチェックリストを特定し、過剰にならない範囲で設計した

## Step 6: アンチパターン検出と修正

コミュニケーションや交渉に潜むアンチパターンを検出し、修正する。

### 6a. アーキテクト人格アンチパターン

| アンチパターン | 症状 | 検出の質問 | 修正アクション |
|:---|:---|:---|:---|
| **コントロールフリーク** | すべての詳細を管理、疑似コードを書く、命名規則まで指示 | 「実装の詳細に踏み込みすぎていないか？」 | 論理コンポーネントと相互作用の定義に集中。内部設計は開発者に委ねる |
| **アームチェア** | コードを書かない、実装を考慮しない、チームと疎遠 | 「最後にコードを書いたのはいつか？チームとの対話頻度は？」 | 技術への関与を増やし、チームの近くに座る |
| **象牙の塔** | 上から命令、意見を聞かない、理由を説明しない | 「判断の理由を常に説明しているか？」 | 理由先行の原則を適用、協働的な言語を使う |
| **偶発的複雑さ** | 不必要に複雑な図面・ドキュメント・設計 | 「この複雑さは本質的か、偶発的か？」 | 4Cs を適用し、明瞭さと簡潔さを追求する |

### 6b. コミュニケーションアンチパターン

| アンチパターン | 例 | 修正 |
|:---|:---|:---|
| **命令形の多用** | 「〜しなさい」「〜する必要がある」 | 「〜を検討した？」「〜はどう思う？」 |
| **理由なき指示** | 「Business 層を通せ」 | 「変更制御のため closed-layered にしている。つまり…」 |
| **冒頭でのコスト言及** | 「それはコストがかかりすぎる」 | 技術的根拠で先に説得し、コストは最後の手段 |
| **公開の場での否定** | 「それはバカげたアイデアだ」 | 「面白い視点ですね。こういう観点からはどうでしょう？」 |
| **個人攻撃** | 「あなたは間違っている」 | 「このアプローチにはこういうリスクがあります」 |

### 6c. コミュニケーション Slop 防止チェック（Distributional Convergence 対策）

LLM が生成するコミュニケーション戦略・交渉スクリプト・プレゼン構成は、ビジネス書やコンサル資料で頻出するフレームワークに収束しやすい。聴衆の組織文化・政治的力学・個人の意思決定スタイルを反映しない「汎用コミュニケーション設計」は、形式的には正しくても聴衆の心に届かない。

| # | コミュニケーション Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| CC-1 | **問題→解決→アクション構成の固定化** | プレゼン構成がすべて「課題提起→解決策→Next Steps」の同一フローになっていないか確認 | 聴衆の意思決定スタイルに応じて構成を変える。データ重視の聴衆には比較評価型、ビジョン重視には将来像→現状ギャップ→ロードマップ型、合意形成型には選択肢提示→議論誘導型 |
| CC-2 | **Power/Interest マトリクス一辺倒** | ステークホルダー分析がすべて2x2マトリクス（権力×関心）で処理されていないか確認 | 組織固有の力学を反映する。公式の権限と非公式の影響力の乖離、キーパーソンの意思決定パターン（データ駆動/直感/合議制）、組織文化（Westrum モデル）を個別に分析 |
| CC-3 | **Win-Win 協調路線の無条件適用** | すべての交渉戦略が「協調的・Win-Win」で設計されていないか確認 | 交渉の文脈に応じてアプローチを変える。時間制約がある場合は NT-4（分割統治）で迅速に合意、技術的対立には NT-6（実証）で決着、政治的抵抗には段階的説得（小さな合意の積み重ね）|
| CC-4 | **SCQA/Minto 構造の画一適用** | メッセージ構造がすべて Situation→Complication→Question→Answer の同一パターンか確認 | 聴衆の認知スタイルに合わせる。エンジニアには結論先行+根拠、経営層には数字先行+ストーリー、現場チームには具体例先行+原則。文化的コンテキスト（日本の根回し文化等）も反映 |
| CC-5 | **アジェンダテンプレートの直写し** | 会議設計がすべて「目的→議題→タイムボックス→Next Steps」の同一テンプレートか確認 | 会議の性質に応じた設計をする。意思決定会議なら事前資料+投票メカニズム、探索会議なら発散→収束のダイヤモンド構造、振り返りなら心理的安全性の確保手順を先行 |
| CC-6 | **Thomas-Kilmann モデルの機械的適用** | コンフリクト対応がすべて Thomas-Kilmann の5スタイル分類で処理されていないか確認 | 対立の根本原因を分析する。技術的対立（データで解決可能）、利害対立（トレードオフ交渉）、価値観対立（共通目的の再定義）、感情的対立（冷却期間+個別対話）を区別し、それぞれに適した介入を設計 |

> **核心原則**: **コミュニケーションは相手の地図で歩くことである** — 自分のフレームワークを押し付けるのではなく、相手の思考様式・組織文化・意思決定パターンに合わせて設計する。「別の組織・別の聴衆への説明にそのまま使い回せるか？」→ 使い回せるなら Communication Slop である。

**チェックリスト**:
- [ ] アーキテクト人格アンチパターンに該当しないか自己診断した
- [ ] コミュニケーションアンチパターンを文面・発言から排除した
- [ ] すべてのメッセージが理由先行・協働的言語になっている
- [ ] 偶発的複雑さを排除し、本質的複雑さのみが残っている
- [ ] コミュニケーション Slop チェック（CC-1〜CC-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Examples

### Example 1: 経営層へのアーキテクチャ刷新提案

```
「マイクロサービスへの移行を経営会議で提案したい」

→ Step 1: 聴衆=経営層（コスト・時間・リスクに関心）
         隠れた懸念:「失敗したらプロジェクトが止まる」
→ Step 2: WHY（現状の課題: デプロイに2週間）→ WHAT（段階的移行）
         → HOW（3フェーズ計画）→ SO WHAT（デプロイ頻度 10 倍）
         定量化:「five nines」→「1日平均1秒のダウンタイム」
→ Step 3: NT-3（定量化）+ NT-4（分割統治: 全体ではなく決済サービスから）
         + NT-5（コストは最後: まず技術的メリットで説得）
→ Step 4: 問題解決型、5-8 枚、1スライド1メッセージ
→ 成果物: プレゼン骨格（slide-gen への入力）+ 交渉スクリプト
```

### Example 2: 開発チームとの技術的合意形成

```
「開発チームが REST を主張しているが、非同期メッセージングが適切」

→ Step 1: 聴衆=開発チーム（実装の明確さ・自律性に関心）
         隠れた懸念:「メッセージングは複雑で経験がない」
→ Step 2: フレーミング=「検討してみて」形式
         理由先行:「パフォーマンスとスケーラビリティのため…」
→ Step 3: NT-6（実証による決着: 両案のベンチマーク実施）
         + NT-8（自己発見: 開発者自身にテストさせる）
         + NT-9（命令→質問: 「メッセージングを検討した？」）
→ Step 5: チーム警告サイン確認（多元的無知がないか）
→ 成果物: 交渉戦略シート + PoC 計画
```

### Example 3: チームコミュニケーションの改善

```
「チームの生産性が落ちている。コミュニケーションを改善したい」

→ Step 5: Elastic Leadership 診断
         チーム熟知度(-20) + サイズ:大(+20) + 経験:ジュニア多(+20)
         + 複雑度:高(+20) + 期間:6ヶ月(-20) = +20（バランス〜積極関与）
→ Step 5b: 警告サイン検出
         ・マージコンフリクト頻発 → プロセスロス検出
         ・会議で全員が黙って同意 → 多元的無知の疑い
→ Step 5c: 制約の「部屋」再設計
         ・コンポーネント内部設計は開発者に委ねる（コントロールフリーク回避）
         ・ライブラリ選定の裁量範囲を明示
→ Step 6: アンチパターン検出
         → アーキテクトがアームチェア化の傾向（チームとの対話不足）
→ 成果物: チーム効果性診断レポート + 改善アクションプラン
```

### Example 4: 他のアーキテクトとの技術交渉

```
「セキュリティアーキテクトが過剰な制約を要求している」

→ Step 1: 聴衆=技術ステークホルダー（技術的正確性に関心）
→ Step 3: NT-2（事前情報収集: セキュリティ要件の根拠を調査）
         + NT-4（分割統治: どの領域に本当にその制約が必要か）
         + NT-6（実証: セキュリティテストの結果で議論）
         + NT-7（冷静さ: 感情的にならない）
→ Step 6: 自分がコントロールフリークになっていないか確認
→ 成果物: 交渉戦略シート + 分割された要件一覧
```

### Example 5: CTO への Kubernetes 移行提案

```
「オンプレの VM 基盤を EKS に移行したい。CTO を説得したい」

→ Step 1: 聴衆=CTO（技術トップ、コスト・リスク・チーム影響に関心）
         隠れた懸念:「移行中にサービス停止が起きたら責任問題」
→ Step 2: WHY（VM 運用コスト年間 X 円、パッチ適用に月 3 日）
         → WHAT（EKS 段階移行、Phase 1 は非本番から）
         → SO WHAT（運用工数 60% 削減、デプロイ頻度 10 倍）
         NT-3 定量化:「スケーラブル」→「ピーク時 3 分で 10→50 Pod 自動拡張」
→ Step 3: NT-4（分割統治: 全システムではなく内部ツールから移行開始）
         + NT-6（実証: 1 サービスの PoC 結果を提示）
         + NT-2（事前情報: CTO が過去に懸念した移行失敗事例を調査）
→ Step 4: 比較評価型、8 枚、VM vs EKS のコスト・運用比較表を中心に
→ 成果物: CTO 向けプレゼン骨格 + PoC 計画 + コスト比較シート
```

### Example 6: AWS → Azure 移行のチーム合意形成

```
「会社方針で Azure に移行するが、チームは AWS に慣れていて抵抗がある」

→ Step 1: 聴衆=開発チーム（実装経験・学習コスト・自律性に関心）
         隠れた懸念:「Azure の経験がなく、生産性が落ちる」「また覚え直し」
→ Step 2: フレーミング=実装チーム向け（「検討してみて」形式）
         理由先行:「会社のセキュリティ基準と既存 AD 統合のため…」
         定量化:「学習期間」→「Azure 研修 2 週間 + Enabling Team 3 ヶ月伴走」
→ Step 3: NT-8（自己発見: チームに Azure/AWS の機能マッピング表を作らせる）
         + NT-9（命令→質問:「Azure Functions と Lambda の違いを調べてみた？」）
         + NT-10（依頼の好意化: 先行調査メンバーへの感謝と評価）
→ Step 5: Elastic Leadership 診断（チーム熟知度-20 + 経験:Azure 未経験+20）
         → 積極関与が必要と判定。Enabling Team 支援を提案
→ Step 6: 自分が象牙の塔になっていないか確認（「会社方針だから」で終わらせない）
→ 成果物: チーム合意形成戦略 + Azure 移行学習ロードマップ + FAQ
```

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| プレゼンが経営層に刺さらない | 技術用語が多い / ビジネスインパクトが不明確 | WHY → SO WHAT を強化し、ROI とリスク軽減を前面に |
| 開発チームが反発する | 命令形の多用 / 理由の不説明 | 理由先行 + 質問形への変換。自己発見を促す |
| 交渉が平行線になる | 議論だけで実証がない | PoC / ベンチマークを実施し、データで決着する |
| 会議で全員が黙る | 多元的無知が発生している | 個別に意見を求め、安全に発言できる環境を作る |
| チームの生産性が低下 | プロセスロス / 制約の過不足 | Elastic Leadership 診断を実施し、関与度を調整 |
| アーキテクトが信頼されない | 象牙の塔 / アームチェア化 | チームの近くに座り、協働的な姿勢を見せる |
| 交渉が感情的になる | 個人攻撃 / 権威主義 | 交渉を中断し、冷静になってから再開。実証で決着 |
| メッセージが複雑すぎる | 偶発的複雑さの混入 | 4Cs チェックを実施。Clear と Concise を強化 |
| ステークホルダーが過大な要求 | バズワードの文字通りの解釈 | バズワード解読 + 定量化 + 分割統治 |
| 改善策が定着しない | チェックリストの形骸化 | ホーソン効果を活用。チーム自身にチェックリストを設計させる |
