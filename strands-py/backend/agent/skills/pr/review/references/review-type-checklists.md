# レビュー種別チェックリスト

> レビュー対象の種別に応じた専用チェック項目を提供する。SKILL.md Step 1 で種別判定後、該当するチェックリストを適用する。

---

## 使い方

1. Step 1（Pre-Flight Check）でレビュー種別を判定する
2. 該当するチェックリストを本 reference から選択する
3. 各チェック項目を順に検証し、重要度に基づいて指摘を記録する
4. Step 5 のレポート出力時、チェック項目の ID（例: `CR-SEC-01`）を指摘の根拠として引用する

**重要度の定義**:
- **Critical**: 即座に修正必須。リリースブロッカー
- **Warning**: 修正必須だが、条件付きで進行可能
- **Suggestion**: 修正推奨。品質向上に寄与するが、ブロッカーではない

## 1. コードレビュー チェックリスト

SKILL.md Step 3（技術仕様レビュー）の 3a-3f を補完する、観点別の詳細チェック項目。

### 1a. セキュリティ

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| CR-SEC-01 | SQL インジェクション対策 | パラメータバインド / ORM の使用を確認 | Critical |
| CR-SEC-02 | XSS 対策 | ユーザー入力のエスケープ / サニタイズ処理を確認 | Critical |
| CR-SEC-03 | 認証・認可の適切な実装 | エンドポイントごとの権限チェックを確認 | Critical |
| CR-SEC-04 | 機密情報のハードコード | API キー・パスワード・トークンが平文で埋め込まれていないか | Critical |
| CR-SEC-05 | CSRF 対策 | 状態変更リクエストにトークン検証があるか | Warning |
| CR-SEC-06 | 依存ライブラリの脆弱性 | 既知の CVE がある依存パッケージを使用していないか | Warning |
| CR-SEC-07 | ログ出力の安全性 | PII・機密情報がログに出力されていないか（Step 3 3d と連動） | Critical |
| CR-SEC-08 | ファイルアップロードの検証 | ファイルタイプ・サイズ制限・パストラバーサル対策 | Warning |

### 1b. パフォーマンス

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| CR-PERF-01 | N+1 クエリ問題 | ループ内のクエリ発行を検出（Step 3 3e と連動） | Warning |
| CR-PERF-02 | 不要なデータ取得 | SELECT * の使用、必要カラムのみ取得しているか | Suggestion |
| CR-PERF-03 | インデックス設計 | WHERE / JOIN / ORDER BY 対象カラムにインデックスがあるか | Warning |
| CR-PERF-04 | メモリ使用量 | 大量データの一括ロード、ストリーム処理の適用可否 | Warning |
| CR-PERF-05 | キャッシュ戦略 | 頻繁にアクセスされる不変データにキャッシュが適用されているか | Suggestion |
| CR-PERF-06 | 非同期処理の活用 | I/O バウンド処理が同期的にブロックしていないか | Suggestion |
| CR-PERF-07 | バッチ処理の最適化 | 大量データ処理にページング / チャンク処理が適用されているか | Warning |

### 1c. 保守性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| CR-MAINT-01 | 単一責務原則 | 1 つの関数/クラスが複数の責務を持っていないか | Warning |
| CR-MAINT-02 | 関数の長さ | 1 関数が 50 行を超えていないか（目安） | Suggestion |
| CR-MAINT-03 | ネストの深さ | if/for のネストが 3 段以上になっていないか（早期 return で浅く保つ） | Suggestion |
| CR-MAINT-04 | マジックナンバー/文字列 | 定数に名前がついているか | Warning |
| CR-MAINT-05 | 命名の明確さ | 変数名・関数名が意図を正確に表現しているか | Suggestion |
| CR-MAINT-06 | 重複コードの排除 | 同一ロジックが複数箇所にコピーされていないか | Warning |
| CR-MAINT-07 | エラーハンドリング | 裸の `except:` / 空の `catch` がないか。具体的な例外を捕捉しているか | Warning |
| CR-MAINT-08 | 型安全性 | 型アノテーション / 型定義が適切に付与されているか（Python: mypy strict、TS: strict mode） | Warning |

### 1d. テスト可能性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| CR-TEST-01 | 外部依存の注入可能性 | DB / API / ファイル I/O がインターフェース経由で注入可能か | Warning |
| CR-TEST-02 | 副作用の分離 | 純粋なビジネスロジックと副作用（I/O）が分離されているか | Warning |
| CR-TEST-03 | テストの存在 | 新規/変更コードに対応するテストが存在するか（Step 3 3f と連動） | Warning |
| CR-TEST-04 | テストの独立性 | テスト間に暗黙の実行順依存がないか | Suggestion |
| CR-TEST-05 | モック戦略 | アプリケーション境界を超える通信のみモック化しているか | Suggestion |
| CR-TEST-06 | テスト名の品質 | テスト名が振る舞いを自然言語で記述しているか | Suggestion |

## 2. ドキュメントレビュー チェックリスト

SKILL.md Step 2（ドキュメントレビュー）の 2a-2d を補完する、観点別の詳細チェック項目。

### 2a. 正確性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| DR-ACC-01 | 数値の計算正確性 | 記載されている計算式を再計算し、結果が一致するか | Critical |
| DR-ACC-02 | 外部参照の正確性 | 引用元の情報と記載内容が一致するか | Warning |
| DR-ACC-03 | 技術用語の正確性 | 用語が業界標準の定義と一致するか | Warning |
| DR-ACC-04 | 日付・バージョンの正確性 | 記載された日付・バージョン番号が最新かつ正確か | Warning |
| DR-ACC-05 | 前提条件の明示 | 計算や結論の前提条件が明記されているか（Step 2 2c と連動） | Warning |
| DR-ACC-06 | 図表と本文の整合 | 図表の数値・ラベルが本文の記述と矛盾しないか | Critical |

### 2b. 完全性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| DR-COMP-01 | 要件の網羅 | RFP / 要件定義の全項目に対応する記述があるか（Step 2 2a と連動） | Critical |
| DR-COMP-02 | 非機能要件の記述 | パフォーマンス・セキュリティ・可用性の要件が定義されているか | Warning |
| DR-COMP-03 | 例外ケースの記述 | 正常系だけでなく異常系・エッジケースの扱いが記述されているか | Warning |
| DR-COMP-04 | 用語集の完備 | ドメイン固有の用語に定義・説明が付与されているか | Suggestion |
| DR-COMP-05 | 前提・制約・リスクの明記 | プロジェクトの前提条件・制約・リスクが章として存在するか | Warning |
| DR-COMP-06 | 付録・参考資料の列挙 | 参照先のドキュメント・URL が漏れなくリストされているか | Suggestion |

### 2c. 一貫性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| DR-CONS-01 | 用語の統一 | 同じ概念に異なる用語が使われていないか（glossary.md 準拠） | Warning |
| DR-CONS-02 | 数値の章間整合 | 各章の数値（工数・金額・人数）が相互に矛盾しないか（Step 2 2b と連動） | Critical |
| DR-CONS-03 | 表記ルールの統一 | 日付形式・数値桁区切り・敬体/常体が統一されているか | Suggestion |
| DR-CONS-04 | 図番・表番の連続性 | 図表の番号が連続しており、欠番・重複がないか | Suggestion |
| DR-CONS-05 | トーン・文体の統一 | 章ごとに文体が変わっていないか（複数著者の痕跡） | Suggestion |
| DR-CONS-06 | スコープの一貫性 | 序章で定義したスコープから逸脱する記述がないか | Warning |

### 2d. 可読性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| DR-READ-01 | 論理構造の明確さ | 章・節の構成が論理的な流れに沿っているか（Step 2 2d と連動） | Warning |
| DR-READ-02 | 要約の存在 | 各章冒頭にサマリー（2-3 行）があるか | Suggestion |
| DR-READ-03 | 図表の活用 | テキストのみで伝わりにくい箇所に図表が配置されているか | Suggestion |
| DR-READ-04 | 1 文の長さ | 80 字を超える文が連続していないか（1 文 1 意を推奨） | Suggestion |
| DR-READ-05 | 専門用語の説明 | 読者の知識レベルを超える専門用語に注釈があるか | Suggestion |
| DR-READ-06 | 行動可能な結論 | 各セクションが「次に何をすべきか」を明示しているか | Warning |

## 3. アーキテクチャレビュー チェックリスト

ソフトウェアアーキテクチャ・システム設計のレビューに特化。software-architecture / flow-architecture スキルの成果物を検証する際に使用する。

### 3a. スケーラビリティ

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| AR-SCAL-01 | 負荷見積もりの明示 | 想定ユーザー数・リクエスト数・データ量が定義されているか | Critical |
| AR-SCAL-02 | 水平スケーリング設計 | ステートレス設計 / 負荷分散の仕組みが考慮されているか | Warning |
| AR-SCAL-03 | データ層のスケーラビリティ | シャーディング / パーティショニング / リードレプリカの検討 | Warning |
| AR-SCAL-04 | ボトルネック特定 | 想定負荷で最初に破綻するコンポーネントが特定されているか | Warning |
| AR-SCAL-05 | 段階的拡張計画 | 成長に合わせた段階的なスケールアップ計画があるか | Suggestion |

### 3b. 障害耐性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| AR-FAIL-01 | 単一障害点の排除 | SPOF（Single Point of Failure）が存在しないか | Critical |
| AR-FAIL-02 | フェイルオーバー設計 | 障害時の自動切り替え機構が定義されているか | Warning |
| AR-FAIL-03 | RPO / RTO の定義 | 復旧目標（Recovery Point/Time Objective）が明記されているか | Critical |
| AR-FAIL-04 | サーキットブレーカー | 外部依存の障害がシステム全体に伝播しない設計か | Warning |
| AR-FAIL-05 | バックアップ戦略 | データのバックアップ頻度・保持期間・復元手順が定義されているか | Warning |
| AR-FAIL-06 | カオスエンジニアリング | 障害注入テストの計画があるか | Suggestion |

### 3c. 運用性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| AR-OPS-01 | オブザーバビリティ設計 | ログ・メトリクス・トレースの収集方針が定義されているか | Warning |
| AR-OPS-02 | デプロイ戦略 | Blue-Green / Canary / Rolling 等のデプロイ方式が定義されているか | Warning |
| AR-OPS-03 | 設定の外部化 | 環境依存の設定が外部化（環境変数 / Config サービス）されているか | Warning |
| AR-OPS-04 | ランブック / 手順書 | 障害対応・定常運用の手順が文書化されているか | Suggestion |
| AR-OPS-05 | SLO / SLI の定義 | サービスレベル目標とその指標が定義されているか | Warning |
| AR-OPS-06 | セキュリティ運用 | 脆弱性スキャン・秘密情報ローテーション・アクセス監査の計画 | Warning |

### 3d. コスト

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| AR-COST-01 | 月次ランニングコスト試算 | クラウドリソースの月次コスト見積もりが提示されているか | Warning |
| AR-COST-02 | コスト最適化ポイント | リザーブド / スポットインスタンス / オートスケーリングの検討 | Suggestion |
| AR-COST-03 | データ転送コスト | リージョン間・AZ 間のデータ転送コストが考慮されているか | Warning |

## 4. 提案書レビュー チェックリスト

bd-skills / drm スキルの成果物（提案書・見積書）を検証する際に使用する。

### 4a. 課題定義の妥当性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| PR-PROB-01 | 課題の具体性 | 「〜が課題」ではなく、定量的な根拠（件数・金額・時間）があるか | Critical |
| PR-PROB-02 | 顧客の声との整合 | ヒアリング結果・RFP と課題定義が整合しているか | Critical |
| PR-PROB-03 | 課題の優先順位 | 複数課題がある場合、優先度の根拠が明示されているか | Warning |
| PR-PROB-04 | 現状分析の深さ | As-Is の分析が表面的でなく、根本原因まで掘り下げているか | Warning |
| PR-PROB-05 | スコープの明確性 | 解決する課題と解決しない課題の線引きが明確か | Warning |

### 4b. 解決策の実現可能性

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| PR-SOL-01 | 技術的実現可能性 | 提案する技術が課題に対して適切か。実績・検証結果があるか | Critical |
| PR-SOL-02 | 体制の妥当性 | 必要スキルセット × 人数がスコープに対して十分か | Warning |
| PR-SOL-03 | スケジュールの実現性 | 工数とスケジュールが整合し、クリティカルパスが特定されているか | Critical |
| PR-SOL-04 | リスク対策 | 主要リスクの洗い出しと対策が記載されているか | Warning |
| PR-SOL-05 | 前提条件の明示 | 顧客側の協力事項・環境提供等の前提が明記されているか | Warning |
| PR-SOL-06 | 代替案の検討 | なぜこのアプローチを選んだかの比較検討があるか | Suggestion |

### 4c. ROI の根拠

| ID | チェック項目 | 検証方法 | 重要度 |
|:---|:---|:---|:---|
| PR-ROI-01 | 定量的効果の提示 | コスト削減額 / 売上増加額 / 時間短縮の具体的な数値があるか | Critical |
| PR-ROI-02 | 効果算出の前提 | 効果の計算式と前提条件が明示されているか | Warning |
| PR-ROI-03 | 投資回収期間 | 投資額に対する回収期間（ペイバック期間）が示されているか | Warning |
| PR-ROI-04 | 比較基準の妥当性 | 「現状維持」との比較が公正に行われているか | Warning |
| PR-ROI-05 | 定性的効果の記述 | 数値化しにくい効果（顧客満足度、ブランド価値等）が補足されているか | Suggestion |

## チェックリスト選択ガイド

Step 1 でレビュー種別を判定した後、以下のマトリクスに従ってチェックリストを組み合わせる。

| レビュー対象 | 必須チェックリスト | 推奨チェックリスト |
|:---|:---|:---|
| ソースコード / PR | 1a, 1b, 1c, 1d | — |
| 提案書 / 見積書 | 4a, 4b, 4c | 2a, 2b, 2c, 2d |
| 設計書 / 仕様書 | 2a, 2b, 2c, 2d | 3a, 3b（技術設計を含む場合） |
| アーキテクチャ設計書 | 3a, 3b, 3c, 3d | 2a, 2b |
| API ドキュメント | 2a, 2b, 2c | 1a（実装コードとクロスレビューする場合） |
| 複合（ドキュメント + コード） | 2a-2d + 1a-1d | 対象に応じて選択 |

## 重要度配分の目安

レビュー種別ごとに、指摘の重要度配分が健全かを検証するためのガイドライン。この配分から大きく逸脱する場合は、レビューの観点に偏りがないか再確認する。

| レビュー種別 | Critical | Warning | Suggestion | 典型的な指摘数 |
|:---|:---|:---|:---|:---|
| コードレビュー（PR） | 0-10% | 30-50% | 40-60% | 5-15 件 |
| ドキュメントレビュー | 5-15% | 40-60% | 30-40% | 8-20 件 |
| アーキテクチャレビュー | 10-20% | 40-60% | 20-40% | 10-25 件 |
| 提案書レビュー | 5-15% | 50-70% | 20-30% | 10-20 件 |

> **注意**: この配分はあくまで目安。高品質な成果物では Critical 0 件 / Warning 0 件もあり得る。配分に合わせるために指摘を「作る」のは RV-1（固定比率カテゴリ分布）に該当する。
