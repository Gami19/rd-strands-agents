# Skill: Incident Response（インシデント対応・SRE 運用設計）

> **「止血 → 原因究明 → 再発防止」のサイクルを組織に実装する**

## strands-py / Strands Agent での利用

SLO・対応フロー・ポストモーテム等は **`file_read` / `file_write`** とプロジェクト workspace（`notes` / `proposal` 等）を正とする。observability・diagram・agent 実装との分担は [references/strands-py-runtime.md](references/strands-py-runtime.md) を参照。

## Instructions

### ワークフロー内の位置

```
monitoring/alerting design → [incident-response] → postmortem → improvement
                                    ↓
                    SLO/SLI 設計、オンコール体制、対応フロー、
                    ポストモーテムテンプレート、プレイブック
```

### 入力

| 入力 | 説明 | 例 |
|:---|:---|:---|
| サービス概要 | 対象サービスのアーキテクチャ・ユーザー規模 | Web API、月間 1000 万リクエスト |
| 現行の監視状況 | 既存のメトリクス・アラート設定 | Datadog, CloudWatch アラーム |
| チーム体制 | エンジニア人数・拠点 | 東京 6 名、大阪 4 名 |
| ビジネス要件 | 可用性要求・収益影響 | 99.9% 可用性、1 分停止 = 10 万円損失 |

## Step 1: SLO/SLI 設計（サービスレベル目標とエラーバジェット）

サービスの信頼性目標を定量的に定義する。

### 1a. SLI の選定

ユーザーが実際に気にする品質指標を特定する。

**サービスタイプ別 SLI マトリクス**:

| サービスタイプ | 推奨 SLI | 測定ポイント |
|:---|:---|:---|
| ユーザー対向 API | 可用性、レイテンシ (p99)、スループット | ロードバランサ or クライアント |
| ストレージ | レイテンシ、可用性、耐久性 | サーバー or クライアント SDK |
| バッチ処理 | スループット、E2E レイテンシ、正確性 | パイプライン出力 |
| ストリーミング | レイテンシ (p99)、エラー率、スループット | クライアント or エッジ |

- パーセンタイル（p50, p95, p99, p99.9）で測定し、平均値は使わない
- 成功リクエストと失敗リクエストのレイテンシを分離する

### 1b. SLO の設定

```
SLO 設定式:
  availability = successful requests / total requests

例:
  SLO: 99.9% (月間)
  月間リクエスト: 10M
  許容エラー数: 10,000 リクエスト/月
  エラーバジェット消費率: (実エラー数 / 10,000) * 100%
```

**設計ルール**:
- 現在の実績値をそのまま SLO にしない
- 内部 SLO は外部 SLO より厳しくする（バッファ確保）
- 四半期ごとに SLO を見直す
- SLO は最小限に絞る（多くても 3-5 個）

### 1c. エラーバジェットポリシー

エラーバジェット枯渇時のアクションを定義する。

```markdown
## エラーバジェットポリシー

| バジェット残量 | アクション |
|:---|:---|
| > 50% | 通常リリースサイクル |
| 20-50% | リリース頻度を半減、追加テスト必須 |
| 5-20% | 重要修正のみリリース、信頼性改善に集中 |
| < 5% | リリース凍結、全リソースを信頼性改善に投入 |
| 0% (枯渇) | プロダクションフリーズ、例外は VP 承認必須 |
```

**チェックリスト**:
- [ ] ユーザー視点の SLI を 2-4 個選定した
- [ ] SLI の測定方法と集計間隔を定義した
- [ ] SLO 目標値をビジネス要件から導出した（現行実績からではない）
- [ ] エラーバジェットを算出し、消費追跡の仕組みを設計した
- [ ] エラーバジェットポリシー（枯渇時のアクション）を文書化した

## Step 2: 監視・アラート設計（Four Golden Signals）

Four Golden Signals に基づいて症状ベースの監視を設計する。

### 2a. Golden Signals の定義

各シグナルについて、対象サービスに即した具体的メトリクスを定義する。

| Signal | メトリクス例 | アラート条件例 |
|:---|:---|:---|
| Latency | HTTP p99 レイテンシ | p99 > 500ms が 5 分間継続 |
| Traffic | HTTP QPS | QPS が過去 1 時間平均の 200% 超過 |
| Errors | HTTP 5xx 率 | 5xx 率 > 1% が 3 分間継続 |
| Saturation | CPU 使用率、メモリ使用率 | CPU > 80% が 10 分間継続 |

### 2b. アラート設計原則

すべてのページングアラートは以下を満たすこと:

1. **緊急** — 即時対応が必要
2. **アクション可能** — 対応者が具体的なアクションを取れる
3. **ユーザー影響あり** — ユーザーに実際の影響が出ている（または差し迫っている）
4. **新規** — ロボット的対応で済むなら自動化すべき

**アンチパターン**:
- 「何か変」レベルのアラートでページしない
- 原因ベースのアラートは確実かつ差し迫った場合のみ
- 同一インシデントに複数アラートを出さない（グルーピング必須）

### 2c. ブラックボックス vs ホワイトボックス

| 方式 | 用途 | 特徴 |
|:---|:---|:---|
| ブラックボックス | ページング | 症状ベース、現在進行中の問題を検知 |
| ホワイトボックス | デバッグ・予防 | 内部メトリクス、差し迫った問題を検知 |

**チェックリスト**:
- [ ] Four Golden Signals それぞれにメトリクスを定義した
- [ ] 各アラートが「緊急・アクション可能・ユーザー影響あり」を満たす
- [ ] アラートの閾値をパーセンタイルベースで設定した（平均値ではない）
- [ ] 関連アラートのグルーピングルールを設定した
- [ ] ダッシュボードに Four Golden Signals を表示した

## Step 3: オンコール体制設計

持続可能なオンコールローテーションを設計する。

### 3a. ローテーション設計

**量的バランス**:
- オンコール時間は全体の **最大 25%** に制限
- 最小チーム規模: シングルサイト **8 名**、マルチサイト **各 6 名**
- Primary + Secondary の 2 層ローテーション

**質的バランス**:
- 1 シフトあたり最大 **2 インシデント**
- 1 インシデントの対応に平均 **6 時間**（根本原因分析 + ポストモーテム含む）
- ページングイベントの時間分布がフラットであること

### 3b. エスカレーションパス

```
L1: Primary On-Call (5分以内にアクノリッジ)
  ↓ アクノリッジなし or 対応不能
L2: Secondary On-Call
  ↓ 対応不能 or 複数チーム関与
L3: Incident Commander 任命 → ICS 発動
  ↓ 重大影響
L4: マネジメントエスカレーション
```

### 3c. 心理的安全性の確保

- 明確なエスカレーションパスを文書化する
- 「一人で抱え込まない」を文化として根付かせる
- 合理的・計画的な意思決定を促す（直感的・反射的対応を避ける）
- オンコール報酬（代休 or 手当）を制度化する
- 運用過少負荷も回避する（四半期 1-2 回はオンコールを経験させる）

### 3d. 訓練

- **Wheel of Misfortune**: 過去のインシデントを役割プレイで追体験
- **DiRT (Disaster Recovery Training)**: 年次の災害復旧訓練
- **ゲームデー**: 意図的な障害注入によるレジリエンス検証

**チェックリスト**:
- [ ] オンコール時間が全体の 25% 以下に収まるローテーションを設計した
- [ ] Primary / Secondary の役割分担を定義した
- [ ] エスカレーションパスを文書化した
- [ ] オンコール報酬制度を確認した
- [ ] 定期訓練（Wheel of Misfortune 等）の計画を策定した

## Step 4: インシデント対応フロー（ICS ベースの役割分担）

ICS (Incident Command System) に基づくインシデント管理を設計する。

### 4a. インシデント宣言基準

以下のいずれかに該当する場合、インシデントを宣言する:

1. **2 チーム以上**の関与が必要
2. 障害が**顧客に可視**
3. **1 時間**の集中分析後も未解決

早めに宣言し、簡単に解決できたら閉じる方が、遅延して大規模化するより良い。

### 4b. ICS 役割

| 役割 | 責務 | 備考 |
|:---|:---|:---|
| **Incident Commander (IC)** | 全体統括、役割割当、障害排除 | 委任していない役割はすべて IC が担う |
| **Operations Lead** | システム操作、運用ツール適用 | インシデント中にシステム変更できる唯一のチーム |
| **Communications Lead** | ステータス更新、ステークホルダー通知 | 最低 4 時間ごとに更新 |
| **Planning Lead** | バグ起票、ハンドオフ調整、長期課題管理 | 食事手配等の後方支援も含む |

### 4c. インシデント対応の 7 原則

1. **止血を最優先**: 根本原因追究よりサービス回復を先に
2. **役割分離**: 全員が自分の役割を明確に理解し、他者の領域に踏み込まない
3. **ライブドキュメント**: インシデント状態ドキュメントをリアルタイムで更新
4. **コマンドポスト**: IRC/Slack チャンネルを設置し、全通信を集約（ログが残る）
5. **明示的ハンドオフ**: 「あなたが IC です、OK?」と口頭確認 + チーム通知
6. **代替手段の定期検討**: 現在のアプローチを継続すべきか定期的に再評価
7. **訓練で定着**: 日常の変更管理にもフレームワークを適用し、筋肉記憶を形成

### 4d. インシデント状態ドキュメント

```markdown
# [インシデントタイトル]: YYYY-MM-DD

## Summary
障害の概要（1-2 文）

## Status
active / mitigated / resolved, incident #XXX

## Command Post
#incident-XXX on Slack

## Command Hierarchy
- IC: @name
  - Ops Lead: @name
  - Comms Lead: @name
  - Planning Lead: @name
- Next IC: TBD

## Exit Criteria
- [ ] 可用性が SLO 内で 30 分以上安定
- [ ] 根本原因の暫定修正が適用済み

## TODO / Bugs
- [ ] アクション項目 (Bug #XXX)

## Timeline (newest first, UTC)
- HH:MM @name — アクションの説明
```

**チェックリスト**:
- [ ] インシデント宣言基準を定義した
- [ ] ICS 4 役割の担当者リストを作成した
- [ ] インシデント状態ドキュメントのテンプレートを準備した
- [ ] コマンドポスト（チャットチャンネル）の命名規則を決めた
- [ ] ハンドオフ手順を文書化した

## Step 5: 緊急対応パターン

3 つの障害誘因パターンに対する対応を事前に設計する。

### 5a. テスト誘発型

- **シナリオ**: 災害復旧テスト、カオスエンジニアリングが予想外の影響を生む
- **対応**: テスト即時中止 → 影響範囲の特定 → 復旧 → テスト範囲の見直し
- **予防**: ロールバック手順の事前テスト、影響範囲を小さく開始して拡大

### 5b. 変更誘発型

- **シナリオ**: 設定変更やデプロイが障害を引き起こす
- **対応**: ロールバック → インシデント宣言 → カナリアプロセスの見直し
- **予防**: カナリアリリースの徹底、「低リスク」変更でも厳格なカナリアを適用

### 5c. プロセス誘発型

- **シナリオ**: 自動化のバグが大規模な意図しない操作を実行
- **対応**: 全自動化の停止 → トラフィック迂回 → 手動復旧
- **予防**: 自動化にサニティチェックを実装、空レスポンスを「全件」と解釈しない

### 5d. 共通対応原則

1. パニックしない — 深呼吸して冷静に
2. 圧倒されたら人を増やす
3. トリガーを引いた人が最も状態を把握している
4. 帯域外通信手段（電話、別系統ツール）を確保する
5. 緩和後に必ず振り返り時間を確保する

**チェックリスト**:
- [ ] 3 類型それぞれの対応手順テンプレートを作成した
- [ ] ロールバック手順をテスト済み
- [ ] 帯域外通信手段を確保した
- [ ] カナリアリリースプロセスを文書化した

## Step 6: ポストモーテム（非難なし文化）

### 6a. ポストモーテムトリガー

以下のいずれかに該当する場合、ポストモーテムを作成する:

- ユーザー可視のダウンタイム/劣化が閾値超過
- データロス発生
- オンコールエンジニアの介入（ロールバック、トラフィック迂回等）
- 解決時間が閾値超過
- 監視の失敗（手動でインシデント発見）
- ステークホルダーの要請

### 6b. 非難なし原則

- 関係者全員が**善意で行動した**と前提する
- 個人ではなく**システムとプロセス**の改善に集中する
- 「人は直せないが、システムは直せる」
- ポストモーテムの多産をスティグマ化しない

**良い例**: 「バックエンドシステムの書き直しにより、再発ページの削減と保守性向上が期待できる」
**悪い例**: 「あの複雑なシステムを書いた人が問題だ」

### 6c. ポストモーテムテンプレート

```markdown
# [タイトル] (incident #XXX)

- **Date**: YYYY-MM-DD
- **Authors**: @name1, @name2
- **Status**: Complete / Action items in progress

## Summary
インシデントの概要（1-2 文）

## Impact
- ユーザー影響: X 件のリクエスト失敗、Y 分間のダウンタイム
- 収益影響: 推定 Z 円

## Root Causes
根本原因の説明（5 Whys テクニックで深掘り）

## Trigger
直接的なトリガー

## Resolution
解決方法の説明

## Detection
検知方法と検知までの時間

## Action Items

| Action Item | Type | Owner | Bug | Status |
|:---|:---|:---|:---|:---|
| 障害検知アラートの追加 | prevent | @name | #123 | TODO |
| プレイブックの更新 | mitigate | @name | — | DONE |
| カスケード障害テストの実施 | process | @name | #456 | TODO |

Type: prevent（再発防止）/ mitigate（影響緩和）/ process（プロセス改善）

## Lessons Learned

### What went well
- 監視が迅速に異常を検知した
- チーム間の連携がスムーズだった

### What went wrong
- ロールバック手順がテストされていなかった
- エスカレーションが遅れた

### Where we got lucky
- 障害が低トラフィック時間帯に発生した

## Timeline (UTC)
- HH:MM イベントの説明

## Supporting Information
- ダッシュボード URL
- ログリンク
```

### 6d. レビューと共有

- シニアエンジニアによるレビュー: データ完全性、影響評価、根本原因の深さ、アクションプランの妥当性
- チーム内共有 → 組織全体への公開
- **Postmortem of the Month**: 月次ニュースレターで秀逸なポストモーテムを紹介
- **読書会**: 過去のポストモーテムをチームで議論
- アクションアイテムの完了追跡を義務化

**チェックリスト**:
- [ ] ポストモーテムトリガー基準を定義した
- [ ] 非難なし原則をチームに周知した
- [ ] ポストモーテムテンプレートを準備した
- [ ] レビュープロセスを定義した
- [ ] アクションアイテム追跡の仕組みを構築した
- [ ] ポストモーテム共有活動（読書会等）を計画した

## Step 7: プレイブック・ランブック生成

障害パターン別の具体的な対応手順書を作成する。

### 7a. プレイブック構成

```markdown
# Playbook: [障害パターン名]

## 概要
障害の説明と想定される影響

## 検知
- アラート名: [alert-name]
- 確認ダッシュボード: [URL]

## トリアージ（5 分以内）
1. 影響範囲の確認: [確認コマンド/手順]
2. 重大度の判定: [基準]
3. インシデント宣言の要否判断

## 緩和（30 分以内）
1. [具体的な手順 1]
2. [具体的な手順 2]
3. ロールバック手順: [手順]

## 根本原因の調査
1. [調査手順 1]
2. [調査手順 2]

## エスカレーション先
- L2: @team-name (#channel)
- L3: @senior-engineer (phone: XXX)
```

### 7b. カスケード障害対策プレイブック

| 対策 | 手順 | タイミング |
|:---|:---|:---|
| 負荷制限発動 | HTTP 503 返却の閾値を下げる | CPU > 90% or エラー率 > 5% |
| 縮退運転 | キャッシュのみ応答、低品質結果の返却 | 正常クラスタ数 < N+1 |
| トラフィック迂回 | 障害クラスタからの排除 | ヘルスチェック 3 回連続失敗 |
| リトライ抑制 | リトライバジェット超過時に即座に失敗返却 | リトライ率 > 20% |
| キャパシティ増強 | オートスケーリング or 手動スケールアウト | 飽和度 > 70% が 10 分継続 |

### 7c. リトライ戦略

- **ランダム化指数バックオフ + ジッター**: `delay = min(base * 2^attempt, max) * random(0.5, 1.5)`
- **リクエストあたりのリトライ上限**: 最大 3-5 回
- **プロセスあたりのリトライバジェット**: 例: 60 回/分超過でリトライ停止
- **多層リトライの増幅回避**: 各層 3 回リトライ = 4^N 回に増幅する危険性
- **リトライ可能/不可能エラーの区別**: 永続エラーや不正リクエストはリトライしない

### インシデント管理 Slop 防止チェック（Distributional Convergence 対策）

インシデント管理の設計が汎用テンプレートの転用に終始すると、対象サービス固有の障害モードやビジネスリスクに対応できず、「マニュアルはあるが使えない」状態に陥る。

| # | インシデント管理 Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| IR-1 | **99.9% 一律 SLO** | すべてのサービスに同じ 99.9% 可用性 SLO を設定し、サービスのビジネスクリティカリティやユーザー期待を反映していない | ビジネス影響（停止 1 分あたりの損失額）とユーザー期待に基づいて SLO を個別設定する（例: 決済 API は 99.99%、管理画面は 99.5%、バッチ処理は 99.0% + E2E レイテンシ SLO） |
| IR-2 | **均一エスカレーション閾値** | エラーバジェットポリシーが全サービスで同じ段階（50%/20%/5%/0%）になっている | サービスのリリース頻度・変更リスクに応じて閾値を調整する（例: 高頻度デプロイの API は 30%/10%/3% と厳しく、安定した内部ツールは 50%/20%/5% で十分） |
| IR-3 | **定型 ICS 役割割当** | インシデント規模やチーム構成に関係なく、毎回同じ 4 役割（IC/Ops/Comms/Planning）をそのまま割当てている | チーム規模とインシデント種別に応じた役割設計にする（例: 3 人チームなら IC + Ops の 2 役割に統合、データ漏洩インシデントならセキュリティリードを追加） |
| IR-4 | **テンプレート語彙 Lessons Learned** | ポストモーテムの Lessons Learned が毎回「監視の強化」「テストの追加」「ドキュメントの更新」等の汎用カテゴリに帰結する | インシデント固有の技術的発見と組織的学びを具体的に記述する（例: 「Connection Pool の枯渇は maxIdleTime=30s が原因。RDS Proxy の導入で根本解決」「Comms Lead の不在で顧客通知が 2 時間遅延。ICS チェックリストに Comms 確認を追加」） |
| IR-5 | **確認→再起動→エスカレーション ランブック** | プレイブックが障害パターンに関係なく「メトリクス確認→サービス再起動→エスカレーション」の 3 ステップで構成されている | 障害パターン固有の調査・緩和手順を設計する（例: メモリリークなら「ヒープダンプ取得→リーク箇所特定→ローリング再起動」、設定ミスなら「変更差分確認→設定ロールバック→カナリア再適用」） |
| IR-6 | **単一経路カスケード分析** | カスケード障害の分析が毎回「サービス A → サービス B → サービス C」の直線的な依存パターンだけを想定している | 対象システム固有の依存関係グラフを作成し、循環依存・共有リソース競合・リトライ増幅などの非線形パターンを特定する（例: 「認証サービスの障害で全 12 サービスが同時にリトライ → 認証サービスのリトライバジェットを 10 req/s に制限」） |

> **核心原則**: **インシデント管理はサービスの「救急医療マニュアル」であり、汎用の「応急手当ガイド」ではない** — プレイブックだけを読んで、対象サービスが何であるか特定できるか？「別のサービスのインシデント管理に差し替えても違和感がないか？」→ 違和感がないならインシデント管理 Slop である。

**チェックリスト**:
- [ ] 主要障害パターンごとのプレイブックを作成した
- [ ] 各プレイブックにトリアージ・緩和・調査の具体的手順がある
- [ ] カスケード障害対策（負荷制限、縮退運転、リトライ戦略）を設計した
- [ ] プレイブックをアラートからリンクした
- [ ] プレイブックの定期レビュー（四半期）をスケジュールした
- [ ] インシデント管理 Slop チェック（IR-1〜IR-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Examples

### Example 1: SLO/エラーバジェット設計

```
「EC サイトの SLO を設計して。月間 500 万リクエスト、99.9% 可用性が目標」

→ Step 1 で SLI を選定:
  - 可用性: リクエスト成功率（HTTP 2xx + 3xx / 全リクエスト）
  - レイテンシ: p99 レスポンスタイム < 800ms
  - エラー率: HTTP 5xx 率
→ Step 1 で SLO を設定:
  - 可用性 SLO: 99.9%（月間、ローリングウィンドウ）
  - エラーバジェット: 5,000 リクエスト/月
  - レイテンシ SLO: p99 < 800ms
→ Step 1 でエラーバジェットポリシーを策定
→ 成果物: SLO 定義書（SLI 定義、目標値、エラーバジェットポリシー）
```

### Example 2: インシデント対応フローの構築

```
「マイクロサービス群のインシデント対応フローを設計して」

→ Step 3 でオンコール体制を設計（8 名ローテーション、Primary/Secondary）
→ Step 4 でインシデント宣言基準を定義
→ Step 4 で ICS 役割分担を設計
→ Step 4 でインシデント状態ドキュメントテンプレートを作成
→ Step 5 で緊急対応パターン（変更誘発型を重点的に）を設計
→ 成果物: インシデント対応ガイド（宣言基準、役割分担、ドキュメント雛形、エスカレーションパス）
```

### Example 3: ポストモーテムテンプレートの作成

```
「非難なしポストモーテムの文化を導入したい」

→ Step 6 でポストモーテムトリガー基準を定義
→ Step 6 で非難なし原則をドキュメント化
→ Step 6 でテンプレートを作成（5 Whys、Action Items、Lessons Learned）
→ Step 6 でレビュープロセスと共有活動を設計
→ 成果物: ポストモーテムテンプレート、文化導入ガイド、読書会運営手順
```

### Example 4: オンコール体制の最適化

```
「現在 4 名でオンコールを回しているが、バーンアウトが問題になっている」

→ Step 3 で現行の量的・質的バランスを評価
  - 4 名 → オンコール比率 50%（上限 25% を大幅超過）
→ Step 3 で最低 8 名への増員を提案
→ Step 3 で心理的安全性の施策を設計
→ Step 2 でアラートノイズの削減を提案（非アクショナブルアラートの除去）
→ 成果物: オンコール改善提案書（人員計画、アラート最適化、報酬制度提案）
```

### Example 5: カスケード障害プレイブックの適用

```
「決済サービスのリトライストームでカスケード障害が発生した。対策を設計して」

→ Step 5 で障害誘因の特定: リトライストーム型（3層 × 各3回 = 81倍増幅）
→ Step 7c でリトライ戦略を再設計:
  - 末端サービスのみリトライ（上流はタイムアウトで即失敗返却）
  - 指数バックオフ + ジッター: delay = min(100ms * 2^n, 30s) * random(0.5, 1.5)
  - プロセスあたりのリトライバジェット: 60回/分上限
→ Step 7b でカスケード対策を多層設計:
  - L1: リトライバジェット超過時の即失敗返却
  - L2: サーキットブレーカー（エラー率 > 50% で Open）
  - L3: 縮退運転（キャッシュフォールバック）
→ Step 6 でポストモーテムを作成、5 Whys で根本原因を深掘り
→ 成果物: カスケード障害対策設計書、リトライ戦略ガイド、ポストモーテム
```

### Example 6: Game Day / DiRT の計画と実施

```
「年次の災害復旧訓練（DiRT）を計画したい。AWS 東京リージョン障害を想定」

→ Step 3d で訓練フレームワークを選択: DiRT（大規模、年次）
→ Step 5 で訓練シナリオを設計:
  - シナリオ: ap-northeast-1a AZ 障害 → マルチ AZ フェイルオーバー検証
  - 影響範囲: ステージング環境 + 本番のカナリアクラスタ（限定）
→ Step 4 で ICS ロールを事前割当:
  - IC: SRE リード、Ops: インフラチーム、Comms: CS リード
→ Step 4c で訓練中のインシデントドキュメント運用を実践
→ Step 7 で発見事項からプレイブック更新:
  - RDS フェイルオーバー時の接続プール再接続手順追加
  - ALB ヘルスチェック閾値の見直し
→ 成果物: DiRT 計画書、訓練報告書、プレイブック更新差分、次回改善ポイント
```

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| アラートが多すぎて対応しきれない | 非アクショナブルアラートが混在 | 各アラートが「緊急・アクション可能・ユーザー影響あり」を満たすか再評価 |
| ポストモーテムが形骸化している | 非難の文化、アクションアイテム未追跡 | 非難なし原則の再徹底、AI 追跡ツールの導入 |
| SLO が現実離れしている | 現行パフォーマンスからの逆算、ビジネス不在 | ビジネスオーナーと共同で SLO を再設定 |
| インシデント対応が属人化 | 手順が文書化されていない | プレイブック作成 + Wheel of Misfortune 訓練 |
| エラーバジェットが常に枯渇 | SLO が厳しすぎる or 根本的な品質問題 | SLO の緩和を検討、または信頼性投資を増加 |
| オンコールのバーンアウト | チーム規模不足、アラートノイズ | 最低 8 名への増員、アラート最適化 |
| カスケード障害が止まらない | 負荷制限・リトライ戦略の未実装 | 負荷テスト + 指数バックオフ + リトライバジェット実装 |
| ロールバックが失敗する | ロールバック手順の未テスト | 定期的なロールバック訓練、自動化 |
| インシデント中に情報が散逸する | コマンドポストの不在 | Slack チャンネル + ライブドキュメントの標準化 |
| 訓練が実施されない | 優先度が低いと認識されている | 四半期 OKR に訓練を組み込む、DiRT の年次実施 |
