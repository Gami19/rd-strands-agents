# Cascade Failure Countermeasure Matrix

> incident-response スキル補足資料 — カスケード障害の誘因・症状・対策マトリクス

## 1. カスケード障害の誘因分類

| 誘因 | 説明 | 典型的なシナリオ |
|:---|:---|:---|
| **サーバー過負荷** | キャパシティ超過でレイテンシ上昇 → タイムアウト → リトライ増幅 | ブラックフライデーのトラフィック急増 |
| **リソース枯渇** | CPU / メモリ / ファイルディスクリプタ / スレッド枯渇 | メモリリークによる GC ストーム |
| **サービス依存関係** | 下流サービスの障害が上流に波及 | DB 遅延 → API タイムアウト → フロントエンド障害 |
| **リトライストーム** | 失敗リトライが指数的に増幅 | 3 層 × 各 3 回リトライ = 81 倍のリクエスト |
| **デプロイメント** | 不良コードの段階的ロールアウトが全クラスタに波及 | カナリアなしの全台デプロイ |

## 2. 症状 → 対策マトリクス

### 即時対応（止血）

| 症状 | 確認指標 | 即時対策 | 実行判断基準 |
|:---|:---|:---|:---|
| レイテンシ急増 | p99 > 閾値 × 3 | 負荷制限（HTTP 503）発動 | CPU > 90% or p99 > 閾値 × 5 が 3 分継続 |
| エラー率急増 | 5xx 率 > 5% | 障害クラスタのトラフィック排除 | ヘルスチェック 3 回連続失敗 |
| スループット低下 | QPS < 通常の 50% | 縮退運転モード移行 | 正常クラスタ数 < N+1 |
| リトライ急増 | リトライ率 > 20% | リトライバジェット超過で即失敗返却 | リトライ率 > 20% が 1 分継続 |
| 飽和度上昇 | CPU > 80% 継続 | オートスケーリング + 手動スケールアウト | 飽和度 > 70% が 10 分継続 |
| 全面停止 | 可用性 < 50% | トラフィック迂回（DR サイト or 静的ページ） | 可用性 < 50% が 5 分継続 |

### 根本対策（再発防止）

| 誘因 | 設計対策 | 実装パターン |
|:---|:---|:---|
| サーバー過負荷 | キャパシティプランニング + 自動スケーリング | N+2 冗長設計、HPA 設定 |
| リソース枯渇 | リソースリミット + プロセス隔離 | K8s resource limits, cgroup |
| サービス依存 | サーキットブレーカー + フォールバック | Resilience4j, Hystrix, Envoy |
| リトライストーム | 指数バックオフ + ジッター + リトライバジェット | `delay = min(base * 2^n, max) * random(0.5, 1.5)` |
| デプロイメント | カナリアリリース + 自動ロールバック | Argo Rollouts, Flagger |

## 3. リトライ戦略設計テンプレート

### 基本設定

```yaml
retry_policy:
  max_attempts: 3
  initial_delay_ms: 100
  max_delay_ms: 30000
  multiplier: 2.0
  jitter_range: [0.5, 1.5]
  retry_budget:
    max_retries_per_minute: 60
    ratio_to_total: 0.1  # total requests の 10% まで
```

### リトライ可否判定

| HTTP ステータス | リトライ | 理由 |
|:---|:---|:---|
| 400 Bad Request | No | クライアントエラー、リトライしても同じ |
| 401 Unauthorized | No | 認証エラー、トークン更新後のみ再試行 |
| 404 Not Found | No | リソース不在 |
| 408 Request Timeout | Yes | サーバー側タイムアウト |
| 429 Too Many Requests | Yes | レートリミット（Retry-After ヘッダに従う） |
| 500 Internal Server Error | Yes | サーバー一時障害の可能性 |
| 502 Bad Gateway | Yes | 上流サーバー障害 |
| 503 Service Unavailable | Yes | 過負荷、Retry-After に従う |
| 504 Gateway Timeout | Yes | 上流タイムアウト |

### 多層リトライの増幅回避

```
■ 悪い例: 各層 3 回リトライ
Frontend (3) → API Gateway (3) → Service A (3) → Service B (3)
最悪ケース: 3^4 = 81 回のリクエスト

■ 良い例: 末端のみリトライ + 上流はタイムアウト
Frontend (1) → API Gateway (1) → Service A (1) → Service B (3)
最悪ケース: 3 回のリクエスト
```

## 4. 縮退運転パターン

| パターン | 説明 | 適用条件 |
|:---|:---|:---|
| **キャッシュフォールバック** | 古いキャッシュデータで応答 | バックエンド障害時、鮮度より可用性優先 |
| **低品質レスポンス** | 推薦なし、デフォルト値で応答 | ML サービス障害時 |
| **機能制限** | 非必須機能（レコメンド、分析等）を無効化 | 部分障害時 |
| **読み取り専用モード** | 書き込み操作を拒否し、読み取りのみ許可 | DB 書き込み障害時 |
| **静的ページ配信** | CDN から静的ページを返却 | 全面障害時の最終手段 |

## 5. Game Day / DiRT チェックリスト

### 計画フェーズ

- [ ] 訓練の目的とスコープを定義（特定のカスケードシナリオ）
- [ ] 影響範囲を本番のサブセットに限定
- [ ] ロールバック手順を事前にテスト
- [ ] 全参加者に日時・シナリオ概要を通知
- [ ] ステークホルダー（CS、BD）に訓練であることを通知

### 実行フェーズ

- [ ] 障害を注入（Chaos Monkey, Litmus, Gremlin 等）
- [ ] ICS ロールを実践（IC, Ops, Comms, Planning）
- [ ] インシデントドキュメントをリアルタイムで記録
- [ ] 緩和策を実行し、復旧時間を測定
- [ ] 想定外の事象をすべて記録

### 振り返りフェーズ

- [ ] 実際の復旧時間 vs 目標時間を比較
- [ ] 発見された問題点をアクションアイテム化
- [ ] プレイブックの更新が必要な箇所を特定
- [ ] 次回 Game Day の改善ポイントを記録
