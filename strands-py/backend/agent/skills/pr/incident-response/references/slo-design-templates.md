# SLO Design Templates

> incident-response スキル補足資料 — SLO/SLI/エラーバジェットの設計テンプレート集

## 1. SLI 定義テンプレート

### リクエストベース SLI

```yaml
sli:
  name: "<service>_<metric>"
  type: request_based
  description: "<What users expect>"
  qualifying_event:
    filter: "<event selection criteria>"
  good_event:
    condition: "<success criteria>"
  bad_event:
    condition: "<failure criteria>"
  measurement_point: "<load_balancer | server | client>"
  aggregation: "<ratio over window>"
```

### サービスタイプ別 SLI 推奨マトリクス

| サービスタイプ | 可用性 SLI | レイテンシ SLI | 品質 SLI |
|:---|:---|:---|:---|
| ユーザー対向 API | HTTP 2xx+3xx 率 | p99 < 閾値 | — |
| ストレージ | Read/Write 成功率 | p99 Read < 閾値 | 耐久性（年間データロス率） |
| バッチ処理 | ジョブ成功率 | E2E 完了時間 < SLA | 出力データ正確性 |
| ストリーミング | イベント処理成功率 | E2E レイテンシ p99 | イベント重複率 |
| 認証/認可 | 認証成功率（正当リクエスト） | p99 < 200ms | — |
| CDN/静的配信 | 2xx 率 | TTFB p95 | キャッシュヒット率 |

## 2. SLO 設定テンプレート

### SLO ドキュメント

```markdown
# SLO: <サービス名> — <SLI 名>

## 概要
- **SLI**: <定義>
- **SLO ターゲット**: <X.XX%>
- **ウィンドウ**: 30 日ローリング
- **測定頻度**: 1 分間隔

## エラーバジェット計算
- 月間リクエスト: <N>
- 許容失敗数: <N × (1 - SLO)>
- 日次バーン上限: <許容失敗数 / 30>

## ステークホルダー
- SLO オーナー: <name>
- ビジネスオーナー: <name>
- レビュー頻度: 四半期
```

### SLO ターゲット早見表

| SLO | 月間ダウンタイム | 年間ダウンタイム | 適用例 |
|:---|:---|:---|:---|
| 99% | 7 時間 12 分 | 3.65 日 | 内部ツール、バッチ |
| 99.5% | 3 時間 36 分 | 1.83 日 | B2B SaaS（低 SLA） |
| 99.9% | 43 分 50 秒 | 8.77 時間 | 一般的な Web サービス |
| 99.95% | 21 分 55 秒 | 4.38 時間 | 決済、認証 |
| 99.99% | 4 分 23 秒 | 52.6 分 | インフラ基盤 |

## 3. エラーバジェットポリシーテンプレート

```markdown
# エラーバジェットポリシー: <サービス名>

## バジェット状態別アクション

| バジェット残量 | 状態 | リリース | 運用 |
|:---|:---|:---|:---|
| > 50% | Green | 通常リリースサイクル | 通常運用 |
| 20-50% | Yellow | リリース頻度半減、追加テスト必須 | アラート閾値の見直し |
| 5-20% | Orange | 重要修正のみ、信頼性改善に集中 | SRE レビュー必須 |
| < 5% | Red | リリース凍結 | 全リソースを信頼性改善に投入 |
| 0% (枯渇) | Critical | プロダクションフリーズ、VP 承認例外のみ | ポストモーテム必須 |

## 例外条件
- セキュリティパッチ: バジェット状態に関わらずリリース可
- 法規制対応: バジェット状態に関わらずリリース可
- 収益直結バグ: Red 以上で VP 承認によりリリース可

## レビュープロセス
- 週次: SLO ダッシュボード確認（SRE + 開発リード）
- 月次: バジェット消費トレンドレビュー（マネジメント含む）
- 四半期: SLO ターゲット見直し（全ステークホルダー）
```

## 4. SLO 設計アンチパターン

| アンチパターン | 問題 | 正しいアプローチ |
|:---|:---|:---|
| 現行パフォーマンスをそのまま SLO に | 改善余地なし、厳しすぎ | ビジネス要件から導出 |
| SLO を大量に設定 | 注意の分散 | 3-5 個に絞る |
| SLI に平均値を使用 | 外れ値が隠れる | パーセンタイル（p99, p95）|
| 固定ウィンドウ（月初リセット） | 月末の行動歪曲 | 30 日ローリング |
| 内部 SLO = 外部 SLO | バッファなし | 内部をより厳しく設定 |
| SLO を「目標」として扱う | 超過も未達も問題 | エラーバジェットとして運用 |
