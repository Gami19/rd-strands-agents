## Step 1: Ops Scope Definition

対象エージェントの運用スコープを定義する。何を計測し、何を改善するかの範囲を明確にしないと、運用の全てが場当たり的になる。

### 1a. 対象エージェントの特定

| 確認項目 | 内容 | 記録 |
|:---|:---|:---|
| **エージェント名** | 運用対象のエージェント識別子 | |
| **アーキテクチャ** | Single / Multi-Agent / ABT / Platform | |
| **自律性レベル** | L0-L3（ai-agents Step 2 で判定済みの値） | |
| **使用 LLM** | モデル名、プロバイダー、バージョン | |
| **デプロイ環境** | クラウド / オンプレ / ハイブリッド | |
| **利用者** | エンドユーザー / 社内チーム / API 消費者 | |

### 1b. 運用目標の定義

| 目標カテゴリ | 質問 | 記録 |
|:---|:---|:---|
| **品質** | 許容できる最低品質は？ | SLA: 正確性 ≥ 90% 等 |
| **コスト** | 月額予算の上限は？ | $XXX / 月 |
| **レイテンシ** | 最大応答時間は？ | p95 ≤ 3 秒 等 |
| **安全性** | ゼロトレランスの項目は？ | PII 漏洩、プロンプトインジェクション |
| **可用性** | 稼働率要件は？ | 99.9% 等 |

### 1c. 評価対象の選定

```
全てのエージェント出力を評価するか？
├─ Yes → コスト増大を許容できるか？
│    ├─ Yes → 全数評価（高コスト・高精度）
│    └─ No → サンプリング評価（統計的代表性を確保）
└─ No → どの条件で評価するか？
     ├─ 新規デプロイ時 → カナリアリリース評価
     ├─ 定期的 → バッチ評価（日次/週次）
     └─ 異常検知時 → トリガーベース評価
```

**チェックリスト**:
- [ ] 対象エージェントの構成情報を確認した
- [ ] 運用目標を 5 カテゴリ（品質・コスト・レイテンシ・安全性・可用性）で定義した
- [ ] 評価の頻度と範囲を決定した
- [ ] ステークホルダーの期待値と合意した

## Step 2: Evaluation Metrics Design

エージェント出力の品質を定量的に測定するメトリクス体系を設計する。
詳細は [evaluation-metrics.md](references/evaluation-metrics.md) を参照。

### 2a. 品質の 6 次元

| # | 次元 | 定義 | 測定方法 | スケール |
|:---|:---|:---|:---|:---|
| Q1 | **正確性 (Accuracy)** | 出力が事実に基づいているか | グラウンディングスコア、ファクトチェック | 1-5 |
| Q2 | **完全性 (Completeness)** | 要求された情報が網羅されているか | ルブリック基準との照合 | 1-5 |
| Q3 | **安全性 (Safety)** | 有害・不適切な出力がないか | コンテンツフィルター、毒性スコア | Pass/Fail |
| Q4 | **一貫性 (Consistency)** | 同一入力に対する出力のブレ | 複数実行の分散測定 | 標準偏差 |
| Q5 | **関連性 (Relevance)** | ユーザーの意図に沿った応答か | 意図分類との一致度 | 1-5 |
| Q6 | **流暢性 (Fluency)** | 自然で読みやすい表現か | 言語品質スコア | 1-5 |

### 2b. 運用メトリクス

| カテゴリ | メトリクス | 測定方法 | 閾値の例 |
|:---|:---|:---|:---|
| **レイテンシ** | p50 / p95 / p99 応答時間 | タイムスタンプ計測 | p95 ≤ 3s |
| **コスト** | リクエスト単価 / 日次コスト | トークン数 × 単価 | ≤ $0.05/req |
| **スループット** | RPM (Requests Per Minute) | カウンター | ≥ 100 RPM |
| **エラー率** | 4xx / 5xx 率 | HTTP ステータス集計 | ≤ 1% |
| **トークン効率** | 出力トークン / 入力トークン比 | API レスポンス解析 | ≤ 2.0 |
| **キャッシュ** | ヒット率 | キャッシュログ | ≥ 30% |

### 2c. ルブリック設計

書籍 Ch.9 の Prompt Flow 評価フレームワークに基づくルブリック定義プロセス:

1. **目的と目標の特定**: エージェントが達成すべきゴールを定義
2. **評価基準の定義**: 品質 6 次元から対象メトリクスを選定
3. **スケールの作成**: 1-5 の数値スケール + 各レベルの説明文
4. **説明の提供**: 各スコアレベルの具体的な判定基準を記述
5. **ルブリックの適用**: 手動評価で基準を検証
6. **総合スコアの算出**: 重み付き平均または単純平均
7. **評価一貫性の確保**: 複数評価者間の一致率を測定
8. **反復改善**: ルブリック自体の精度を定期的に見直す

**チェックリスト**:
- [ ] 品質 6 次元の中から対象メトリクスを選定した
- [ ] 各メトリクスの測定方法とスケールを定義した
- [ ] 運用メトリクス（レイテンシ・コスト・スループット・エラー率）を定義した
- [ ] ルブリックを作成し、手動評価で検証した
- [ ] 各メトリクスの閾値（SLO）を設定した

## Step 3: Evaluation Pipeline Construction

評価を自動化するパイプラインを構築する。手動評価は初期校正には有効だが、継続的運用にはスケールしない。

### 3a. パイプラインアーキテクチャ

```
エージェント出力
     ↓
[入力ログ収集] → 入力プロンプト + コンテキストを記録
     ↓
[出力ログ収集] → 生成テキスト + メタデータ（トークン数、レイテンシ）を記録
     ↓
[自動評価] → ルブリックベース LLM 評価（グラウンディング）
     ↓
[スコア集約] → メトリクスごとの平均・分散・トレンドを算出
     ↓
[アラート判定] → 閾値違反時に通知
     ↓
[ダッシュボード更新] → 可視化
```

### 3b. グラウンディング評価の実装

書籍 Ch.9 の LLM グラウンディング評価パターンに基づく:

| 評価パターン | 説明 | 適用場面 |
|:---|:---|:---|
| **LLM-as-Judge** | 別の LLM がルブリックに基づいて出力をスコアリング | 汎用的な品質評価 |
| **Reference-based** | 参照回答との一致度を算出 | 正解がある QA タスク |
| **Self-evaluation** | 同一 LLM が自身の出力を評価 | 低コスト・迅速な評価 |
| **Cross-agent** | 別エージェントが出力を検証 | マルチエージェント構成 |
| **Human-in-the-Loop** | 人間がサンプルをレビュー | 高精度のベースライン構築 |

### 3c. バッチ評価の設計

| 項目 | 設計内容 |
|:---|:---|
| **テストデータセット** | 代表的な入力 50-200 件 + 期待出力 |
| **バリアント管理** | プロンプト変種・モデル変種のラベル管理 |
| **並列実行** | Prompt Flow のワーカー並列化 |
| **結果集約** | メトリクス別の平均・分散・比較レポート |

**チェックリスト**:
- [ ] 評価パイプラインのアーキテクチャを設計した
- [ ] グラウンディング評価のパターンを選定した
- [ ] 評価プロンプト（ルブリック込み）を作成した
- [ ] テストデータセットを準備した（最低 50 件）
- [ ] バッチ評価の実行環境を構築した
- [ ] 評価結果の集約・レポート形式を定義した

## Step 4: Guardrail Design

エージェントの安全性を 4 層のガードレールで防御する。ガードレールは「あった方がいい」ではなく「なければ運用できない」必須インフラ。
詳細は [guardrail-patterns.md](references/guardrail-patterns.md) を参照。

### 4a. 4 層ガードレール概要

| 層 | 防御対象 | 主要パターン |
|:---|:---|:---|
| **入力** | 悪意ある入力・過大入力 | プロンプトインジェクション検出、PII マスキング、入力長制限、レート制限 |
| **出力** | 不正確・有害な出力 | ハルシネーション検出、有害出力フィルタリング、出力構造検証、情報漏洩防止 |
| **コスト** | 予算超過・無限ループ | トークン予算上限、セッション/日次/月次予算、反復上限（推奨 10-20） |
| **安全性** | 危険なアクション実行 | アクション承認ゲート、ホワイトリスト、Kill switch、ロールバック機構 |

各パターンの詳細な実装方法は [guardrail-patterns.md](references/guardrail-patterns.md) を参照。

### 4b. ガードレール選定フロー

```
エージェントの自律性レベルは？
├─ L0-L1 → 入力ガードレール（4a）のみ必須
├─ L2 → 入力（4a）+ 出力（4b）+ アクション承認ゲート
└─ L3 → 全 4 層（4a-4d）必須
     └─ 特に反復上限・Kill switch・コスト上限は絶対必須
```

**チェックリスト**:
- [ ] 自律性レベルに応じたガードレール層を選定した
- [ ] 入力ガードレールを設計した（プロンプトインジェクション、PII）
- [ ] 出力ガードレールを設計した（ハルシネーション、有害出力）
- [ ] コストガードレールを設計した（予算上限、反復制限）
- [ ] L3 の場合、Kill switch とロールバック機構を設計した
- [ ] ガードレールの偽陽性率を推定し、許容範囲か確認した

## Step 5: Monitoring & Observability

エージェントの稼働状況をリアルタイムで把握する監視体制を構築する。監視なき運用は夜間の無灯火走行に等しい。
詳細は [monitoring-guide.md](references/monitoring-guide.md) を参照。

### 5a. 監視の 3 本柱

| 柱 | 収集対象 | ツール例 |
|:---|:---|:---|
| **Traces（トレース）** | リクエスト → LLM 呼び出し → ツール実行 → 応答の完全な経路 | AgentOps, LangSmith, LangFuse |
| **Metrics（メトリクス）** | レイテンシ、トークン使用量、エラー率、コスト | Prometheus, CloudWatch, Datadog |
| **Logs（ログ）** | 入力/出力テキスト、エラー詳細、デバッグ情報 | ELK Stack, CloudWatch Logs |

### 5b. 監視ツール統合

書籍 Ch.4 の AgentOps パターンに基づく。ツール選定と統合の詳細は [monitoring-guide.md](references/monitoring-guide.md) を参照。

| 収集対象 | 説明 | 実装 |
|:---|:---|:---|
| **セッショントレース** | 全行動を時系列で記録 | `agentops.init()` / LangSmith tracing |
| **LLM 呼び出し** | プロンプト・応答・トークン数・レイテンシ | プロバイダー SDK のフック |
| **ツール実行** | ツール名・入出力・実行時間 | アクションデコレーター |
| **コスト** | 実行ごとの推定コスト | モデル別単価テーブル |
| **エラー** | 例外・タイムアウト・レート制限 | エラーハンドラーのフック |

### 5c. ダッシュボード設計

| パネル | 表示内容 | 更新頻度 |
|:---|:---|:---|
| **Overview** | 総リクエスト数、成功率、平均レイテンシ、累積コスト | リアルタイム |
| **Quality** | 品質 6 次元のスコア推移（時系列グラフ） | 評価実行ごと |
| **Cost** | 日次/週次コスト推移、モデル別内訳 | 日次 |
| **Errors** | エラー種別の内訳、トップエラーランキング | リアルタイム |
| **Guardrails** | ガードレール発動回数、偽陽性率 | リアルタイム |

### 5d. アラート設計

| アラート | 条件 | 重大度 | 通知先 |
|:---|:---|:---|:---|
| **品質劣化** | 評価スコア平均が SLO を下回る | Warning | 運用チーム |
| **コスト超過** | 日次コストが予算の 80% を超過 | Warning | 運用チーム + 管理者 |
| **エラー急増** | エラー率が 5% を超過 | Critical | 運用チーム |
| **レイテンシ劣化** | p95 レイテンシが SLO の 2 倍 | Warning | 運用チーム |
| **ガードレール突破** | 安全性ガードレールが 1 回でも突破 | Critical | 運用チーム + セキュリティ |
| **予算枯渇** | 月次予算の 95% を消費 | Critical | 管理者 |

**チェックリスト**:
- [ ] 監視の 3 本柱（Traces / Metrics / Logs）の収集方法を設計した
- [ ] 監視ツール（AgentOps / LangSmith / LangFuse 等）を選定した
- [ ] ダッシュボードのパネル構成を設計した
- [ ] アラート条件と重大度を定義した
- [ ] アラート通知先とエスカレーションフローを定義した
- [ ] 監視のオーバーヘッド（コスト・レイテンシ）を見積もった

## Step 6: Cost Optimization

トークンコストはエージェント運用の最大の変動費。書籍 Ch.4 では単一ジョーク生成に $0.50 以上かかる事例が報告されている。計測なき最適化は不可能。

### 6a. コスト分析

| 分析項目 | 手法 | ツール |
|:---|:---|:---|
| **トークン内訳** | 入力/出力トークンの比率分析 | AgentOps コストパネル |
| **モデル別コスト** | モデルごとの単価 × 使用量 | Provider 課金ダッシュボード |
| **機能別コスト** | 推論 / ツール呼び出し / 評価の分離 | カスタムタグ付きログ |
| **ユーザー別コスト** | ヘビーユーザーの特定 | セッションログ集計 |

### 6b. 最適化施策

| 施策 | 効果 | 実装難易度 | リスク |
|:---|:---|:---|:---|
| **プロンプト圧縮** | 入力トークン 20-40% 削減 | 低 | 品質劣化の可能性 |
| **セマンティックキャッシング** | 類似クエリの再利用で 30-50% 削減 | 中 | 陳腐化リスク |
| **モデルカスケード** | 簡単なタスクに安価なモデルを割当 | 中 | ルーティング精度 |
| **出力トークン制限** | max_tokens 設定で過剰生成防止 | 低 | 応答の途中切断 |
| **バッチ処理** | リアルタイム不要な処理をバッチ化 | 低 | レイテンシ増加 |
| **Fine-tuning** | 小型モデルの特化学習 | 高 | 学習データの準備コスト |

### 6c. コスト最適化の判断フロー

```
現在のコストは予算内か？
├─ Yes → 品質は SLO を満たしているか？
│    ├─ Yes → 現状維持（定期的にモニタリング）
│    └─ No → モデルアップグレードを検討
└─ No → コスト超過の主因は？
     ├─ 入力トークン過多 → プロンプト圧縮
     ├─ 出力トークン過多 → max_tokens 制限
     ├─ リクエスト数過多 → セマンティックキャッシング
     ├─ 高価なモデル → モデルカスケード / ダウングレード
     └─ 自律エージェントの反復過多 → 反復上限の引き下げ
```

**チェックリスト**:
- [ ] 現在のコスト構造を分析した（トークン内訳・モデル別・機能別）
- [ ] コスト最適化施策を 1 つ以上選定した
- [ ] 選定した施策の品質影響を評価した（A/B テストまたは事前検証）
- [ ] 最適化後の期待コスト削減率を見積もった
- [ ] 最適化施策のロールバック計画を策定した

## Step 7: Feedback Loop & Continuous Improvement

エージェントは静的なシステムではない。フィードバックを収集し、評価し、改善を反映する継続的サイクルを設計する。

### 7a. フィードバック源の設計

| 源 | 収集方法 | データ形式 | 適用場面 |
|:---|:---|:---|:---|
| **Environmental** | API レスポンス、ツール実行結果 | 構造化ログ | 自動フィードバック |
| **Human (explicit)** | 評価 UI（良い/悪い）、修正テキスト | ラベル付きデータ | ユーザー対面型 |
| **Human (implicit)** | リトライ率、セッション離脱率 | 行動ログ | 間接的品質指標 |
| **Self-generated** | LLM の自己評価、信頼度スコア | 評価スコア | 自律的改善 |
| **Cross-agent** | 他エージェントによる検証 | 評価レポート | マルチエージェント |

### 7b. Human-in-the-Loop / Human-on-the-Loop 設計

| パターン | 説明 | 適用 |
|:---|:---|:---|
| **Human-in-the-Loop** | 全ての出力を人間が確認・承認 | L2、高リスクタスク |
| **Human-on-the-Loop** | 人間は監視し、異常時のみ介入 | L3、低リスク・高頻度 |
| **Human-off-the-Loop** | 完全自律、定期レビューのみ | L3、信頼性が実証済み |

```
初期デプロイ: Human-in-the-Loop（全承認）
     ↓ 信頼性スコア ≥ 95% を 2 週間維持
     ↓
移行期: Human-on-the-Loop（異常時のみ介入）
     ↓ 信頼性スコア ≥ 98% を 1 ヶ月維持
     ↓
成熟期: Human-off-the-Loop（定期レビュー）
```

### 7c. A/B テスト設計

| 設計項目 | 内容 |
|:---|:---|
| **仮説** | 「新プロンプト B は旧プロンプト A より正確性が 10% 向上する」 |
| **トラフィック分割** | 50:50 または段階的（10% → 50% → 100%） |
| **評価メトリクス** | Step 2 で定義した品質 6 次元 + コスト |
| **最小サンプルサイズ** | 統計的有意差（p < 0.05）に必要なリクエスト数 |
| **実験期間** | サンプルサイズに到達するまでの想定日数 |
| **意思決定基準** | 勝者判定の基準（メトリクスの重み付き比較） |

### 7d. 改善サイクル

```
Plan:  品質メトリクスの分析 → 改善仮説の立案
  ↓
Do:    プロンプト修正 / モデル変更 / ガードレール調整
  ↓
Check: A/B テスト → 評価パイプラインでスコア比較
  ↓
Act:   勝者をデプロイ / 仮説棄却 → 新しい仮説へ
  ↓
（反復）
```

**チェックリスト**:
- [ ] フィードバック源を特定し、収集方法を設計した
- [ ] Human-in-the-Loop / on-the-Loop のパターンを選定した
- [ ] 信頼性に基づく人間介入レベルの移行基準を定義した
- [ ] A/B テストの設計（仮説・分割・メトリクス・サンプルサイズ）を完了した
- [ ] PDCA 改善サイクルの実行ルールを定義した

### エージェント運用 Slop 防止チェック（Distributional Convergence 対策）

エージェント運用設計が汎用テンプレートの転用に終始すると、対象エージェント固有のリスクや特性を見逃し、「運用している体裁」だけが整い実質的な品質保証が機能しない。

| # | エージェント運用 Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| AO-1 | **三種の神器メトリクス** | 評価メトリクスが毎回 accuracy / latency / cost の 3 つだけで、エージェント固有の品質次元がない | エージェントのタスク特性に応じた固有メトリクスを追加する（例: RAG なら「引用元の正確性」「情報の鮮度」、コード生成なら「構文正当性」「テスト通過率」、対話なら「文脈保持率」「トーン一貫性」） |
| AO-2 | **汎用ガードレール症候群** | 入力/出力のガードレールが全エージェントで同一パターン（入力長制限 + 毒性フィルター + PII マスキング）になっている | エージェントの権限・ツール・ドメインに応じた固有ガードレールを設計する（例: 金融エージェントなら「取引金額の上限チェック」、採用エージェントなら「保護属性への言及フィルター」） |
| AO-3 | **キャッシュ・バッチ万能薬** | コスト最適化が毎回「セマンティックキャッシング導入」と「バッチ処理化」の 2 施策に帰結する | コスト構造の実データに基づいて主因を特定し、エージェント固有の施策を選定する（例: マルチエージェントなら「不要なエージェント間通信の削減」、長文処理なら「チャンク戦略の最適化」） |
| AO-4 | **五大メトリクスダッシュボード** | 監視ダッシュボードがエージェントの種類に関係なく同じ 5 パネル（リクエスト数、レイテンシ、エラー率、コスト、成功率）で構成される | エージェントのタスク特性に応じた固有パネルを追加する（例: ツール使用エージェントなら「ツール別呼び出し頻度・成功率」、マルチエージェントなら「エージェント間メッセージフロー」） |
| AO-5 | **対称 A/B テスト構造** | A/B テストが毎回同じ構造（50:50 分割、品質 6 次元で比較、p < 0.05 で判定）で、実験仮説がエージェント固有でない | エージェントの改善課題に即した固有の仮説・メトリクス・判定基準を設計する（例: 「ツール選択の正確性を改善する Few-shot の効果」「特定ドメインの幻覚率を下げるグラウンディング戦略の効果」） |
| AO-6 | **三段階エスカレーションプレイブック** | インシデント対応が毎回同じ「確認→再起動→エスカレーション」パターンで、エージェント固有の障害モードに対応していない | エージェント固有の障害モードを分類し、それぞれに対応プレイブックを設計する（例: 「ツール呼び出しの無限ループ」「幻覚の連鎖増幅」「コンテキストウィンドウ枯渇」「プロンプトインジェクションの検知後対応」） |

> **核心原則**: **運用設計はエージェントの「カルテ」であり、汎用の「健康診断フォーム」ではない** — 運用設計書だけを読んで、対象エージェントが何をするエージェントか特定できるか？「別のエージェントの運用設計に差し替えても違和感がないか？」→ 違和感がないなら Agent Ops Slop である。

**チェックリスト**:
- [ ] 運用ランブックを作成した
- [ ] Ops レディネスチェックリストの全項目を確認した
- [ ] セルフレビューを実施し、問題を修正した
- [ ] ステークホルダーに運用計画を共有し、承認を得た
- [ ] エージェント運用 Slop チェック（AO-1〜AO-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Examples

### Example 1: チャットボット評価パイプライン構築

```
「カスタマーサポートチャットボットの品質を自動評価するパイプラインを作って」

→ Step 1: 対象=FAQ チャットボット（L2）、品質 SLO=正確性 90%、コスト=$500/月
→ Step 2: Q1 正確性 + Q2 完全性 + Q5 関連性の 3 次元を採用
→ Step 3: LLM-as-Judge パターン、テストデータ 100 件、バッチ評価を週次で実行
   ルブリック: 正確性=FAQ 原文との一致度、完全性=質問に対する情報網羅度
→ 成果物: 評価パイプライン設計書 + ルブリック定義 + テストデータセット
```

### Example 2: 自律エージェントのガードレール設計

```
「L3 自律リサーチエージェントにガードレールを設計して」

→ Step 1: 対象=L3 リサーチエージェント、Kill switch 必須、反復上限必須
→ Step 4: 4 層全てを設計
   入力: プロンプトインジェクション検出 + 入力長 4096 トークン上限
   出力: ハルシネーション検出（グラウンディングスコア ≥ 3）+ 有害出力フィルター
   コスト: 反復上限 10 回 + セッション予算 $1 + 月次予算 $200
   安全性: Web アクセスのドメインホワイトリスト + Kill switch
→ 成果物: 4 層ガードレール設計書 + テストシナリオ（正常系 + 攻撃系）
```

### Example 3: トークンコスト 50% 削減

```
「マルチエージェントの月額コストが $2,000 を超えているので半減したい」

→ Step 1: 対象=CrewAI 3 エージェント構成、月額 $2,100、コスト目標 $1,000
→ Step 6: コスト分析→ Agent A がコストの 60% を消費（長文プロンプト + 高頻度）
   施策 1: Agent A のプロンプト圧縮（-30%、品質影響を A/B テストで検証）
   施策 2: Agent B をGPT-4 → GPT-4o-mini に変更（-40%、品質劣化リスク低）
   施策 3: セマンティックキャッシング導入（ヒット率 35% 推定で -20%）
→ Step 7: 施策 2 → 3 → 1 の順に段階適用、各段階で A/B テスト
→ 成果物: コスト最適化計画書（施策・期待効果・リスク・ロールバック計画）
```

### Example 4: 監視ダッシュボード設計

```
「エージェントの稼働状況を可視化するダッシュボードを設計して」

→ Step 1: 対象=BD 支援エージェント（L2）、監視チーム 3 名
→ Step 5: AgentOps を監視ツールに選定
   Overview パネル: 日次リクエスト数、成功率、平均レイテンシ、累積コスト
   Quality パネル: 正確性・完全性スコアの週次トレンド
   Cost パネル: モデル別コスト内訳、日次コスト推移
   Errors パネル: エラー種別サンバースト図、直近 24h のエラーログ
   アラート: エラー率 > 5%（Critical）、p95 > 5s（Warning）、日次コスト > $30（Warning）
→ 成果物: ダッシュボード設計書 + アラートルール定義
```

### Example 5: プロンプト A/B テスト

```
「新しいプロンプトの効果を A/B テストで検証したい」

→ Step 2: 評価メトリクス=正確性 + 完全性 + レイテンシ + トークンコスト
→ Step 3: LLM-as-Judge で自動評価、テストデータ 200 件
→ Step 7: A/B テスト設計
   仮説: 「CoT プロンプト B は Q&A プロンプト A より正確性が 15% 向上する」
   トラフィック分割: 50:50
   最小サンプル: 500 リクエスト（効果量 0.15、検出力 0.8 で算出）
   期間: 5 日間（日次 100 リクエストの想定）
   意思決定: 正確性 +10% 以上かつコスト増 ≤ 20% なら B を採用
→ 成果物: A/B テスト設計書 + 結果レポートテンプレート
```

### Example 6: 運用ランブック作成

```
「エージェント障害時の対応手順書を作って」

→ Step 1: 対象=本番チャットボット（L2）、SLA=99.9%、対応チーム=SRE 2 名
→ Step 5: 監視設計（アラート 6 種 + エスカレーション 3 段階）
→ Step 8: 運用ランブック
   正常時: 毎朝ダッシュボード確認、週次の評価パイプライン実行
   障害レベル 1（品質劣化）: 評価ログ確認→プロンプト原因調査→ロールバック判断
   障害レベル 2（サービス停止）: Kill switch 発動→フォールバック→RCA
   障害レベル 3（安全性侵害）: 即時停止→セキュリティチーム通知→インシデント対応
   定期メンテ: 月次ルブリック見直し、四半期テストデータ更新
→ 成果物: 運用ランブック（障害対応手順 + エスカレーションマトリクス）
```

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| 評価スコアがドメインの品質を反映しない | ルブリックの基準がドメインに合っていない | ドメイン専門家と協力してルブリックを再設計。Few-shot で評価基準の具体例を提供 |
| グラウンディングスコアが常に低い | 参照データが不十分、または評価プロンプトが厳しすぎる | 参照データを拡充し、評価スケールの説明文を見直す。手動評価とのキャリブレーションを実施 |
| ガードレールが正常入力をブロックする（偽陽性） | 検出ルールが過度に厳格 | 閾値を緩和し、偽陽性のログを収集して検出ルールを調整。ホワイトリストの導入を検討 |
| ガードレールが攻撃を通過させる（偽陰性） | 検出パターンが攻撃手法をカバーしていない | 最新の攻撃パターン集（OWASP LLM Top 10）で検出ルールを更新。多層防御を導入 |
| 監視ツールのオーバーヘッドでレイテンシ増加 | トレース収集が同期的に実行されている | 非同期でのログ送信に変更。サンプリングレートを導入（全リクエストの 10-50%） |
| トークンコストが予算を超過 | プロンプトの肥大化、または自律エージェントの反復過多 | Step 6 のコスト分析を実施し、主因を特定。プロンプト圧縮・モデルカスケード・反復上限を適用 |
| キャッシュヒット率が低い | クエリの多様性が高い、またはキャッシュキーの設計が不適切 | セマンティックキャッシング（エンベディング類似度ベース）に変更。キャッシュ TTL を調整 |
| A/B テストの結果が統計的に有意でない | サンプルサイズが不足、または効果量が小さい | 実験期間を延長してサンプルを増やす。効果量の期待値を下方修正し、必要サンプルサイズを再計算 |
| フィードバックループが収束しない | 改善基準が曖昧、または前回との差分が未追跡 | 定量的な品質スコアを導入し、3 回連続で改善幅が閾値未満なら打ち切り。改善履歴を記録 |
| Human-in-the-Loop がボトルネック化 | レビュー待ちキューが滞留 | Human-on-the-Loop への移行を検討。信頼度スコアに基づく自動承認ルールを導入 |
| 評価パイプラインの実行コストが本体を超える | 全数評価を実施している、または評価 LLM が高価 | サンプリング評価に切り替え（統計的代表性を維持しつつ 10-20% に削減）。評価 LLM を安価なモデルに変更 |
| 複数モデル間で評価基準がブレる | モデルカスケードで異なるモデルに同一ルブリックを適用 | モデル別のルブリック校正を実施。各モデルの出力特性に応じた評価基準の微調整 |
