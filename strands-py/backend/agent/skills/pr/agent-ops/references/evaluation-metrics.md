## 1. 品質メトリクス（6 次元）

### Q1: 正確性 (Accuracy)

出力が事実に基づいているかを測定する。エージェント品質の最も基本的な次元。

| 測定手法 | 説明 | 実装方法 |
|:---|:---|:---|
| **グラウンディングスコア** | 参照データとの一致度（1-5） | LLM-as-Judge でルブリック評価 |
| **ファクトチェック** | 個別の事実主張の検証 | 主張抽出 → 参照 DB 照合 |
| **Reference-based 評価** | 正解との ROUGE/BERTScore | 正解付きテストデータで算出 |
| **人間評価** | ドメイン専門家による正確性判定 | サンプリング + 評価 UI |

**ルブリック例（1-5 スケール）**:

| スコア | 説明 |
|:---:|:---|
| 1 | 事実と矛盾する情報を含む。明確なハルシネーション |
| 2 | 部分的に正確だが、重要な誤りを含む |
| 3 | 概ね正確だが、細部に不正確な点がある |
| 4 | ほぼ正確。軽微な曖昧さのみ |
| 5 | 完全に正確。参照データと一致し、誤りなし |

---

### Q2: 完全性 (Completeness)

要求された情報が網羅されているかを測定する。

| 測定手法 | 説明 | 実装方法 |
|:---|:---|:---|
| **要件カバレッジ率** | 要求項目の充足率 | チェックリスト照合（自動/手動） |
| **情報量スコア** | 期待される情報量との比較 | トークン数 + 意味的カバレッジ |
| **欠落検出** | 含まれるべきだが欠落した情報 | ルブリック内の必須項目チェック |

**ルブリック例（1-5 スケール）**:

| スコア | 説明 |
|:---:|:---|
| 1 | 要求の大部分に未回答。情報が著しく不足 |
| 2 | 主要な要求に部分的に回答。重要な欠落あり |
| 3 | 主要な要求にはおおむね回答。補足的な情報が不足 |
| 4 | ほぼ全ての要求に回答。軽微な追加情報のみ不足 |
| 5 | 全ての要求に完全に回答。必要十分な情報を提供 |

---

### Q3: 安全性 (Safety)

有害・不適切な出力がないかを測定する。Pass/Fail の二値判定が基本。

| 測定手法 | 説明 | 実装方法 |
|:---|:---|:---|
| **毒性スコア** | 攻撃的・差別的表現の検出 | Perspective API / Azure Content Safety |
| **PII 検出** | 個人情報の出力有無 | NER + パターンマッチング |
| **プロンプトリーク** | システムプロンプトの漏洩有無 | 既知パターンの照合 |
| **有害行動誘導** | 危険な行動を促す内容の有無 | コンテンツ分類器 |

**判定基準**:

| 結果 | 条件 |
|:---|:---|
| **Pass** | 全安全性チェックをクリア |
| **Fail** | いずれかのチェックで違反を検出 |
| **Warning** | グレーゾーン（人間レビューが必要） |

---

### Q4: 一貫性 (Consistency)

同一または類似の入力に対する出力のブレを測定する。

| 測定手法 | 説明 | 実装方法 |
|:---|:---|:---|
| **再現性テスト** | 同一入力を N 回実行し、出力の分散を測定 | temperature=0 でも変動を確認 |
| **意味的類似度** | 複数出力間のコサイン類似度 | エンベディングベース |
| **構造的一貫性** | 出力フォーマットの安定性 | JSON Schema バリデーション |
| **論理的一貫性** | 矛盾する主張がないか | 自己矛盾検出プロンプト |

**測定手順**:
1. 同一入力で 5-10 回の実行
2. 各出力のエンベディングを計算
3. ペアワイズコサイン類似度の平均を算出
4. 標準偏差が閾値内か確認（推奨: σ ≤ 0.1）

---

### Q5: 関連性 (Relevance)

ユーザーの意図に沿った応答かを測定する。

| 測定手法 | 説明 | 実装方法 |
|:---|:---|:---|
| **意図一致度** | ユーザーの質問意図と応答の方向性 | 意図分類器 + 応答分類器の一致率 |
| **トピック適合度** | 応答がトピックから逸脱していないか | トピックモデリング |
| **情報密度** | 関連情報と非関連情報の比率 | 文単位の関連性スコアリング |

---

### Q6: 流暢性 (Fluency)

自然で読みやすい表現かを測定する。

| 測定手法 | 説明 | 実装方法 |
|:---|:---|:---|
| **言語品質スコア** | 文法・語彙・構成の品質 | LLM-as-Judge |
| **可読性指標** | 読みやすさの定量指標 | Flesch-Kincaid 等 |
| **構造化品質** | 見出し・リスト・段落の適切さ | 構造解析 |

## 2. 運用メトリクス

### レイテンシメトリクス

| メトリクス | 定義 | 計算方法 | 推奨閾値 |
|:---|:---|:---|:---|
| **TTFB** | Time to First Byte | リクエスト送信 → 最初のバイト受信 | ≤ 500ms |
| **p50 レイテンシ** | 中央値応答時間 | 全リクエストの 50 パーセンタイル | ≤ 1s |
| **p95 レイテンシ** | 95 パーセンタイル応答時間 | 全リクエストの 95 パーセンタイル | ≤ 3s |
| **p99 レイテンシ** | 99 パーセンタイル応答時間 | 全リクエストの 99 パーセンタイル | ≤ 10s |

### コストメトリクス

| メトリクス | 定義 | 計算方法 |
|:---|:---|:---|
| **リクエスト単価** | 1 リクエストあたりの推定コスト | (入力トークン × 入力単価) + (出力トークン × 出力単価) |
| **日次コスト** | 1 日の累積コスト | 全リクエスト単価の合計 |
| **ユーザー単価** | ユーザーあたりの月間コスト | 月次コスト / アクティブユーザー数 |
| **トークン効率** | 出力/入力トークン比 | 出力トークン数 / 入力トークン数 |

### 信頼性メトリクス

| メトリクス | 定義 | 計算方法 | 推奨閾値 |
|:---|:---|:---|:---|
| **成功率** | 正常応答の割合 | 成功リクエスト / 全リクエスト | ≥ 99% |
| **エラー率** | エラー応答の割合 | エラーリクエスト / 全リクエスト | ≤ 1% |
| **タイムアウト率** | タイムアウトの割合 | タイムアウト数 / 全リクエスト | ≤ 0.1% |
| **可用性** | サービス稼働率 | 稼働時間 / 全時間 | ≥ 99.9% |

## 3. 評価パターン比較

| パターン | コスト | 精度 | スケーラビリティ | 適用場面 |
|:---|:---|:---|:---|:---|
| **LLM-as-Judge** | 中 | 高 | 高 | 汎用的な品質評価。最もバランスが良い |
| **Reference-based** | 低 | 最高 | 高 | 正解がある QA タスク。テストデータ準備が必要 |
| **Self-evaluation** | 低 | 中 | 高 | 低コスト・迅速な評価。バイアスに注意 |
| **Cross-agent** | 中-高 | 高 | 中 | マルチエージェント構成。相互検証 |
| **Human-in-the-Loop** | 高 | 最高 | 低 | ベースライン構築。定期的なキャリブレーション |
| **Batch comparison** | 中 | 高 | 高 | プロファイル変種の比較。A/B テスト |

### 評価パターン選定フロー

```
正解データがあるか？
├─ Yes → Reference-based 評価（最も精度が高い）
│    └─ 定期的に Human-in-the-Loop でキャリブレーション
└─ No → 評価予算は十分か？
     ├─ Yes → LLM-as-Judge（別モデル推奨）
     │    └─ マルチエージェントなら Cross-agent を追加
     └─ No → Self-evaluation（バイアス監視必須）
```
