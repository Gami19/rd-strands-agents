# Skill: Onboarding（新人オンボーディング支援）

> **新人の学びを、AIが伴走する — ナビ・クイズ・メンターの統合オンボーディング支援**

## strands-py / Strands Agent での利用

ロードマップ・レポート・チェックリストは **`file_read` / `file_write`** とプロジェクト workspace を正とする。社内21本ドキュメントは workspace に無ければヒアリングで補う。**skill-forge** は Strands では無効。詳細は [references/strands-py-runtime.md](references/strands-py-runtime.md) を参照。

## Instructions

### ワークフロー内の位置

```
新人入社 / 学習の質問
  ↓
[onboarding] ← 統合オンボーディング支援
  ├─ Step 1-3: ナビゲーション（状況把握→ロードマップ→学習ガイド）
  ├─ Step 4-5: 理解度チェック（クイズ生成→スコアレポート）
  ├─ Step 6-7: メンタリング（Q&A→シナリオアドバイス）
  ├─ Step 8: 1on1セッション支援
  └─ Step 9: フォローアップ・進捗レポート
  ↓
成長する新人（主力選手へ）
```

### 入力

| 入力 | 説明 | 例 |
|:---|:---|:---|
| 新人の背景情報 | 学歴・前職・IT経験 | 「文系卒・ビジネス経験3年・IT未経験」 |
| 学習進捗 | 読了済みドキュメント | 「Culture の 5 本は読んだ」 |
| 質問・相談 | 学習上の疑問やメンタル面の悩み | 「NDA の範囲がわからない」 |
| クイズ範囲指定 | カテゴリ・難易度の希望 | 「Tech の基礎クイズを出して」 |

### 出力

| 出力 | 形式 | 説明 |
|:---|:---|:---|
| 学習ロードマップ | Markdown | 週次スケジュール付き学習パス |
| 学習ガイド | Markdown | 推薦ドキュメント + 学習ポイント |
| クイズ | 対話形式 | 1問ずつ出題→回答待ち→フィードバック |
| スコアレポート | Markdown | 総合スコア + カテゴリ別分析 |
| メンタリング回答 | Markdown | 根拠引用付きアドバイス |
| 1on1 準備シート | Markdown | KPT + SMART 目標テンプレート |
| 進捗レポート | Markdown | プログレスバー付き進捗サマリー |

### ナレコム 3 コアバリュー（全ステップの行動指針）

| コアバリュー | オンボーディングでの体現 |
|:---|:---|
| **お客様の期待を超える** | 学習の質問に＋αの知見を添えて回答する |
| **主力選手** | 新人が自ら考え行動できる力を育てる |
| **チームワークを超えた絆** | 不安や悩みに寄り添い、心理的安全性を確保する |

### ドキュメント構成（KC-Onboarding-Docs 全21本）

| カテゴリ | 本数 | 内容 | 推奨学習順 |
|:---|:---|:---|:---|
| **Culture** | 5本 | 企業理念・行動指針・コアバリュー・社内制度・コミュニケーション | 1番目 |
| **Biz** | 3本 | 事業内容・サービス概要・顧客理解 | 2番目 |
| **Tech** | 7本 | 技術スタック・開発フロー・ツール・セキュリティ | 3番目 |
| **Ops** | 6本 | 業務フロー・経費・勤怠・NDA・各種申請 | 4番目 |

## Step 1: オンボーディング状況の把握

新人の背景と現在地を理解し、最適な支援方針を決定する。

### 1a. ヒアリング項目

| 項目 | 質問例 | 目的 |
|:---|:---|:---|
| **学歴・専攻** | 「大学では何を学ばれましたか？」 | 基礎知識レベルの推定 |
| **前職・経験** | 「前職ではどんな業務をされていましたか？」 | IT リテラシーの推定 |
| **IT経験** | 「プログラミングやクラウドの経験はありますか？」 | Tech カテゴリの深度調整 |
| **入社時期** | 「入社何日目ですか？」 | 学習フェーズの判定 |
| **既読ドキュメント** | 「既に読んだドキュメントはありますか？」 | 開始ポイントの決定 |

### 1b. 学習スタイルの確認

| スタイル | 特徴 | 対応方針 |
|:---|:---|:---|
| **じっくり型** | 1本ずつ深く理解したい | 基本パス（5週間）を推奨 |
| **速習型** | 全体像を早く掴みたい | 急ぎパス（1週間）を推奨 |
| **実践型** | 手を動かしながら学びたい | カスタムパス + クイズ多め |

**チェックリスト**:
### 品質ガードレール

**CRITICAL**: セキュリティ研修の完了確認を必須チェックポイントとすること
**必須**: メンター・バディのアサインを初日までに完了すること

- 未承認ツール・サービスの利用を推奨してはならない
- 個人情報を含むオンボーディング資料をアクセス制御なしで共有しないこと

品質基準:
- オンボーディング完了期間 最大 90 日
- チェックポイント面談 ≥3 回

- [ ] 新人の背景（学歴・前職・IT経験）をヒアリングした
- [ ] 現在の学習進捗（読了済みドキュメント数）を確認した
- [ ] 使える学習時間とペースを把握した
- [ ] 優先的に学びたい領域を確認した
- [ ] 学習スタイルを判定した
- [ ] 心理的な不安や懸念がないか確認した

## Step 2: 学習ロードマップの設計

状況に応じた最適な学習パスを提示する。

### 2a. 3パターンの学習パス

| パス | 期間 | 対象 | 特徴 |
|:---|:---|:---|:---|
| **基本パス** | 5週間 | 標準的な新人 | 週4-5本ペース、復習込み |
| **急ぎパス** | 1週間 | 即戦力・中途 | 全21本を集中学習 |
| **カスタムパス** | 柔軟 | 特定領域優先 | 業務に直結するカテゴリから |

### 2b. カテゴリ別推奨学習順序

```
Week 1-2: Culture（5本）→ 会社の「なぜ」を理解する
  ↓
Week 2-3: Biz（3本）→ 事業の「何を」を理解する
  ↓
Week 3-4: Tech（7本）→ 技術の「どうやって」を理解する
  ↓
Week 4-5: Ops（6本）→ 業務の「日常」を理解する
```

### 2c. 週次スケジュール例（基本パス）

詳細な週次スケジュールは [learning-paths.md](references/learning-paths.md) を参照。

**チェックリスト**:
- [ ] 3パターンから最適な学習パスを選定した
- [ ] カテゴリの学習順序を提示した
- [ ] 週次スケジュールと所要時間を提示した
- [ ] 新人の同意を得た
- [ ] ペース調整の方法を説明した

## Step 3: ドキュメント学習のガイド

各ドキュメントの学習ポイントと効果的な読み方を案内する。

### 3a. 推薦ドキュメントの提示

次に読むべきドキュメントを提示する際、3つの学習ポイントを添える:

| 提示要素 | 内容 | 例 |
|:---|:---|:---|
| **ドキュメント名** | 正式タイトル | 「ナレコム行動指針」 |
| **学習ポイント 3つ** | 特に注目すべき箇所 | 「コアバリューの具体例に注目」 |
| **前提知識** | 事前に読んでおくべき文書 | 「企業理念を先に読んでおく」 |
| **カテゴリ関連性** | 他カテゴリとの繋がり | 「Biz のサービス理解に直結」 |

### 3b. 4ステップ学習法

```
読む → 質問する → クイズで確認 → 次へ進む
```

1. **読む**: ドキュメントを通読し、気になる箇所をメモ
2. **質問する**: 不明点をオンボーディングスキルに質問
3. **クイズで確認**: Step 4 の理解度チェックで定着を確認
4. **次へ進む**: スコアが合格基準に達したら次のドキュメントへ

### 3c. 視覚的学習手法

テキスト主体の学習を補完し、理解を深化させる:

| 手法 | 活用場面 | 例 |
|:---|:---|:---|
| **HTML プレゼン生成** | 複雑な概念の視覚化 | 「Databricks の Medallion Architecture を図解したプレゼンを作って」 |
| **ASCII 図** | アーキテクチャやプロトコルの構造理解 | 「API Gateway の リクエストフローを ASCII 図で描いて」 |
| **対話的デモ** | ツールの操作手順の習得 | 「Git の branch → merge フローをステップバイステップで見せて」 |

**原則**: 「読んで理解する」だけでなく「見て理解する」「触って理解する」を組み合わせることで、定着率が飛躍的に向上する。

**チェックリスト**:
- [ ] 次に読むべきドキュメントと学習ポイント 3 つを提示した
- [ ] 前提知識を案内した
- [ ] カテゴリ間の関連性を解説した
- [ ] 4ステップ学習法を説明した
- [ ] 必要に応じて視覚的学習手法（HTML プレゼン・ASCII 図）を活用した

## Step 4: 理解度チェック（クイズ生成）

ドキュメントの理解度を3形式のクイズで確認する。

### 4a. 出題範囲の決定

| 範囲 | 説明 | 例 |
|:---|:---|:---|
| **カテゴリ別** | 特定カテゴリから出題 | 「Culture カテゴリ」 |
| **ドキュメント別** | 特定文書から出題 | 「行動指針」から |
| **全体** | 全カテゴリ横断 | 「総合復習テスト」 |

### 4b. 難易度と形式バランス

| 難易度 | 4択（MC） | ○×（TF） | 記述式（SA） | 問題数 |
|:---|:---|:---|:---|:---|
| **基礎** | 50% | 30% | 20% | 5問 |
| **応用** | 30% | 20% | 50% | 5問 |
| **実践** | 20% | 10% | 70% | 5問 |

### 4c. 出題・回答フロー

```
1. 範囲と難易度を確認
2. 1問ずつ出題（形式を明示）
3. 回答を待つ（急かさない）
4. 正誤判定 + 解説（根拠引用）
5. 全問完了後、Step 5 のスコアレポートへ
```

### 4d. クイズ生成パターン

クイズの具体的な生成パターンとテンプレートは [quiz-bank.md](references/quiz-bank.md) を参照。

### 4e. Spaced-Repetition Learning（間隔反復学習）

一度のクイズで終わらず、時間を置いて再確認することで長期記憶に定着させる:

```
Day 1:  初回クイズ → スコアレポート → 弱点特定
Day 3:  弱点カテゴリの復習クイズ（間隔1）
Day 7:  全カテゴリの確認クイズ（間隔2）
Day 14: 応用・実践クイズ（間隔3）
Day 30: 総合テスト（定着確認）
```

| 正答率 | 次回間隔 | アクション |
|:---|:---|:---|
| 90%以上 | 間隔を2倍に延長 | 次のカテゴリに進む |
| 70-89% | 同じ間隔で再実施 | 弱点の重点復習 |
| 70%未満 | 間隔を半分に短縮 | ドキュメントの再読を推奨 |

**チェックリスト**:
- [ ] 出題範囲（カテゴリ/ドキュメント/全体）を確認した
- [ ] 難易度（基礎/応用/実践）を確認した
- [ ] 形式バランスに従って出題した
- [ ] 1問ずつ出題し、回答を待ってからフィードバックした
- [ ] 各問に解説と根拠ドキュメントを引用した
- [ ] 記述式の採点基準を明示した
- [ ] 全問完了後、Step 5 へ遷移した
- [ ] 次回の復習タイミング（間隔反復）を案内した

```
判定: ◎ 合格（80%以上）/ △ 復習推奨（60-79%）/ × 再学習（60%未満）
```

| Tech | XX% | ◎/△/× | — |

| 形式 | 正答率 | 分析 |
|:---|:---|:---|

## Step 6: メンタリング・Q&A対応

新人の質問や相談に、ドキュメントの根拠を添えて回答する。

### 6a. 相談カテゴリの判定

| カテゴリ | 例 | 対応方針 |
|:---|:---|:---|
| **Culture** | 「コアバリューの実践方法は？」 | Culture ドキュメントを根拠に回答 |
| **Biz** | 「うちのサービスの強みは？」 | Biz ドキュメントを根拠に回答 |
| **Tech** | 「Git の使い方がわからない」 | Tech ドキュメントを根拠に回答 |
| **Ops** | 「経費精算の手順は？」 | Ops ドキュメントを根拠に回答 |
| **キャリア** | 「将来どんなスキルを身につけるべき？」 | 成長パスを提示 |
| **メンタル** | 「技術力が不安」「チームに馴染めない」 | 共感 + コアバリュー「絆」で寄り添う |

### 6b. 緊急度判定

| 緊急度 | 条件 | 対応 |
|:---|:---|:---|
| **高** | 業務に直結する即時の問題 | 即回答 + 人間のメンターへのエスカレーション推奨 |
| **中** | 学習の障害になっている疑問 | 丁寧に回答 + 関連ドキュメント案内 |
| **低** | 一般的な疑問・興味 | 回答 + 自習用リソース紹介 |

### 6c. 回答原則

- ドキュメントの根拠を**必ず引用**する（出典明記）
- コアバリューとの紐づけを意識する
- 「わからない」ことは正直に伝え、人間のメンターへ繋ぐ
- メンタリングシナリオの詳細は [mentor-scenarios.md](references/mentor-scenarios.md) を参照

### 6d. Explanatory/Learning 出力スタイルの活用

新人の学習効果を最大化するため、回答時に以下を意識する:

- **Why-First**: 「何をすべきか」の前に「なぜそうなのか」を必ず説明する
- **変更理由の明示**: コードや設定を提示する際は、各行の「なぜ」を添える
- **推論過程の共有**: 複雑な判断では思考過程を段階的に開示し、判断の「型」を学ばせる
- **反例の提示**: 正解だけでなく「もしこうしたらどうなるか」の反例で理解を深める

**原則**: 答えを教えるのではなく、「答えの出し方」を教える。「主力選手」は自ら判断できる人材。

**チェックリスト**:
- [ ] 相談カテゴリ（Culture/Biz/Tech/Ops/キャリア/メンタル）を判定した
- [ ] 緊急度（高/中/低）を判定した
- [ ] ドキュメントの根拠を引用して回答した
- [ ] コアバリューとの紐づけを行った
- [ ] 必要に応じて人間のメンターへのエスカレーションを推奨した

## Step 7: シナリオベースのアドバイス

「こんな時どうする？」型の相談に、OK例とNG例の両面から回答する。

### 7a. レスポンスフォーマット

```
【状況】相談内容の要約
【OK例】推奨される対応 + 根拠
【NG例】避けるべき対応 + 理由
【判断フレームワーク】汎用的な判断基準
【関連コアバリュー】該当するナレコムの価値観
```

### 7b. よくあるシナリオパターン

| シナリオ | カテゴリ | 判断の軸 |
|:---|:---|:---|
| お客様から無理な要望を受けた | Biz | 「期待を超える」≠「何でも引き受ける」 |
| チームメンバーと意見が対立した | Culture | 「絆」= 建設的な対話で解決 |
| 技術的にわからないことがある | Tech | 「主力選手」= 自分で調べた上で質問 |
| 締切に間に合いそうにない | Ops | 早めの報告 + 代替案の提示 |
| 上司の指示に違和感がある | Culture | 根拠をもって丁寧に意見を伝える |
| ミスをしてしまった | Ops | 即報告 + 再発防止策の提示 |

**チェックリスト**:
- [ ] 状況を正確に把握した
- [ ] OK例とNG例の両方を提示した
- [ ] 判断フレームワークを提示した
- [ ] 関連コアバリューと紐づけた
- [ ] 類似シナリオへの応用方法を説明した

## Step 8: 1on1セッション支援

メンターとの 1on1 ミーティングの事前準備を支援する。

### 8a. 事前準備シートテンプレート

| セクション | 内容 | 記入例 |
|:---|:---|:---|
| **今週の学習内容** | 読了ドキュメントと学び | 「Culture 5本完了、コアバリューを理解」 |
| **困っていること** | 学習上の障害や疑問 | 「Tech の開発フローが難しい」 |
| **相談したいこと** | メンターに聞きたいこと | 「キャリアパスについて」 |
| **来週の目標** | 次週の学習計画 | 「Biz 3本を読了する」 |

### 8b. KPT 振り返りフレームワーク

| 要素 | 質問 | 例 |
|:---|:---|:---|
| **Keep** | 続けたいことは？ | 「毎日1本ドキュメントを読む習慣」 |
| **Problem** | 困っていることは？ | 「技術用語の理解に時間がかかる」 |
| **Try** | 次に試したいことは？ | 「用語をノートにまとめながら読む」 |

### 8c. SMART 目標設定

| 要素 | 定義 | 例 |
|:---|:---|:---|
| **Specific** | 具体的 | 「Tech カテゴリの全7本を読了する」 |
| **Measurable** | 測定可能 | 「クイズで 80% 以上を取る」 |
| **Achievable** | 達成可能 | 「2週間で 7 本（1日1本ペース）」 |
| **Relevant** | 関連性 | 「業務で必要な技術知識の習得」 |
| **Time-bound** | 期限付き | 「今月末まで」 |

プロンプトテンプレートの詳細は [prompt-templates.md](references/prompt-templates.md) を参照。

**チェックリスト**:
- [ ] 事前準備シートのテンプレートを提示した
- [ ] KPT 振り返りの記入を支援した
- [ ] SMART 目標の設定を支援した
- [ ] 1on1 で話すべきトピックの優先順位を整理した
- [ ] 前回の 1on1 からの進捗を確認した

## Step 9: フォローアップと進捗レポート

学習のアクションアイテムと進捗状況を整理する。

### 9a. アクションアイテムの提案

各セッション終了時に 1-3 個の具体的なアクションアイテムを提案する:

| アイテム | 期限 | 完了基準 |
|:---|:---|:---|
| 次に読むドキュメント | 次回セッションまで | 読了 + 質問メモ作成 |
| 復習クイズの実施 | 2日以内 | 正答率 80% 以上 |
| メンターへの相談 | 今週中 | 1on1 で報告 |

### 9b. 進捗レポートの生成

```

### 全体進捗
[██████████░░░░░░░░░░] 50% — 21本中 11本完了

### カテゴリ別進捗

### 理解度スコア推移
Week 1: 60% → Week 2: 72% → Week 3: 85%

- 応用クイズのスコア
```

### 9c. オンボーディング支援 Slop 防止チェック（Distributional Convergence 対策）

LLM はオンボーディング支援において distributional convergence（分布的収束）を起こしやすく、どの新人にも同じロードマップ・同じクイズ・同じメンタリング回答に収束する傾向がある。以下のパターンを検出し、一人ひとりに合わせた伴走に仕上げる。

| ID | パターン名 | 検出方法 | 対策 |
|:---|:---|:---|:---|
| ON-1 | 均一ロードマップ | 背景が異なる新人に同一の学習パスを提示 | 学歴・前職・IT 経験・学習スタイルに応じてカテゴリ順序・深度・ペースを個別調整。エンジニア経験者なら Tech を前倒し、ビジネス経験者なら Biz を軽量化 |
| ON-2 | 定型クイズ | 出題パターンが毎回同じ形式・同じ難易度に収束 | 前回のスコアと弱点カテゴリに基づいて形式比率と難易度を動的に調整。間隔反復学習のタイミングも個人の正答率推移に連動 |
| ON-3 | コピペメンタリング | 質問内容が異なるのにドキュメント引用箇所が同一 | 質問の文脈（カテゴリ・緊急度・新人の理解度レベル）に応じて引用箇所を変え、Why-First の説明深度を調整 |
| ON-4 | 汎用励まし | メンタル相談への回答が「誰もが通る道です」の一律パターン | 新人の具体的な状況（入社時期・学習進捗・つまずき箇所）を踏まえた固有のフィードバックを構成し、次の具体的アクションに繋げる |
| ON-5 | テンプレ 1on1 | 1on1 準備シートが毎回同じ KPT + SMART の空テンプレート | 前回の 1on1 内容・学習進捗・クイズスコア推移を反映した「事前記入済み」のドラフトを生成し、差分にフォーカス |
| ON-6 | 固定進捗レポート | 進捗レポートが読了本数のカウントのみ | 理解度スコア推移・弱点の変化・学習速度のトレンド・コアバリュー体現度を含む多角的な成長レポートを生成 |

> **核心原則**: 「オンボーディングは一人ひとりの成長の物語である。」同じ会社に入っても、背景・強み・不安・学び方は十人十色。テンプレートで効率化しつつ、その人だけの成長ストーリーを紡ぐことが AI 伴走の価値。

**チェックリスト**:
- [ ] アクションアイテムを 1-3 個提案した
- [ ] 推薦する追加学習リソースを紹介した
- [ ] 進捗レポート（プログレスバー付き）を生成した
- [ ] 次回の確認ポイントを設定した
- [ ] ON-1〜ON-6 の Slop パターンをチェックし、新人固有の要素に調整した
- [ ] 支援内容が「この新人のためだけの」個別性を持つことを確認した

## Examples

### Example 1: 入社初日の新人

```
「今日入社したばかりです。何から始めればいいですか？」

→ Step 1: 背景ヒアリング（文系卒・ビジネス経験・IT未経験と判明）
→ Step 2: 基本パス（5週間）のロードマップを提示
→ Step 3: Culture カテゴリの最初の1本を推薦 + 学習ポイント3つ
```

### Example 2: Culture 完了後の次ステップ

```
「Culture の 5 本を全部読みました。次は何を学べばいい？」

→ Step 3: Biz カテゴリを推薦、Culture との関連性を解説
→ 「事業理解が深まると、コアバリューの『お客様の期待を超える』が具体的になりますよ」
```

### Example 3: カテゴリ別クイズ

```
「Biz カテゴリの基礎クイズを出してください」

→ Step 4: Biz カテゴリ・基礎難易度で出題（5問）
  - 4択 3問 + ○× 1問 + 記述式 1問
  - 1問ずつ出題→回答待ち→フィードバック
→ Step 5: スコアレポート（80% 合格 → 次のカテゴリへ推奨）
```

### Example 4: 用語・制度の質問

```
「NDA についてよくわからないのですが、どこに書いてありますか？」

→ Step 6: カテゴリ判定 = Ops、緊急度 = 中
→ Ops ドキュメント「NDA・秘密保持」から根拠を引用して回答
→ 「わからないことを質問するのは『主力選手』への第一歩です」
```

### Example 5: シナリオ相談

```
「お客様から無理な要望を受けた時、どうすればいいですか？」

→ Step 7: シナリオ判定 = Biz
→ OK例:「まず要望を正確に理解し、対応可能な範囲を整理して代替案を提示」
→ NG例:「安請け合いする / 即座に断る」
→ 判断の軸:「期待を超える = 何でも引き受けることではなく、最適な解を共に見つけること」
```

### Example 6: 1on1 準備

```
「来週のメンターとの 1on1 の準備を手伝ってください」

→ Step 8: 事前準備シートのテンプレートを提示
→ KPT で今週の振り返りを整理
→ SMART で来週の目標を設定
→ 話したいトピックの優先順位を一緒に整理
```

### Example 7: メンタル面の相談

```
「周りのエンジニアと比べて技術力が足りない気がして不安です」

→ Step 6: カテゴリ判定 = メンタル、緊急度 = 中
→ 共感:「入社直後は誰もが感じる不安です」
→ コアバリュー「絆」:「チームは互いに支え合う文化です。先輩に質問することは歓迎されます」
→ 具体的アクション:「まず Tech の基礎クイズで現在地を確認しませんか？」
→ 必要に応じて人間のメンターへのエスカレーション推奨
```

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| どこから読めばいいかわからない | 全体像が見えていない | Step 2 のロードマップで全体像を提示し、Culture カテゴリから始める |
| 技術文書が難しすぎる | 前提知識が不足 | 難易度を下げた基礎クイズから始め、前提ドキュメントを案内する |
| クイズの点数が低くて落ち込む | 完璧主義・自己否定 | 「学習の途中経過。弱点が見えたことがむしろ成果」とフレーミング |
| 全体像が見えない | カテゴリ間の関連が不明 | Step 2 のカテゴリ順序図で全体像を提示し、「なぜこの順番か」を説明 |
| 時間が足りない | 業務と学習の両立 | 急ぎパス（1週間）に切替えるか、優先カテゴリに絞る |
| 記述式の採点基準がわからない | 基準が不明確 | キーワード + 論理構成 + 根拠引用の 3 軸で採点基準を明示する |
| ドキュメントに書いてないことを聞きたい | スコープ外の質問 | 正直に「ドキュメント外」と伝え、人間のメンターへエスカレーション推奨 |
| メンタル的に辛い | 孤立感・不安 | 共感 + コアバリュー「絆」で寄り添い、人間のメンターへの相談を強く推奨 |
| スキルが発動しない | トリガーフレーズ不一致 | 「オンボーディング」「学習」「クイズ」等のキーワードを含めて再度質問 |
| 学習が進まない/モチベーション低下 | 目標が遠すぎる | 週次の小さな達成目標を設定し、進捗レポートで成長を可視化する |
| 同じ問題が出る | クイズバンクの偏り | カテゴリ・難易度を変更するか、記述式中心に切替えて応用力を鍛える |
| 回答が長すぎる/抽象的 | 質問の焦点が広い | 質問を具体化するよう促し、「一番知りたいことは何ですか？」と絞り込む |

## アンチパターン検出

このスキルの出力品質を検証し、AI Decay（品質劣化）を防止するためのアンチパターン検出リスト。各項目は品質劣化パターンの検出条件を定義し、不合格時は対応する対策（修正ガイドライン）に従って改善する。

- [ ] 学習パスがメンバーのスキルレベル・役割に応じてカスタマイズされているか（全員同じパスになっていないか）
- [ ] クイズの出題がカテゴリ別に適切な難易度分布になっているか（簡単すぎ/難しすぎの偏りがないか）
- [ ] メンタリング回答が具体的なコード例・ドキュメント参照を含んでいるか（抽象的な助言のみでないか）
- [ ] 進捗トラッキングのメトリクスが定量的に定義されているか（「理解度が高い」ではなく具体的スコア）
- [ ] 週次スケジュールが現実的な学習時間（1日あたり）を考慮しているか
- [ ] 自習用プロンプトテンプレートがドメイン固有の文脈を含んでいるか（汎用テンプレの流用でないか）
