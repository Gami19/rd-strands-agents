# テスト手法リファレンス

> Anthropic 公式ガイドに基づく 3 種テスト（トリガー/ファンクショナル/パフォーマンス）の
> 詳細手法、テンプレート、メトリクス定義。

---

## 1. トリガーテスト（Triggering Tests）

### 目的

スキルが正しいタイミングで発動し、無関係なタスクでは発動しないことを検証する。
description フィールドの精度を直接測定するテスト。

### テストケーステンプレート

#### Should-trigger テスト（6 件以上）

| # | テスト入力 | 種別 | 期待結果 | 実際の結果 | Pass/Fail |
|:---|:---|:---|:---|:---|:---|
| T-1 | description 内のトリガーフレーズそのまま | 直接一致 | 発動する | | |
| T-2 | トリガーの言い換え（口語） | 口語 | 発動する | | |
| T-3 | トリガーの言い換え（略語） | 略語 | 発動する | | |
| T-4 | トリガーの言い換え（状況説明型） | 状況説明 | 発動する | | |
| T-5 | 関連するドメイン用語 | ドメイン | 発動する | | |
| T-6 | 英語での同等リクエスト | 英語 | 発動する | | |

#### Should-NOT-trigger テスト（3 件以上）

| # | テスト入力 | 除外理由 | 期待結果 | 実際の結果 | Pass/Fail |
|:---|:---|:---|:---|:---|:---|
| NT-1 | Do NOT use for に記載のケース | 明示的除外 | 発動しない | | |
| NT-2 | 隣接スキルのトリガー | 責務境界 | 発動しない | | |
| NT-3 | 完全に無関係なタスク | 無関係 | 発動しない | | |

### 合格基準

| メトリクス | 基準値 | 判定方法 |
|:---|:---|:---|
| Should-trigger 発動率 | **90% 以上** | 6 件中 5 件以上で発動 |
| Paraphrase 発動率 | **70% 以上** | 言い換え表現でも発動 |
| Should-NOT-trigger 誤発動率 | **0%** | 誤発動 0 件 |

### Undertriggering の修正戦略

スキルが発動すべき場面で発動しない場合:

| 兆候 | 原因 | 対策 |
|:---|:---|:---|
| ユーザーが手動でスキル名を指定する | トリガーフレーズが不足 | description に類義語・口語・略語を追加 |
| 技術用語で発動しない | 英語/日本語の混在が未対応 | 両言語のバリエーションを網羅 |
| 曖昧なリクエストで発動しない | description が具体的すぎる | 状況説明型トリガー（「〇〇で困っている」等）を追加 |
| ドメイン固有の表現で発動しない | 専門用語が description に未記載 | ドメインの業界用語・俗称を追加 |

### Overtriggering の修正戦略

スキルが発動すべきでない場面で発動する場合:

| 兆候 | 原因 | 対策 |
|:---|:---|:---|
| 無関係なタスクで発動 | description が広すぎる | Do NOT use for を強化、除外条件を具体化 |
| 隣接スキルと競合 | 責務境界が曖昧 | 両スキルの description で相互に Do NOT use for を明記 |
| 一般的なキーワードで発動 | トリガーが汎用的すぎる | ドメイン固有の修飾語を組み合わせる |

## 2. ファンクショナルテスト（Functional Tests）

### 目的

スキルが正しい出力を生成し、ワークフローが正常に完了することを BDD 形式で検証する。

### BDD テストテンプレート

```
テスト: [テスト名]

Given（前提条件）:
  - [条件1]
  - [条件2]

When（実行アクション）:
  - [アクション]

Then（期待結果）:
  - [結果1]
  - [結果2]
  - [結果3]
```

### 推奨テストケース

| # | テスト名 | 検証対象 | 優先度 |
|:---|:---|:---|:---|
| F-1 | **基本フロー完走** | 全ステップを正常に通過する | 必須 |
| F-2 | **エントリーポイント分岐** | 中間ステップからの開始が正しく動作する | 必須 |
| F-3 | **エラーハンドリング** | 不正入力時にエラーメッセージが出力される | 必須 |
| F-4 | **エッジケース** | 最小入力/最大入力での挙動が適切 | 推奨 |
| F-5 | **出力フォーマット** | 出力が指定のフォーマットに準拠している | 必須 |
| F-6 | **references/ 整合** | SKILL.md のリンクと references/ ファイルが一致する | 必須 |

### テスト結果記録テンプレート

```markdown
## ファンクショナルテスト結果: [スキル名] v[X.Y.Z]

| # | テスト名 | 結果 | 備考 |
|:---|:---|:---|:---|
| F-1 | 基本フロー完走 | Pass / Fail | |
| F-2 | エントリーポイント分岐 | Pass / Fail | |
| F-3 | エラーハンドリング | Pass / Fail | |
| F-4 | エッジケース | Pass / Fail | |
| F-5 | 出力フォーマット | Pass / Fail | |
| F-6 | references/ 整合 | Pass / Fail | |

合格基準: F-1, F-2, F-3, F-5, F-6 が全て Pass（必須 5 件）
```

### テストケース設計のコツ

| ポイント | 説明 |
|:---|:---|
| **境界値を含める** | ステップ数の上限/下限、description の文字数上限（1024 文字）等 |
| **ネガティブケース** | 存在しないファイルパス、空の入力、不正なフォーマット |
| **依存スキルのモック** | 委譲先スキル（pdf-convert, distill, review）が未導入の場合の挙動 |
| **冪等性の確認** | 同じ入力で 2 回実行しても同等の出力が得られるか |

## 3. パフォーマンス比較テスト（Performance Comparison）

### 目的

スキルの有無で成果物品質・効率を比較し、スキルの付加価値を定量化する。

### メトリクス定義

| メトリクス | 計測方法 | 改善目標 | 備考 |
|:---|:---|:---|:---|
| **メッセージ往復数** | 同一タスクをスキルなし/ありで実行し比較 | 30% 以上削減 | ユーザーの手間を直接反映 |
| **ユーザー補正回数** | ユーザーが出力を修正・やり直しした回数 | 50% 以上削減 | 品質の間接指標 |
| **品質チェック通過率** | quality-checklist.md の通過項目数 / 全項目数 | 90% 以上 | 客観的品質指標 |
| **API/ツール呼び出し回数** | ツール使用の総数（MCP 統合スキルの場合） | 20% 以上削減 | 効率指標 |
| **トークン消費量** | 同一タスクでの総トークン数 | 参考記録 | 削減目標は設定しない |

### 比較記録テンプレート

```markdown
## パフォーマンス比較: [スキル名] v[X.Y.Z]

### テストタスク
[具体的なタスク記述]

### ベースライン（スキルなし）
- メッセージ往復数: N 回
- ユーザー補正回数: N 回
- 品質チェック通過率: N/40 (N%)
- 備考: [自由記述]

### スキルあり
- メッセージ往復数: N 回
- ユーザー補正回数: N 回
- 品質チェック通過率: N/40 (N%)
- 備考: [自由記述]

### 改善率
| メトリクス | ベースライン | スキルあり | 改善率 | 目標達成 |
|:---|:---|:---|:---|:---|
| メッセージ往復数 | N | N | N% 削減 | Yes/No |
| ユーザー補正回数 | N | N | N% 削減 | Yes/No |
| 品質通過率 | N% | N% | +N% | Yes/No |
```

### vibes-based assessment の位置づけ

> 「Aim for rigor but accept that there will be an element of vibes-based assessment.」
> — Anthropic, The Complete Guide to Building Skills for Claude

定量化が困難な品質向上は、以下の定性記録テンプレートで捕捉する:

| 定性指標 | スキルなし | スキルあり | 評価 |
|:---|:---|:---|:---|
| ドメイン用語の正確性 | [記述] | [記述] | 向上 / 同等 / 低下 |
| 出力の構造的一貫性 | [記述] | [記述] | 向上 / 同等 / 低下 |
| ユーザー体験の滑らかさ | [記述] | [記述] | 向上 / 同等 / 低下 |
| 次ステップの自然な提示 | [記述] | [記述] | 向上 / 同等 / 低下 |

> **注意**: 定性評価は主観的だが、複数回の実行で一貫した傾向が見られれば信頼性がある。
> 定量メトリクスと定性評価の両方を記録し、総合的に判断する。

## テスト実施フロー

```
1. トリガーテスト（Step 9a）
   └─ description の発動精度を確認
   └─ 不合格 → description を修正して再テスト

2. ファンクショナルテスト（Step 9b）
   └─ 基本フロー・エラーハンドリングを確認
   └─ 不合格 → ステップ/チェックリストを修正

3. パフォーマンス比較（Step 9c）
   └─ ベースラインとの比較で付加価値を確認
   └─ 改善率が目標未達 → ステップの詳細化/自動化を検討
```

### テスト結果の解釈

| 結果パターン | 解釈 | アクション |
|:---|:---|:---|
| 3 種全て合格 | スキルは実用品質 | リリース可能 |
| トリガー不合格 | description の精度が低い | トリガーフレーズの追加・修正 |
| ファンクショナル不合格 | ワークフローに欠陥がある | ステップ/チェックリストの見直し |
| パフォーマンス未達 | スキルの付加価値が不十分 | ステップの詳細化、references/ の充実 |
| トリガー + ファンクショナル不合格 | スキルの設計に根本的問題 | Step 4 の設計案からやり直し |
