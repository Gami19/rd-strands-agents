# OMM Evaluation Template

> observability スキル補足資料 — Observability Maturity Model（OMM）の評価テンプレート

## 1. OMM 5 能力評価シート

各能力を **未着手 / 初期 / 発展 / 成熟** の 4 段階で評価する。

### 能力 1: 障害への回復力（Resilience to Failure）

| 評価段階 | 基準 |
|:---|:---|
| **未着手** | 障害は顧客報告で初めて認識。MTTR > 24 時間。オンコール体制なし |
| **初期** | 基本的なアラートあり。MTTR 4-24 時間。属人的な対応 |
| **発展** | SLO ベースアラート導入。MTTR 1-4 時間。プレイブックあり |
| **成熟** | 予測バーンアラート。MTTR < 1 時間。自動復旧の仕組みあり |

**評価チェックリスト**:
- [ ] アラート疲れは発生していないか（週 10 件以下が目安）
- [ ] オンコールの持続可能性は確保されているか（25% ルール）
- [ ] 過去 3 ヶ月の MTTR トレンドは改善傾向か
- [ ] 障害検知の手段（自動 vs 手動 vs 顧客報告）の比率

**現状評価**: _______ / スコア理由: _______________________________

---

### 能力 2: 高品質コードの提供（Delivering High-Quality Code）

| 評価段階 | 基準 |
|:---|:---|
| **未着手** | 本番デプロイが恐怖。バグは顧客報告で発見。デバッグに数日 |
| **初期** | CI/CD あり。ステージングでテスト。デバッグに数時間 |
| **発展** | フィーチャーフラグ活用。ODD 実践開始。デバッグに分単位 |
| **成熟** | 本番デバッグが日常。開発者が自発的にテレメトリを追加 |

**評価チェックリスト**:
- [ ] 開発者は本番でのデバッグにオブザーバビリティツールを使えるか
- [ ] 「奇妙な振る舞い」の調査に要する時間は（分/時間/日）
- [ ] フィーチャーフラグでリリースとデプロイを分離しているか
- [ ] コードレビューに計装のレビューが含まれるか

**現状評価**: _______ / スコア理由: _______________________________

---

### 能力 3: 技術的負債の管理（Managing Technical Debt）

| 評価段階 | 基準 |
|:---|:---|
| **未着手** | 負債の可視化なし。「火消し」が日常。コアビジネス作業 < 30% |
| **初期** | 負債の一覧あり。四半期で優先度付け。コアビジネス作業 30-50% |
| **発展** | テレメトリで負債の影響を定量化。計画的に償却。コアビジネス 50-70% |
| **成熟** | 負債が SLO バーンレートに直結。自動的に優先順位決定。コアビジネス > 70% |

**評価チェックリスト**:
- [ ] チームの時間のうちコアビジネス作業の割合は
- [ ] 反復的な手作業（toil）の量は定量化されているか
- [ ] 技術的負債の返済がスプリント計画に組み込まれているか
- [ ] 負債がユーザー体験に与える影響を測定できるか

**現状評価**: _______ / スコア理由: _______________________________

---

### 能力 4: リリースケイデンス（Release Cadence）

| 評価段階 | 基準 |
|:---|:---|
| **未着手** | 月次以下のリリース。コミット→本番に数週間。手動デプロイ |
| **初期** | 週次リリース。コミット→本番に数日。半自動デプロイ |
| **発展** | 日次リリース。コミット→本番に数時間。自動デプロイ |
| **成熟** | 日に複数回リリース。コミット→本番に分単位。プログレッシブデリバリー |

**評価チェックリスト**:
- [ ] コミットから本番までのリードタイムは
- [ ] デプロイ頻度は（日/週/月/四半期）
- [ ] デプロイの変更失敗率は
- [ ] 失敗デプロイからのリカバリー時間は

**現状評価**: _______ / スコア理由: _______________________________

---

### 能力 5: ユーザー行動の理解（Understanding User Behavior）

| 評価段階 | 基準 |
|:---|:---|
| **未着手** | ユーザー行動データなし。PM が開発に機能要望を口頭伝達 |
| **初期** | 基本的な分析ツール（GA 等）あり。PM が限定的にデータアクセス |
| **発展** | テレメトリにユーザー行動データ統合。PM が自分でクエリ実行 |
| **成熟** | リアルタイムでユーザーコホート分析。A/B テストが日常 |

**評価チェックリスト**:
- [ ] PM/ビジネス側がオブザーバビリティツールにアクセスできるか
- [ ] ユーザー行動の変化を何分/時間/日で検知できるか
- [ ] フィーチャーの利用状況をテレメトリで測定できるか
- [ ] ユーザーフィードバックとテレメトリデータを紐付けられるか

**現状評価**: _______ / スコア理由: _______________________________

### 優先順位の決定基準

1. **ビジネスインパクト**: 改善による収益/コスト効果が最大の能力
2. **現状の痛み**: チームの日常業務に最も影響している能力
3. **依存関係**: 他の能力改善の前提となる能力（通常は「障害への回復力」が基盤）

## 3. 改善ロードマップテンプレート

```markdown
## フェーズ 1: 基盤構築（月 1-3）
- [ ] 最優先能力の改善施策を開始
- [ ] OTel 自動計装を最重要サービスに導入
- [ ] SLO を 1-2 個定義し、ダッシュボード構築

## フェーズ 2: 拡張（月 4-6）
- [ ] 計装を全サービスに展開
- [ ] SLO ベースアラートに移行開始
- [ ] 開発チームへの ODD トレーニング実施

## フェーズ 3: 最適化（月 7-12）
- [ ] サンプリング戦略の導入でコスト最適化
- [ ] 全能力の再評価と次期目標設定
- [ ] オブザーバビリティの文化的定着度の評価
```
