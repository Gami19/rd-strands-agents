ビジネスコンテキストを把握し、SEO戦略の土台を築く。

### 1a. コンテキストファイルの確認

`.claude/product-marketing-context.md` が存在する場合、まず読み込む。既にカバーされている情報は再質問しない。

### 1b. 不足情報のヒアリング

| 確認項目 | 質問 |

### 1c. 目標設定

| ビジネス目標 | SEO指標 | 計測方法 |

## Step 2: 技術SEO監査 — サイト基盤の健全性を検証する

サイトの技術的基盤を体系的に監査し、検索エンジンが正しくクロール・インデックスできる状態を確保する。

### 2a. クローラビリティ

| 監査項目 | 確認内容 | 合格基準 |
|:---|:---|:---|
| robots.txt | 意図しないブロックがないか | 重要ページが `Allow` |
| XMLサイトマップ | 存在・形式・更新頻度 | 全正規URLを含み最新 |
| サイト構造 | 重要ページへのクリック数 | ホームから3クリック以内 |
| クロールバジェット | パラメータURL・無限スクロール | 不要URLの制御が適切 |

### 2b. インデックス状態

| 監査項目 | 確認内容 | 合格基準 |
|:---|:---|:---|
| インデックス数 | `site:domain.com` の結果数 | 期待値と一致 |
| canonical タグ | 全ページに正しい canonical | 自己参照・重複なし |
| noindex | 意図しない noindex がないか | 重要ページは全て index |
| リダイレクト | チェーン・ループがないか | 1ホップ以内 |

### 2c. Core Web Vitals

| 指標 | 合格基準 | 改善手法 |
|:---|:---|:---|
| LCP（Largest Contentful Paint） | < 2.5秒 | 画像最適化・CDN・プリロード |
| INP（Interaction to Next Paint） | < 200ms | JS最適化・メインスレッド負荷削減 |
| CLS（Cumulative Layout Shift） | < 0.1 | 画像サイズ指定・フォント最適化 |

### 2d. モバイル対応・セキュリティ

- レスポンシブデザインであること（別ドメインのm.サイトは非推奨）
- タップターゲット: 48x48px 以上
- HTTPS全ページ適用・有効なSSL証明書
- 混在コンテンツなし

詳細な技術SEO監査項目は [references/technical-seo-checklist.md](references/technical-seo-checklist.md) を参照。

- [ ] インデックス状態と canonical 設定を検証した

- [ ] モバイル対応・HTTPS を確認した

## Step 3: AI検索最適化 — 次世代検索エンジンへの対応

Google AI Overviews、ChatGPT、Perplexity 等のAI検索エンジンに引用される戦略を設計する。

### 3a. AI検索プラットフォーム別の最適化方針

| プラットフォーム | 検索基盤 | 重視シグナル |
|:---|:---|:---|
| Google AI Overviews | Google 自社インデックス | E-E-A-T・構造化データ・トピック権威性 |
| ChatGPT | Bing ベース | ドメイン権威性・鮮度・コンテンツ構造適合度 |
| Perplexity | 独自 + Google | FAQスキーマ・自己完結パラグラフ・PDF公開 |
| Copilot | Bing | ページ速度・LinkedIn/GitHub連携・IndexNow |
| Claude | Brave Search | 事実密度・引用元明記・数値データ |

### 3b. コンテンツの引用されやすさ向上

AI検索で引用されるコンテンツの特徴:

| 手法 | 効果 | 実装方法 |
|:---|:---|:---|
| 統計データの引用 | +37-40% 可視性向上 | 出典付きの具体的数値を含める |
| 専門家の引用 | +25-30% | 氏名・肩書き・所属を明記 |
| 権威ある文体 | +25% | 実証的・具体的な記述 |
| 自己完結パラグラフ | 高い抽出率 | 40-60語で単独で意味が通る文を書く |

### 3c. AIボットのアクセス制御

robots.txt で以下のボットを許可する:

```
User-agent: GPTBot           # OpenAI（ChatGPT）
User-agent: ChatGPT-User     # ChatGPT ブラウジング
User-agent: PerplexityBot    # Perplexity
User-agent: ClaudeBot        # Anthropic Claude
User-agent: Google-Extended   # Google AI Overviews
User-agent: Bingbot          # Microsoft Copilot
Allow: /
```

詳細は [references/ai-search-optimization.md](references/ai-search-optimization.md) を参照。

## Step 4: コンテンツ戦略設計 — 検索意図に応えるコンテンツ計画

キーワードリサーチに基づき、トピッククラスターとコンテンツカレンダーを設計する。

### 4a. キーワードリサーチ

購買ファネル別にキーワードを分類する:

| ステージ | キーワード修飾語 | コンテンツ形式 |
|:---|:---|:---|
| 認知 | 「とは」「方法」「ガイド」 | 定義記事・ハウツー |
| 検討 | 「おすすめ」「比較」「vs」 | 比較記事・リスト |
| 決定 | 「料金」「レビュー」「導入事例」 | 価格比較・ケーススタディ |
| 実装 | 「テンプレート」「設定方法」「使い方」 | チュートリアル・テンプレート |

### 4b. トピッククラスター設計

```
ピラー（Hub）: /[メイントピック]/
├── スポーク 1: /blog/[サブトピック-1]/
├── スポーク 2: /blog/[サブトピック-2]/
├── スポーク 3: /blog/[サブトピック-3]/
└── 関連ピラーへの内部リンク
```

### 4c. コンテンツカレンダー

| 週 | コンテンツ | ターゲットKW | 形式 | ファネル位置 |
|:---|:---|:---|:---|:---|
| 1 | [タイトル] | [KW] | ブログ | 認知 |
| 2 | [タイトル] | [KW] | 比較記事 | 検討 |
| 3 | [タイトル] | [KW] | ケーススタディ | 決定 |
| 4 | [タイトル] | [KW] | ハウツー | 実装 |

### 4d. コンテンツ品質基準

| 基準 | 要件 |
|:---|:---|
| E-E-A-T | 実体験・専門性・権威性・信頼性を示す |
| 検索意図適合 | クエリの意図に正確に応える |
| AI引用適合 | 統計データ・専門家引用・自己完結パラグラフを含む |
| 鮮度 | 「最終更新日」を明記、競合トピックは四半期更新 |

## Step 5: 構造化データ実装 — JSON-LD スキーマの設計

検索エンジンとAIシステムがコンテンツを正確に理解できるよう構造化データを設計する。

### 5a. ページ種別ごとのスキーマ選定

| ページ種別 | 推奨スキーマ | リッチリザルト |
|:---|:---|:---|
| トップページ | Organization + WebSite | サイトリンク検索ボックス |
| ブログ記事 | Article / BlogPosting | 著者・日付表示 |
| 商品ページ | Product + AggregateRating | 価格・レビュー星評価 |
| FAQページ | FAQPage | FAQ折りたたみ表示 |
| ハウツー記事 | HowTo | ステップ表示 |
| イベント | Event | 日時・場所表示 |
| ローカルビジネス | LocalBusiness | 地図・営業時間 |
| パンくず | BreadcrumbList | パンくずリスト表示 |

### 5b. 実装方式

- **形式**: JSON-LD（Google 推奨）
- **配置**: `<head>` 内または `<body>` 末尾
- **複数スキーマ**: `@graph` で結合

### 5c. バリデーション

| ツール | 用途 |
|:---|:---|
| Google Rich Results Test | リッチリザルト対応確認 |
| Schema.org Validator | スキーマ構文検証 |
| Search Console 拡張レポート | エラー・警告の監視 |

**注意**: `web_fetch` や `curl` では JSON-LD を検出できない。CMS プラグイン（Yoast、RankMath等）がクライアントサイド JS で注入するため。検証には必ず Rich Results Test またはブラウザの DevTools を使用すること。

具体的な JSON-LD 実装例は [references/schema-examples.md](references/schema-examples.md) を参照。

**チェックリスト**:
- [ ] 各ページ種別に適切なスキーマを選定した
- [ ] JSON-LD コードを生成した
- [ ] Rich Results Test で検証しエラーがないことを確認した
- [ ] 既存のスキーマとの整合性を確認した

## Step 6: 計測基盤設計 — GA4 イベント設計・GTM 実装

データに基づく意思決定を可能にする計測基盤を設計する。

### 6a. イベント設計の原則

| 原則 | 説明 |
|:---|:---|
| 意思決定のために計測する | 虚栄メトリクスは追わない |
| 命名規則を統一する | `object_action` 形式（例: `signup_completed`） |
| PIIを含めない | 個人情報はイベントパラメータに入れない |
| コンセント対応 | GDPR/個人情報保護法に準拠 |

### 6b. 推奨イベント設計

マーケティングサイト向けの基本イベント:

| イベント名 | 説明 | パラメータ |
|:---|:---|:---|
| `cta_clicked` | CTA ボタンクリック | button_text, location |
| `form_submitted` | フォーム送信完了 | form_name, form_location |
| `signup_completed` | ユーザー登録完了 | method, plan, source |
| `demo_requested` | デモ申し込み | company_size, industry |
| `pricing_viewed` | 料金ページ閲覧 | source |
| `purchase_completed` | 購入完了 | plan, value, currency |

### 6c. データレイヤーとGTM 設計

```javascript
// dataLayer 初期化（GTMコンテナより前に配置）
window.dataLayer = window.dataLayer || [];

// イベント発火
dataLayer.push({
  'event': 'form_submitted',
  'formName': 'contact',
  'formLocation': 'footer'
});
```

GTM コンテナ構造:
- **タグ**: GA4 設定タグ + イベントタグ + 広告ピクセル
- **トリガー**: ページビュー / クリック / カスタムイベント
- **変数**: データレイヤー変数 / カスタム JS 変数

### 6d. UTM パラメータ戦略

| パラメータ | 用途 | 例 |
|:---|:---|:---|
| `utm_source` | 流入元 | google, newsletter |
| `utm_medium` | 媒体 | cpc, email, social |
| `utm_campaign` | キャンペーン名 | spring_sale_2026 |
| `utm_content` | クリエイティブ区別 | hero_cta, sidebar_banner |

詳細は [references/ga4-event-library.md](references/ga4-event-library.md) を参照。

## Step 7: A/Bテスト設計 — 仮説構築から統計的検定まで

データに基づく改善サイクルを回すための実験設計を行う。

### 7a. 仮説の構造化

```
[観察/データ] に基づき、
[変更内容] を実施すれば、
[期待する結果] が [対象ユーザー] に対して生じると考える。
[計測指標] で効果を判定する。
```

**良い仮説の例**:
「ヒートマップでCTAが見つけにくいという行動が観察されるため、CTAボタンを大きくしコントラストを高めれば、新規訪問者のCTAクリック率が15%以上向上すると考える。ページビューからサインアップ開始への遷移率で効果を判定する。」

### 7b. 指標の設計

| 指標種別 | 役割 | 例 |
|:---|:---|:---|
| 主要指標 | テスト判定に使用する唯一の指標 | CTA クリック率 |
| 副次指標 | 主要指標の解釈を補助する | 滞在時間・ページ遷移数 |
| ガードレール指標 | 悪化を検知する安全弁 | サポート問い合わせ数・離脱率 |

### 7c. サンプルサイズ計算

| 基準CVR | 10%リフト検出 | 20%リフト検出 | 50%リフト検出 |
|:---|:---|:---|:---|
| 1% | 380,000/群 | 97,000/群 | 16,000/群 |
| 3% | 120,000/群 | 31,000/群 | 5,200/群 |
| 5% | 72,000/群 | 18,000/群 | 3,100/群 |
| 10% | 34,000/群 | 8,700/群 | 1,500/群 |

**テスト期間の計算**:
```
期間（日）= サンプル数 x バリアント数 / 日次トラフィック
```
最低1週間（曜日変動の吸収）、最大8週間（新規性効果の減衰防止）。

### 7d. テスト実行のルール

- 途中でバリアントを変更しない
- 途中結果を見て早期終了しない（覗き見問題）
- 95%信頼水準・80%検出力をデフォルトとする
- 結果は必ず文書化する

詳細は [references/ab-test-calculator.md](references/ab-test-calculator.md) を参照。

**チェックリスト**:
- [ ] 構造化された仮説を記述した
- [ ] 主要指標・副次指標・ガードレール指標を定義した
- [ ] サンプルサイズを計算しテスト期間を算出した
- [ ] テスト実行ルールをチームに共有した

## Step 8: パフォーマンスレポート — ダッシュボード設計と改善サイクル

継続的な改善を推進するレポーティング体制を設計する。

### 8a. KPIダッシュボード設計

| カテゴリ | KPI | データソース | 更新頻度 |
|:---|:---|:---|:---|
| オーガニック流入 | セッション数・ユーザー数 | GA4 | 週次 |
| 検索順位 | ターゲットKW の平均順位 | Search Console | 週次 |
| CTR | 検索結果でのクリック率 | Search Console | 週次 |
| コンバージョン | CV数・CVR | GA4 | 週次 |
| Core Web Vitals | LCP・INP・CLS | CrUX / PageSpeed | 月次 |
| AI検索可視性 | AI回答での引用頻度 | 手動 / Otterly AI | 月次 |
| インデックス状態 | インデックス済みページ数 | Search Console | 月次 |

### 8b. レポートテンプレート

```markdown
# SEO 月次レポート: [YYYY年MM月]

## エグゼクティブサマリー
- オーガニック流入: XX,XXX（前月比 +X%）
```

### 8d. SEO Analytics Slop 防止チェック（Distributional Convergence 対策）

LLM が SEO 施策を生成する際にも distributional convergence が起きる。全サイトに対して「同じ監査項目・同じコンテンツ戦略・同じ KPI 構成」が出力されることを防ぐ。

| # | SEO Analytics Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| SE-1 | **技術SEO監査の画一化** | 全サイトで同じ推奨事項（robots.txt確認→サイトマップ→Core Web Vitals）が同じ順序・同じ重要度で出力 | サイト種別（SaaS/EC/メディア/コーポレート）に応じた監査の重点を変える。ECなら商品ページのインデックス効率、メディアなら鮮度管理が優先 |
| SE-2 | **コンテンツ戦略の収束** | 全案件で同じトピッククラスター構造、同じカレンダーフォーマット | 業界の検索動向とコンテンツギャップから逆算。競合が強い領域を避け、ニッチの権威を先に確立する戦略も検討 |
| SE-3 | **構造化データのコピペ** | JSON-LD スキーマがどのサイトでも同じテンプレート | ページの実際のコンテンツと目的から必要なスキーマを導出。「このページで何のリッチリザルトを狙うか」を明確にしてから実装 |
| SE-4 | **AI検索最適化の汎用化** | 全サイトに同じ robots.txt 設定と「統計データを含めよ」アドバイス | プロダクトのAI検索での現状引用状況を調査してから施策を設計。プラットフォーム別の重視シグナルに基づいて優先度を変える |
| SE-5 | **KPI ダッシュボードの画一化** | 毎回同じ7指標（セッション/順位/CTR/CV/CWV/AI可視性/インデックス）の同じ構成 | ビジネスの成長段階に応じた指標を選定。立ち上げ期はインデックス効率、成長期はCV効率、成熟期はリテンションとブランドクエリ |
| SE-6 | **レポートテンプレの固定** | 月次レポートが毎回同じ構造で、分析の深さが変わらない | レポートは「意思決定を促す」もの。毎月のフォーカスポイント（今月は技術面、来月はコンテンツ等）を変えて掘り下げる |
| SE-7 | **A/Bテスト設計の公式化** | 仮説テンプレートが同じ構文、サンプルサイズ計算が同じ表から参照 | 仮説はSEOデータ（Search Console、ヒートマップ）の具体的な観察から導出。「なぜこの変更が効くと考えるか」の根拠を個別に記述 |
| SE-8 | **技術的推奨事項の収束** | Core Web Vitals の改善案が毎回「画像最適化→JS削減→フォント最適化」 | サイトの技術スタック（Next.js/WordPress/Shopify等）に応じた具体的な改善手段を提示。PageSpeed Insights の個別項目から優先度を導出 |

> **Why**: SEO 施策が均質化すると、全クライアントに同じ推奨事項が出力される。サイトの技術スタック・ビジネスモデル・成長段階に応じた差別化された戦略だけが、実際に検索順位を動かす。
>
> **核心原則**: **同じフレームワークでも、入力データが変われば出力は変わるべき** — SEO はデータドリブンの領域。汎用テンプレートではなく、個別のデータ（Search Console、GA4、競合分析）から導出された施策だけが有効。

**チェックリスト**:
- [ ] KPIダッシュボードの構造を設計した
- [ ] レポートテンプレートを作成した
- [ ] レポートの更新頻度と担当を決定した
- [ ] 改善サイクルのプロセスを定義した
- [ ] SEO Analytics Slop チェック（SE-1〜SE-8）に 2 つ以上該当しない
- [ ] 推奨事項がサイト種別・技術スタック・成長段階に基づいて個別化されている

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| `web_fetch` でスキーマが検出できない | CMSプラグインがクライアントサイドJSで注入するため `<script>` タグが取得できない | Rich Results Test またはブラウザ DevTools で確認する |
| インデックス数が期待より少ない | canonical の不整合・noindex の誤設定・クロールバジェット不足 | Search Console のインデックスカバレッジレポートで原因特定 |
| Core Web Vitals が不合格 | 未最適化画像・過剰なJS・レイアウトシフト | PageSpeed Insights で個別項目を確認し優先順位をつけて改善 |
| AI検索で引用されない | コンテンツが汎用的・構造化データなし・AIボットブロック | 統計データ追加・スキーマ実装・robots.txt 許可設定 |
| GA4 でイベントが表示されない | トリガー設定ミス・GTM未公開・コンセントモードでブロック | DebugView で確認→トリガー条件→GTMプレビューモードで検証 |
| A/Bテストが有意差に達しない | サンプルサイズ不足・効果量が小さい・テスト期間が短い | MDE を見直し、より大胆な変更を検討。トラフィックが少ない場合は定性判断も検討 |
| GTM のタグが二重発火する | 複数コンテナの競合・トリガー条件の重複 | タグ発火オプションを確認し、トリガーの排他条件を設定 |
| UTM パラメータが計測されない | リダイレクトでパラメータが欠落・パラメータ名の大文字小文字不一致 | リダイレクトチェーン確認・パラメータは全て小文字に統一 |
| コンテンツカレンダーが実行されない | リソース不足・優先順位の不明確さ | 週1本に絞り、最もインパクトの大きいキーワードから着手 |
| 順位が突然下がった | アルゴリズム更新・技術的問題・競合の動き | Search Console で変動範囲特定→技術監査→コンテンツ品質見直し |
| 構造化データにエラーが出る | 必須プロパティの欠落・値の形式不正 | Rich Results Test でエラー内容確認→Google のドキュメントで必須項目を再確認 |
| レポートの数値が合わない | データソース間のフィルタ差異・サンプリング | GA4 のデータフィルタ設定確認・BigQuery エクスポートで正確な数値を取得 |
| 技術SEO監査が全サイトで同じ結果になる（SE-1） | サイト種別・技術スタックを考慮せず汎用チェックリストのみ適用 | サイト種別（SaaS/EC/メディア/コーポレート）と技術スタック（Next.js/WordPress/Shopify等）に応じた監査重点を再設定し、優先順位を差別化する |
| コンテンツ戦略が競合と同じクラスター構造になる（SE-2） | キーワードツールの提案をそのまま採用し、コンテンツギャップ分析を省略 | コンテンツギャップ分析で競合が弱い領域を特定し、ニッチの権威を先に確立する差別化戦略を採用する |
| A/Bテスト仮説が毎回同じパターンになる（SE-7） | 仮説テンプレートに当てはめるだけで、個別のデータ観察が不足 | Search Console のクエリデータとヒートマップから具体的な観察を起点にし、「なぜこの変更が効くか」の根拠を個別に記述する |
