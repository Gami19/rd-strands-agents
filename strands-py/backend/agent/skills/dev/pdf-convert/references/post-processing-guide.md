# 変換後処理ガイド

> Markdown 出力の品質改善手順、テーブル整形、画像パス管理、ヘッダー/フッター除去、ページ番号処理、目次再構成のリファレンス

---

## 1. Markdown 出力の品質改善手順

### 変換後処理の標準フロー

```
Step 1: 構造チェック（見出し階層の確認）
  ↓
Step 2: ヘッダー/フッター/ページ番号の除去
  ↓
Step 3: テーブルの整形と検証
  ↓
Step 4: 画像パスの整理
  ↓
Step 5: 目次の再構成（必要に応じて）
  ↓
Step 6: フロントマターの記入
  ↓
Step 7: 最終品質チェック
```

### 見出し階層の修正パターン

| 問題 | 検出方法 | 修正方法 |
|:---|:---|:---|
| 見出しレベルの飛び（# → ###） | 元文書の目次と比較 | 中間レベル（##）を補完 |
| 全てが同一レベル（## のみ） | 見出しの内容で判断 | 章・節・項に応じてレベルを調整 |
| 見出しがプレーンテキスト化 | 太字 + 改行のパターンを検索 | `**見出し**` → `## 見出し` に変換 |
| 見出し番号の重複 | 連番を確認 | 元文書の番号体系に合わせて修正 |

### 修正マーキングの規約

変換後の手動修正箇所には HTML コメントで修正内容を記録する。後続の distill スキルが修正箇所を認識し、信頼度を判断するため。

```markdown
<!-- fixed: heading level adjusted from ### to ## -->
## 2. システム要件

<!-- fixed: merged cell in table -->
| 項目 | 要件 | 備考 |

<!-- removed: header/footer -->

<!-- added: figure text from original document -->
図 3: システム構成図（出典: RFP p.15）

<!-- fixed: OCR misrecognition "期首" → "期末" -->
```

## 2. テーブルの整形パターン

### Markdown テーブルの基本規則

```markdown
| ヘッダー1 | ヘッダー2 | ヘッダー3 |
|:---|:---|:---|
| データ1 | データ2 | データ3 |
| データ4 | データ5 | データ6 |
```

### よくあるテーブル問題と修正

#### パターン A: 列ズレの修正

```markdown
<!-- Before: columns misaligned -->
| 項目 | 要件 | 備考 |
|:---|:---|:---|
| CPU | 8コア以上必要 |
| メモリ | 32GB | 推奨64GB |

<!-- After: fixed alignment -->
<!-- fixed: column alignment -->
| 項目 | 要件 | 備考 |
|:---|:---|:---|
| CPU | 8 コア以上 | 必要 |
| メモリ | 32GB | 推奨 64GB |
```

#### パターン B: セル結合の表現

```markdown
<!-- Markdown does not support cell merging natively -->
<!-- Use HTML comments to indicate merged cells -->

<!-- fixed: merged cell spanning 3 columns -->
| カテゴリ | 項目 | 値 |
|:---|:---|:---|
| **ハードウェア要件** | | |
| | CPU | 8 コア以上 |
| | メモリ | 32GB 以上 |
| **ソフトウェア要件** | | |
| | OS | RHEL 8+ |
| | DB | PostgreSQL 14+ |
```

#### パターン C: 数値テーブルの整合性確認

```markdown
<!-- Verify: total should match sum of rows -->
| 項目 | Q1 | Q2 | Q3 | Q4 | 合計 |
|:---|---:|---:|---:|---:|---:|
| 売上 | 100 | 120 | 130 | 150 | 500 |
| 原価 | 60 | 72 | 78 | 90 | 300 |
| 粗利 | 40 | 48 | 52 | 60 | 200 |

<!-- data-validation: 100+120+130+150=500 ✓ -->
```

#### パターン D: 長いテーブルの分割

```
6 行を超えるテーブルは、スライド化時に問題になる。
以下の基準で分割を検討する:

  - 10 行以下: そのまま維持（distill で要約可能）
  - 11-20 行: 論理的な区切りで 2 テーブルに分割
  - 21 行以上: カテゴリごとに分割 + サマリーテーブルを追加
```

## 3. 画像パスの管理

### 画像の出力構造

```
projects/<案件名>/10_inbox/
├── RFP.md                    ← 変換結果
├── RFP_images/               ← 抽出された画像
│   ├── image_001.png
│   ├── image_002.png
│   └── image_003.png
└── 仕様書.md
```

### Markdown 内の画像参照

```markdown
<!-- Docling の標準出力形式 -->
![](RFP_images/image_001.png)

<!-- 推奨: キャプションと元文書の参照を追加 -->
![図1: システム構成概要（出典: RFP p.8）](RFP_images/image_001.png)
```

### 画像パス管理のルール

| ルール | 説明 |
|:---|:---|
| 相対パスを使用 | `./RFP_images/image_001.png`（絶対パスは避ける） |
| ファイル名は元文書ベース | `RFP_images/`（元ファイル名 + `_images/`） |
| キャプションを付与 | 元文書の図番号・タイトルを `![キャプション](パス)` に記載 |
| 不要な画像を除去 | 装飾画像・背景画像・ロゴなど、情報価値のない画像は参照を削除 |
| 前後の説明文を保持 | 図表の前後にある説明文・凡例は削除しない |

## 4. ヘッダー/フッターの除去

### 検出パターン

| パターン | 検出方法 | 例 |
|:---|:---|:---|
| **繰り返しテキスト** | 全ページに同一テキストが出現 | 「株式会社XX　提案書」 |
| **ページ番号パターン** | `- N -` / `N / M` / `Page N` の正規表現 | `- 3 -`、`12 / 45` |
| **日付パターン** | 全ページに同一日付が出現 | 「2026年2月28日」 |
| **機密表示** | 全ページに「Confidential」等 | 「社外秘」「CONFIDENTIAL」 |
| **章タイトルの繰り返し** | ヘッダーに章名が繰り返される | 「第2章 システム要件」 |

### 除去の手順

```markdown
1. 変換結果を通読し、繰り返しパターンを特定

2. ヘッダー/フッターの候補行をリストアップ
   例:
   - 「株式会社XX　提案書」（全ページに出現）
   - 「- {N} -」（ページ番号）
   - 「2026年2月28日」（日付）
   - 「Confidential」（機密表示）

3. 該当行を削除し、マーキングを付与
   <!-- removed: header "株式会社XX　提案書" -->
   <!-- removed: page number -->

4. 削除後に前後の文脈が崩れていないか確認
```

### 除去してはいけないもの

| 要素 | 理由 |
|:---|:---|
| 脚注（Footnote） | 本文の補足情報。ヘッダー/フッターと混同しやすい |
| 参照番号 | `[1]`、`※1` 等の参照は本文の一部 |
| 改ページ後の章タイトル | 見出しとして必要 |
| 表の列ヘッダー（ページまたぎ） | テーブル構造の一部 |

## 5. ページ番号の処理

### ページ番号パターンと除去

| パターン | 正規表現 | 例 |
|:---|:---|:---|
| ハイフン囲み | `^- \d+ -$` | `- 3 -` |
| スラッシュ区切り | `^\d+ / \d+$` | `12 / 45` |
| Page N | `^Page \d+$` | `Page 7` |
| 全角数字 | `^[０-９]+$` | `１２` |
| 丸囲み | `^[\u2460-\u2473]$` | 丸囲み数字 |

### ページ参照の保持

元文書のページ番号への参照は、トレーサビリティのために保持する。

```markdown
<!-- Page references are preserved for traceability -->

本仕様の詳細は第3章（原文 p.15-23）を参照。

<!-- 出典の記法 -->
> 出典: RFP p.23, §3.2.1
```

## 6. 目次の再構成

### 目次の扱い

| パターン | 対応方法 |
|:---|:---|
| 元文書の目次がそのまま変換された | 変換された目次を削除し、Markdown の見出しから自動生成を推奨 |
| 目次が崩れて変換された | 削除して、見出し構造から再構成 |
| 目次が存在しない | 見出し構造が正しければ不要。必要なら追加 |

### 目次の再構成テンプレート

```markdown
## 目次

- [1. はじめに](#1-はじめに)
- [2. システム要件](#2-システム要件)
  - [2.1 機能要件](#21-機能要件)
  - [2.2 非機能要件](#22-非機能要件)
- [3. 提案内容](#3-提案内容)
```

### 目次の自動生成が不要なケース

- distill スキルが目次を独自に構築する場合
- 文書が短い（10 ページ以下）場合
- フラットな見出し構造（## のみ）の場合

## 7. 最終品質チェックリスト

変換後処理の完了前に、以下を確認する。

| # | チェック項目 | 確認方法 |
|:---|:---|:---|
| 1 | 見出し階層が元文書と一致する | 元 PDF の目次と Markdown の `#` 構造を比較 |
| 2 | テーブルの列数・行数が元文書と一致する | テーブルごとにカウント比較 |
| 3 | 数値データが元文書と一致する | 金額・日付・パーセンテージをサンプリング照合 |
| 4 | ヘッダー/フッター/ページ番号が除去されている | 繰り返しパターンの検索 |
| 5 | 画像パスが正しく機能する | Markdown プレビューで画像表示を確認 |
| 6 | 修正箇所に `<!-- fixed: -->` マーキングがある | HTML コメントの検索 |
| 7 | フロントマターが完備している | `source`、`converted_at`、`confidence` 等 |
| 8 | 文字化け・文字欠けがない | 全文通読（最低限サンプリング） |
| 9 | 箇条書きのインデントが保持されている | ネストされたリストの確認 |
| 10 | 不要な空行・改行が整理されている | 3 行以上の連続空行がないか |
