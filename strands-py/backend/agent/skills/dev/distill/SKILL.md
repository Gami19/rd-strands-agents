# Skill: Distill（知識蒸留 & 構造化）

> **混沌を秩序に変え、暗黙知を形式知に鋳造する**

## Instructions

### ワークフロー内の位置

```
pdf-convert → [distill] → 成果物作成 → review
                 ↑                           ↓
            10_inbox/ の生データ        40_decision_log/ に記録
```

### プロジェクトディレクトリ構造

本スキルは番号付きディレクトリ構造に従って動作する:

| 番号 | ディレクトリ | 位置づけ |
|:---|:---|:---|
| **10** | `10_inbox/` | 生データ（入力）— 顧客要望・メモ・RFP |
| **20** | `20_notes/` | 構造化された知識（出力）— 蒸留した要件 |
| **40** | `40_decision_log/` | 意思決定の記録 — エージェントが即座にコンテキスト復元可能 |
| **50** | `50_snapshots/` | マイルストーン記録 — 変更前の状態を保存 |

### 入力

| ソース | パス |
|:---|:---|
| ソース資料（生データ） | `10_inbox/` |
| 各成果物の標準フォーマット | [references/output-examples.md](references/output-examples.md) を参照 |
| 既存仕様（更新時） | `20_notes/` |

### 出力

| 成果物 | パス | 説明 |
|:---|:---|:---|
| 用語集 | `20_notes/glossary.md` | ドメイン固有用語の体系化定義 |
| ドメインモデル | `20_notes/domain-model.md` | エンティティ・属性・関係の構造定義 |
| ビジネスルール | `20_notes/business-rules.md` | 検証可能なルール・制約の網羅的抽出 |
| 非機能要件 | `20_notes/non-functional-requirements.md` | パフォーマンス・セキュリティ・可用性の定量定義 |
| API仕様（該当時） | `20_notes/api-spec.md` | OpenAPI 準拠のインターフェース定義 |
| データモデル（該当時） | `20_notes/data-model.md` | 物理データ設計・パーティション戦略 |
| 技術選定根拠（該当時） | `20_notes/tech-decisions.md` | 技術選定の評価マトリクス・ADR |
| トレーサビリティ | `20_notes/traceability.md` | 要件 → 仕様の追跡マトリクス |

各成果物の具体的な出力例は [references/output-examples.md](references/output-examples.md) を参照。

## Step 1: ソース資料の収集と分析

`10_inbox/` の全資料を精読し、対象ドメインの課題・制約条件・評価基準を抽出する。

### @ ファイル参照パターンの活用

大きな知識ソースを効率的にコンテキストに注入する:

| パターン | 使用場面 | 例 |
|:---|:---|:---|
| `@skills-machine/10_inbox/filename.md` | 特定の知識ソースを指定 | `@skills-machine/10_inbox/Laws-of-UX.md を蒸留して` |
| `@skills-machine/10_inbox/` | inbox 全体の構成を俯瞰 | `inbox にどんなソースがあるか確認して` |
| 章単位の指定 | 大きなソースを分割蒸留 | `@filename.md の第3章だけを蒸留して` |

**原則**: コンテキストウィンドウには限りがある。大きな知識ソース（500行超）は章単位で分割して蒸留し、精度を保つ。

### 初回分析のフレームワーク

資料を精読する際、以下の 5 つの軸で情報を分類する:

| 軸 | 抽出する情報 | 記録方法 |
|:---|:---|:---|
| **Why** | ビジネス課題、プロジェクトの目的 | 箇条書きで明文化 |
| **Who** | ステークホルダー、ユーザーペルソナ | 役割と責任範囲を定義 |
| **What** | 機能要件、業務フロー | ドメインモデル・ビジネスルールに構造化 |
| **How** | 技術制約、アーキテクチャ要件 | 非機能要件・技術選定に反映 |
| **When** | スケジュール、マイルストーン | 優先度付けの根拠として記録 |

**チェックリスト**:
- [ ] inbox の全資料を読み込んだか
- [ ] 対象ドメインの根本的な課題（Why）を理解できているか
- [ ] 評価基準・優先度を把握しているか
- [ ] 制約条件（期間、予算、技術要件、法規制）を特定しているか
- [ ] 不明点・矛盾点のリストを作成したか
- [ ] 資料間の重複・矛盾を検出しマーキングしたか

## Step 2: 用語集（Glossary）の策定

ソース資料に出現する専門用語を体系化する。出力例は [references/output-examples.md](references/output-examples.md) 参照。

### 用語抽出の戦略

1. **ドメイン固有用語**: 業界・業務に特有の用語（最優先）
2. **多義語**: 文脈によって意味が変わる用語（定義の固定が必須）
3. **略語・頭字語**: 正式名称との対応を明確化
4. **和製英語**: 日本語環境で誤解されやすい英語由来の用語

### 用語間の関係マッピング

```
上位概念 ──→ 下位概念   例: データ基盤 → データレイク、DWH
同義語   ←→ 同義語     例: 校務支援システム = 校務システム
関連語   --- 関連語     例: 学籍情報 --- 指導要録
```

**チェックリスト**:
- [ ] ドメイン固有の用語が網羅されているか
- [ ] 略語の正式名称が定義されているか
- [ ] 同義語・類義語が統一されているか
- [ ] 和英対訳が必要な場合、対応が記載されているか
- [ ] 技術用語と業務用語の区別が明確か
- [ ] 用語間の階層関係（上位-下位）が整理されているか

## Step 3: ドメインモデルの構造化

ビジネスドメインを構造化し、主要エンティティを特定する。出力例は [references/output-examples.md](references/output-examples.md) 参照。

### エンティティ抽出の手順

1. **名詞抽出**: ソース資料から固有名詞・業務名詞を抽出
2. **エンティティ候補**: 属性を持ち、一意に識別されるものを選定
3. **属性定義**: 各エンティティの属性・データ型・制約を定義
4. **関係定義**: エンティティ間のカーディナリティ（1:1, 1:N, N:M）を確定
5. **制約定義**: 主キー・外部キー・一意制約・NOT NULL を明示

### データ型の選定指針

| カテゴリ | 推奨型 | 避ける型 | 理由 |
|:---|:---|:---|:---|
| 金額 | DECIMAL(15,2) | FLOAT/DOUBLE | 浮動小数点の丸め誤差 |
| 日時 | TIMESTAMP WITH TZ | VARCHAR | ソート・計算の正確性 |
| 識別子 | UUID / BIGINT | INT | 将来のスケーラビリティ |
| フラグ | BOOLEAN | INT(0/1) | 意味の明確性 |
| 列挙値 | ENUM / CHECK | VARCHAR | 値の制約保証 |

**チェックリスト**:
- [ ] 主要エンティティが特定されているか
- [ ] エンティティの属性と制約が明確か
- [ ] エンティティ間の関係（1:1, 1:N, N:M）は定義されているか
- [ ] 主キー・外部キー・一意制約が定義されているか
- [ ] データ型と精度（文字列長、数値精度）が明記されているか
- [ ] 法規制に基づく保存期間・削除要件があるか
- [ ] 後続の成果物で参照可能な形式になっているか

## Step 4: ビジネスルールの抽出

ソース資料からビジネスルール・制約を体系的に抽出する。出力例は [references/output-examples.md](references/output-examples.md) 参照。

### ルール分類体系

| 分類 | 説明 | 例 |
|:---|:---|:---|
| **制約ルール** | データの整合性を保証 | 「年齢は 0 以上 150 以下」 |
| **導出ルール** | 他のデータから算出 | 「合計金額 = 単価 × 数量 × (1 + 税率)」 |
| **アクションルール** | 条件を満たした時の処理 | 「在庫が 10 未満なら発注通知」 |
| **権限ルール** | アクセス制御 | 「管理者のみ削除可能」 |
| **時間ルール** | 時間に関する制約 | 「申請後 3 営業日以内に承認」 |

### ルール記述の品質基準

各ルールは以下の要素を**すべて**含むこと:

```markdown
| ID | BR-XXX-YYY |
| 名称 | ルールの端的な名称 |
| 分類 | 制約/導出/アクション/権限/時間 |
| 条件 | IF <条件> THEN <アクション> |
| 検証ロジック | 擬似コード or 数式 |
| 例外 | 例外ケースの列挙 |
| 出典 | 10_inbox/[ファイル名] §[セクション] |
| 依存 | 依存する他のルール ID |
```

**チェックリスト**:
- [ ] すべてのビジネスルールに一意IDが付与されているか
- [ ] 各ルールの根拠（ソース資料の該当箇所）が明記されているか
- [ ] 例外ケースは記述されているか
- [ ] 検証ロジックが擬似コード or 数式で記述されているか
- [ ] ルール間の依存関係が明確か
- [ ] ルールの分類（制約/導出/アクション/権限/時間）が適切か
- [ ] 後続の成果物で参照可能な形式になっているか

## Step 5: 技術仕様の策定（該当案件のみ）

非機能要件、API 仕様、データモデル、技術選定根拠を定義する。

### 非機能要件の定量化テンプレート

| カテゴリ | 指標 | 目標値 | 測定方法 |
|:---|:---|:---|:---|
| パフォーマンス | 応答時間（P95） | < 200ms | APM ツール |
| パフォーマンス | スループット | > 1000 req/s | 負荷テスト |
| 可用性 | 稼働率 | 99.9% | 月次 SLA レポート |
| 可用性 | RPO / RTO | 1h / 4h | DR 訓練 |
| セキュリティ | 認証方式 | OAuth 2.0 + MFA | セキュリティ監査 |
| 拡張性 | 同時接続数 | 10,000 | 負荷テスト |

### 技術選定の評価マトリクス

技術選定が必要な場合、以下のテンプレートで比較評価する:

| 評価軸 | 重み | 選択肢A | 選択肢B | 選択肢C |
|:---|---:|---:|---:|---:|
| 機能充足度 | 30% | 8 | 9 | 7 |
| 運用負荷 | 25% | 7 | 6 | 9 |
| コスト | 20% | 9 | 5 | 7 |
| チーム習熟度 | 15% | 8 | 4 | 6 |
| エコシステム | 10% | 7 | 9 | 5 |
| **加重平均** | **100%** | **7.85** | **6.55** | **7.10** |

選定結果は `40_decision_log/` に ADR として記録する。

**チェックリスト**:
- [ ] パフォーマンス要件（応答時間、スループット、同時接続数）が定量的に定義されているか
- [ ] セキュリティ要件（認証方式、暗号化、監査ログ）が定義されているか
- [ ] 可用性要件（SLA、RPO/RTO、DR 構成）が定義されているか
- [ ] API 仕様が OpenAPI 形式またはそれに準ずる形式で記述されているか
- [ ] データモデルのパーティション・クラスタリング戦略が定義されているか
- [ ] 技術選定の根拠が評価マトリクスで論理的に説明されているか
- [ ] 選定結果が `40_decision_log/` に記録されているか

## Step 6: 仕様の妥当性確認

inbox の原本と notes の蒸留結果を照合し、情報の欠落や誤解がないか検証する。

### 検証の 3 層アプローチ

| 層 | 検証内容 | 方法 |
|:---|:---|:---|
| **網羅性** | inbox の全要件が notes に反映されているか | トレーサビリティマトリクスで全数チェック |
| **整合性** | notes 内の各文書が互いに矛盾していないか | 用語集準拠チェック + 相互参照検証 |
| **十分性** | 後続工程に必要な情報が揃っているか | story-map / review の入力要件と照合 |

### トレーサビリティマトリクス

| 要件ID | ソース（10_inbox） | 蒸留先（20_notes） | カバー状態 |
|:---|:---|:---|:---|
| REQ-001 | Request.md §2.1 | domain-model.md E-001 | カバー済 |
| REQ-002 | Request.md §3.2 | business-rules.md BR-001 | カバー済 |
| REQ-003 | Material.md §1 | — | **未カバー（要確認）** |

### 知識蒸留 Slop 防止チェック（Distributional Convergence 対策）

知識蒸留は「ソース資料固有の知見を構造化する」ことが本質であり、LLM がソース資料を読み飛ばして一般知識や教科書的な定義で穴埋めすると、蒸留結果がソースと乖離した「偽の構造化」になる。

| # | 蒸留 Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| DI-1 | **分類体系の画一化** — どのソース資料でも同じ3階層（大分類→中分類→小分類）の分類木を生成し、ソース固有の分類構造を無視する | 蒸留結果の分類体系が inbox のソース資料の構造（章立て・セクション構成）を反映しているか確認する | ソース資料の固有構造を尊重する。技術書なら原著の章構成を活かし、RFP なら要件カテゴリの粒度に合わせる。「標準3階層」に無理に嵌めない |
| DI-2 | **表層的な用語抽出** — ソース資料の表面に現れる用語だけを拾い、文脈に埋め込まれた暗黙の概念・前提条件・業界慣習を見落とす | 用語集に「ソース資料の特定セクションでしか使われない独自概念」「著者が定義した造語」が含まれているか確認する | ソース資料の文脈を深く読み、著者独自の概念定義・暗黙の前提・業界固有のニュアンスを用語として抽出する。一般辞書にある定義の転記ではなく、ソース文脈での意味を記載する |
| DI-3 | **一般論への置換** — ソース資料の具体的な原則・知見を「品質を確保する」「適切に設計する」等の汎用的な教訓に置き換えてしまう | 抽出された原則・パターンを読み、それが inbox のソースを読まなくても書ける内容か確認する | ソース資料に記載された具体的な数値・条件・事例・著者の主張をそのまま構造化する。「品質を意識する」ではなく「著者は X の条件下で Y が Z% 改善したと報告している」と記述する |
| DI-4 | **既知パターンへの強制マッピング** — ソース資料のパターンを GoF デザインパターンや TOGAF 等の既知フレームワークに無理やり対応づけ、ソース固有の知見を失う | 蒸留結果のパターン名がすべて既知フレームワークの名称になっていないか確認する。ソース独自のパターン・アンチパターンが抽出されているか確認する | ソース資料が提示する固有のパターン・手法・フレームワークをそのまま抽出する。既知パターンとの対比は付記として記載し、ソース固有の名称と定義を主体にする |
| DI-5 | **出典切断** — 蒸留結果の各項目からソース資料の具体的なセクション・ページへの参照が欠落し、蒸留の根拠が追跡不能になる | 用語集・ドメインモデル・ビジネスルールの各項目に `出典: 10_inbox/[ファイル名] §[セクション]` が記載されているか確認する | すべての蒸留結果に出典を付与する。「§2.3」「p.45」「Table 3」レベルの具体的な参照を記載し、トレーサビリティマトリクスで全要件の追跡を担保する |
| DI-6 | **出力構造の固定化** — ソース資料が技術書でも RFP でも議事録でも、常に同じ成果物セット（glossary + domain-model + business-rules）を出力し、ソースの性質に応じた取捨選択をしない | 出力された成果物の種類がソース資料の性質に適合しているか確認する。不要な成果物が空の定型文で埋められていないか確認する | ソース資料の性質に応じて出力成果物を取捨選択する。技術書の蒸留ならパターン集・原則集が主体、RFP なら要件・ビジネスルールが主体、議事録なら決定事項・アクションアイテムが主体になる |

> **核心原則**: **ソースの声を聴く蒸留** — 蒸留結果はソース資料の「声」を構造化したものであり、LLM の一般知識で水増しした合成物ではない。「別のソース資料を蒸留しても同じ notes が出力されるか？」→ されるなら蒸留 Slop である。

**チェックリスト**:
- [ ] inbox の全要件が notes に反映されているか
- [ ] 蒸留過程で情報が失われていないか
- [ ] notes 内で矛盾する記述がないか
- [ ] 用語集の用語が notes 全体で統一的に使用されているか
- [ ] ビジネスルール間に論理的矛盾がないか
- [ ] 技術仕様が非機能要件と整合しているか
- [ ] 後続の成果物作成に十分な情報量があるか
- [ ] トレーサビリティマトリクスで未カバー要件がゼロか
- [ ] 蒸留 Slop チェック（DI-1〜DI-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Step 7: インクリメンタル蒸留（差分更新）

既存の `20_notes/` がある状態で追加資料を蒸留する場合のワークフロー。

### 差分更新の手順

1. **変更前スナップショット**: `50_snapshots/` に現在の notes を保存
2. **差分抽出**: 新資料と既存 notes を比較し、新規・変更・削除の差分を特定
3. **影響分析**: 差分が他の notes 文書に及ぼす影響を評価
4. **段階的更新**: 影響範囲の小さい文書から順に更新
5. **整合性再検証**: Step 6 の検証を差分対象に限定して実行

### 差分管理テーブル

| 項目 | 変更種別 | 影響範囲 | 対応状態 |
|:---|:---|:---|:---|
| エンティティ E-005 追加 | 新規 | domain-model, business-rules | 対応済 |
| BR-003 条件変更 | 変更 | business-rules, api-spec | 対応中 |
| G-TECH-002 定義修正 | 変更 | glossary のみ | 未対応 |

**チェックリスト**:
- [ ] 変更前の notes が `50_snapshots/` に保存されているか
- [ ] 新規・変更・削除の差分が明確に分類されているか
- [ ] 差分の影響範囲が評価されているか
- [ ] 更新後に notes 全体の整合性が維持されているか
- [ ] 変更履歴が `40_decision_log/` に記録されているか

---

## Quality Gates

以下のすべてを満たした場合のみ、distill フェーズを完了とする:

**基本ゲート**:
- すべての inbox 資料が分析済み
- 用語集が作成され、全用語に ID が付与済み
- ドメインモデルが対象ドメインの業務を正確に表現
- ビジネスルールが網羅的に抽出済み（全ルールに ID、出典、検証ロジック）
- notes の各ドキュメントが相互参照可能
- トレーサビリティマトリクスで全要件の追跡が可能

**技術仕様ゲート（該当案件のみ）**:
- 非機能要件が定量的に定義済み
- API 仕様が入出力含め定義済み（該当時）
- データモデルが正規化・パーティション戦略含め設計済み（該当時）
- 技術選定根拠が論理的に文書化済み（該当時）

## Examples

### Example 1: 基本的な知識蒸留

```
「inbox の資料を notes に蒸留して」

→ Step 1: inbox の知識ソース確認（@skills-machine/10_inbox/ で構成把握）
→ Step 1: 5軸フレームワーク（Why/Who/What/How/When）で情報分類
→ Step 2: 用語集の抽出（ドメイン固有用語 → 定義 → 用法 → 和英対訳）
→ Step 3: ドメインモデルの構築（エンティティ → 属性 → 関係 → 制約）
→ Step 4: ビジネスルールの構造化（ID付き → 分類 → 条件 → 検証ロジック）
→ Step 6: トレーサビリティマトリクスで全要件のカバー確認
→ 出力: 20_notes/ に 5 ファイル（glossary.md, domain-model.md, business-rules.md, traceability.md）
```

### Example 2: 技術仕様の策定

```
「技術仕様（非機能要件、データモデル）を策定して」

→ Step 1: inbox の技術要件関連資料を分析
→ Step 2: 用語集に技術用語を追加
→ Step 5: 非機能要件を定量化テンプレートで定義（応答時間 P95 < 200ms 等）
→ Step 5: データモデルを正規化・パーティション戦略含め設計
→ Step 5: 技術選定は評価マトリクスで比較、結果を 40_decision_log/ に記録
→ 出力: 20_notes/ に non-functional-requirements.md, data-model.md, tech-decisions.md
```

### Example 3: 用語集の作成

```
「用語集を作成して」

→ Step 1: inbox の全資料から専門用語を抽出
→ Step 2: ドメイン固有用語 → 多義語 → 略語 → 和製英語の順に整理
→ Step 2: 用語間の階層関係（上位-下位）をマッピング
→ Step 2: 同義語・類義語を統一し、標準用語を確定
→ 出力: 20_notes/glossary.md（全用語に ID・定義・和英対訳・出典を付与）
```

### Example 4: 複雑な RFP からのビジネスルール抽出

```
「RFP の業務要件からビジネスルールを網羅的に抽出して」

→ Step 1: RFP の全章を精読し、明示的・暗黙的なルールを洗い出し
→ Step 4: 制約/導出/アクション/権限/時間の5分類でルールを構造化
→ Step 4: 各ルールに検証ロジック（擬似コード）と例外ケースを定義
→ Step 4: ルール間の依存関係グラフを作成
→ Step 6: RFP の要件番号とルール ID のトレーサビリティを確認
→ 出力: 20_notes/business-rules.md（50+ルール、全てに出典・検証ロジック付き）
```

### Example 5: 章単位の分割蒸留（大規模ソース）

```
「この 800 行の技術仕様書を章ごとに分割して蒸留して」

→ Step 1: 目次を確認し、6 章構成を把握。各章のテーマと行数を整理
→ Step 1: 章ごとに @ 参照で個別にコンテキストに注入（1章 ≤ 300行を目安に分割）
→ Step 2-4: 各章から用語・エンティティ・ルールを抽出し、中間成果物に蓄積
→ Step 6: 全章の蒸留結果を統合し、章間の矛盾・重複を解消
→ Step 6: トレーサビリティマトリクスで全章の要件カバーを確認
→ 出力: 20_notes/ に統合された成果物（分割蒸留の結合点を明示）
```

### Example 6: 既存仕様の差分更新

```
「追加の顧客要望が inbox に入りました。notes を差分更新して」

→ Step 7: 50_snapshots/ に現在の notes をスナップショット保存
→ Step 1: 新資料を読み込み、既存 notes との差分を特定
→ Step 7: 差分管理テーブルで新規/変更/削除を分類
→ Step 7: 影響範囲を評価（glossary 2件、domain-model 1件、business-rules 3件）
→ Step 2-4: 影響範囲の小さい文書から順に更新
→ Step 6: 差分対象に限定した整合性再検証
→ 出力: 更新された 20_notes/ + 40_decision_log/ に変更理由を記録
```

### Example 7: 非機能要件の定義（SaaS 基盤）

```
「SaaS 基盤の非機能要件を定義して」

→ Step 1: inbox から性能・セキュリティ・運用の要件を抽出
→ Step 5: 非機能要件を定量化テンプレートで定義（SLA 99.9%, RPO 1h, RTO 4h）
→ Step 5: マルチテナント要件（データ分離、リソース制限）を定義
→ Step 5: セキュリティ要件（OAuth 2.0 + MFA、暗号化、監査ログ）を定義
→ Step 5: 技術選定は評価マトリクスで 3 候補を比較し、ADR に記録
→ 出力: 20_notes/non-functional-requirements.md + 40_decision_log/ADR-001-tech-selection.md
```

## Troubleshooting

| 問題 | 原因 | 対処 |
|:---|:---|:---|
| inbox の資料が不十分 | 情報不足 | 質問リストを作成し、追加情報の収集を依頼 |
| 用語が業界固有で理解困難 | ドメイン知識の不足 | 業界標準文書、規格、ガイドラインを参照 |
| 要件間の矛盾を発見 | ソース資料内の不整合 | 矛盾を明示し、確認を依頼。ADR に仮定を記録 |
| 技術選定で迷う | 複数の有力候補 | 評価マトリクスを作成し、ADR に判断根拠を記録 |
| 既存仕様の更新時に diff が不明 | 差分管理の不備 | `50_snapshots/` に変更前の状態を保存してから更新 |
| コンテキストが長すぎて蒸留精度が下がる | 知識ソースが巨大（500行超） | 章・セクション単位で分割して蒸留し、後から統合する |
| 用語数が 100 超で管理困難 | 大規模ドメイン | カテゴリ分類（業務/技術/略語）で glossary を構造化。ID 体系で検索性を確保 |
| エンティティ間の関係が複雑すぎる | N:M 関係の多用 | 中間テーブルで N:M を 1:N に分解。diagram スキルで ER 図を生成し可視化 |
| 仕様とソースの乖離が大きい | 蒸留過程での情報変質 | トレーサビリティマトリクスの未カバー率を確認し、乖離箇所を個別に原本と照合 |
| API 仕様の粒度が定まらない | エンドポイント設計の判断材料不足 | ユースケースベースで API を洗い出し、CRUD マトリクスで網羅性を検証 |
| 差分更新で既存仕様との整合が崩れる | 影響分析の不足 | Step 7 の差分管理テーブルで影響範囲を可視化し、依存する文書を先に確認 |
