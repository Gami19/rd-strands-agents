# 蒸留品質の評価基準

> 蒸留の良し悪しを測る物差し。主観的な「良さそう」を排除し、再現可能な品質判定を実現する。

---

## 概要

蒸留結果の品質は 4 つの軸で評価する。
各軸に 3 段階の判定基準を設け、蒸留完了時のセルフチェックと、
review スキルによる第三者レビューの両方で使用する。

---

## 評価軸 1: 忠実性（Fidelity）

原著の意図・ニュアンス・主張が正しく保持されているか。

### 判定基準

| 評価 | 基準 | 具体的な指標 |
|:---|:---|:---|
| **優** | 原著の主張・数値・条件がすべて正確に反映され、著者固有の概念名が原語で保持されている | 出典参照が全項目に付与済み。著者造語が原語で記載。数値に測定条件が併記 |
| **良** | 主要な主張は正確だが、一部の条件・例外ケースが省略されている | 出典参照が 80% 以上の項目に付与。著者造語の一部が意訳されている |
| **要改善** | 原著の主張が一般論に置換されている、または出典が追跡できない | 出典参照が 50% 未満。「適切に設計する」等の汎用表現が多い。DI-3 に該当 |

### チェック項目

- [ ] 原著の具体的な数値（人数、割合、閾値）が保持されているか
- [ ] 著者固有の概念名が原語で記載されているか（例: Five Pillars, Architecture Quantum）
- [ ] 原著の「must/should/may」の強度が蒸留結果に反映されているか
- [ ] 蒸留結果の各項目に `出典: 10_inbox/[ファイル名] §[セクション]` が付与されているか
- [ ] 原著が示す例外ケース・制約条件が省略されていないか

### Before/After 比較例

**Before（原文）**:

> Teams should be kept small, ideally between 5 and 9 members (following Dunbar's grouping: 5 for close, 15 for deep trust, 50 for mutual trust, 150 for stable relationships). Beyond this size, communication overhead increases quadratically per Brooks's Law.

**After（忠実な蒸留）**:

| 原則 | 内容 | 出典 |
|:---|:---|:---|
| チーム規模の制約 | チームは 5-9 名が理想。Dunbar's Number に基づく段階: 5 名（親密）、15 名（深い信頼）、50 名（相互信頼）、150 名（安定的関係）。これを超えると Brooks's Law により通信オーバーヘッドが二次関数的に増大する | §4.2 |

**After（忠実性 "要改善" の例）**:

| 原則 | 内容 |
|:---|:---|
| チーム規模 | チームは小さく保つべきである。大きすぎるとコミュニケーションが困難になる |

問題点: 具体的な数値（5-9 名、Dunbar's Number の各段階）が消失し、Brooks's Law への言及が脱落。出典なし。

## 評価軸 2: 構造化度（Structure）

情報が検索・参照しやすい形に再構成されているか。

### 判定基準

| 評価 | 基準 | 具体的な指標 |
|:---|:---|:---|
| **優** | 情報が一貫した ID 体系で管理され、テーブル・階層構造で即時参照可能。相互参照リンクが完備 | 全項目に ID 付与。テーブル/リスト形式が 70% 以上。notes 間の相互参照あり |
| **良** | 主要な情報は構造化されているが、ID 体系や相互参照が部分的に欠如 | 主要項目に ID 付与。テーブル/リスト形式が 50% 以上 |
| **要改善** | 散文形式が主体で、情報の検索・参照が困難 | ID 未付与。散文が 70% 以上。相互参照なし |

### チェック項目

- [ ] 用語集の全用語に一意 ID（G-XXX-YYY 形式）が付与されているか
- [ ] ドメインモデルのエンティティに一意 ID（E-XXX 形式）が付与されているか
- [ ] ビジネスルールに一意 ID（BR-XXX-YYY 形式）が付与されているか
- [ ] テーブル形式で比較・一覧化されるべき情報がテーブルになっているか
- [ ] notes 内の各文書が相互参照で接続されているか（例: ビジネスルールからエンティティ ID を参照）
- [ ] 見出し階層が 3 段以内に収まっているか（深すぎるネストは可読性を損なう）

### Before/After 比較例

**Before（原文の散文）**:

> メダリオンアーキテクチャでは、まずブロンズ層に生データを取り込みます。次にシルバー層でクレンジングと標準化を行います。最後にゴールド層でビジネス要件に合わせた集計テーブルを作成します。各層間の移動にはDelta Live Tablesを使うのが推奨です。

**After（構造化された蒸留）**:

| 層 | ID | 目的 | データ品質 | 移行手段 |
|:---|:---|:---|:---|:---|
| Bronze | ML-001 | 生データの忠実な取り込み | 未加工 | Auto Loader / COPY INTO |
| Silver | ML-002 | クレンジング・標準化・重複排除 | 検証済み | Delta Live Tables（推奨） |
| Gold | ML-003 | ビジネス要件に合わせた集計・結合 | ビジネス品質 | Delta Live Tables / Scheduled Job |

出典: inbox/databricks-guide.md §3.1

## 評価軸 3: 密度（Density）

冗長な説明を省きつつ、重要な判断基準を漏らしていないか。

### 判定基準

| 評価 | 基準 | 具体的な指標 |
|:---|:---|:---|
| **優** | 原著の本質的な知見がすべて保持され、冗長な説明・繰り返し・修辞が除去されている。圧縮率 60-80% | 原著 100 行 → 蒸留 20-40 行。判断基準・条件・数値がすべて保持 |
| **良** | 主要な知見は保持されているが、一部に冗長な説明が残っている。圧縮率 40-60% | 原著 100 行 → 蒸留 40-60 行。一部に散文的な説明が残存 |
| **要改善** | 冗長な説明がそのまま残っている、または重要な判断基準が省略されている | 原著 100 行 → 蒸留 80 行以上（薄い圧縮）、または 10 行以下（過剰圧縮で情報喪失） |

### チェック項目

- [ ] 同じ概念の重複説明が排除されているか
- [ ] 修辞的な表現（「言うまでもなく」「周知の通り」等）が除去されているか
- [ ] 原著の具体例のうち、最も代表的な 1-2 例が保持されているか（全例の転記は不要）
- [ ] 判断に必要な条件・閾値・数値が省略されていないか
- [ ] 蒸留結果がソース資料の 20-40% の行数に収まっているか（圧縮率 60-80% が目安）

### 密度の判断指針

| 情報の種類 | 保持する | 除去する |
|:---|:---|:---|
| 判断基準（条件 → 選択肢） | 全件保持 | -- |
| 具体的な数値・閾値 | 全件保持 | -- |
| 著者固有のパターン名 | 全件保持 | -- |
| 具体例・事例 | 最も代表的な 1-2 例 | 残りの類似例 |
| 前提条件の説明 | 簡潔に要約 | 詳細な歴史的経緯 |
| 修辞・装飾的表現 | -- | 全件除去 |
| 他章への導入・まとめ | -- | 全件除去 |

## 評価軸 4: 再利用性（Reusability）

蒸留結果が複数のスキルやプロジェクトで活用できるか。

### 判定基準

| 評価 | 基準 | 具体的な指標 |
|:---|:---|:---|
| **優** | 蒸留結果が特定のプロジェクトに依存せず、同ドメインの別案件でもそのまま参照可能 | ドメイン固有の原則・パターンが汎用的な形で構造化。ID 体系で個別参照可能 |
| **良** | 蒸留結果の大部分は再利用可能だが、一部にプロジェクト固有の記述が混在 | 70% 以上の項目がプロジェクト非依存。固有記述に注記あり |
| **要改善** | 蒸留結果が特定プロジェクトの文脈に強く依存し、他で使えない | プロジェクト名・固有名詞がハードコード。抽象化されていない |

### チェック項目

- [ ] 蒸留結果のパターン・原則がプロジェクト名に依存していないか
- [ ] ID 体系により、他のスキルや notes から個別項目を参照できるか
- [ ] 蒸留結果が skills/ のスキル定義に取り込み可能な粒度か
- [ ] プロジェクト固有の情報（社名、日付、金額等）が原則・パターンから分離されているか

## 総合評価テーブル

4 軸の評価を集計し、蒸留全体の品質を判定する。

| 総合判定 | 条件 | アクション |
|:---|:---|:---|
| **合格** | 4 軸すべてが「良」以上、かつ 2 軸以上が「優」 | 蒸留完了。後続工程に進む |
| **条件付き合格** | 「要改善」が 1 軸のみ、他が「良」以上 | 該当軸のみ改善し再評価 |
| **差し戻し** | 「要改善」が 2 軸以上 | ソース資料を再読し、蒸留をやり直す |

### 評価記録テンプレート

蒸留完了時に以下のテーブルを `40_decision_log/` に記録する。

```markdown
## 蒸留品質評価: [ソース名]

| 評価軸 | 評価 | 根拠 |
|:---|:---|:---|
| 忠実性 | 優/良/要改善 | [具体的な根拠] |
| 構造化度 | 優/良/要改善 | [具体的な根拠] |
| 密度 | 優/良/要改善 | [具体的な根拠] |
| 再利用性 | 優/良/要改善 | [具体的な根拠] |
| **総合** | 合格/条件付き/差し戻し | |

評価日: YYYY-MM-DD
評価者: [名前/エージェント]
ソース: 10_inbox/[ファイル名]
蒸留先: 20_notes/[ディレクトリ]
```

## Slop 検出との連携

本評価基準は SKILL.md の蒸留 Slop 防止チェック（DI-1〜DI-6）を補完する。

| 評価軸 | 検出する Slop パターン |
|:---|:---|
| 忠実性 | DI-3（一般論への置換）、DI-5（出典切断） |
| 構造化度 | DI-1（分類体系の画一化）、DI-4（既知パターンへの強制マッピング） |
| 密度 | DI-2（表層的な用語抽出）、DI-3（一般論への置換） |
| 再利用性 | DI-6（出力構造の固定化） |

**運用ルール**: 総合評価が「差し戻し」の場合、DI-1〜DI-6 のチェックも再実施し、
Slop パターンの該当有無を特定した上で蒸留をやり直す。
