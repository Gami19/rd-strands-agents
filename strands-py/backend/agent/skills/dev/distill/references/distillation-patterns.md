# 知識ソース種別 蒸留パターン集

> ソースの性質を見極め、最適な蒸留戦略を選択する。

---

## 概要

知識ソースは一様ではない。書籍と RFP では情報の構造も密度も異なる。
ソース種別ごとに「何を抽出し、どう構造化するか」を事前に定めることで、
蒸留精度を高め、DI-6（出力構造の固定化）を防止する。

---

## Pattern 1: 書籍（体系的知識）

### 入力の特徴

- 章・節で体系的に構成されている
- 原則 → パターン → 実例 の階層構造を持つ
- 著者独自の概念・造語・フレームワークが含まれる
- 1 冊で 500 行超になることが多く、分割蒸留が前提

### 蒸留の焦点

| 抽出対象 | 説明 | 優先度 |
|:---|:---|:---|
| **著者固有の原則** | 著者が提唱する独自の法則・原則・フレームワーク名 | 最高 |
| **パターン/アンチパターン** | 著者が名付けた設計パターンと、やってはいけない例 | 高 |
| **判断基準テーブル** | 「いつ A を使い、いつ B を使うか」の条件分岐 | 高 |
| **数値・閾値** | 著者が示す具体的な数値（「5 人」「150 人」「99.9%」等） | 中 |
| **用語の著者定義** | 一般用語に著者が独自の意味を与えているもの | 中 |

### 出力構造

```
20_notes/<domain>/
├── principles.md    # 原則の階層化（最上位 → 派生原則）
├── patterns.md      # パターン集（名前 → 文脈 → 解決策 → トレードオフ）
├── glossary.md      # 著者固有の用語定義（一般辞書定義との差異を明記）
└── decisions.md     # 判断基準テーブル（条件 → 選択肢 → 推奨）
```

### よくある落とし穴

| 落とし穴 | 具体例 | 回避策 |
|:---|:---|:---|
| 章構成のコピー | 原著の目次をそのまま notes の見出しにする | スキルのステップで参照する単位に再構成する |
| 原則の薄め蒸留 | 「品質を意識する」「適切に設計する」 | 原著の具体的条件・数値・事例を保持する |
| 造語の見落とし | Five Pillars, Architecture Quantum を一般用語として扱う | 著者がイタリック体/太字/定義文で導入した語を抽出する |
| 過剰な一般化 | 9 つのアーキテクチャスタイルを 3 つに集約 | 著者の分類粒度を尊重し、全パターンを保持する |

### 実証例

- `software-architecture/` — 9 スタイルの比較表、Architecture Quantum の定義を保持
- `ai-agents/` — Five Pillars を原著の階層構造で蒸留、L0-L3 自律度スペクトラムを保持
- `flow-architecture/` — Wardley Mapping x DDD x Team Topologies の三位一体構造を保持

## Pattern 2: 技術ドキュメント（API/CLI/プラットフォーム）

### 入力の特徴

- 機能領域別に構成されている（認証、ストレージ、コンピュート等）
- コマンド・プロパティ・設定値の列挙が多い
- バージョン依存の情報が含まれる
- ユースケースやサンプルコードが散在している

### 蒸留の焦点

| 抽出対象 | 説明 | 優先度 |
|:---|:---|:---|
| **設定パラメータの判断基準** | デフォルト値、推奨値、変更すべき条件 | 最高 |
| **ユースケースパターン** | 典型的な利用シナリオとその構成 | 高 |
| **制約・制限** | API リミット、サイズ制限、互換性の注意点 | 高 |
| **ベストプラクティス** | 公式が推奨する構成・設定 | 中 |
| **トラブルシューティング** | よくあるエラーと対処法 | 中 |

### 出力構造

```
20_notes/<platform>/
├── architecture-patterns.md   # 構成パターン（条件 → 推奨構成）
├── configuration-guide.md     # 設定パラメータ一覧（パラメータ → 推奨値 → 根拠）
├── use-case-catalog.md        # ユースケース別の実装パターン
└── constraints.md             # 制約・制限・互換性情報
```

### よくある落とし穴

| 落とし穴 | 具体例 | 回避策 |
|:---|:---|:---|
| パラメータの列挙だけ | 設定値を一覧化するが判断基準がない | 「いつ変更するか」「どの値を選ぶか」の条件を付与する |
| バージョン情報の欠落 | 特定バージョンでしか有効でない情報を普遍的に記述 | バージョン適用範囲を明記する |
| サンプルコードの丸写し | 公式ドキュメントのコード例をそのまま転記 | パターンとして抽象化し、適用条件を明示する |

### 実証例

- `databricks/` — Medallion Architecture の段階別設計判断、コンピュート選定ガイド
- `unity-catalog/` — Securable Objects の階層構造、権限マトリクス

## Pattern 3: ガイド/フレームワーク（プロセス定義）

### 入力の特徴

- 段階的なプロセスやフェーズが定義されている
- チェックリスト・判断フローが含まれる
- 役割と責任の定義がある
- 成果物のテンプレートや例が付属している

### 蒸留の焦点

| 抽出対象 | 説明 | 優先度 |
|:---|:---|:---|
| **プロセス分解** | フェーズ → ステップ → タスクの階層化 | 最高 |
| **判断フロー** | 分岐条件と選択肢のデシジョンツリー | 高 |
| **チェックリスト** | 各フェーズのゲート条件・品質基準 | 高 |
| **役割・責任マトリクス** | RACI や DACI 形式の責任分担 | 中 |
| **成果物テンプレート** | 各フェーズの出力物のひな型 | 中 |

### 出力構造

```
20_notes/<process>/
├── process-overview.md      # フェーズ全体図と各フェーズの目的・成果物
├── decision-flows.md        # 判断フロー・分岐条件のデシジョンツリー
├── checklists.md            # フェーズ別のゲート条件・品質基準
└── templates.md             # 成果物テンプレート集
```

### よくある落とし穴

| 落とし穴 | 具体例 | 回避策 |
|:---|:---|:---|
| プロセスの抽象化しすぎ | 6 フェーズを「計画→実行→確認」の 3 段階に集約 | 原著のフェーズ粒度を保持する |
| 判断基準の脱落 | フローだけ残して分岐条件を省略 | IF-THEN 形式で条件を明文化する |
| テンプレートの空洞化 | テンプレートの枠だけ残して記入例を省略 | 記入例を 1 つ以上保持する |

### 実証例

- `drm/` — DRM 6 フェーズの各フェーズ定義、ヒアリングフレームワーク
- `skill-forge/` — スキル鋳造プロセス、品質チェックリスト、テストメソドロジー

## Pattern 4: 研究論文/レポート（実証的知見）

### 入力の特徴

- 仮説 → 実験/調査 → データ → 結論の構造を持つ
- 数値データ（統計値、ベンチマーク結果）が含まれる
- 先行研究への参照が多い
- 著者の主張と客観的データが混在している

### 蒸留の焦点

| 抽出対象 | 説明 | 優先度 |
|:---|:---|:---|
| **主張と根拠の対** | 結論とそれを支持するデータの紐付け | 最高 |
| **数値データ** | 改善率、閾値、ベンチマーク結果 | 高 |
| **適用条件** | 結論が有効な前提条件・スコープ | 高 |
| **限界と制約** | 著者が認めている研究の限界 | 中 |
| **実務への示唆** | 研究結果から導かれる実践的アクション | 中 |

### 出力構造

```
20_notes/<topic>/
├── findings.md         # 主張-根拠ペア（主張 → データ → 適用条件 → 限界）
├── metrics.md          # 数値データの構造化テーブル
└── implications.md     # 実務への示唆とアクションアイテム
```

### よくある落とし穴

| 落とし穴 | 具体例 | 回避策 |
|:---|:---|:---|
| 結論のみ抽出 | 「X は Y より効果的」だけ記録し、条件・データを省略 | 必ず根拠データと適用条件をセットで抽出する |
| 数値の文脈切断 | 「35% 改善」の母集団・条件を省略 | N 数、測定条件、比較対象を明記する |
| 相関と因果の混同 | 相関関係を因果関係として記述 | 原著の表現を尊重し、「相関」「因果」を区別する |

## 言語変換時の蒸留注意点（英語 → 日本語）

知識ソースの多くは英語で書かれている。日本語への蒸留時に以下の品質が劣化しやすい。

| 注意点 | 具体例 | 対策 |
|:---|:---|:---|
| **固有名詞の安易な翻訳** | Architecture Quantum を「アーキテクチャ量子」と翻訳 | 著者の造語・固有概念は原語を保持し、括弧内で説明を付与する |
| **ニュアンスの喪失** | "should" と "must" の区別が消える | 原著の強度（must/should/may）を日本語でも区別する（「必須」「推奨」「任意」） |
| **文化的コンテキストの脱落** | 英語圏特有の比喩・事例を省略 | 日本語読者向けに補足説明を付記するか、等価な日本語事例を追加する |
| **略語の放置** | DORA, SLO, ADR をそのまま使用 | 初出時に正式名称と日本語説明を併記する |
| **和製英語との混同** | "service" の日本語的用法と原著の意味が異なる | 用語集で原著での定義を明記し、日本語慣用との差異を注記する |

### 蒸留時の表記ルール

1. **著者固有の概念名**: 原語を保持（例: Five Pillars、Wardley Map）
2. **一般的な技術用語**: カタカナ表記 + 初出時に英語併記（例: マイクロサービス（Microservices））
3. **略語**: 初出時に `正式名称（略語）` 形式（例: Service Level Objective（SLO））
4. **定義文**: 日本語で記述し、原文のニュアンスが重要な箇所は原文を引用として併記

## パターン選択フローチャート

```
ソース資料を受け取る
  │
  ├─ 体系的な章構成がある？ ──→ Yes ──→ Pattern 1（書籍）
  │                             No
  │                             │
  ├─ API/CLI/設定値が主体？ ──→ Yes ──→ Pattern 2（技術ドキュメント）
  │                             No
  │                             │
  ├─ プロセス/フェーズ定義？ ──→ Yes ──→ Pattern 3（ガイド/フレームワーク）
  │                             No
  │                             │
  └─ データ/統計が主体？ ─────→ Yes ──→ Pattern 4（研究論文/レポート）
                                No
                                │
                                └─ 複合型: 主要な特徴で判断し、補助パターンを組み合わせる
```

**複合型の例**: 技術書の中に API リファレンスが含まれる場合、Pattern 1 を主軸に Pattern 2 の出力構造を部分的に取り入れる。

## Slop 防止との関係

本パターン集は SKILL.md の蒸留 Slop 防止チェック（DI-1〜DI-6）と連動する。

| Slop パターン | 本パターン集での対策 |
|:---|:---|
| DI-1（分類体系の画一化） | パターン別に異なる出力構造を定義 |
| DI-2（表層的な用語抽出） | 各パターンの「蒸留の焦点」で抽出対象を明示 |
| DI-3（一般論への置換） | 「よくある落とし穴」で薄め蒸留を警告 |
| DI-4（既知パターンへの強制マッピング） | 著者固有の概念名・分類粒度の保持を明記 |
| DI-5（出典切断） | 全パターンで出典参照の維持を前提 |
| DI-6（出力構造の固定化） | 4 パターン + 複合型で出力構造を可変化 |
