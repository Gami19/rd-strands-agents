# Skill: Data Architecture（データアーキテクチャ選定・設計・レビュー）

> **アーキテクチャは技術選定ではなく、ビジネス要件から逆算して設計せよ**

## Instructions

### ワークフロー内の位置

```
ビジネス要件 → [data-arch] → 実装設計 → diagram → review
                  ↓
             アーキテクチャ設計
             ・アーキテクチャ選定
             ・データモデリング
             ・チーム構成
             ・テクノロジー選定
```

### 入力

| 入力 | 説明 | 例 |
|:---|:---|:---|
| ビジネス要件 | データ活用の目的・ユースケース | 「全社のBI基盤を構築したい」 |
| 現状のデータ環境 | 既存システム・データソースの情報 | オンプレSQL Server + CSVファイル群 |
| 制約条件 | 予算、スケジュール、スキルセット | 「6ヶ月以内、Azure環境」 |
| 比較・評価対象 | 特定のアーキテクチャの評価依頼 | 「Data Mesh は我々に適切か」 |

### 出力

| 出力 | 形式 | 説明 |
|:---|:---|:---|
| アーキテクチャ推奨 | Markdown | 推奨アーキテクチャと根拠 |
| 比較評価表 | Markdown テーブル | 候補アーキテクチャの多軸比較 |
| 設計ドキュメント | Markdown + 図 | アーキテクチャの論理設計・物理設計 |
| ADS レポート | Markdown | 設計セッションの結果と推奨事項 |

## Step 1: ビジネス要件の把握

対象組織のデータ要件を以下の 6 軸で分析する。

| 軸 | 確認項目 | 参照 |
|:---|:---|:---|
| **Volume** | 現在のデータ量と予想成長率 | [architecture-comparison.md](references/architecture-comparison.md) |
| **Variety** | データ種別（構造化/半構造化/非構造化）の割合 | [architecture-comparison.md](references/architecture-comparison.md) |
| **Velocity** | リアルタイム処理の要件（バッチ/ストリーミング） | [data-ingestion-guide.md](references/data-ingestion-guide.md) |
| **Veracity** | データ品質・ガバナンスの現状と要件 | [data-ingestion-guide.md](references/data-ingestion-guide.md) |
| **Value** | BI / データサイエンス / ML のユースケース | [architecture-comparison.md](references/architecture-comparison.md) |
| **組織成熟度** | データ成熟度（Reactive / Informative / Predictive / Transformative） | [people-and-technology.md](references/people-and-technology.md) |

## Step 2: アーキテクチャ候補の絞り込み

ビジネス要件に基づき、6 種のアーキテクチャから候補を絞り込む。

### Decision Framework

```
START
  ↓
データの種類は？
  ├─ 構造化データのみ → RDW を検討
  │    └─ 規模は？
  │         ├─ 小〜中規模 → RDW
  │         └─ 大規模 → MDW or Data Lakehouse
  │
  ├─ 構造化 + 非構造化 → MDW / Data Lakehouse / Data Fabric
  │    └─ メタデータ駆動の自動化が必要？
  │         ├─ Yes → Data Fabric
  │         └─ No → MDW or Data Lakehouse
  │              └─ Delta Lake / Spark エコシステム？
  │                   ├─ Yes → Data Lakehouse
  │                   └─ No → MDW
  │
  └─ 大規模組織で分散チーム → Data Mesh を検討
       └─ 4原則を満たせるか？
            ├─ Yes → Data Mesh（Hub-and-Spoke 推奨）
            └─ No → 集中型 + ドメインオーナーシップ
```

### 6 種アーキテクチャの比較

| 項目 | RDW | Data Lake | MDW | Data Fabric | Data Mesh | Data Lakehouse |
|:---|:---|:---|:---|:---|:---|:---|
| **集中/分散** | 集中 | 集中 | 集中 | 集中 | 分散 | 集中 |
| **データ種別** | 構造化のみ | 全種別 | 全種別 | 全種別 | 全種別 | 全種別 |
| **スキーマ** | オンライト | オンリード | 両方 | 両方 | 両方 | 両方 |
| **レイテンシ** | バッチ | バッチ | バッチ+RT | バッチ+RT | バッチ+RT | バッチ+RT |
| **コスト** | 中〜高 | 低 | 中〜高 | 中〜高 | 高 | 低〜中 |
| **難易度** | 中 | 低〜中 | 中〜高 | 高 | 非常に高 | 中 |
| **最適規模** | 小〜中 | 中〜大 | 中〜大 | 大 | 大（分散組織） | 中〜大 |

### 各アーキテクチャの最適ユースケース

| アーキテクチャ | 最適なユースケース |
|:---|:---|
| **RDW** | 構造化データ中心のBI・レポーティング。小〜中規模 |
| **Data Lake** | 大量の非構造化データ保存。データサイエンス。低コスト |
| **MDW** | 構造化+非構造化の統合。多様な分析ニーズ |
| **Data Fabric** | メタデータ駆動の統合。セルフサービス。複数システムの仮想統合 |
| **Data Mesh** | 大規模組織。明確なドメイン境界。分散チーム |
| **Data Lakehouse** | DW+DLの統合。コスト最適化。Databricks/Spark エコシステム |

**チェックリスト**:
- [ ] Decision Framework に沿って候補を 1〜3 個に絞り込んだ
- [ ] 各候補のメリット・デメリットを評価した
- [ ] 組織の規模・成熟度と候補の適合性を確認した
- [ ] 段階的移行パスを検討した

## Step 3: ADS（Architecture Design Session）の実施

アーキテクチャ選定を構造的に行うための設計セッションを実施する。

### 3a. 準備フェーズ

| 準備項目 | 内容 |
|:---|:---|
| **参加者選定** | データアーキテクト、ビジネスステークホルダー、データエンジニア、DBA |
| **事前タスク** | 現行アーキテクチャの文書化、データソースインベントリ |
| **ホワイトボード準備** | 現状（As-Is）と将来（To-Be）のアーキテクチャ図 |

### 3b. 発見フェーズ

4 カテゴリ・32 問の質問テンプレートで要件を深掘りする（詳細は [ads-question-templates.md](references/ads-question-templates.md) を参照）。

| カテゴリ | 質問数 | 主な確認項目 |
|:---|:---|:---|
| **データソース** | 8問 | ソース数・種類・データ量・増加率・鮮度要件 |
| **処理要件** | 8問 | バッチ/リアルタイム・ETL/ELT・品質・MDM |
| **分析要件** | 8問 | レポーティング・DS/ML・セルフサービスBI・データ共有 |
| **非機能要件** | 8問 | セキュリティ・コンプライアンス・SLA・予算・スキルセット |

### 3c. セッション実施

- 推奨時間: 1日〜数日
- ホワイトボードで As-Is → To-Be の図を作成
- 各候補アーキテクチャの Pros/Cons を議論

### 3d. フォローアップ

- 推奨アーキテクチャの文書化
- 実装ロードマップの作成
- ステークホルダーへの最終プレゼンテーション

**チェックリスト**:
- [ ] 適切な参加者が選定されている
- [ ] データソースインベントリが完了している
- [ ] 4 カテゴリ（ソース / 処理 / 分析 / 非機能）の質問に回答を得た
- [ ] As-Is / To-Be の図が作成されている
- [ ] 推奨アーキテクチャの根拠が文書化されている

## Step 4: データモデリング手法の選定

選定したアーキテクチャに適したデータモデリング手法を決定する。

### 主要モデリング手法

| 手法 | 特徴 | 適用 |
|:---|:---|:---|
| **3NF（第三正規形）** | 冗長性排除、データ整合性重視 | EDW のステージング/統合層（Inmon アプローチ） |
| **ディメンショナル（スタースキーマ）** | クエリ性能重視、ビジネスユーザーに直感的 | プレゼンテーション層、データマート（Kimball アプローチ） |
| **Data Vault** | 柔軟性・監査性重視。Hub/Link/Satellite | 大規模 EDW のステージング、ソース変更が頻繁な環境 |
| **スキーマオンリード** | スキーマ未定義でデータ投入 | Data Lake の Raw 層 |

### Kimball vs Inmon

| 項目 | Kimball（ボトムアップ） | Inmon（トップダウン） |
|:---|:---|:---|
| **モデリング** | ディメンショナル（スタースキーマ） | 3NF（第三正規形） |
| **開始点** | データマートから → 統合 | EDW から → データマート派生 |
| **利点** | 迅速なデリバリー、直感的 | 一貫性、企業全体の統合ビュー |
| **課題** | データの重複可能性 | 初期投資大、デリバリー遅い |
| **推奨** | 小〜中規模、迅速な価値創出 | 大規模、データ統合が最重要 |

### ハイブリッドモデル（推奨）

実務ではハイブリッドが一般的:
- **ステージング層**: Raw データ（スキーマオンリード or 3NF）
- **統合層**: Data Vault or 3NF
- **プレゼンテーション層**: スタースキーマ（Kimball）

### SCD（Slowly Changing Dimensions）

| タイプ | 説明 | 履歴保持 | 用途 |
|:---|:---|:---|:---|
| **Type 0** | 変更なし（固定値） | 不変 | 生年月日等の不変属性 |
| **Type 1** | 上書き（古い値は消失） | なし | 修正データ、履歴不要の属性 |
| **Type 2** | 新行追加（有効期間で管理） | 完全 | 住所、ステータス等の変遷追跡 |
| **Type 3** | 現在値+前回値の 2 列 | 限定的 | 直近の変更のみ追跡 |

**チェックリスト**:
- [ ] アーキテクチャに適したモデリング手法を選定した
- [ ] Kimball / Inmon / ハイブリッドの選択根拠を明確にした
- [ ] SCD の戦略を決定した
- [ ] ファクト/ディメンションの粒度を定義した
- [ ] 命名規約を策定した

## Step 5: データ取り込み・処理パターンの設計

### ETL vs ELT

| 項目 | ETL | ELT |
|:---|:---|:---|
| **処理順序** | Extract → Transform → Load | Extract → Load → Transform |
| **変換場所** | 中間サーバー | ターゲットシステム内 |
| **適用** | RDW、オンプレミス | クラウド、Data Lake |
| **スケーラビリティ** | 中 | 高（MPPエンジン活用） |
| **推奨** | レガシー環境、複雑な変換 | クラウドネイティブ、大量データ |

### バッチ vs リアルタイム処理

| 項目 | バッチ処理 | リアルタイム処理 |
|:---|:---|:---|
| **頻度** | 定期（日次、時間次） | 連続（ストリーミング） |
| **レイテンシ** | 分〜時間 | ミリ秒〜秒 |
| **コスト** | 低 | 高 |
| **ツール例** | ADF, SSIS, Glue | Kafka, Event Hubs, Spark Streaming |

### アーキテクチャパターン

| パターン | 特徴 | 適用 |
|:---|:---|:---|
| **Lambda** | バッチ層 + スピード層の並列処理 | バッチ+リアルタイム両方が必要 |
| **Kappa** | ストリーム処理のみで統一 | リアルタイム中心、シンプルさ重視 |
| **Polyglot Persistence** | データ種別ごとに最適なストアを使い分け | 多様なデータ種別への対応 |

### Data Lake の層構造

```
Raw Layer（ブロンズ）→ Conformed（シルバー）→ Cleansed（ゴールド）→ Presentation
    ↓
  Sandbox（データサイエンス用）
```

**チェックリスト**:
- [ ] ETL / ELT の選択根拠を明確にした
- [ ] バッチ / リアルタイム処理の要件を定義した
- [ ] アーキテクチャパターン（Lambda / Kappa）を選定した
- [ ] Data Lake の層構造を定義した
- [ ] データガバナンス（品質チェック、リネージ）を組み込んだ

## Step 6: テクノロジースタックの選定

### プラットフォーム選択の指針

| 選択肢 | 推奨場面 | 注意点 |
|:---|:---|:---|
| **クラウド（PaaS推奨）** | 新規構築、スケーラビリティ必要 | ベンダーロックインの検討 |
| **オンプレミス** | 極低レイテンシ、規制要件、既存リース | スケーラビリティ限界、高コスト |
| **ハイブリッド** | 段階的移行、規制データの一部オンプレ保持 | 複雑性増大 |
| **マルチクラウド** | データ主権要件、CSP障害リスク分散 | コスト増大、相互運用性課題 |

### クラウドサービスモデル

| モデル | 管理範囲 | 推奨度 |
|:---|:---|:---|
| **IaaS** | OS 以上を顧客管理 | VM 必要時のみ |
| **PaaS** | アプリとデータのみ | **推奨**（DWH に最適） |
| **SaaS** | 設定のみ | 最も簡単。カスタマイズは限定的 |

### 主要クラウドプロバイダー比較

| 項目 | Azure | AWS | GCP |
|:---|:---|:---|:---|
| **市場シェア** | 23% | 32% | 10% |
| **分析製品** | Synapse Analytics, Fabric | Redshift, Athena | BigQuery |
| **強み** | Microsoft 連携 | サービス幅 | ML・データ分析 |
| **学習曲線** | Microsoft 製品経験者に易 | 多サービスで急勾配 | ユーザーフレンドリー |

### 主要ソフトウェアフレームワーク

| フレームワーク | 特徴 | 適用 |
|:---|:---|:---|
| **Databricks** | Spark ベースの統合分析プラットフォーム。Delta Lake。Data Lakehouse。パフォーマンス最適化（Data Skipping, Z-order, Caching 等） | バッチ+ストリーミング統合、ML |
| **Snowflake** | フルマネージド DWH。コンピュート/ストレージ分離 | DWH 中心、Data Lake 機能も |
| **Hadoop** | 分散処理フレームワーク。HDFS + MapReduce | レガシー大規模バッチ処理（採用減少傾向） |

**チェックリスト**:
- [ ] クラウド / オンプレミス / ハイブリッドの方針を決定した
- [ ] サービスモデル（IaaS / PaaS / SaaS）を選定した
- [ ] CSP を選定し、主要サービスをマッピングした
- [ ] 分析エンジン（Databricks / Snowflake 等）を選定した
- [ ] コストの概算見積もりを作成した

## Step 7: チーム構成と組織設計

### 集中型アーキテクチャのチーム構成

| ロール | 責任 | 必須度 |
|:---|:---|:---|
| **データアーキテクト** | 全体設計、技術選定 | 必須 |
| **データエンジニア** | パイプライン構築、ETL/ELT | 必須 |
| **データスチュワード** | データ品質・メタデータ管理 | 推奨 |
| **DBA** | DB管理、パフォーマンス最適化 | 必須 |
| **データアナリスト** | レポート、分析 | 必須 |
| **データサイエンティスト** | ML モデル、高度分析 | ユースケース依存 |
| **BA** | 要件定義、ビジネス翻訳 | 推奨 |
| **PM** | プロジェクト管理 | 必須 |

### Data Mesh のチーム構成

| ロール | 責任 |
|:---|:---|
| **ドメインチーム** | ドメイン固有のデータプロダクト開発・運用 |
| **データプロダクトオーナー** | データプロダクトのロードマップ・品質管理 |
| **プラットフォームチーム** | 共通インフラ・ツールの開発・提供 |
| **ガバナンスチーム** | 組織横断のポリシー策定・自動化 |

### プロジェクト失敗の主要落とし穴

| 落とし穴 | 防止策 |
|:---|:---|
| ビジネス要件の収集不足 | ADS の実施。ステークホルダーとの徹底対話 |
| データ品質の軽視 | 品質チェックをパイプラインに組み込む |
| 経験不足の実装パートナー | 実績確認。POC の実施 |
| ナレッジトランスファーの欠如 | トレーニング計画の策定 |
| プロジェクト途中の予算削減 | 現実的な予算計画。クリティカル領域の保護 |
| ソースデータ構造のそのまま反映 | ビジネスニーズに基づくモデリング |
| IT とビジネスのコミュニケーション不足 | 定期合同ミーティング。BAの活用 |

**チェックリスト**:
- [ ] 必要なロールを特定し、人材の確保計画を立てた
- [ ] データアーキテクチャに精通した人物が少なくとも 1 名いる
- [ ] プロジェクトチャンピオン（経営スポンサー）を確保した
- [ ] ナレッジトランスファー計画を策定した
- [ ] 80% 効率のプロジェクト計画（20% バッファ）を作成した

## Step 8: Data Mesh 適性評価（該当する場合）

Data Mesh の導入を検討する場合、以下の適性評価を実施する。

### Dehghani の 4 原則チェック

| 原則 | 確認項目 | 評価 |
|:---|:---|:---|
| **Domain Ownership** | ドメイン境界が明確か？ドメインチームにデータエンジニアリング能力があるか？ | Yes/No |
| **Data as a Product** | データプロダクト思考を各ドメインに定着させられるか？ | Yes/No |
| **Self-Serve Platform** | セルフサービスプラットフォームを構築・運用できるか？ | Yes/No |
| **Federated Governance** | 連合的ガバナンスのフレームワークを策定できるか？ | Yes/No |

### 組織適性アセスメント

| 評価項目 | 高適性 | 低適性 |
|:---|:---|:---|
| **組織規模** | 数百〜数千人。複数事業部 | 小規模。単一チーム |
| **ドメインの明確性** | ビジネスドメインが明確に定義 | ドメイン境界が曖昧 |
| **技術成熟度** | クラウドネイティブ、DevOps 文化 | レガシー中心 |
| **データ文化** | データドリブンな意思決定が浸透 | データ活用が限定的 |
| **人材** | 各ドメインにデータエンジニアがいる | 中央チームに集中 |

### 5 つの神話に注意

1. Data Mesh は**銀の弾丸ではない** → すべての組織に適用できるわけではない
2. Data Lake / DW を**置き換えるものではない** → 補完するもの
3. DW プロジェクトの失敗を**解決するものではない** → 根本原因は別
4. **すべてを分散化すべきではない** → Hub-and-Spoke が現実的
5. データ仮想化 = Data Mesh **ではない** → 仮想化は一要素

### 推奨: Hub-and-Spoke モデル

```
                ┌──────────────────┐
                │  Central Platform │
                │  (Hub)           │
                │  ・共通インフラ    │
                │  ・ガバナンス     │
                │  ・標準・テンプレート│
                └──────┬───────────┘
         ┌─────────────┼─────────────┐
    ┌────┴────┐   ┌────┴────┐   ┌────┴────┐
    │ Domain A │   │ Domain B │   │ Domain C │
    │ (Spoke)  │   │ (Spoke)  │   │ (Spoke)  │
    └──────────┘   └──────────┘   └──────────┘
```

**チェックリスト**:
- [ ] 4 原則の充足度を評価した
- [ ] 組織適性アセスメントを実施した
- [ ] 5 つの神話を理解し、現実的な期待値を設定した
- [ ] Hub-and-Spoke モデルの実現可能性を検討した
- [ ] 段階的導入のロードマップを作成した

### Data Architecture Slop 防止チェック（Distributional Convergence 対策）

データアーキテクチャ設計は LLM が最も均質な出力を生みやすい領域の一つである。指示がなければ「とりあえず Medallion Architecture + Delta Lake + Star Schema」という安全な組み合わせに収束し、ビジネス要件・データ特性・組織成熟度を無視した汎用設計になる。

| # | Data Architecture Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| DA-1 | **Medallion Architecture の思考停止採用** | ユースケースを問わず Bronze/Silver/Gold を提案 | 6 Vs 分析結果から最適な層構造を導出。Kappa / Lambda / Hub-and-Spoke も候補に含める |
| DA-2 | **Data Mesh の流行追従** | チーム成熟度・組織文化を評価せずに Data Mesh を推奨 | Step 8 の適性評価を実施。Hub-and-Spoke → Data Fabric → Data Mesh の段階的移行を検討 |
| DA-3 | **Star Schema 一辺倒のモデリング** | 全テーブルを事実テーブル + 次元テーブルに押し込む | データの性質に応じて 3NF / Data Vault / CDM / ワイドテーブルを使い分ける |
| DA-4 | **ETL vs ELT の表面的選択** | 最新トレンドだからと ELT を無条件推奨 | データ量・変換複雑度・レイテンシ要件・既存インフラから判断。ハイブリッドも選択肢 |
| DA-5 | **業種非依存のガバナンス設計** | GDPR / HIPAA / 金融規制等の業種固有要件を無視 | 業界固有のコンプライアンス要件を Step 1 で明確化し、ガバナンス設計に直接反映 |
| DA-6 | **技術スタック推奨の固定化** | 毎回同じ組み合わせ（Databricks + Delta Lake + Unity Catalog）を推奨 | ADS の結果とチームのスキルセットから技術を選定。Snowflake / BigQuery / Fabric も候補に |
| DA-7 | **汎用的なデータ品質ルール** | 完全性・正確性・一貫性の 3 項目で終わる | ビジネスドメイン固有の品質ルール（例: 金融の T+1 整合性、医療の患者 ID 一意性）を定義 |
| DA-8 | **3 層構造図のコピペ** | どの案件でも同じ Source → Processing → Serving の図 | データの流れ・変換ポイント・消費パターンを案件固有に描く。リアルタイム層・ML 層の有無も反映 |

> **核心原則**: **アーキテクチャは処方箋であり、市販薬ではない** — 患者（ビジネス）の症状（要件）を診断してから処方する。「別の業種・別の規模の企業に同じ設計図を渡しても違和感がないか？」→ 違和感がないなら Data Architecture Slop である。

**最終チェックリスト**:
- [ ] 9 ステップ（要件 → 絞込み → ADS → モデリング → 取込み → 技術 → チーム → Mesh評価 → 出力）を実施した
- [ ] ビジネス要件に基づく選定根拠が明確である
- [ ] 比較評価が多軸で実施されている
- [ ] 実装ロードマップが段階的に策定されている
- [ ] リスクと緩和策が文書化されている
- [ ] ステークホルダーへの説明資料が準備されている
- [ ] Data Architecture Slop チェック（DA-1〜DA-8）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| アーキテクチャが決まらない | 要件が曖昧 | Step 3 の ADS を実施し、質問リストで要件を明確化 |
| Data Lake がデータスワンプ化 | ガバナンス不足 | メタデータカタログ導入、層構造の厳格な運用 |
| DWH のパフォーマンスが悪い | モデリング or インデックス不足 | Step 4 のスタースキーマ最適化、パーティショニング |
| Data Mesh 導入が進まない | 組織の準備不足 | Step 8 の適性評価。Hub-and-Spoke で段階的に |
| ETL が遅すぎる | バッチサイズ or 変換の非効率 | ELT への移行、増分ロードの導入 |
| コストが想定を超過 | リソースの過剰プロビジョニング | PaaS 活用、自動スケーリング、リザーブドインスタンス |
| チーム間の連携が悪い | IT とビジネスの溝 | BA の配置、定期合同ミーティング |
| Kimball vs Inmon で議論が終わらない | 二者択一の誤解 | ハイブリッドモデルの提案。各層で最適手法を使い分け |
| リアルタイム要件が後から追加された | 初期要件の見落とし | Lambda Architecture で既存バッチ層に Speed 層を追加 |
| ベンダーロックインが心配 | 特定 CSP への過度な依存 | オープンフォーマット（Parquet, Delta）の採用 |
