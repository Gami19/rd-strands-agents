# Data Modeling Guide Reference

> データモデリング手法の詳細リファレンス

## 1. モデリング手法の全体像

```
ソース層           統合/ステージング層        プレゼンテーション層
（スキーマオンリード） （3NF / Data Vault）     （スタースキーマ）
                                              ↓
                                           BI / 分析
```

## 2. リレーショナルモデリング（3NF）

### 正規化の段階

| 段階 | ルール | 排除するもの |
|:---|:---|:---|
| **1NF** | 各列にアトミック値のみ | 繰り返しグループ |
| **2NF** | 1NF + 非キー列が主キー全体に完全依存 | 部分関数従属性 |
| **3NF** | 2NF + 非キー列間の従属性なし | 推移的関数従属性 |

### 3NF の適用

- **用途**: EDW の統合層（Inmon アプローチ）、OLTP システム
- **利点**: データの一貫性、冗長性排除、更新アノマリー防止
- **欠点**: 結合（JOIN）が多くクエリ性能に影響、ビジネスユーザーに直感的でない

## 3. ディメンショナルモデリング（Kimball）

### スタースキーマ

中心にファクトテーブル、周囲にディメンションテーブルが放射状に接続する構造。

```
          ┌──────────┐
          │ Dim_Date  │
          └─────┬────┘
                │
┌──────────┐  ┌─┴──────────┐  ┌──────────┐
│Dim_Product├──┤ Fact_Sales  ├──┤Dim_Store │
└──────────┘  └─┬──────────┘  └──────────┘
                │
          ┌─────┴────┐
          │Dim_Customer│
          └──────────┘
```

### ファクトテーブル

| 種類 | 説明 | 例 |
|:---|:---|:---|
| **Transaction Fact** | 個々のトランザクション | 売上明細 |
| **Periodic Snapshot** | 定期的なスナップショット | 月末在庫残高 |
| **Accumulating Snapshot** | プロセスの進捗 | 注文ライフサイクル |

### ディメンションテーブル

主要なディメンション:
- **Time（時間）**: 日付、月、四半期、年
- **Product（製品）**: カテゴリ、サブカテゴリ、ブランド
- **Customer（顧客）**: 地域、セグメント、属性
- **Geography（地理）**: 国、地域、都市
- **Organization（組織）**: 部門、チーム

### SCD（Slowly Changing Dimensions）

| タイプ | 方法 | 履歴 | 実装の複雑さ |
|:---|:---|:---|:---|
| **Type 0** | 変更なし（固定値） | 不変 | 最低 |
| **Type 1** | 上書き | なし | 低 |
| **Type 2** | 新行追加（有効開始日/終了日 + 現在フラグ） | 完全 | 中 |
| **Type 3** | 現在値 + 前回値の 2 列 | 直近のみ | 低〜中 |

### スノーフレークスキーマ

スタースキーマのディメンションテーブルをさらに正規化した構造。
- **利点**: ストレージ効率
- **欠点**: 結合増加によるクエリ性能低下
- **実務**: スタースキーマが一般的に推奨される

## 4. Data Vault

Dan Linstedt が考案した柔軟なモデリング手法。

### 3つの構成要素

| 要素 | 内容 | 例 |
|:---|:---|:---|
| **Hub** | ビジネスキー + ロードメタデータ | Customer Hub（顧客ID） |
| **Link** | Hub 間のリレーションシップ | Customer-Order Link |
| **Satellite** | コンテキスト情報 + 履歴 | Customer Satellite（名前、住所、変更履歴） |

### Data Vault の利点

- **柔軟性**: ソースシステムの変更に強い
- **監査性**: 全データの来歴と変更履歴を完全保持
- **スケーラビリティ**: 新規ソースの追加が容易
- **並列ロード**: Hub/Link/Satellite を独立してロード可能

### Data Vault の適用

- **用途**: 大規模 EDW のステージング/統合層
- **組合せ**: Data Vault（統合層）+ スタースキーマ（プレゼンテーション層）

## 5. Kimball vs Inmon

### 詳細比較

| 項目 | Kimball（ボトムアップ） | Inmon（トップダウン） |
|:---|:---|:---|
| アプローチ | データマートから → 統合 | EDW から → データマート |
| モデリング | ディメンショナル（スタースキーマ） | 3NF（第三正規形） |
| デリバリー速度 | 速い（週〜月単位で価値創出） | 遅い（EDW 構築に数ヶ月〜） |
| データの一貫性 | 低〜中（個別のコンフォーマンスに依存） | 高（統合 EDW が single source of truth） |
| 初期投資 | 低〜中 | 高 |
| ユーザー親和性 | 高い（直感的なスタースキーマ） | 低い（3NF は技術者向け） |
| スケーラビリティ | 高（データマートの追加） | 高（EDW の拡張） |
| メンテナンス | 中〜高（コンフォーマンス維持） | 中（EDW の一元管理） |

### 5つの神話の解体

1. **「Kimball はデータマートだけ」** → Kimball も EDW を作る（Bus Architecture）
2. **「Inmon は 3NF だけ」** → Inmon もプレゼンテーション層にスタースキーマを使う
3. **「Kimball はスケールしない」** → Conformed Dimensions で企業規模にスケール
4. **「Inmon は遅すぎる」** → 初期は遅いが長期的には整合性の恩恵
5. **「二者択一」** → ハイブリッドが実務の標準

### ハイブリッドモデル（推奨）

```
ソース → ステージング → 統合層（3NF or Data Vault）→ プレゼンテーション層（スタースキーマ）→ BI
                                    ↕
                              データマート
```

## 6. CDM（Common Data Model）

業界標準のデータモデル。Microsoft CDM 等。

- エンティティの標準定義を提供
- 異なるシステム間のデータ統合を容易にする
- 業界固有のテンプレート（ヘルスケア、金融等）
