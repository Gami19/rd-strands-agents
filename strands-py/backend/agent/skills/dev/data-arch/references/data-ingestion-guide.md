# Data Ingestion Guide Reference

> データ取り込み・処理パターンの詳細リファレンス

## 1. ETL vs ELT

### 詳細比較

| 項目 | ETL | ELT |
|:---|:---|:---|
| 正式名 | Extract, Transform, Load | Extract, Load, Transform |
| 処理順序 | 抽出 → 変換 → ロード | 抽出 → ロード → 変換 |
| 変換場所 | 中間 ETL サーバー | ターゲット DWH / Data Lake 内 |
| スケーラビリティ | 中間サーバーに制約 | MPP エンジン活用で高スケーラビリティ |
| 適用環境 | オンプレミス、RDW | クラウド、Data Lake、Data Lakehouse |
| データ量 | 中規模まで効率的 | 大規模データに適合 |
| 変換の柔軟性 | ロード前に変換（品質保証） | ロード後に変換（生データも保持） |
| 主要ツール | SSIS, Informatica, Talend | dbt, Spark SQL, Azure Synapse |

### ELT の推奨理由（クラウド環境）

1. **スケーラビリティ**: クラウド DWH の MPP エンジンで大量データを高速変換
2. **生データ保持**: Raw データを保持できるため、後から変換ロジックの変更が可能
3. **コスト**: 中間 ETL サーバーが不要
4. **シンプルさ**: 抽出とロードを統合し、変換を SQL ベースで実行

### Reverse ETL

データウェアハウスからオペレーショナルシステムへデータを逆流させるプロセス。

**ユースケース**:
- DWH のインサイトを CRM（Salesforce 等）に還元
- ML モデルの予測結果を運用システムに反映
- 顧客セグメントをマーケティングツールに配信

**主要ツール**: Census, Hightouch, Airbyte

## 2. バッチ処理 vs リアルタイム処理

### 詳細比較

| 項目 | バッチ処理 | マイクロバッチ | リアルタイム処理 |
|:---|:---|:---|:---|
| 頻度 | 日次〜時間次 | 分次 | 連続（ストリーミング） |
| レイテンシ | 分〜時間 | 秒〜分 | ミリ秒〜秒 |
| 複雑性 | 低 | 中 | 高 |
| コスト | 低 | 中 | 高 |
| データ量 | 大量の一括処理 | 中量の頻繁処理 | 少量の連続処理 |
| エラーハンドリング | リトライが容易 | リトライ可能 | 複雑（順序保証等） |

### バッチ処理ツール

| ツール | プラットフォーム | 特徴 |
|:---|:---|:---|
| Azure Data Factory | Azure | ノーコード/ローコード ETL |
| AWS Glue | AWS | サーバーレス ETL |
| Apache Airflow | OSS | ワークフローオーケストレーション |
| dbt | OSS | SQL ベースの変換 |

### リアルタイム処理ツール

| ツール | プラットフォーム | 特徴 |
|:---|:---|:---|
| Apache Kafka | OSS | 分散メッセージキュー |
| Azure Event Hubs | Azure | マネージド Kafka 互換 |
| Amazon Kinesis | AWS | マネージドストリーミング |
| Spark Structured Streaming | Databricks | マイクロバッチ + ストリーミング |
| Apache Flink | OSS | 低レイテンシストリーム処理 |

## 3. アーキテクチャパターン

### Lambda Architecture

```
                    ┌─────────────────┐
                    │   Data Source    │
                    └──┬──────────┬───┘
                       │          │
              ┌────────┴──┐   ┌──┴────────┐
              │ Batch Layer│   │Speed Layer │
              │ (全データ   │   │(リアルタイム│
              │  バッチ処理)│   │ ストリーム) │
              └────────┬──┘   └──┬────────┘
                       │          │
                    ┌──┴──────────┴───┐
                    │  Serving Layer   │
                    │ (統合ビュー)      │
                    └──────────────────┘
```

**利点**: バッチの正確性 + リアルタイムの速度を両立
**欠点**: 2つのコードベースの維持が必要（バッチ + ストリーム）

### Kappa Architecture

```
                    ┌─────────────────┐
                    │   Data Source    │
                    └──────┬──────────┘
                           │
                    ┌──────┴──────────┐
                    │  Stream Layer    │
                    │ (全データを       │
                    │  ストリームとして  │
                    │  処理)           │
                    └──────┬──────────┘
                           │
                    ┌──────┴──────────┐
                    │  Serving Layer   │
                    └──────────────────┘
```

**利点**: シンプル（単一コードベース）
**欠点**: 大量の履歴データの再処理が難しい場合がある

### 選定ガイド

| 条件 | 推奨パターン |
|:---|:---|
| バッチ + リアルタイム両方必要 | Lambda |
| リアルタイム中心、シンプルさ重視 | Kappa |
| バッチのみ | 従来型 ETL/ELT |
| 多様なデータストア | Polyglot Persistence |

## 4. RDW へのデータ投入方法

| 方法 | 仕組み | 適用 |
|:---|:---|:---|
| **タイムスタンプ** | 更新日時カラムで差分検出 | 最もシンプル。更新日時が信頼できる場合 |
| **CDC** | トランザクションログから変更検出 | リアルタイム性が必要。データ量が多い場合 |
| **パーティショニング** | テーブル分割で並列処理 | 大量データの一括投入 |
| **トリガー** | DB トリガーで変更検知 | 即時反映。ただし性能影響に注意 |

## 5. データガバナンス

### 成熟度モデル

| レベル | 名称 | 特徴 |
|:---|:---|:---|
| 1 | **Initial** | ガバナンスなし。アドホック対応 |
| 2 | **Managed** | 基本ポリシー策定。一部チームで実施 |
| 3 | **Defined** | 組織横断のガバナンスプログラム |
| 4 | **Measured** | メトリクスによる品質管理。KPI 設定 |
| 5 | **Optimized** | 継続的改善。自動化されたポリシー適用 |

### データ品質の6次元

| 次元 | 説明 |
|:---|:---|
| **完全性** | 必要なデータが欠損していないか |
| **正確性** | データが正確で信頼できるか |
| **一貫性** | 複数ソース間でデータが矛盾していないか |
| **適時性** | データが十分に最新か |
| **一意性** | 重複データがないか |
| **有効性** | データが定義されたルール・形式に準拠しているか |

### データカタログ

- メタデータの管理と検索
- データリネージ（データの来歴追跡）
- データディクショナリ（用語と定義の管理）
- ビジネスユーザー向けのセルフサービス発見

**主要ツール**: Azure Purview, AWS Glue Data Catalog, Google Data Catalog, Apache Atlas
