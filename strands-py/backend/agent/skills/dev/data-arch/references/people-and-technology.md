## 2. プロジェクト失敗の 12 の落とし穴

| # | 落とし穴 | 防止策 |
|:---|:---|:---|
| 1 | **ビジネス要件の収集不足** | ADS の実施。ステークホルダーとの徹底的な対話。要件を文書化 |
| 2 | **データ品質の軽視** | データ品質チェックをパイプラインに組み込む。品質ダッシュボード |
| 3 | **経験不足のコンサル雇用** | 実績とリファレンスの確認。POC で実力を検証 |
| 4 | **オフショア開発への丸投げ** | タイムゾーン・言語の障壁を事前評価。品質管理の仕組み |
| 5 | **プロジェクト所有権のコンサル委譲** | 内部 PM を配置。進捗レポートの透明性確保 |
| 6 | **ナレッジトランスファーの欠如** | トレーニング計画を契約に含める。内部チームの OJT |
| 7 | **プロジェクト途中の予算削減** | 現実的な予算計画。クリティカル領域の保護。段階的デリバリー |
| 8 | **厳格なデッドラインへの固執** | 柔軟なスケジュール。アジャイルアプローチの採用 |
| 9 | **ソースデータ構造のそのまま反映** | ビジネスニーズに基づくモデリング。ソース依存の排除 |
| 10 | **パフォーマンス問題の放置** | 移行前のベンチマーク。パフォーマンステストの義務化 |
| 11 | **アーキテクチャの過剰/過少設計** | バランスの取れた設計。段階的拡張。YAGNI 原則 |
| 12 | **IT とビジネスのコミュニケーション不足** | BA の配置。定期合同ミーティング。共通用語の整理 |

## 3. 成功のためのヒント

### 投資を惜しまない
- 長期的な ROI を重視
- データアーキテクチャに精通した人材を少なくとも 1 名確保
- ツールの選定にも投資（安物買いの銭失い防止）

### ユーザーを巻き込む
- 早期からエンドユーザーを参加させる
- アジャイルアプローチで頻繁にリリース
- 「クイックウィン」で成果を見せる

### 新しいレポートで付加価値を
- 既存レポートの単純再作成ではなく新しいインサイトを提供
- KPI ダッシュボードなど未体験の価値を見せる

### プロトタイプを構築させる
- エンドユーザーに BI ツールのプロトタイプを作らせる
- 「百聞は一見に如かず」の体験を提供

### プロジェクトチャンピオンを見つける
- 経営層のスポンサーを確保
- 予算確保、部門間調整、変革推進の支援者

### 80% 効率のプロジェクト計画
- チームメンバーの稼働を 80% で計画
- 20% は休暇、会議、学習、予期せぬ事態のバッファ

## 4. テクノロジー選定ガイド

### プラットフォーム選択

| 条件 | 推奨 |
|:---|:---|
| 新規構築 | クラウド（PaaS） |
| 既存オンプレの移行 | クラウド PaaS（段階的移行） |
| 極低レイテンシ要件 | オンプレミス |
| 規制要件でクラウド不可 | オンプレミス or プライベートクラウド |
| データ主権要件 | マルチクラウド or リージョン指定 |

### クラウドサービスモデル選定

| モデル | 管理範囲 | 適用 |
|:---|:---|:---|
| **IaaS** | OS, ミドルウェア, アプリ, データ | 最大の制御が必要。レガシーアプリの移行 |
| **PaaS** | アプリ, データ | **推奨**。DWH に最適。VM 管理不要 |
| **SaaS** | 設定のみ | 最も簡単。カスタマイズは限定的 |

PaaS が DWH に推奨される理由:
1. VM 管理不要
2. パッチ適用が自動
3. DR 設定が容易
4. バックアップが自動
5. スケーリングが容易

### ソフトウェアフレームワーク

#### Hadoop
- 2005年 Apache Foundation 開発
- HDFS（分散ファイルシステム）+ MapReduce（分散処理）+ YARN（リソース管理）
- エコシステム: Hive, Pig, HBase, Spark, Presto
- **現状**: 採用減少傾向。Spark やクラウドネイティブに移行

#### Databricks
- Apache Spark ベースの統合データ分析プラットフォーム
- Delta Lake（ACID トランザクション、タイムトラベル、スキーマ強制）
- Data Lakehouse アーキテクチャの推進者
- MLlib, GraphX, Spark Streaming のサポート
- Python, Java, Scala, SQL, R 対応

#### Snowflake
- フルマネージドのクラウド DWH
- コンピュートとストレージの完全分離（独立スケーリング）
- Data Lake としても機能（CSV, JSON, Parquet 等を直接取り込み）
- 自動スキーマ推論
- Data Mesh 対応（ドメイン別アカウント/仮想 WH）

### マルチクラウド

**利点**:
- 各 CSP の強みを活用
- データ主権対応
- ベンダーロックインの軽減

**欠点**:
- 運用複雑性の増大
- コスト増加（人材、ツール、転送料）
- 相互運用性の課題
- セキュリティリスクの増大

**著者の見解**: ほとんどの企業は単一 CSP に集中し、PaaS/SaaS を最大活用すべき。マルチクラウドのメリットは限定的。
