# Jeff Patton 型ユーザーストーリーマッピング 方法論リファレンス

> SKILL.md から Progressive Disclosure で分離した理論的背景と実践ガイドです。

---

## 1. ユーザーストーリーマッピングとは

Jeff Patton が提唱したユーザーストーリーマッピングは、
**フラットなバックログに「物語」と「構造」を与える手法** である。

従来のフラットバックログの問題:
- 全体像が見えない（木を見て森を見ず）
- 優先度を付けても「何を捨てたか」が分からない
- リリース計画が機能ベースになり、ユーザー価値が断片化する

ストーリーマッピングの解決策:
- **横軸（左→右）**: ユーザーの活動を時系列で並べ、「物語」を形成する
- **縦軸（上→下）**: 各活動の詳細度・優先度で並べ、「深さ」を表現する
- **水平スライス**: リリース単位で横に切り、「段階的な価値提供」を計画する

---

## 2. Patton の3層構造

ストーリーマップは3つの階層で構成される:

```
レベル1: アクティビティ（Activities）
  ├── ユーザーの大目的。ストーリーマップの「背骨（Backbone）」を形成する。
  ├── 例: 「商品を探す」「購入する」「配送を追跡する」
  └── 粒度の目安: 「〜するために、まとまった時間をかける活動」

レベル2: ユーザータスク（User Tasks）
  ├── アクティビティを達成するための具体的な手順。
  ├── 例: 「キーワードで検索する」「カートに入れる」「配送状況を確認する」
  └── 粒度の目安: 「1回の操作セッションで完了する手順」

レベル3: ユーザーストーリー（User Stories）
  ├── タスクの具体的な実現方法。最も詳細な粒度。
  ├── 例: 「カテゴリで絞り込む」「価格帯で並び替える」
  └── 粒度の目安: 「1スプリントで実装・テスト可能な単位」
```

### 各レベルの判別基準

| 質問 | アクティビティ | タスク | ストーリー |
|:---|:---|:---|:---|
| 完了に要する時間は？ | 数日〜数週間 | 数分〜数時間 | 数時間〜数日（開発工数） |
| 独立して価値を提供？ | 他のアクティビティと連携が必要 | アクティビティ内で意味を持つ | 単体でテスト可能 |
| ユーザーの意識は？ | 「今日は〜をする」 | 「次に〜をやる」 | 「〜という方法で」 |
| 分解可能？ | タスクに分解 | ストーリーに分解 | これ以上分解すると価値が失われる |

## 3. Walking Skeleton（歩く骸骨）

Walking Skeleton は Alistair Cockburn が提唱し、Patton がストーリーマッピングに統合した概念である。

**定義**: システムの主要なアーキテクチャ要素を薄くエンドツーエンドで貫通する、
最小限の実装。「骨格だけだが、歩くことはできる」状態。

### Walking Skeleton の特徴

1. **エンドツーエンド**: すべてのアクティビティを横断する
2. **最小限**: 各アクティビティから最も基本的なタスクを1つだけ選ぶ
3. **動作する**: デプロイして実際に動作させることができる
4. **アーキテクチャを検証**: 技術的なリスクを早期に検出できる

### Walking Skeleton と MVP の違い

| 項目 | Walking Skeleton | MVP |
|:---|:---|:---|
| 目的 | アーキテクチャの検証 | ビジネス仮説の検証 |
| 対象 | 開発チーム内部 | エンドユーザー |
| 品質 | 動作するが粗い | ユーザーが使える品質 |
| 範囲 | 最小限の機能 | 最小限の「価値ある」機能 |

KC-Prop-Foundry のストーリーマップでは、Walking Skeleton を基盤として
MVP（R1）を構築する。R1 = Walking Skeleton + ユーザーが実際に業務を遂行できるストーリー群。

## 4. Slice the Cake（ケーキの切り方）原則

Patton の最も重要な原則の1つ。リリース計画を「縦切り」ではなく「横切り」で行う。

### 正しい切り方（横切り = Slice the Cake）

```
         ACT-001     ACT-002     ACT-003     ACT-004
         ─────────────────────────────────────────────
R1       [薄い層]    [薄い層]    [薄い層]    [薄い層]    ← 全アクティビティを横断
         ─────────────────────────────────────────────
R2       [次の層]    [次の層]    [次の層]    [次の層]    ← さらに厚みを追加
         ─────────────────────────────────────────────
R3       [最後の層]  [最後の層]  [最後の層]  [最後の層]  ← 完成へ
```

### 間違った切り方（縦切り = Feature-by-Feature）

```
         ACT-001     ACT-002     ACT-003     ACT-004
         ─────────────────────────────────────────────
R1       [完全実装]                                      ← 1つのアクティビティだけ完璧
         ─────────────────────────────────────────────
R2                   [完全実装]                           ← 次のアクティビティ
         ─────────────────────────────────────────────
R3                               [完全実装]  [完全実装]   ← 残り
```

**縦切りの問題**:
- R1 だけでは業務が完結しない（データは入力できるが分析・レポートはできない）
- ユーザーから「使えない」と言われるリスクが高い
- 後半のリリースまで価値が提供できない

**横切りの利点**:
- R1 の時点で「粗いが一通り使える」状態になる
- 各リリースで独立した価値を提供できる
- ユーザーフィードバックを早期に得られる

## 5. 提案書への活用

KC-Prop-Foundry のワークフローにおいて、ストーリーマップは提案書の以下の要素に直結する:

### 5-1: スケジュール根拠

ストーリーマップのリリーススライス（R1/R2/R3）が、提案書のフェーズ分割の根拠となる。

```
ストーリーマップ        提案書
R1 (MVP)           →   Phase 1: 基本機能構築（3ヶ月）
R2 (Growth)        →   Phase 2: 機能拡張（3ヶ月）
R3 (Maturity)      →   Phase 3: 高度化（3ヶ月）
```

### 5-2: 見積根拠

ストーリーの規模（S/M/L/XL）をリリースごとに集計し、工数見積の根拠とする。

```
規模の目安:
S = 1〜2人日  → 小さなストーリー（画面の微修正、バリデーション追加等）
M = 3〜5人日  → 標準的なストーリー（CRUD 画面、基本機能）
L = 5〜10人日 → 大きなストーリー（外部連携、複雑なビジネスロジック）
XL = 10人日〜 → 要分割（エピック相当、このままでは見積困難）
```

### 5-3: 価値訴求

ストーリーマップの物語構造が、提案書の「なぜこの順序で開発するのか」の説得力を生む。

- DRM Phase 2 で「この MVP で顧客は何ができるか」を明確に説明できる
- 競合他社の「機能リスト型」提案に対し、「ユーザー価値型」提案で差別化できる

## 6. アンチパターン集

### AP-1: 「全部入り症候群」

**症状**: R1（MVP）にほぼ全ストーリーを詰め込んでしまう。
**原因**: 顧客の「全部必要」という要求をそのまま受け入れている。
**対処**: 「すべて作りますが、段階的に届けます」と説明する。R1 で業務が回る最小限を合意する。

### AP-2: 「技術タスクの混入」

**症状**: 「DBを設計する」「APIを実装する」がストーリーとして並んでいる。
**原因**: 開発者視点でマッピングしている。
**対処**: すべてのストーリーを「〜として、〜したい。なぜなら〜」で記述し直す。技術タスクはストーリーの実装詳細に移す。

### AP-3: 「縦切りリリース」

**症状**: R1 で「ユーザー管理だけ完璧に作る」計画になっている。
**原因**: Slice the Cake 原則の未適用。
**対処**: 各リリースが全アクティビティを横断しているかチェックする。

### AP-4: 「ペルソナなきストーリー」

**症状**: ストーリーの「〜として」が省略され、誰のためのストーリーか不明。
**原因**: ペルソナ定義をスキップしている。
**対処**: Step 1 のペルソナ定義に戻り、全ストーリーにペルソナを紐付ける。

### AP-5: 「受入基準なき承認」

**症状**: ストーリーに受入基準がなく、「正しく動作すること」としか書かれていない。
**原因**: 「完了」の定義が曖昧。
**対処**: 具体的な条件（入力値、期待出力、エラー時の動作）を受入基準に記述する。

### AP-6: 「巨大ストーリー」

**症状**: XL（10人日超）のストーリーが複数ある。
**原因**: 分解が不十分。
**対処**: INVEST の S 基準に従い、受入基準ごとに分割する。1ストーリー = 1つの受入基準セット。

### AP-7: 「バックボーンなき詳細化」

**症状**: 全体の流れ（バックボーン）を定義する前に、個別ストーリーを細かく書き始める。
**原因**: 全体像の把握をスキップしている。
**対処**: 必ず Step 2（バックボーン構築）→ Step 3（タスク）→ Step 4（ストーリー）の順に進める。

## 7. PO 実務との統合（認定プロダクトオーナー研修知見）

KC-Prop-Foundry のストーリーマッピングは、提案書作成のためのツールであるが、
その本質は PO（プロダクトオーナー）実務と同じ構造を持つ。

### 7-1: 優先順位付けの複数レンズ

ストーリーマップ内のストーリー優先順位付けには、以下の複数レンズを併用する:

| レンズ | 用途 | ストーリーマップでの適用 |
|:---|:---|:---|
| **価値 x 労力（4象限）** | 高価値・低労力から着手 | R1 に入れるストーリーの選定 |
| **MoSCoW** | Must/Should/Could/Won't で投資配分 | リリーススライスの境界線を引く |
| **狩野モデル** | 当たり前/性能/魅力品質のバランス | MVP に「当たり前品質 + 少しの魅力品質」を入れる |
| **WSJF** | 遅延コスト / ジョブサイズ | 複数ストーリーの定量的な順序付け |

### 7-2: 狩野モデルとMVP設計

MVP（R1）の設計では、狩野モデルの3品質を意識的にバランスさせる:

- **当たり前品質**: なければ不満。セキュリティ、基本認証、データ保全等。これがないと使えない。
- **性能品質**: 多いほど満足度が上がる。検索速度、レポート種類等。段階的に追加可能。
- **魅力品質**: 期待を超える驚き。ダッシュボード自動生成、AI分析等。差別化要素。

```
R1（MVP）= 当たり前品質（Must）100% + 性能品質（Should）一部 + 魅力品質（Could）少し
R2 = 性能品質の拡充 + 魅力品質の追加
R3 = 魅力品質の本格化 + 性能品質の完成
```

### 7-3: PBI の縦割りスライシング

ストーリーが大きすぎる（XL）場合の分割は、「データ/ルール/ターゲットを絞る」方法が有効:

- **データを絞る**: 全データではなく、主要なデータタイプのみ
- **ルールを絞る**: 全ビジネスルールではなく、最重要ルールのみ
- **ターゲットを絞る**: 全ユーザーではなく、主要ペルソナのみ
- **地域/範囲を絞る**: 全拠点ではなく、パイロット拠点のみ

いずれの場合も「価値は残る」ことが条件。価値が残らない分割は技術タスク分割であり、
ユーザーストーリーの分割としては不適切。

### 7-4: 「No」の力

すべてのストーリーを R1 に入れてはならない。PO にとって最も重要な言葉は「No」である。
ストーリーマップは「何をやるか」だけでなく、「何を（今は）やらないか」を可視化するツール。
