# ストーリーマッピング ファシリテーションガイド

> ワークショップの準備から MVP スライシングまで、ストーリーマッピングセッションを成功させるためのファシリテーション実践リファレンス

---

## 1. ワークショップ準備チェックリスト

### 事前準備（1-2 週間前）

| 項目 | 内容 | 完了 |
|:---|:---|:---|
| **目的の明確化** | ワークショップのゴールと成果物を定義 | [ ] |
| **参加者の選定** | PO・開発者・デザイナー・ステークホルダーの招集 | [ ] |
| **事前資料の配布** | inbox 資料のサマリー、ドメイン用語集、ビジョンステートメント案 | [ ] |
| **タイムボックスの設計** | 全体時間と各ステップの配分を決定 | [ ] |
| **ツールの準備** | 付箋・マーカー・模造紙（対面）/ Miro・FigJam（リモート） | [ ] |
| **ファシリテーターの確認** | メインファシリテーター + 書記の役割分担 | [ ] |

### 参加者の役割定義

| 役割 | 人数 | 主な責任 | ワークショップでの動き |
|:---|:---|:---|:---|
| **プロダクトオーナー (PO)** | 1 名 | ビジョンの承認、優先順位の最終判断 | ビジョンステートメントを提示、スライシング時の判断 |
| **ドメインエキスパート** | 1-2 名 | 業務知識の提供、用語の正確性確認 | フィーチャー/エピック洗い出し時に業務視点を補完 |
| **開発者** | 2-4 名 | 技術的実現可能性の評価、規模見積もり | ストーリー分解、規模感の評価、技術制約の指摘 |
| **デザイナー** | 1 名 | ユーザー体験の視点、ペルソナの具体化 | ペルソナ定義、ユーザージャーニーの補強 |
| **ステークホルダー** | 1-2 名 | ビジネス制約・予算・スケジュールの共有 | リリース計画へのインプット、優先度の合意 |
| **ファシリテーター** | 1 名 | 進行管理、議論の整理、合意形成の促進 | 全ステップの進行、停滞時の介入、時間管理 |
| **書記** | 1 名 | 議論の記録、成果物のデジタル化 | 付箋の内容を記録、写真撮影、Markdown 化 |

### 推奨タイムボックス

| ステップ | 時間 | 内容 |
|:---|:---|:---|
| アイスブレイク + 目的共有 | 15 分 | ゴールの確認、グラウンドルールの設定 |
| Step 1: ペルソナ抽出 | 30-45 分 | ペルソナの洗い出しと優先度付け |
| Step 2: ビジョンステートメント | 20-30 分 | ビジョンの策定と合意 |
| 休憩 | 10 分 | — |
| Step 3: フィーチャー洗い出し | 45-60 分 | バックボーンの構築 |
| Step 4: エピック分解 | 30-45 分 | 各フィーチャー下のエピック配置 |
| 昼休憩（終日の場合） | 60 分 | — |
| Step 5: ストーリー分解 | 60-90 分 | エピックからストーリーへの分解 |
| Step 6: リリーススライシング | 45-60 分 | MVP 定義とリリース計画 |
| Step 7: 整合性確認 + まとめ | 20-30 分 | 全体レビューと次のアクション |

## 2. ファシリテーションの段階的手順

### Phase A: 場のセットアップ（最初の 15 分）

```
1. ゴールの共有
   「今日のワークショップで、[プロダクト名] のストーリーマップを完成させます。
    MVP の範囲を合意し、リリース計画の骨格を作ることがゴールです。」

2. グラウンドルールの設定
   - 発言は平等（役職に関係なく）
   - 批判ではなく「Yes, and...」で積み上げる
   - タイムボックスを守る（ファシリテーターが管理）
   - 完璧を求めない（後で修正できる）

3. 付箋の色分けルール（対面の場合）
   - 黄色: フィーチャー
   - 青色: エピック
   - ピンク: ストーリー
   - 緑色: 技術タスク / 非機能要件
   - 赤色: 疑問点 / 要確認事項
```

### Phase B: 発散フェーズ（ペルソナ → フィーチャー → エピック）

```
ファシリテーションのコツ:

1. 個人ワーク → グループ共有の順で進める
   → まず各自 5 分で付箋に書き出し
   → その後 1 人ずつ発表・貼り出し
   → 重複を統合、補足を追加

2. 「もう出ない」と感じたら、視点を変える質問を投げる
   - 「このユーザーが一番困っている瞬間は？」
   - 「この機能がなかったら、ユーザーはどう回避する？」
   - 「競合はこの課題をどう解決している？」

3. フィーチャーの並びは左→右の時系列で
   → 「ユーザーが最初にやることは何？」から始める
   → 「その次は？」「最後にやることは？」で左→右に展開
```

### Phase C: 収束フェーズ（ストーリー → スライシング）

```
1. ストーリー分解は「幅優先」で
   → 全エピックを浅く分解してから、重要なエピックを深掘り
   → 1 つのエピックに時間をかけすぎない（タイムボックス）

2. MVP スライシング時の合意形成
   → PO が「これは R1 に必要」と判断したものを上に移動
   → 開発者が「これは技術的に先に必要」と指摘したものを調整
   → ステークホルダーが予算・時間の制約を共有

3. 「R1 に入れすぎ問題」が発生したら
   → 「もし R1 の期間が半分になったら、何を外す？」と質問
   → 優先度の議論を強制的に引き出す
```

## 3. よくある停滞パターンと打破テクニック

| 停滞パターン | 症状 | 打破テクニック |
|:---|:---|:---|
| **沈黙の壁** | 参加者が発言しない | 個人ワーク（Silent Brainstorming）で付箋に書かせてから共有。「1 人 3 枚以上」とルール化 |
| **技術論争** | 開発者同士が実装方法で議論 | 「How は後で。今は What に集中しましょう」とリダイレクト。Parking Lot に記録 |
| **HiPPO 支配** | 最上位者の意見に全員が従う | 「他の方はどう思いますか？」と明示的に指名。匿名投票（ドット投票）で意見を可視化 |
| **スコープクリープ** | 「ついでにこれも」と範囲が膨張 | ビジョンステートメントに立ち返る。「これはビジョンの実現に必要？」と問う |
| **粒度の不一致** | フィーチャーとストーリーが混在 | 「これは 1 スプリントで完了する？」の質問で粒度を判定。Yes → ストーリー、No → エピック |
| **完璧主義** | 受入基準の詳細にこだわりすぎる | 「R1 のストーリーだけ詳細化。R2/R3 は粗くてOK」とスコープを絞る |
| **対立の膠着** | PO と開発者が優先度で対立 | 「データで決めましょう」と定量化に誘導。ユーザーインタビューや A/B テストの提案 |
| **エネルギー低下** | 午後の集中力切れ | 休憩 + スタンドアップ形式に変更。小さな成功を可視化（「ここまでで X 個のストーリーが完成」） |

### Parking Lot（駐車場）テクニック

```
ワークショップ中に出た「今は扱わないが重要な論点」を記録する場所。

使い方:
  1. 模造紙の隅 / Miro の別エリアに「Parking Lot」ゾーンを作る
  2. 議論が脱線したら「重要な論点なので Parking Lot に記録しましょう」
  3. ワークショップの最後に Parking Lot を確認し、次のアクションを決定
  4. 各項目に担当者と期限を設定

典型的な Parking Lot 項目:
  - 技術的な制約の詳細調査
  - 外部 API の仕様確認
  - セキュリティ要件の詳細化
  - 非機能要件の数値目標設定
```

## 4. リモート vs 対面のツール選択

### 比較マトリクス

| 観点 | 対面 | リモート |
|:---|:---|:---|
| **推奨ツール** | 模造紙 + 付箋 + マーカー | Miro / FigJam / Mural |
| **参加者上限** | 8-12 名（物理スペース依存） | 6-8 名（画面共有の限界） |
| **適する場面** | 初回のストーリーマップ、複雑なドメイン | リファインメント、リモートチーム |
| **成果物の保存** | 写真撮影 → Markdown 化が必要 | デジタルネイティブで保存 |
| **エネルギー管理** | 自然な休憩・雑談が発生 | 意図的な休憩設計が必須 |
| **セッション長** | 4-8 時間（終日可能） | 2-3 時間（集中力の限界） |

### リモートセッション固有のテクニック

| テクニック | 説明 |
|:---|:---|
| **タイマーの画面共有** | 全員が見える位置にカウントダウンタイマーを配置 |
| **ブレイクアウトルーム** | 3-4 人の小グループで発散 → 全体で共有 |
| **リアクション活用** | 「同意なら Thumbs Up」「質問なら ? を貼って」等の非言語コミュニケーション |
| **45 分ルール** | 45 分ごとに 5 分の休憩を強制。画面疲れを防止 |
| **事前テンプレート** | Miro のフレームに事前にフィーチャー列・リリースラインを配置しておく |

## 5. MVP スライシングのファシリテーション技法

### ドット投票（Dot Voting）

```
目的: 参加者全員の意見を定量的に可視化し、優先順位の合意を効率的に形成する

手順:
  1. 全ストーリーを壁 / ボードに貼り出す
  2. 各参加者にドット（シール）を配布
     - PO: 5 個（優先度の重み付けが大きい）
     - 開発者: 3 個
     - その他: 2 個
  3. 各自がMVPに必須と思うストーリーにドットを貼る
  4. ドット数の多い順に R1 候補として議論

注意:
  - ドット投票は「議論の出発点」であり「最終決定」ではない
  - PO が最終判断権を持つことを事前に合意しておく
```

### Walking Skeleton ファースト

```
目的: エンドツーエンドの薄いスライスを最初に特定する

手順:
  1. 各フィーチャーから「最も基本的なエピック」を 1 つずつ選ぶ
  2. 選んだエピックをつなげて、端から端まで価値が流れるか確認
  3. Walking Skeleton の各エピックから最小のストーリーを選ぶ
  4. これが R1 の核（Must）になる

ファシリテーションの問いかけ:
  - 「ユーザーが最低限の業務を完了できるパスはどれ？」
  - 「このフィーチャーで一番シンプルなシナリオは？」
  - 「技術的に先に構築が必要な基盤はどれ？」
```

### 「もし期間が半分になったら？」テスト

```
目的: 優先度の議論が停滞した時に、本当に必要なものを炙り出す

手順:
  1. 現在の R1 候補を一覧にする
  2. 「もし期間が半分になったら、何を残しますか？」と問う
  3. 残ったものが「真の Must」
  4. 外れたものは Should に格下げ

バリエーション:
  - 「予算が半分になったら？」
  - 「チームが半分になったら？」
  - 「3 ヶ月後にデモが必要になったら？」
```

### 狩野モデルの可視化

```
目的: MVP に「当たり前品質 + 少しの魅力品質」のバランスを意識させる

手順:
  1. R1 候補のストーリーを 3 列に分類
     - 当たり前品質（なければクレーム）
     - 性能品質（あれば満足度が比例上昇）
     - 魅力品質（あれば感動、なくても不満なし）
  2. 当たり前品質が 100% カバーされているか確認
  3. 魅力品質から 1-2 個を R1 に追加（差別化要素）
  4. 全体のバランスを視覚的に確認

ファシリテーションの問いかけ:
  - 「これがなかったら、ユーザーは怒る？ → 当たり前品質」
  - 「これがあったら、ユーザーは驚く？ → 魅力品質」
  - 「これの品質が上がるほど、満足度が上がる？ → 性能品質」
```
