# Skill: Story Map（Jeff Patton 型ユーザーストーリーマッピング）

> **ビジョンから始め、フィーチャー → エピック → ストーリーへ分解し、「誰に・何を・どの順で届けるか」を一枚の地図にする**

CRITICAL: 本スキル実行時は、対象案件の `projects/<案件名>/` を明示すること。

## Instructions

### ワークフロー内の位置

```
pdf-convert → distill → [story-map] → proposal 作成 → drm → review → 提出
                ↑            ↑                                    ↓
          10_inbox/ の生データ  20_notes/ の蒸留仕様       40_decision_log/ に記録
```

story-map は **distill（要件蒸留）の後工程**として位置づけられる。
distill が「What（何が必要か）」をドメイン視点で構造化するのに対し、
story-map は「Who × What × When（誰に・何を・いつ届けるか）」をユーザー視点で地図化する。

**distill の成果物が存在しない場合**: inbox のみからストーリーマップを作成可能。
ただし、distill の成果物（用語集、ドメインモデル、ビジネスルール）が存在する場合は必ず参照し、
用語の統一・要件の網羅性を担保すること。

### 入力

| ソース | パス | 必須/任意 |
|:---|:---|:---|
| 顧客資料（生データ） | `projects/<案件名>/10_inbox/` | 必須 |
| 用語集 | `projects/<案件名>/20_notes/01-glossary.md` | 推奨 |
| ドメインモデル | `projects/<案件名>/20_notes/02-domain-model.md` | 推奨 |
| ビジネスルール | `projects/<案件名>/20_notes/03-business-rules.md` | 推奨 |
| 非機能要件 | `projects/<案件名>/20_notes/non-functional-requirements.md` | 任意 |
| ドキュメントテンプレート | [references/output-examples.md](references/output-examples.md) | 必須 |

### 出力

| 成果物 | パス | 説明 |
|:---|:---|:---|
| ストーリーマップ | `projects/<案件名>/20_notes/05-story-map.md` | ビジョン・フィーチャー・エピック・ストーリーの全体地図 |
| ペルソナ定義 | `projects/<案件名>/20_notes/04-personas.md` | ユーザー像の構造化定義 |
| リリース計画 | `projects/<案件名>/20_notes/06-release-plan.md` | リリーススライスと段階的デリバリー計画 |

各成果物の具体的な出力例は [references/output-examples.md](references/output-examples.md) を参照。
Jeff Patton 型手法の理論的背景は [references/patton-methodology.md](references/patton-methodology.md) を参照。

### 4層構造（本スキルの中核）

```
ビジョンステートメント   ← プロダクト全体の Why / What / Who を1文で定義
       ↓
フィーチャー（Feature）  ← ビジョンを実現する大きな価値の塊（バックボーン）
       ↓
エピック（Epic）         ← フィーチャーを構成する実現単位（1スプリントに収まらない）
       ↓
ユーザーストーリー       ← エピックを分解した実装可能な最小単位（1スプリントで完了）
```

### ID 体系

| レベル | ID形式 | 例 |
|:---|:---|:---|
| ペルソナ | `P-NNN` | `P-001` |
| フィーチャー | `FT-NNN` | `FT-001` |
| エピック | `EP-NNN-NNN` | `EP-001-001`（FT-001 配下の1番目のエピック） |
| ストーリー | `US-NNN-NNN-NNN` | `US-001-001-001`（EP-001-001 配下の1番目のストーリー） |
| リリース | `R[N]` | `R1`, `R2`, `R3` |

## Step 1: ペルソナの抽出と定義

`10_inbox/` の全資料（および `20_notes/` が存在する場合はその内容）を精読し、
システムに関わるすべてのユーザー像（ペルソナ）を抽出・構造化する。

**ペルソナ定義の構造**:

```markdown
### P-001: [ペルソナ名]

| 項目 | 内容 |
|:---|:---|
| **ID** | P-001 |
| **名前** | [代表的な名前] |
| **役割** | [組織内の役割・職種] |
| **目標** | [このシステムで達成したいこと（成果で記述）] |
| **課題（ペイン）** | 1. [課題1] 2. [課題2] 3. [課題3] |
| **得たい状態（ゲイン）** | 1. [ゲイン1] 2. [ゲイン2] |
| **利用頻度** | [日次/週次/月次/不定期] |
| **技術リテラシー** | [高/中/低] |
| **顧客タイプ** | [エンドユーザー/調達者・購入者/影響者/内部ステークホルダー] |
| **出典** | 10_inbox/[ファイル名] §[セクション] |
```

**顧客の4象限**（PO研修知見）:

| 象限 | 定義 | 優先度判断への影響 |
|:---|:---|:---|
| エンドユーザー | 実際にシステムを使う人 | 使いづらければ黙って離脱する。UX に直結 |
| 調達者・購入者 | 支払い・契約の意思決定者 | リスク回避・ROI を重視。提案書の説得力に直結 |
| 影響者 | 現場運用・問い合わせ対応者 | 運用性・保守性の要求を持つ |
| 内部ステークホルダー | 組織内の支援者・反対者 | 社内合意形成に影響 |

**チェックリスト**:
- inbox に登場するすべてのユーザータイプを網羅したか
- 4象限（エンドユーザー/調達者/影響者/内部SH）を意識して抽出したか
- 各ペルソナの目標が「行動」ではなく「成果」で記述されているか
- ペイン（課題）とゲイン（得たい状態）が明確か
- 優先ペルソナ（Primary Persona）を特定したか
- ペルソナは3〜5人に収まっているか（多すぎる場合はサブペルソナに）

## Step 2: ビジョンステートメントの策定

ペルソナとinbox の分析を踏まえ、プロダクト全体の方向性を **ビジョンステートメント** として言語化する。
ビジョンステートメントは、以降のフィーチャー・エピック・ストーリーすべての判断基準となる「北極星」。

**ビジョンステートメント・テンプレート**:

```markdown
## ビジョンステートメント

**【対象顧客】** で
**【ニーズ / 課題】** を抱える人向けの
**【プロダクト名 / プロジェクト名】** は **【プロダクト分野】** であり、
**【最大の利点 / 使うべき理由】** を提供する。
**【競合や代替手段】** とは違って、
我々のプロダクトは **【差別化ポイント】** である。
```

**記述例**:

```markdown
## ビジョンステートメント

**教育委員会管内の教員・管理者** で
**児童の学習データが分散し、個別最適な指導ができない** という課題を抱える人向けの
**教育データ利活用基盤** は **教育データプラットフォーム** であり、
**学校・教委をまたいだデータ統合と可視化により、エビデンスに基づく教育改善** を提供する。
**各校バラバラの Excel 管理や既存の校務支援システム単体** とは違って、
我々のプロダクトは **メダリオンアーキテクチャによる統合分析と、教員が即座に使えるダッシュボードを両立** する。
```

**チェックリスト**:
- Who（対象顧客）が具体的か → ペルソナ（Step 1）と整合しているか
- Why（ニーズ/課題）が inbox の事実に基づいているか
- What（提供価値）が「機能」ではなく「顧客が得る成果」で書かれているか
- How（差別化）が競合・代替手段と明確に区別されているか
- チーム全員がこの1文で「何を作るか」を説明できるか

**ビジョンから下位構造への接続**:

```
ビジョンステートメント（なぜやるか・何を実現するか）
  ↓ 「この価値を実現するために必要な大きな機能群は何か？」
フィーチャー（Step 3）
  ↓ 「各フィーチャーを構成する価値の塊は？」
エピック（Step 4）
  ↓ 「各エピックを1スプリントで実装可能な単位に分解すると？」
ストーリー（Step 5）
```

## Step 3: フィーチャーの洗い出し（バックボーン構築）

ビジョンを実現するために必要な **フィーチャー（Feature）** を洗い出す。
フィーチャーはストーリーマップの「背骨（Backbone）」を形成し、左から右に時系列で並ぶ。

**フィーチャーの定義**:
- ユーザーに提供する **大きな価値の塊**
- 複数のエピックで構成される
- ユーザーの活動・目的に対応する（技術的な分類ではない）

**フィーチャー定義の構造**:

```markdown
### FT-001: [フィーチャー名]

| 項目 | 内容 |
|:---|:---|
| **ID** | FT-001 |
| **フィーチャー名** | [ユーザー視点の価値を表す名称] |
| **説明** | [このフィーチャーが提供する価値] |
| **対象ペルソナ** | P-001, P-002 |
| **ビジョンとの関連** | [ビジョンのどの部分を実現するか] |
| **出典** | 10_inbox/[ファイル名] §[セクション] |
```

**バックボーン表現形式**:

```markdown
## ストーリーマップ バックボーン（フィーチャー）

| FT-001 | FT-002 | FT-003 | FT-004 | FT-005 |
|:---:|:---:|:---:|:---:|:---:|
| **データ準備・登録** | **データ統合・取込** | **データ分析・可視化** | **レポート・報告** | **運用管理** |
| P-001, P-004 | P-001, P-004 | P-001, P-002, P-003 | P-002, P-003 | P-004 |
| → | → | → | → | → |
```

**チェックリスト**:
- フィーチャーは **ユーザー視点の価値** で記述されているか（技術的な機能名ではない）
- フィーチャーの並びが左→右の時系列・因果関係で自然に読めるか
- 3〜7個のフィーチャーに収まっているか
- ビジョンステートメントのすべての価値が、いずれかのフィーチャーにマッピングされるか
- inbox / notes の全要件がいずれかのフィーチャーに属するか

**アンチパターン**:
- 技術レイヤーで分類する（例: ×「フロントエンド」「バックエンド」「DB」）
- フィーチャーの粒度が不揃い（1つだけ巨大、他が細かすぎる）
- 時系列が成立しない並び順

### 代替手法: EventStorming によるフィーチャー発見

inbox の要件が複雑・大規模でフィーチャーの洗い出しが困難な場合、Alberto Brandolini の EventStorming を活用する。

| ステップ | 内容 | 対応する付箋色 |
|:---|:---|:---|
| 1. Domain Events | ドメインで起きる出来事を過去形で列挙 | オレンジ |
| 2. Commands | イベントを引き起こすアクション | 青 |
| 3. Aggregates | コマンドを処理するドメインオブジェクト | 黄 |
| 4. Bounded Contexts | 関連する Aggregate をグルーピング | ピンク枠 |
| 5. → Feature 変換 | Bounded Context をフィーチャーにマッピング | FT-NNN |

**EventStorming → Story Map 変換ルール**:
- Bounded Context → フィーチャー（FT-NNN）
- Aggregate → エピック（EP-NNN-NNN）
- Command + Event ペア → ユーザーストーリー（US-NNN-NNN-NNN）

**使い分け**:
| 手法 | 推奨場面 |
|:---|:---|
| 直接フィーチャー洗い出し（Step 3 標準） | 要件が明確、小〜中規模 |
| EventStorming 経由 | 要件が複雑、レガシーシステム、ドメイン知識が分散 |

> **参考**: EventStorming の詳細な手法分類（Big Picture / Process Modeling / Software Design）は flow-architecture スキルの知見に基づく。

## Step 4: エピックの分解

各フィーチャーの下に、**エピック（Epic）** を配置する。
エピックは **1スプリントでは完成できない大きな価値の塊** であり、
ストーリーマップの「肋骨（Ribs）」にあたる。

**エピックの定義**:
- フィーチャーを達成するための構成要素
- 複数のユーザーストーリーに分解される
- それ自体がユーザーに意味のある価値を持つ

**エピック定義の構造**:

```markdown
### EP-001-001: [エピック名]

| 項目 | 内容 |
|:---|:---|
| **ID** | EP-001-001 |
| **フィーチャー** | FT-001: [フィーチャー名] |
| **エピック名** | [ユーザーに提供する価値の塊] |
| **ペルソナ** | P-001: [ペルソナ名] |
| **説明** | [このエピックで実現すること] |
| **完了条件** | [このエピックが完了したと言える状態] |
| **規模感** | [ストーリー数の見込み: 3〜8件] |
| **出典** | 10_inbox/[ファイル名] §[セクション] |
```

**Walking Skeleton の特定**:

Step 4 の重要なサブタスクとして、**Walking Skeleton** を特定する。
各フィーチャーから最も基本的なエピックを1つずつ選び、
エンドツーエンドで価値を提供できる最薄のスライスを定義する。

```markdown
## Walking Skeleton（最小限のエンドツーエンドパス）

| FT-001 | FT-002 | FT-003 | FT-004 | FT-005 |
|:---:|:---:|:---:|:---:|:---:|
| EP-001-001 | EP-002-001 | EP-003-001 | EP-004-001 | EP-005-001 |
| 学籍情報の手動登録 | CSV一括取込 | 基本集計ダッシュボード | 定型レポート出力 | アカウント管理 |
```

**チェックリスト**:
- 各フィーチャーに最低2つ以上のエピックがあるか
- エピックは1スプリントでは完了しない粒度か（小さすぎればストーリーに格下げ）
- エピック自体がユーザーに意味のある価値を持つか
- エピックの完了条件が明確に定義されているか
- 主要ペルソナのエピックが優先的にマッピングされているか
- Walking Skeleton が特定されているか

## Step 5: ユーザーストーリーの分解

各エピックの下に、具体的な **ユーザーストーリー** を配置する。
ストーリーは **1スプリントで実装・テスト可能な最小の価値単位** であり、
INVEST 基準を満たすこと。

**ストーリー記述形式（Connextra 形式）**:

```markdown
### US-001-001-001: [ストーリータイトル]

| 項目 | 内容 |
|:---|:---|
| **ID** | US-001-001-001 |
| **エピック** | EP-001-001: [エピック名] |
| **ペルソナ** | P-001: [ペルソナ名] |
| **ストーリー** | [ペルソナ名]として、[行動]したい。なぜなら[価値・理由]だから。 |
| **受入基準** | 1. [条件1] 2. [条件2] 3. [条件3] |
| **規模** | S / M / L / XL |
| **リリース** | R1 / R2 / R3 |
| **ビジネスルール参照** | BR-XXX-NNN（該当時） |
| **ドメインモデル参照** | E-NNN（該当時） |
| **出典** | 10_inbox/[ファイル名] §[セクション] |
```

**受入基準の書き方（Gherkin 風）**:

```
前提（Given）: [初期状態]
実行（When）: [ユーザーの操作]
結果（Then）: [期待される結果]
```

**INVEST 基準チェック**:

| 基準 | 説明 | チェック |
|:---|:---|:---|
| **I**ndependent | 他のストーリーと独立して実装・テスト可能 | 依存関係が最小限か |
| **N**egotiable | 実装方法は交渉可能（What を定義、How は柔軟） | 実装詳細を含んでいないか |
| **V**aluable | ユーザーまたはビジネスに価値を提供 | 「なぜなら」が明確か |
| **E**stimable | 規模を見積もり可能 | S/M/L/XL で判断できるか |
| **S**mall | 1スプリントで完了可能 | XL の場合は分割を検討 |
| **T**estable | 受入基準で完了を検証可能 | 受入基準が具体的か |

**ストーリーの垂直優先順位**:

ストーリーマップ内では、各エピック列の中でストーリーを **上から下に優先度順** に配置する。

```
高優先度（MVP / R1）  ← 上
  ↓
中優先度（R2）
  ↓
低優先度（R3）        ← 下
```

**規模の目安**:

| 規模 | 開発工数目安 | 粒度の基準 | 扱い |
|:---|:---|:---|:---|
| **S** | 1〜2人日 | 小さな改修、バリデーション追加、UI 微修正 | そのまま実装 |
| **M** | 3〜5人日 | 標準的な CRUD 画面、基本的なビジネスロジック | そのまま実装 |
| **L** | 5〜10人日 | 外部連携、複雑なビジネスロジック、ダッシュボード | 分割を検討 |
| **XL** | 10人日超 | エピック相当。複数の独立した価値を含む | 必ず分割する |

**XL ストーリーの分割テクニック（縦割りスライシング）**:
- **データを絞る**: 全データではなく主要なデータタイプのみ
- **ルールを絞る**: 全ビジネスルールではなく最重要ルールのみ
- **ターゲットを絞る**: 全ユーザーではなく主要ペルソナのみ
- **地域/範囲を絞る**: 全拠点ではなくパイロット拠点のみ

いずれの場合も「価値が残る」ことが条件。

**チェックリスト**:
- すべてのストーリーが Connextra 形式（〜として、〜したい、なぜなら〜）で記述されているか
- 受入基準が具体的かつテスト可能か
- ストーリーの規模が S〜L に収まっているか（XL は分割）
- ビジネスルール（BR-XXX）との相互参照があるか
- ドメインモデルのエンティティとの関連が明確か
- inbox の出典が明記されているか
- 「なぜなら」が欠落していないか（価値が不明なストーリーは排除）

**アンチパターン**:
- 技術タスクをストーリーにする（例: ×「DBテーブルを作成する」→ ○「管理者として、教員データを登録したい」）
- 受入基準が曖昧（例: ×「正しく動作すること」→ ○「入力値が不正な場合、エラーメッセージが表示される」）
- 「なぜなら」が欠落している（価値不明のストーリー）

## Step 6: リリーススライスの策定

ストーリーマップを水平にスライスし、段階的なリリース計画を策定する。
各スライスは独立してデプロイ可能な価値の塊であり、提案書のスケジュール・見積の根拠となる。

**スライス戦略（Patton の「Slice the Cake」原則）**:

ケーキを水平に切るように、全フィーチャーを横断する薄いスライスで段階的にリリースする。
1つのフィーチャーだけを深掘りする「縦切り」はアンチパターンである。

**リリーススライス定義**:

```markdown
## R1: MVP（Minimum Viable Product）

| 項目 | 内容 |
|:---|:---|
| **目標** | [このリリースで達成するビジネス成果] |
| **ペルソナ** | [対象ペルソナ] |
| **期間** | [想定期間] |
| **ストーリー数** | [N件] |
| **規模合計** | [S×a + M×b + L×c] |

### 含まれるストーリー

| フィーチャー | エピック | ストーリーID | ストーリータイトル | 規模 |
|:---|:---|:---|:---|:---|
| FT-001 | EP-001-001 | US-001-001-001 | [タイトル] | M |
| FT-002 | EP-002-001 | US-002-001-001 | [タイトル] | S |
```

**MVP の品質バランス（狩野モデル）**:

| 品質タイプ | MVP での扱い | 例 |
|:---|:---|:---|
| **当たり前品質** | 必須（Must）。なければ成立しない | 認証、データ保全、基本操作 |
| **性能品質** | 一部（Should）。段階的に追加 | 検索速度、レポート種類 |
| **魅力品質** | 少し（Could）。差別化要素 | ダッシュボード、AI分析 |

```
R1（MVP）= 当たり前品質 100% + 性能品質の一部 + 魅力品質を少し
R2 = 性能品質の拡充 + 魅力品質の追加
R3 = 魅力品質の本格化 + 性能品質の完成
```

**スライス策定の原則**:

1. **R1（MVP）**: Walking Skeleton + 最低限の業務遂行に必要なストーリー
   - すべてのフィーチャーから最低1つのストーリーを含む
   - 主要ペルソナが基本業務を完了できる
   - 「使えないが完璧」よりも「粗いが使える」を優先
   - 当たり前品質（Must）+ 魅力品質を少し混ぜる

2. **R2（成長リリース）**: 業務効率化・利便性向上のストーリー
   - セカンダリペルソナの対応
   - 例外処理・エラーハンドリングの充実
   - 外部連携・パフォーマンス最適化

3. **R3（成熟リリース）**: 高度な機能・差別化要素
   - 分析・レポーティングの高度化
   - 自動化・AI/ML 連携
   - 完全自動化

**MoSCoW によるリリース割当**:

| 分類 | 定義 | リリース割当 |
|:---|:---|:---|
| **Must** | なければ成立しない | R1 |
| **Should** | 重要だが時間がなければ延期可能 | R1〜R2 |
| **Could** | 余裕があれば | R2〜R3 |
| **Won't（今回）** | 将来必要でも今はやらない | スコープ外 |

**提案書連携**:

| リリース | 提案書の章 | マッピング内容 |
|:---|:---|:---|
| R1 | スケジュール・工程表 | フェーズ1の開発範囲・期間 |
| R1 | 見積 | MVP の工数・コスト |
| R2/R3 | スケジュール・工程表 | フェーズ2以降の開発範囲・期間 |
| 全体 | 開発方針 | 段階的デリバリーの根拠（ストーリーマップに基づく） |

**チェックリスト**:
- 各リリーススライスが独立してデプロイ・検証可能か
- R1（MVP）がすべてのフィーチャーを横断しているか（縦切りになっていないか）
- R1 で主要ペルソナが基本的な業務を完了できるか
- 狩野モデルのバランスが意識されているか（当たり前品質 + 少しの魅力品質）
- 各リリースのビジネス価値が明確に説明できるか
- MoSCoW で分類済みか
- 提案書のスケジュール・見積と整合性があるか

## Step 7: 整合性の検証と相互参照

ストーリーマップ全体を inbox の原本および notes の蒸留結果と照合し、
情報の欠落・矛盾・過不足がないか検証する。

**検証項目**:

### 7-1: 網羅性の検証

inbox の全要件がストーリーマップのいずれかのストーリーにマッピングされているか。

```markdown
## 要件カバレッジマトリクス

| inbox 要件 | 出典 | ストーリーID | リリース | ステータス |
|:---|:---|:---|:---|:---|
| [要件1] | 10_inbox/Request.md §2.1 | US-001-001-001 | R1 | カバー済 |
| [要件2] | 10_inbox/Request.md §3.2 | US-002-001-002 | R2 | カバー済 |
| [要件3] | 10_inbox/Material_list.md §1 | — | — | 未カバー（要確認） |
```

### 7-2: notes との整合性

| 検証観点 | 方法 |
|:---|:---|
| 用語の統一 | ストーリーマップ内の用語が `01-glossary.md` と一致しているか |
| エンティティの網羅 | `02-domain-model.md` の全エンティティがいずれかのストーリーに関連するか |
| ビジネスルールの反映 | `03-business-rules.md` の全ルールがストーリーの受入基準に反映されているか |
| 非機能要件の考慮 | `non-functional-requirements.md` がリリース計画の制約として反映されているか |

### 7-3: 内部整合性

| 検証観点 | チェック |
|:---|:---|
| ID の一意性 | すべての ID（P-NNN, FT-NNN, EP-NNN-NNN, US-NNN-NNN-NNN）が一意か |
| 参照の健全性 | ストーリーが参照するエピック・フィーチャー・ペルソナが実在するか |
| 優先度の一貫性 | 高優先度ストーリーが依存する低優先度ストーリーがないか |
| リリース割当の完全性 | すべてのストーリーにリリース（R1/R2/R3）が割り当てられているか |
| ビジョンとの整合 | 全フィーチャーがビジョンステートメントの価値に紐づいているか |

### 7-4: DEEP チェック（バックログ品質）

| 基準 | チェック |
|:---|:---|
| **D**etailed appropriately | 上位（R3）は粗く、直近（R1）は詳細か |
| **E**stimated | 直近のストーリーは規模（S/M/L）が見積もられているか |
| **E**mergent | 学習・変化で項目が更新され続ける仕組みがあるか |
| **P**rioritized | 価値の順に並んでいるか。なぜその順か説明できるか |

### 7-5: ストーリーマッピング Slop 防止チェック（Distributional Convergence 対策）

ストーリーマッピングは「案件固有のユーザー課題を構造化する」ことが本質であり、LLM がどの案件でも同じ構造・同じ表現のストーリーマップを生成すると、案件固有の価値提案やリスクが見えなくなる。

| # | ストーリーマッピング Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| SM-1 | **フィーチャー名の汎用動詞化** — フィーチャー名が「データを管理する」「情報を確認する」「設定を行う」等の汎用動詞で、案件ドメインの固有概念が名称に含まれない | フィーチャー名だけを読み、どの案件のストーリーマップか特定できるか確認する | フィーチャー名にドメイン固有の名詞を含める。「管理する」ではなく「学籍データを統合する」「在庫引当を最適化する」等、ドメインの業務動詞・対象名詞を使用する |
| SM-2 | **ストーリーの金太郎飴化** — すべてのストーリーが「[ペルソナ]として、[対象]を[CRUD操作]したい、なぜなら[業務効率化]だから」の同一パターンで、案件固有のペイン・業務文脈が反映されていない | ストーリーの「なぜなら」部分を通読し、すべてが「業務効率を上げたい」「ミスを減らしたい」等の汎用理由になっていないか確認する | ストーリーの「なぜなら」にペルソナ固有のペイン・業務コンテキストを反映する。inbox のソース資料から具体的な課題・数値（「月末の集計作業に3日かかる」等）を引用する |
| SM-3 | **MVP の定型化** — MVP（R1）に常に「ログイン」「ダッシュボード」「CRUD画面」「CSV出力」を含め、案件の本質的な価値提案に基づかない標準機能セットを並べる | R1 のストーリー一覧を読み、案件のビジョンステートメントが掲げる差別化ポイントがR1に含まれているか確認する | MVP はビジョンの差別化ポイントを最小限で体現するスライスにする。「ログイン+CRUD」ではなく「案件固有の価値を検証できる最小セット」を Walking Skeleton から導出する |
| SM-4 | **リリース計画の等間隔化** — R1/R2/R3 が常に同じスプリント数・同じ構造（基本→拡張→高度化）になり、案件固有のリスク・依存関係・外部制約が計画に反映されない | リリース計画の期間・構成が案件の納期制約・外部依存（API連携先のスケジュール等）を反映しているか確認する | リリース計画を案件固有の制約から導出する。外部 API の開放時期、法改正の施行日、顧客の検収サイクルなど、案件固有のマイルストーンを計画の骨格にする |
| SM-5 | **受入基準の定型句化** — 受入基準が「正常に登録できること」「エラーメッセージが表示されること」「一覧が表示されること」等の定型句で、ビジネスルールの具体的な検証条件が含まれない | 受入基準を読み、notes の business-rules.md に定義された具体的なルール（BR-XXX）の検証条件が含まれているか確認する | 受入基準に inbox・notes から導出した具体的な条件を記述する。「エラーメッセージが表示される」ではなく「年齢が0未満または150超の場合、BR-001に基づきエラーコード E-AGE-001 を返す」と記述する |
| SM-6 | **MoSCoW 分布の固定化** — Must:Should:Could:Won't の比率が常に同じ（例: 40:30:20:10）になり、案件のリスクプロファイルや顧客の優先度が反映されない | MoSCoW の分布が案件固有の優先度（inbox の要件優先度・顧客の期待・競合状況）を反映しているか確認する | MoSCoW の分類を inbox の顧客要求優先度から導出する。「顧客が明示的に Must と指定した要件」「競合に対抗するために必要な差別化要件」等、案件固有の根拠で分類する |

> **核心原則**: **案件の DNA を刻むストーリーマップ** — ストーリーマップはその案件でしか生まれない固有のユーザー課題・業務文脈・価値提案を構造化した「案件の DNA」である。「別の案件のストーリーマップに差し替えても違和感がないか？」→ 違和感がないならストーリーマッピング Slop である。

**最終成果物として、要件カバレッジマトリクスを作成する。**

**チェックリスト（7-1〜7-5 統合）**:
- [ ] inbox の全要件がストーリーマップにカバーされている（カバレッジ100%）
- [ ] notes（glossary, domain-model, business-rules）との相互参照が健全
- [ ] 要件カバレッジマトリクスが作成済み
- [ ] DEEP チェックをパス
- [ ] ストーリーマッピング Slop チェック（SM-1〜SM-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Quality Gates

以下のすべてを満たした場合のみ、story-map フェーズを完了とする:

**ペルソナゲート**:
- inbox に登場するすべてのユーザータイプがペルソナとして定義済み
- 4象限（エンドユーザー/調達者/影響者/内部SH）を意識して抽出済み
- Primary Persona が特定済み

**ビジョンゲート**:
- ビジョンステートメントが策定済み
- Who / Why / What / 差別化が明確に記述済み
- ペルソナの課題（ペイン）と整合

**フィーチャーゲート**:
- 3〜7個のフィーチャーで構成されたバックボーンが定義済み
- フィーチャーがユーザー視点の価値で記述されている
- ビジョンの全価値がフィーチャーにマッピング済み

**エピックゲート**:
- 各フィーチャーに最低2つ以上のエピックが定義済み
- Walking Skeleton が特定済み
- エピックの完了条件が明確

**ストーリーゲート**:
- すべてのストーリーが Connextra 形式（〜として、〜したい、なぜなら〜）で記述済み
- すべてのストーリーに受入基準が定義済み
- すべてのストーリーに規模（S/M/L/XL）とリリース（R1/R2/R3）が割当済み
- INVEST 基準を満たさないストーリーが 0 件

**リリースゲート**:
- R1（MVP）がすべてのフィーチャーを横断
- R1 で主要ペルソナが基本業務を完了可能
- 狩野モデルのバランス（当たり前品質 + 魅力品質少し）が意識されている
- MoSCoW 分類済み

**トレーサビリティゲート**:
- inbox の全要件がストーリーマップにカバーされている（カバレッジ100%）
- notes（glossary, domain-model, business-rules）との相互参照が健全
- 要件カバレッジマトリクスが作成済み
- DEEP チェックをパス

## Troubleshooting

| 問題 | 原因 | 対処 |
|:---|:---|:---|
| ビジョンが曖昧で方向性が定まらない | 顧客の Why が不明 | inbox の背景・目的セクションを精読。不足なら顧客にヒアリング |
| ペルソナが特定できない | inbox にユーザー情報が不足 | 質問リストを作成し、顧客にヒアリング。仮ペルソナを設定し ADR に記録 |
| フィーチャーの粒度が定まらない | 業務フローの全体像が不明 | 顧客の現行業務フローをヒアリング。inbox の業務フロー図を参照 |
| エピックが大きすぎる | 分解が不十分 | ユーザーに意味のある価値の単位まで分解。3〜8ストーリーに分解できるサイズを目安に |
| ストーリーが巨大すぎる（XL以上） | 縦割りスライシング未実施 | データ/ルール/ターゲットを絞って分割 |
| リリーススライスが縦切りになる | Slice the Cake 原則の未適用 | 各リリースが全フィーチャーを横断するよう再構成 |
| notes との用語が不一致 | glossary 未参照 | `20_notes/01-glossary.md` を再読し、全ストーリーの用語を統一 |
| inbox に矛盾する要件がある | RFP 内の不整合 | 矛盾を明示し、顧客に確認。仮定を ADR に記録 |
| ストーリーの依存関係が複雑で INVEST の I（独立性）が維持できない | ストーリー分割の粒度が粗い | 依存関係を受入基準として表現し、INVEST 違反ストーリーをスコープから切り出す |
| フィーチャー数が 7 を超えてバックボーンが肥大化する | ユーザー目的の統合不足 | 類似フィーチャーをユーザー目的単位で統合し、7±2 に収める |
| 既存システム置き換えで現行機能の棚卸しが膨大 | 現行機能を全てストーリー化しようとしている | 「現状維持/改善/廃止」の3分類でふるい分け、改善対象のみストーリー化 |
