# Skill: Data Validation

> **Garbage In, Garbage Out を防ぐ最後の砦**

CRITICAL: 本スキル実行時は、対象案件の `projects/<案件名>/` を明示すること。

## Instructions

### ワークフロー内の位置

```
pdf-convert → [data-validation] → distill → review
                    ↑ ↓
              10_inbox/ or 20_notes/ or 30_proposal/ のデータ検証
```

### プロジェクトディレクトリ構造

| 番号 | ディレクトリ | 検証対象 |
|:---|:---|:---|
| **10** | `10_inbox/` | PDF 変換後のテーブルデータ |
| **20** | `20_notes/` | スキーマ定義・ビジネスルールの論理整合性 |
| **30** | `30_proposal/` | 見積金額・工数・SLA 等の数値整合性 |
| **40** | `40_decision_log/` | 検証で発見した問題と対応判断を記録 |

### 入力

| ソース | パス | 検証対象 |
|:---|:---|:---|
| inbox のテーブルデータ | `projects/<案件名>/10_inbox/*.md` | PDF 変換された Markdown テーブル |
| notes のスキーマ定義 | `projects/<案件名>/20_notes/domain-model.md` | エンティティ定義の論理的整合性 |
| notes のビジネスルール | `projects/<案件名>/20_notes/business-rules.md` | ルール間の矛盾・欠落 |
| proposal の見積・数値 | `projects/<案件名>/30_proposal/*.md` | 工数、金額、SLA の整合性 |
| CSV データ | 任意のパス | 生データの品質チェック |

### 出力

- **検証レポート**: 検出された問題の一覧（Markdown 形式）
- **修正済みデータ（オプション）**: 自動修正可能な問題の修正版
- **40_decision_log/**: 検証判断の記録

## Step 1: 検証対象の特定

検証すべきデータの種類と場所を特定し、検証の厳密度（strict / lenient）を決定する。

### 検証種別の判定フロー

```
検証対象は？
  ├── PDF 変換後のテーブル → Step 2: テーブル構造検証
  ├── CSV / Excel データ  → Step 3: データ品質検証
  ├── ドメインモデル定義   → Step 4: スキーマ整合性検証
  ├── ビジネスルール       → Step 5: ルール整合性検証
  └── 提案書内の数値       → Step 6: 数値整合性検証
```

### 厳密度の選択

| 厳密度 | 適用場面 | 基準 |
|:---|:---|:---|
| **strict** | 提出前の最終検証、本番データ | 全ルール適用、例外なし |
| **lenient** | 作業中のドラフト、探索的分析 | Critical/Major のみ検出 |

**チェックリスト**:
- [ ] 検証対象のファイルパスを特定したか
- [ ] 検証種別（テーブル/データ品質/スキーマ/ルール/数値）を判定したか
- [ ] 厳密度（strict/lenient）を決定したか
- [ ] 検証基準（元文書、スキーマ定義等）を確認したか

---

## Step 2: テーブル構造検証（PDF 変換後）

`pdf-convert` で生成された Markdown テーブルの品質を確認する。

### 検証手順

1. **構造チェック**: Markdown テーブルのシンタックスが正しいか
2. **列数チェック**: 全行の列数がヘッダーと一致するか
3. **値チェック**: セルの値が妥当か（空セル、数値形式等）
4. **原本照合**: 元 PDF のテーブルとサンプリング比較

### テーブル品質メトリクス

| メトリクス | 計算方法 | 基準値 |
|:---|:---|:---|
| 列数一致率 | 一致行数 / 全行数 | 100% |
| 空セル率 | 空セル数 / 全セル数 | < 5%（意図的な空を除く） |
| 数値フォーマット一致率 | 正しい形式の数値 / 全数値 | 100% |
| 日付フォーマット一致率 | 正しい形式の日付 / 全日付 | 100% |

**チェックリスト**:
- [ ] テーブルのヘッダー行が正しく認識されているか
- [ ] 列数が全行で一致しているか
- [ ] セル結合が崩れて空セルになっていないか
- [ ] 数値データの桁区切り・小数点が正しいか
- [ ] 日付データのフォーマットが統一されているか
- [ ] 元の PDF と Markdown テーブルの値が一致するか（サンプリング検証）

## Step 3: データ品質検証（CSV / Excel）

### プロファイリング

まずデータの全体像を把握する:

| 分析項目 | 内容 | 目的 |
|:---|:---|:---|
| レコード数 | 全行数 + ユニーク行数 | 重複の有無 |
| カラム数 | 全カラム + 型推定 | スキーマの概要 |
| 欠損率 | カラムごとの NULL/空文字率 | データ品質の全体像 |
| 値の分布 | 数値: min/max/mean/median、文字列: top N | 異常値の検出 |
| カーディナリティ | カラムごとのユニーク値数 | ID カラムの特定 |

### 検証ルール

| カテゴリ | チェック内容 | 深刻度 |
|:---|:---|:---|
| **型整合** | 数値カラムに文字列が混入していないか | Critical |
| **必須チェック** | NOT NULL フィールドに欠損がないか | Critical |
| **一意性** | ID カラムに重複がないか | Critical |
| **範囲チェック** | 数値が妥当な範囲か（負の年齢、未来の日付等） | Major |
| **参照整合** | FK 値が参照先に存在するか | Major |
| **フォーマット** | 日付・電話番号・メール等の形式チェック | Minor |
| **文字コード** | 全行が統一エンコーディング（UTF-8 推奨）か | Minor |

**チェックリスト**:
- [ ] 全行でカラム数が一致しているか
- [ ] 必須フィールドに NULL / 空文字がないか
- [ ] データ型が期待通りか
- [ ] 数値の範囲は妥当か
- [ ] 一意制約を満たしているか
- [ ] 文字コードは統一されているか
- [ ] 欠損値のパターンに規則性はあるか

---

## Step 4: スキーマ整合性検証（ドメインモデル）

`20_notes/domain-model.md` のエンティティ定義を論理的に検証する。

### 検証マトリクス

| チェック項目 | 検証方法 | 不合格の影響 |
|:---|:---|:---|
| 主キー存在 | 全エンティティに PK があるか | データの一意識別不可 |
| 外部キー参照 | FK の参照先エンティティが存在するか | 参照整合性の崩壊 |
| データ型適切性 | 金額に DECIMAL、ID に UUID 等 | 精度誤差・拡張性問題 |
| NOT NULL 妥当性 | 必須属性に NOT NULL があるか | データ品質低下 |
| 関係の論理性 | 1:1, 1:N, N:M が業務に合致するか | モデル誤り |
| 循環参照 | FK の参照チェーンが循環していないか | 無限ループ・デッドロック |
| 正規化レベル | 第 3 正規形以上が適切か | 更新異常・データ冗長 |

**チェックリスト**:
- [ ] すべてのエンティティに主キー（PK）が定義されているか
- [ ] 外部キー（FK）の参照先エンティティが存在するか
- [ ] データ型がドメインに適切か（例: 金額に INTEGER ではなく DECIMAL）
- [ ] NOT NULL / UNIQUE 制約が適切に設定されているか
- [ ] エンティティ間の関係（1:1, 1:N, N:M）が論理的に正しいか
- [ ] 循環参照が発生していないか
- [ ] 正規化のレベルは適切か

---

## Step 5: ルール整合性検証（ビジネスルール）

`20_notes/business-rules.md` のルール間の論理的整合性を検証する。

### 矛盾検出のパターン

| パターン | 例 | 検出方法 |
|:---|:---|:---|
| **直接矛盾** | BR-001「必須」vs BR-005「任意」 | 同一属性への条件比較 |
| **間接矛盾** | BR-002 + BR-003 の組み合わせで不可能な状態 | ルールの連鎖検証 |
| **デッドルール** | 条件が常に false で実行されないルール | 条件の論理評価 |
| **循環依存** | BR-010 → BR-020 → BR-010 | 依存グラフの巡回検出 |
| **欠落ルール** | エンティティに対応するルールがない | domain-model との照合 |

### ルール間の依存グラフ

```
BR-001 ──→ BR-003 ──→ BR-007
  │                      ↑
  └──→ BR-005 ──→ BR-006 ┘
```

循環が検出された場合は `40_decision_log/` に記録し、解消方法を提示する。

**チェックリスト**:
- [ ] すべてのルールに一意 ID が付与されているか
- [ ] ルール間に直接矛盾がないか
- [ ] ルール間に間接矛盾（組み合わせで不可能な状態）がないか
- [ ] ルールが参照するエンティティ・属性が domain-model.md に存在するか
- [ ] 出典（RFP の該当箇所）が明記されているか
- [ ] 例外ケースが網羅されているか
- [ ] ルールの優先順位が定義されているか（矛盾時の解決策）

## Step 6: 数値整合性検証（提案書内）

`30_proposal/` 内の数値（工数、金額、SLA 目標値等）が整合しているか検証する。

### 検証の 3 層

| 層 | 内容 | 例 |
|:---|:---|:---|
| **算術整合** | 計算が正しいか | 人員 × 単価 × 期間 = 見積金額 |
| **論理整合** | 数値間の関係が成立するか | 工数 ≤ 人員 × 稼働日数 |
| **実現可能性** | 目標が技術的に実現可能か | SLA 99.9% が設計で担保されるか |

### 積算検証チェックシート

| 検証項目 | 算式 | 判定 |
|:---|:---|:---|
| 工数合計 | Σ(工程別工数) = 総工数 | 一致/不一致 |
| 金額合計 | Σ(人員 × 単価 × 期間) = 見積額 | 一致/不一致 |
| バッファ率 | バッファ / 本体工数 | 10-20% が妥当 |
| 消費税 | 本体価格 × 税率 = 税額 | 一致/不一致 |
| 人月単価 | 総額 / 総人月 | 市場相場と比較 |

**チェックリスト**:
- [ ] 工数の合計が各章で一致しているか
- [ ] 金額の積算根拠と合計が一致しているか
- [ ] SLA 目標値が技術章のアーキテクチャで実現可能か
- [ ] スケジュールの工程間に論理的な依存関係が成立しているか
- [ ] 人員計画 × 単価 × 期間 = 見積金額が成立しているか
- [ ] 消費税・諸経費の計算が正しいか
- [ ] リスクバッファが計上されているか（10-20% が目安）

## Step 7: 検証結果のレポート

レポートは以下の構造で出力する。詳細なテンプレートと指摘例は [references/report-templates.md](references/report-templates.md) を参照。

### レポート構造

1. **メタ情報**: 案件名、検証対象、検証種別、検証日、厳密度
2. **結果サマリー**: 総合判定（Valid / Conditional / Invalid）、深刻度別件数
3. **検出された問題**: 問題の一覧（深刻度、箇所、影響、修正案）
4. **推奨アクション**: Critical → 即修正、Major → 修正後再検証、Minor → 修正推奨
5. **判断記録**: `40_decision_log/` への記録事項

### 深刻度レベル

| レベル | 定義 | 対応 | 例 |
|:---|:---|:---|:---|
| Critical | データ損失・誤計算リスク | 即座に修正必須 | 見積金額の合計不一致 |
| Major | 論理的矛盾 | 修正後に再検証 | ビジネスルールの矛盾 |
| Minor | 品質改善 | 修正推奨 | 日付フォーマットの不統一 |
| Info | 推奨事項 | 参考情報 | カラム名の命名規則提案 |

### 総合判定

| 判定 | 条件 | 次のアクション |
|:---|:---|:---|
| **Valid** | Critical 0 件 + Major 0 件 | 後続工程に進行可能 |
| **Conditional** | Critical 0 件 + Major 1 件以上 | 修正後に再検証 |
| **Invalid** | Critical 1 件以上 | 即座に修正、全面再検証 |

### データ検証 Slop 防止チェック（Distributional Convergence 対策）

データ検証の出力が汎用テンプレートの貼り付けに終始すると、ドメイン固有の品質問題を見逃し、「検証したつもり」で致命的な欠陥が後工程に流出する。

| # | データ検証 Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| DV-1 | **万能正規表現パターン** | すべてのデータ型に同じ正規表現パターン（`^\d+$` 等）を適用している | ドメイン固有のバリデーション式を設計する（例: 学籍番号は `^[A-Z]\d{7}$`、注文番号は `^ORD-\d{8}-[A-Z]{2}$`） |
| DV-2 | **均一閾値症候群** | 全カラムに同じ割合ベースの閾値（欠損率 5% 以下、外れ値 3σ 等）を機械的に適用 | ドメインの業務特性に基づく閾値を設定する（例: 緊急連絡先の欠損は 0% 必須、備考欄の欠損は 80% でも許容） |
| DV-3 | **汎用エラーレポート** | 検証ドメインに関係なく「型不一致」「NULL 違反」「範囲外」の 3 カテゴリでしか報告しない | ドメイン固有のエラーカテゴリを定義する（例: 「SLA 目標と設計の乖離」「工程間依存の論理破綻」「税率適用の時期不整合」） |
| DV-4 | **構造偏重スキーマ検証** | PK/FK/型の構造だけ確認し、ビジネス上の意味制約を無視している | 業務不変条件を検証に含める（例: 「受注日 ≤ 納品日」「割引率 ≤ 上限値」「承認者 ≠ 申請者」） |
| DV-5 | **NULL/NOT NULL 二値クロス検証** | クロスフィールド検証が NULL チェックのみで、値の意味的関係を検証していない | ドメイン固有の条件付き整合性を検証する（例: 「status=完了 なら completion_date が存在」「discount_type=定率 なら discount_value ≤ 100」） |
| DV-6 | **四次元コピペ品質メトリクス** | 完全性・正確性・一貫性・適時性の 4 次元を毎回そのまま適用し、ドメイン固有の品質軸を追加していない | 対象ドメインに固有の品質軸を追加する（例: 医療データなら「匿名化度」、金融データなら「監査証跡の完全性」、教育データなら「学年コード体系の一貫性」） |

> **核心原則**: **検証ルールはデータの「方言」を反映する処方箋** — 汎用的な健康診断では患者固有の疾患を見逃す。「別のドメインのデータに差し替えても同じ検証ルールで通るか？」→ 通るなら Data Validation Slop である。

**チェックリスト**:
- [ ] レポートの全セクションが記入されているか
- [ ] 深刻度の分類が基準に基づいているか
- [ ] 各問題に修正案が提示されているか
- [ ] 総合判定が条件に基づいているか
- [ ] `40_decision_log/` に判断事項を記録したか
- [ ] データ検証 Slop チェック（DV-1〜DV-6）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Examples

### Example 1: PDF 変換後のテーブル検証

```
「data-validation スキルで、education-data-platform の inbox にある RFP のテーブルデータを検証して」

→ Step 1: 10_inbox/RFP.md 内のテーブルを特定（12 テーブル）
→ Step 2: 列数一致率・空セル率・数値フォーマットを検証
→ Step 2: 元 PDF と照合し、テーブル 3 の列ズレを検出（Critical）
→ Step 7: レポート出力 — Conditional（Critical 0, Major 1: テーブル 3 の列ズレ）
→ 結果: 列ズレ箇所を修正指示付きで報告
```

### Example 2: 提案書の数値整合性チェック

```
「data-validation スキルで、education-data-platform の提案書内の見積金額・工数・SLA の数値整合性を検証して」

→ Step 1: 30_proposal/ 内の見積書・提案書を特定
→ Step 6: 積算検証 — 人員 × 単価 × 期間 = 見積金額の照合
→ Step 6: SLA 99.9% がアーキテクチャ章の DR 構成で担保されるか検証
→ Step 7: レポート出力 — Valid（Minor 2: 端数処理の不統一、バッファ率 8% で低め）
→ 結果: 金額整合性は合格。バッファ率の引き上げを推奨
```

### Example 3: CSV データの品質検証

```
「data-validation スキルで、生徒データ CSV を検証して。学籍番号の一意性と必須フィールドの欠損をチェックして」

→ Step 1: CSV ファイルを特定、strict モードで検証
→ Step 3: プロファイリング — 5,000 行 × 15 カラム、欠損率 2.3%
→ Step 3: 一意性チェック — 学籍番号の重複 3 件を検出（Critical）
→ Step 3: 必須チェック — 保護者連絡先の欠損 115 件（Major）
→ Step 7: レポート出力 — Invalid（Critical 1, Major 1）
→ 結果: 重複学籍番号 3 件と欠損パターンを詳細レポートで報告
```

### Example 4: 見積書の数値整合性検証

```
「提案書の見積金額と工数の整合性をチェックして」

→ Step 1: 30_proposal/ 内の見積書から工数テーブル・単価テーブル・合計金額を抽出
→ Step 6: 積算根拠の検証 — 人員数 × 単価 × 期間 = 小計が各工程で成立しているか照合
→ Step 6: リスクバッファの妥当性確認 — バッファ率 15% が計上済み
→ Step 6: 消費税・諸経費・管理費の計算精度を検証（端数処理の一貫性含む）
→ Step 7: 検出された不整合を深刻度付きでレポート出力
→ 結果: Valid — 積算整合性は問題なし。端数処理の切り捨て/四捨五入の統一を推奨（Info）
```

### Example 5: CSV データのスキーマ検証

```
「このCSVのカラム型と値の範囲を検証して」

→ Step 1: CSV ファイルを読み込み、ヘッダー行からカラム一覧を抽出
→ Step 3: プロファイリング — 各カラムの型推定・値分布・欠損率を算出
→ Step 3: 値の範囲チェック — 数値カラムの min/max/平均、日付カラムの範囲
→ Step 3: 異常値の検出 — 外れ値（IQR 基準）、型混在、フォーマット違反を特定
→ Step 7: カラムごとの型適合率・値範囲・異常値件数をまとめたレポートを出力
→ 結果: Conditional — 日付カラムに 3 種類のフォーマット混在（Major）を検出
```

### Example 6: 複数ドキュメント間のクロスチェック

```
「提案書と見積書の金額・工数が一致しているか検証して」

→ Step 1: 照合対象のドキュメントを特定（30_proposal/提案書.md と 30_proposal/見積書.md）
→ Step 6: 各ドキュメントから金額・工数・人員数を抽出し、対照表を作成
→ Step 6: ドキュメント間で同一であるべき数値の一致チェック（合計金額、総工数、期間）
→ Step 6: 参照関係の整合性 — 提案書の機能が見積書の工数項目に含まれているか（漏れ・余剰の検出）
→ Step 7: 不一致箇所をマッピング表で可視化し、レポート出力
→ 結果: Conditional — 提案書と見積書で管理費率が異なる（Major: 15% vs 12%）
```

## Troubleshooting

| 問題 | 原因 | 対処 |
|:---|:---|:---|
| CSV の文字化け | 文字コード不一致（Shift_JIS vs UTF-8） | UTF-8 に変換してから検証 |
| テーブルの列数が動的 | セル結合の PDF 変換崩れ | 元 PDF を参照して手動修正後に再検証 |
| スキーマの循環参照 | 設計ミス | 関係を見直し、中間エンティティで解決 |
| ルール間矛盾が解決不能 | RFP 自体に矛盾 | 顧客に確認し、`40_decision_log/` に仮定を記録 |
| 大量データで検証が遅い | データ量過多 | サンプリング検証に切り替え（母集団の 10% or 1000 行） |
| 浮動小数点の比較で誤検出 | IEEE 754 の丸め誤差 | 許容誤差（epsilon = 1e-10）を設定して比較 |
| 日付フォーマットが混在 | 入力元が複数システム | 正規化ルール（ISO 8601 統一）を定義してから検証 |
| BOM 付き UTF-8 で先頭カラムが認識失敗 | BOM の除去漏れ | BOM を除去してから検証（`sed -i '1s/^\xEF\xBB\xBF//'`） |
| SLA 目標値が技術的に実現不可能 | 要件と設計の乖離 | 20_notes/non-functional-requirements.md と照合し、実現可能なラインを提示 |
| ルール連鎖で予期しない状態が発生 | 間接矛盾 | ルール依存グラフを作成し、全パスを検証 |
| サンプリング検証で偏りが出る | サンプル選択の偏り | 層別サンプリング（各カテゴリから均等抽出）に切り替え |

## Scripts

| スクリプト | 用途 | 実行方法 |
|:---|:---|:---|
| [validate_csv.py](scripts/validate_csv.py) | CSV/TSV の自動バリデーション | `python scripts/validate_csv.py data.csv --schema schema.json` |
| [sample-schema.json](scripts/sample-schema.json) | スキーマ定義のサンプル | 参考用 |

### validate_csv.py の機能

- **構造検証**: ヘッダー存在チェック、列数一致性、重複ヘッダー検出
- **データ品質検証**: NULL/空値チェック、重複行検出、外れ値検出（IQR 法）、型推定・プロファイリング
- **スキーマ検証**（`--schema` 指定時）: カラム名・型・NOT NULL・UNIQUE・範囲・正規表現・列挙値の検証
- **ルール検証**（`--rules` 指定時）: カスタムルール（範囲・正規表現・クロスカラム比較等）の適用
- **JSON レポート出力**: M-5 レポート形式（メタ情報・サマリー・カラムプロファイル・問題一覧・推奨アクション）
- **日本語対応**: UTF-8 BOM 検出、cp932（Shift_JIS）フォールバック
- **終了コード**: 0 = Valid, 1 = Conditional, 2 = Invalid, 3 = エラー

### 使用例

```bash
# 基本検証（構造 + データ品質）
python scripts/validate_csv.py data.csv

# スキーマ付き検証
python scripts/validate_csv.py data.csv --schema scripts/sample-schema.json

# カスタムルール付き、レポート出力先指定
python scripts/validate_csv.py data.csv --rules rules.json --output report.json

# TSV ファイル、lenient モード
python scripts/validate_csv.py data.tsv --delimiter tab --severity lenient
```
