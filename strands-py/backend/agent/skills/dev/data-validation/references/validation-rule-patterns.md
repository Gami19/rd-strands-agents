# データ種別 検証ルールパターン集

> データの「方言」を見抜く処方箋 -- 種別ごとの検証ルールで Validation Slop を排除する。

---

## 概要

本リファレンスは、SKILL.md Step 3（データ品質検証）および Step 6（数値整合性検証）で適用する
検証ルールを **データ種別ごと** に体系化したものである。
汎用ルールの機械的適用（DV-1/DV-2）を防ぎ、ドメイン固有の検証設計を支援する。

---

## 1. 数値データ

| パターン | 検出方法 | 判定基準 | 具体例 |
|:---|:---|:---|:---|
| **範囲チェック** | `min ≤ value ≤ max` で境界判定 | ドメイン固有の上下限を定義。汎用閾値（0〜999999）は禁止 | 年齢: `6 ≤ age ≤ 100`、消費税率: `0.08 ≤ rate ≤ 0.10` |
| **異常値検出（IQR 法）** | Q1 - 1.5×IQR 未満 / Q3 + 1.5×IQR 超過 | validate_csv.py の `IQR_MULTIPLIER = 1.5` 準拠 | 月額売上が Q3+1.5×IQR を超過 → 入力ミスか季節要因か確認 |
| **異常値検出（Z-score）** | `|z| > 3` を閾値とする | 正規分布を仮定できるデータに限定 | 試験スコア分布で z=4.2 の値 → 採点ミスの疑い |
| **単位整合性** | 同一カラム内で桁数の分布を確認 | 同一意味の数値が 1000 倍/100 倍ずれていないか | 金額列に `50000`（円）と `50`（千円）が混在 |
| **丸め誤差許容** | `abs(a - b) < epsilon` で比較 | IEEE 754 対策: `epsilon = 1e-10`（SKILL.md Troubleshooting 準拠） | `0.1 + 0.2 ≠ 0.3` 問題への対処 |
| **桁区切り一貫性** | 正規表現で形式を判定 | カンマ区切り（`1,000`）とピリオド区切り（`1.000`）の混在検出 | PDF 変換で `1,000` → `1.000` に誤変換（report-templates.md 指摘例参照） |
| **ゼロ値の妥当性** | ゼロが業務上ありえるか確認 | 「未入力のゼロ」と「正当なゼロ」を区別 | 単価 0 円 → 無償提供か入力漏れかの判定 |

### 数値検証の設計指針

```
ドメイン固有の範囲を定義する手順:
1. 業務担当者に「ありえない値」のヒアリング
2. 過去データの分布から P1/P99 を算出
3. 上記を統合してカラム固有の min/max を設定
4. sample-schema.json の min/max フィールドに反映
```

## 2. 日付データ

| パターン | 検出方法 | 判定基準 | 具体例 |
|:---|:---|:---|:---|
| **範囲チェック** | 業務上の有効期間で判定 | 「未来の日付」「創業前の日付」を異常とする | 契約開始日が `2099-01-01` → 入力ミス |
| **順序整合性** | `開始日 ≤ 終了日` の論理チェック | クロスカラム比較（DV-5 対策） | `contract_start > contract_end` → Critical |
| **営業日判定** | 土日祝日カレンダーとの照合 | 業務イベント日が非営業日でないか | 納品日が祝日 → Warning |
| **タイムゾーン一貫性** | UTC/JST/ローカル時間の混在検出 | 同一カラム内でオフセットが統一されているか | `2026-02-28T09:00+09:00` と `2026-02-28T00:00Z` が混在 |
| **フォーマット統一** | validate_csv.py の `DATE_PATTERNS` 5 形式で判定 | ISO-8601 / JP-slash / JP-era / US-slash / EU-dot の混在検出 | `2026/02/28` と `02/28/2026` が同一カラムに混在 → Major |
| **和暦整合性** | 元号の切替日と照合 | `平成32年` のように存在しない和暦を検出 | `平成32年4月1日` → `令和2年` に修正すべき |
| **閏年/月末チェック** | カレンダー上の存在チェック | `2025-02-29` は閏年でないため不正 | 月末処理で `4月31日` が生成されていないか |

### 日付検証の設計指針

```
フォーマット正規化の優先順位:
1. ISO-8601（YYYY-MM-DD）を標準とする
2. 入力元システムの形式を特定し、変換ルールを定義
3. 和暦は西暦に正規化してから検証
4. validate_csv.py の DATE_PATTERNS に未対応の形式は手動追加
```

## 3. テキストデータ

| パターン | 検出方法 | 判定基準 | 具体例 |
|:---|:---|:---|:---|
| **フォーマット検証** | ドメイン固有の正規表現で照合 | DV-1 対策: `^\d+$` のような万能パターン禁止 | 学籍番号: `^[A-Z]\d{7}$`、部署コード: `^[A-Z]{2,5}-\d{3}$` |
| **エンコーディング検証** | validate_csv.py の `detect_encoding` で判定 | UTF-8 統一が基本。cp932 フォールバックは Warning | BOM 付き UTF-8 で先頭カラム認識失敗 → BOM 除去 |
| **長さ制約** | `min_length ≤ len(value) ≤ max_length` | DB カラム定義と一致すること | 郵便番号が 8 文字（ハイフン含む）: `^\d{3}-\d{4}$` |
| **全角/半角混在** | Unicode カテゴリで判定 | 同一カラム内で全角数字と半角数字が混在していないか | 電話番号に `０３-１２３４-５６７８` と `03-1234-5678` が混在 |
| **制御文字検出** | ASCII 0x00-0x1F（改行・タブ除く）の存在チェック | 制御文字はデータ破損の兆候 | CSV セル内に `\x00`（NULL バイト）が混入 |
| **トリミング不備** | 先頭/末尾の空白検出 | 意図しない空白がキー照合に影響 | ID カラムに `" ABC001"` → 参照整合性エラーの原因 |
| **HTML/特殊文字混入** | `<`, `>`, `&` の存在チェック | PDF 変換時のエスケープ漏れ | テーブルセルに `&amp;` がそのまま残存 |

### テキスト検証の設計指針

```
正規表現パターンの設計手順:
1. サンプルデータから有効値の形式を特定
2. ドメイン固有のパターンを正規表現で定義
3. sample-schema.json の "pattern" フィールドに設定
4. 例外値（旧形式など）がある場合は許容パターンを追加
```

## 4. カテゴリデータ

| パターン | 検出方法 | 判定基準 | 具体例 |
|:---|:---|:---|:---|
| **有効値リスト** | `value in enum_list` で照合 | sample-schema.json の `"enum"` フィールドで定義 | 成績: `["A", "B", "C", "D", "F"]` 以外は不正 |
| **相互排他** | 同一レコード内の複数カテゴリ値の論理チェック | 同時に成立しえない値の組み合わせを検出 | `status=退職済` かつ `is_active=true` → 矛盾 |
| **階層整合性** | 親カテゴリと子カテゴリの包含関係を検証 | 子の値が親の分類体系に属するか | `大分類=食品` に `小分類=パソコン` → 階層不整合 |
| **表記ゆれ** | 類似値のクラスタリング（レーベンシュタイン距離等） | 同一カテゴリの別表記を検出 | `東京都` / `東京` / `とうきょう` が混在 |
| **欠損カテゴリ** | マスタデータと突合 | 定義済みカテゴリがデータに一度も出現しないケース | 47 都道府県のうち 3 県のデータが皆無 → データ欠損の疑い |
| **コード値の廃止** | マスタの有効期間と照合 | 廃止済みコードが使用されていないか | 旧部署コード `D-001` が 2024 年以降のデータに出現 |

### カテゴリ検証の設計指針

```
有効値リスト（enum）の管理:
1. マスタテーブルまたは定義書から正式な値リストを取得
2. sample-schema.json の "enum" に設定
3. 大文字/小文字の正規化ルールを定義
4. 定期的にマスタとの同期を実施
```

## 5. 関係データ

| パターン | 検出方法 | 判定基準 | 具体例 |
|:---|:---|:---|:---|
| **参照整合性（FK）** | 子テーブルの FK 値が親テーブルに存在するか | 孤立レコード = Critical | `order.customer_id=999` だが `customer` テーブルに ID=999 不在 |
| **1:N 整合性** | 親 1 件に対する子の件数を集計 | 異常な件数（0 件/過多）を検出 | 顧客 1 人に対し注文 10,000 件 → データ重複の疑い |
| **M:N 整合性** | 中間テーブルの両端 FK を検証 | 片側孤立（一方の FK のみ存在）を検出 | `student_course` に存在するが `course` テーブルに該当科目なし |
| **循環参照検出** | 参照チェーンの深さ優先探索 | SKILL.md Step 4 のスキーマ検証と連携 | `A.parent_id → B → C → A` の循環 → 無限ループリスク |
| **カスケード影響範囲** | 親レコード削除時の影響子レコード数を算出 | 影響範囲が閾値超過なら Warning | 1 顧客削除で 50,000 件の関連レコードに影響 |
| **時系列参照整合** | 参照先の有効期間内に参照元が収まるか | 廃止済みマスタへの参照を検出 | 2026 年の注文が 2024 年に廃止済みの商品コードを参照 |

### 関係データ検証の設計指針

```
参照整合性チェックの手順:
1. ER 図から FK 関係を一覧化
2. 各 FK について参照先の存在チェックを実行
3. 孤立レコードの件数と影響を評価
4. SKILL.md Step 4 のチェックリストと統合して報告
```

## 6. 集計データ

| パターン | 検出方法 | 判定基準 | 具体例 |
|:---|:---|:---|:---|
| **合計一致** | `Σ(明細) = 合計行` の算術検証 | 丸め誤差許容（epsilon）を考慮した一致判定 | 工数小計の合計 ≠ 総工数 → SKILL.md Step 6 の積算検証 |
| **比率合計 100%** | `Σ(構成比) = 100%` の検証 | 丸め後の合計が 99.9%〜100.1% の範囲か | 市場シェアの合計が 102.3% → 重複カウントの疑い |
| **前期比の妥当性** | `当期値 / 前期値` の変動率チェック | ±50% 以上の変動は Warning（業種別に閾値調整） | 売上前年比 +300% → 合併・特需か入力ミスか確認 |
| **クロス集計整合** | 行合計と列合計の交差点が一致するか | ピボットテーブルの整合性 | 部門別×月別の売上合計が総計行と不一致 |
| **累積値の単調増加** | `cumulative[i] ≤ cumulative[i+1]` | 累積値が減少していないか | 累積売上が 3 月に前月比マイナス → 修正仕訳漏れ |
| **消費税整合** | `本体 × 税率 = 税額`（端数処理方式を考慮） | 切り捨て/四捨五入/切り上げの方式統一 | SKILL.md Step 6 の積算検証チェックシート準拠 |
| **バッファ率妥当性** | `バッファ / 本体工数` の比率チェック | 10-20% が目安（SKILL.md Step 6 準拠） | バッファ率 3% → リスク過小評価 → Warning |

### 集計データ検証の設計指針

```
数値整合性チェックの手順（SKILL.md Step 6 連携）:
1. 提案書内の全テーブルから数値を抽出
2. 算術整合（計算の正しさ）を検証
3. 論理整合（数値間の関係）を検証
4. 実現可能性（SLA 等の目標値）を検証
5. ドキュメント間のクロスチェック（Example 6 参照）
```

## データ種別と深刻度のマッピング

検証ルール適用時の深刻度判定ガイド。SKILL.md Step 7 の深刻度レベルと連動する。

| データ種別 | Critical（即修正） | Major（再検証） | Minor（改善推奨） |
|:---|:---|:---|:---|
| 数値 | 算術不一致、単位混在 | 異常値（IQR 超過） | 桁区切り不統一 |
| 日付 | 存在しない日付 | 順序逆転、フォーマット混在 | 和暦の旧表記 |
| テキスト | 制御文字混入、エンコーディング破損 | フォーマット違反 | 全角半角混在、トリミング不備 |
| カテゴリ | 有効値リスト外（PK/FK 関連） | 相互排他違反 | 表記ゆれ、廃止コード |
| 関係 | FK 参照先不在、循環参照 | 1:N 件数異常 | カスケード影響範囲超過 |
| 集計 | 合計不一致 | 比率合計逸脱、前期比異常 | バッファ率逸脱 |

## validate_csv.py との対応

本リファレンスのルールパターンと `scripts/validate_csv.py` の機能マッピング。

| 本リファレンスのパターン | validate_csv.py の対応機能 | 補足 |
|:---|:---|:---|
| 数値の範囲チェック | `--schema` の `min`/`max` | sample-schema.json で定義 |
| 異常値検出（IQR 法） | `validate_data_quality` 内の IQR 判定 | `IQR_MULTIPLIER = 1.5` |
| フォーマット検証（正規表現） | `--schema` の `pattern` | sample-schema.json の `"pattern"` |
| 有効値リスト | `--schema` の `enum` | sample-schema.json の `"enum"` |
| 型整合チェック | `infer_column_type` による自動型推定 | 5 形式の日付パターン対応 |
| 一意性チェック | `--schema` の `unique` | ID カラムの重複検出 |
| カスタムルール | `--rules` でクロスカラム比較等 | JSON 形式でルール定義 |
