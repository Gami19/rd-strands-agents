# Hive Metastore → Unity Catalog 移行チェックリスト

Hive Metastore（HMS）から Unity Catalog（UC）への移行を安全かつ段階的に実行するための
包括的なチェックリスト。

---

## 1. 移行アプローチの概要

| アプローチ | 説明 | リスク | 所要期間 |
|:---|:---|:---|:---|
| **Soft Migration** | HMS Federation で UC に接続。コード変更なし | 低 | 2-4 週 |
| **Hard Migration** | SYNC / CLONE / CTAS でデータを移行 | 中 | 4-12 週 |
| **段階的移行（推奨）** | Soft → 並行運用 → Hard の 3 段階 | 低〜中 | 8-20 週 |

---

## 2. Phase 0: 事前評価

### 2.1 UCX アセスメントの実行

- [ ] Databricks Labs の UCX ツールをインストールした
- [ ] Assessment Workflow を全ワークスペースで実行した
- [ ] テーブルインベントリ（HMS テーブルの一覧）を取得した
- [ ] テーブルごとのフォーマット（Delta / Parquet / CSV 等）を確認した
- [ ] テーブルごとのサイズとアクセス頻度を把握した
- [ ] 非互換コード（ノートブック、ジョブ）の一覧を取得した

### 2.2 現状の権限マッピング

- [ ] 既存のテーブル ACL を一覧化した
- [ ] テーブル ACL → UC GRANT への変換マッピングを作成した
- [ ] ワークスペースローカルグループの使用箇所を特定した
- [ ] サービスプリンシパルの使用状況を確認した

### 2.3 移行スコープの確定

- [ ] 移行対象のデータベース/テーブルを確定した
- [ ] 移行の優先順位を決定した（重要度 × 複雑度）
- [ ] 移行対象外（アーカイブ / 廃止予定）のテーブルを特定した
- [ ] 外部テーブルと Managed テーブルを分類した

## 3. Phase 1: 準備（2-4 週）

### 3.1 Unity Catalog の基盤構築

- [ ] Metastore を作成した（または既存を確認した）
- [ ] Metastore をワークスペースに関連付けた
- [ ] Metastore の Managed Storage Location を設定した
- [ ] Storage Credential を作成した（Azure: Managed ID / AWS: IAM Role）
- [ ] External Location を作成した（HMS の External テーブルが参照するパス）

### 3.2 カタログ/スキーマの作成

- [ ] UC 側のカタログ構造を設計・作成した
- [ ] スキーマ構造を設計・作成した
- [ ] 命名規則を策定し、適用した
- [ ] Managed Storage の割り当てを確認した

### 3.3 グループとアイデンティティの移行

- [ ] Identity Federation を有効化した
- [ ] アカウントレベル SCIM プロビジョニングを設定した
- [ ] ワークスペースレベルグループ → アカウントレベルグループの移行を完了した
- [ ] UCX Group Migration Workflow を実行した
- [ ] サービスプリンシパルをアカウントレベルに登録した

### 3.4 権限の事前設定

- [ ] UC 側のグループに権限を付与した（GRANT）
- [ ] 権限マトリクスに基づいてテスト用の権限を設定した
- [ ] Metastore 管理者を設定した

---

## 4. Phase 2: 並行運用（4-8 週）

### 4.1 Soft Migration（HMS Federation）の実行

- [ ] HMS Federation を有効化した
- [ ] HMS のテーブルが UC 経由でアクセス可能であることを確認した
- [ ] 既存のノートブック/ジョブがそのまま動作することを検証した
- [ ] 新規テーブルの作成先を UC に切り替えた

### 4.2 並行運用期間のテスト

- [ ] 主要なパイプラインが正常に動作することを確認した
- [ ] クエリの実行結果が HMS 直接アクセスと一致することを検証した
- [ ] パフォーマンスの劣化がないことを確認した
- [ ] 権限設定が正しく動作することを検証した

### 4.3 問題の記録と対処

- [ ] 発見された問題を一覧化した
- [ ] 各問題の対処方法を決定した
- [ ] Phase 3 で必要なコード修正箇所を特定した

## 5. Phase 3: 本移行（4-12 週）

### 5.1 テーブルの移行

#### External テーブル（SYNC コマンド推奨）

```sql
-- External テーブルの同期
SYNC TABLE prod.bronze.customers FROM hive_metastore.bronze.customers;
```

- [ ] すべての External テーブルに SYNC を実行した
- [ ] SYNC 後のテーブルが正しくアクセスできることを確認した

#### Managed テーブル（CLONE 推奨）

```sql
-- Managed テーブルの Deep Clone
CREATE TABLE prod.silver.orders DEEP CLONE hive_metastore.silver.orders;
```

- [ ] Managed テーブルの CLONE を実行した
- [ ] CLONE 後のデータが正しいことを検証した（行数、スキーマ）

#### SYNC / CLONE が使えない場合（CTAS）

```sql
-- CTAS によるテーブルコピー
CREATE TABLE prod.gold.daily_sales AS
SELECT * FROM hive_metastore.gold.daily_sales;
```

- [ ] CTAS が必要なテーブルを特定し、実行した

### 5.2 ビューの移行

- [ ] ビューの定義を UC 参照に書き換えた
- [ ] 書き換え後のビューが正しく動作することを検証した

### 5.3 コードの修正

- [ ] ノートブック内の `hive_metastore.` 参照を UC 参照に更新した
- [ ] ジョブの設定を UC 対応に更新した
- [ ] `spark.sql("USE hive_metastore")` を削除/更新した
- [ ] `spark.catalog.setCurrentDatabase()` を UC 対応に修正した
- [ ] テスト環境でコード修正の動作確認を完了した

### 5.4 パーティション関連の対処

- [ ] HMS パーティション直接操作コマンドの使用箇所を特定した
- [ ] UC でサポートされるパーティション操作に書き換えた
- [ ] パーティション関連のジョブが正常動作することを確認した

## 6. Phase 4: HMS リタイア（2-4 週）

### 6.1 HMS 参照の完全除去

- [ ] すべてのノートブック・ジョブから `hive_metastore` 参照を除去した
- [ ] HMS Federation を無効化した（または段階的に縮小した）
- [ ] HMS 経由のアクセスがないことを監査ログで確認した

### 6.2 クリーンアップ

- [ ] HMS の不要なテーブルメタデータを削除した
- [ ] 不要な Storage Credential / External Location を削除した
- [ ] 移行完了を関係者に通知した

### 6.3 ドキュメント更新

- [ ] 運用手順書を UC ベースに更新した
- [ ] チーム向けの UC 利用ガイドを作成・配布した
- [ ] Terraform で UC リソースを管理する体制を確立した

## 7. 移行時の重要な注意事項

| 注意事項 | 説明 | 対処 |
|:---|:---|:---|
| **Time Travel 非移行** | CLONE / CTAS では Delta Lake のヒストリーは移行されない | 移行前にヒストリーが必要な場合はアーカイブを作成 |
| **パーティション操作** | HMS のパーティション直接操作コマンドは UC 非サポート | UC サポートのパーティション操作に書き換え |
| **テーブル統計情報** | CLONE では統計情報が移行されない場合がある | 移行後に `ANALYZE TABLE` を実行 |
| **テーブルプロパティ** | 一部のテーブルプロパティは UC 非互換 | UCX レポートで非互換プロパティを確認 |
| **ワークスペースローカルグループ** | UC の GRANT で使用不可 | アカウントレベルグループに移行 |
| **個人のオーナーシップ** | 移行時にオーナーが個人ユーザーになりがち | 移行後にグループオーナーシップに変更 |

## 8. 移行方法の比較表

| 方法 | 対象 | データコピー | ヒストリー | 推奨シーン |
|:---|:---|:---|:---|:---|
| **SYNC** | External テーブル | メタデータのみ | N/A | 外部テーブルの移行（推奨） |
| **DEEP CLONE** | Managed テーブル | データ全体 | 非移行 | Managed テーブルの移行 |
| **SHALLOW CLONE** | Managed テーブル | メタデータのみ | 非移行 | 一時的な並行運用 |
| **CTAS** | 任意 | データ全体 | 非移行 | SYNC / CLONE 非対応時 |

---

## 9. ロールバック手順

万が一、移行に問題が発生した場合のロールバック手順:

1. **Soft Migration 中**: HMS Federation を無効化するだけで元に戻る
2. **Hard Migration 中（External テーブル）**: UC 側のテーブルを DROP（データは残る）
3. **Hard Migration 中（Managed テーブル）**: HMS 側のテーブルを引き続き使用し、コード修正を戻す
4. **コード修正後**: Git のバージョン管理で以前のコードに戻す

**重要**: ロールバックに備えて、移行期間中は HMS 側のテーブルを削除しない。
