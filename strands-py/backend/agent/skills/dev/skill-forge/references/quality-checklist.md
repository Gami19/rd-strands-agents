# 品質チェックリスト

> スキル完成時に全項目を通過させる品質ゲート。5 領域 40 項目。

---

## 1. Anthropic ガイド準拠（10 項目）

Anthropic 公式『The Complete Guide to Building Skill for Claude』に基づく。

| # | チェック項目 | 合格基準 |
|:---|:---|:---|
| A-1 | **Progressive Disclosure** | frontmatter → SKILL.md 本文 → references/ の 3 段階で情報を開示している |
| A-2 | **description のトリガー** | `Use when user says「...」` 形式でトリガーフレーズが 6 個以上ある |
| A-3 | **description の除外** | `Do NOT use for:` で誤発動防止の境界条件を 2 個以上明示している |
| A-4 | **description の文字数** | 1024 文字以内（YAML `>` 記法） |
| A-5 | **チェックリスト駆動** | 全ステップにチェックリストが含まれている |
| A-6 | **Examples 必須** | 3 個以上の具体的な使用例がある（金型基準: 6 個以上） |
| A-7 | **Troubleshooting 必須** | 問題/原因/解決策のテーブルが 5 項目以上ある（金型基準: 10 項目以上） |
| A-8 | **Related Skills** | 他スキルとの連携が明示されている |
| A-9 | **スキルパターン適合** | Anthropic 5 パターン（P1-P5）のいずれかに分類され、対応する設計指針（skill-patterns.md）が反映されている |
| A-10 | **テスト 3 種実施** | トリガーテスト（発動精度）・ファンクショナルテスト（BDD）・パフォーマンス比較の 3 種が実施されている |

## 2. リポジトリ規約準拠（6 項目）

CLAUDE.md の Skill Development Conventions に基づく。

| # | チェック項目 | 合格基準 |
|:---|:---|:---|
| R-1 | **フォルダ名** | kebab-case（例: `my-new-skill`） |
| R-2 | **必須ファイル** | `SKILL.md`（大文字小文字厳密）が存在する |

| R-4 | **文字コード** | UTF-8 |
| R-5 | **frontmatter 構造** | `name`, `description`, `metadata.author`, `metadata.version`, `metadata.category` が存在する |
| R-6 | **author** | `KC-Prop-Foundry` 固定 |

## 4. コンテンツ品質（9 項目）

| # | チェック項目 | 合格基準 |
|:---|:---|:---|
| C-1 | **キャッチフレーズ** | 1 行で本質を表現する引用ブロックがある |
| C-2 | **Instructions** | ワークフロー図（ASCII）+ 入力/出力テーブルがある |
| C-3 | **ステップの独立性** | 各ステップが明確な目的を持ち、順序が論理的 |
| C-4 | **Examples の多様性** | 異なるユースケース/シナリオをカバーしている |
| C-5 | **Troubleshooting の実用性** | 実際に遭遇しうる問題を扱っている |

| C-7 | **既存スキルとの境界** | 他スキルとの責務分担が明確で重複がない |
| C-8 | **日本語の自然さ** | 不自然な直訳や用語のブレがない |
| C-9 | **references/ 設計適合** | reference-design-matrix.md のセルフテスト 3 項目（ST-1〜ST-3）に合格している |

## 5. Distributional Convergence 防止 — Skill Slop（9 項目）

LLM がスキルを生成する際の均質化（distributional convergence）を検出する。

| # | チェック項目 | 合格基準 |
|:---|:---|:---|
| D-1 | **キャッチフレーズの固有性** | ドメイン固有の比喩・動詞を使っている。「高品質な」「効率的に」で始まらない |
| D-2 | **Examples の具体性** | 6 例中 4 例以上にドメイン固有の名詞（書籍名・ツール名・業界用語）が含まれる |
| D-3 | **チェックリストのドメイン適合** | 各ステップのチェック項目にドメイン固有の判断基準が 1 つ以上ある |
| D-4 | **ステップ粒度のメリハリ** | 核心工程と補助工程で行数・チェックリスト数に差がある（均一でない） |
| D-5 | **Troubleshooting の実践性** | 実際に遭遇しうるドメイン固有の問題が 5 項目以上ある |
| D-6 | **references/ の固有性** | 知識ソースの形態に応じた固有の reference 構成になっている |
| D-7 | **Related Skills の連携具体性** | 連携シナリオが Step レベルで記述されている（「参照」だけで終わらない） |
| D-8 | **description のトリガー多様性** | 命令形・口語・略語・状況説明型を含む。6 個以上のバリエーションがある |
| D-9 | **パターン分布の意識** | Sequential パターン偏重（60%超）を認識し、Iterative / Domain-specific パターンの適用可能性を検討している |

## 使い方

### 全項目チェック

```
A-1 ~ A-10: □□□□□□□□□□  (10/10)
R-1 ~ R-6:  □□□□□□        (6/6)
M-1 ~ M-6:  □□□□□□        (6/6)
C-1 ~ C-9:  □□□□□□□□□    (9/9)
D-1 ~ D-9:  □□□□□□□□□    (9/9)
合計: 40/40
```

### 不合格時の対応

| 不合格レベル | 対応 |
|:---|:---|
| A 領域（Anthropic ガイド） | **必須修正** — ガイドの根幹に関わるため修正必須 |
| R 領域（リポジトリ規約） | **必須修正** — リポジトリの一貫性に関わるため修正必須 |
| M 領域（金型基準） | **推奨修正** — 基準範囲を大きく逸脱する場合は修正 |
| C 領域（コンテンツ品質） | **推奨修正** — 品質向上のため修正を推奨 |
| D 領域（Distributional Convergence） | **推奨修正** — 3 項目以上不合格の場合はスキルの差別化が不足 |
