## description の書き方ガイド

### 構成

```
[日本語概要 — 1-2文]
Use when user says「...」「...」...（6-10個のトリガー）。
Do NOT use for: [除外1]（→ [代替]）、[除外2]（→ [代替]）。
```

### ルール

1. **1024 文字以内**（YAML の `>` 記法で折り返し）
2. **日本語概要**: スキルの目的を端的に
3. **トリガー**: ユーザーが言いそうな自然な表現を網羅（6-10個）
4. **除外**: 誤発動を防ぐ境界条件（2-3個）
5. **代替スキルの明示**: `（→ skill-name）` 形式で誘導

### 良い例

```yaml
description: >
  データアーキテクチャの選定・設計・レビューを、書籍『Deciphering Data Architectures』の
  フレームワークに基づいて支援する。6種のアーキテクチャの比較評価、ADS の実施、
  データモデリング手法の選定を包括的にガイドする。
  Use when user says「データアーキテクチャを選定して」「DWHを設計して」
  「Data Meshを導入すべきか」「データレイクとDWHどちらを使うべきか」。
  Do NOT use for: 単純な SQL クエリチューニング（→ DBA タスク）、
  データ品質の数値検証（→ data-validation スキル）。
```

### 悪い例

```yaml
# NG: 概要が英語
description: >
  A skill for data architecture design and review.

# NG: トリガーがない
description: >
  データアーキテクチャの設計を支援する。

# NG: 除外がない（誤発動リスク）
description: >
  データに関するあらゆる質問に回答する。
```
