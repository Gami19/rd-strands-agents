# Skill: Marketing Context（マーケティングコンテキスト構築）

> **戦略なき施策は弾を込めずに撃つようなもの — まずコンテキストを固めよ**

## Instructions

### ワークフロー内の位置

```
情報収集 → [marketing-context] → ad-creative / marketing-copy / seo-analytics / growth-ops
                  ↓
             marketing-context.md
             ・プロダクト情報
             ・ターゲットペルソナ
             ・ブランドボイス
             ・競合分析
             ・カスタマージャーニー
             ・KPI・目標
```

### 入力

| 入力 | 説明 | 例 |
|:---|:---|:---|
| プロダクト情報 | 製品・サービスの基本情報 | LP、README、製品仕様書 |
| 既存マーケ資料 | 現在のマーケティング素材 | ブランドガイドライン、過去の広告 |
| ユーザーの口頭説明 | 対話形式での情報提供 | 「BtoBのSaaSで、中小企業向け」 |
| コードベース | プロダクトの技術情報 | package.json、README.md |

### 出力

| 出力 | 形式 | 説明 |
|:---|:---|:---|
| marketing-context.md | Markdown | 全情報を統合した構造化ドキュメント |
| ペルソナシート | Markdown テーブル | ICP と 3 層ペルソナの定義 |
| ブランドボイスガイド | Markdown | トーン・スタイル・NG 表現の定義 |
| 競合ポジショニングマップ | ASCII / Markdown | 差別化の可視化 |

## Step 1: 既存情報の収集と棚卸し

プロダクトに関する既存情報を体系的に収集する。

### 1a. 自動収集（コードベースがある場合）

以下のファイルを自動的に読み取り、初期ドラフトを作成する:

- `README.md` — プロダクト概要、技術スタック
- `package.json` / `pyproject.toml` — 依存関係、説明文
- LP・ランディングページ — USP、CTA、キービジュアル
- 既存の `marketing-context.md` — 過去のコンテキスト情報
- `.claude/` 配下のドキュメント — プロジェクト固有の情報

### 1b. 対話形式での収集（コードベースがない場合）

セクションごとに質問を投げかけ、情報を段階的に収集する。
一度にすべてを聞かず、1 セクションずつ確認する。

### 1c. 既存コンテキストの更新

既に `marketing-context.md` が存在する場合:
1. 現在の内容を要約して提示する
2. 更新が必要なセクションを確認する
3. 該当セクションのみを更新する

**チェックリスト**:
- [ ] 利用可能な情報ソースをすべて特定した
- [ ] 自動収集とユーザー入力の両方を活用した
- [ ] 既存コンテキストの有無を確認した
- [ ] 情報の鮮度と信頼性を評価した

## Step 2: プロダクト情報の整理

製品・サービスの核心情報を構造化する。

### 2a. 基本情報

| 項目 | 記載内容 | 質問例 |
|:---|:---|:---|
| **製品名** | 正式名称とサブブランド | 「正式な製品名は？」 |
| **カテゴリ** | 顧客が検索する棚（市場カテゴリ） | 「顧客はどんな言葉で検索する？」 |
| **ワンライナー** | 1 文で伝えるプロダクトの価値 | 「エレベーターピッチは？」 |
| **プロダクトタイプ** | SaaS / EC / マーケットプレイス 等 | 「ビジネスモデルは？」 |

### 2b. USP（独自の価値提案）

1. **主要な USP を 3 つ**特定する
2. 各 USP に対して「なぜそれが顧客にとって重要か」を記述する
3. 競合と比較した際の明確な差別化ポイントを添える

### 2c. 価格・プラン体系

| 項目 | 詳細 |
|:---|:---|
| 料金モデル | サブスクリプション / 従量課金 / ライセンス |
| プラン構成 | Free / Starter / Pro / Enterprise 等 |
| 価格帯 | 各プランの月額 / 年額 |
| 主力プラン | 最も推奨するプランとその理由 |

**チェックリスト**:
- [ ] ワンライナーが 1 文で価値を伝えている
- [ ] USP が 3 つ明確に定義されている
- [ ] 市場カテゴリが顧客の検索行動と一致している
- [ ] 価格体系が競合と比較可能な形で整理されている

## Step 3: ターゲットペルソナの定義

ICP（理想顧客プロファイル）と具体的なペルソナを定義する。
詳細なテンプレートは [persona-templates.md](references/persona-templates.md) を参照。

### 3a. ICP の定義

| 項目 | BtoB の場合 | BtoC の場合 |
|:---|:---|:---|
| **セグメント** | 業種・企業規模・ステージ | 年齢層・ライフステージ |
| **意思決定者** | 役職・部門 | 本人 / 家族 |
| **予算規模** | 年間 IT 予算 | 可処分所得 |
| **導入障壁** | 承認プロセス・既存ツール | 習慣・認知 |

### 3b. 3 層ペルソナの設計

| 層 | 名称 | 目的 |
|:---|:---|:---|
| **Primary** | メインターゲット | 最も注力するべき顧客像 |
| **Secondary** | セカンダリターゲット | 成長フェーズで拡大する顧客像 |
| **Negative** | 非ターゲット（アンチペルソナ） | 意図的に除外する顧客像 |

各ペルソナには以下を定義する:
- **名前と肩書き**（架空だがリアリティのあるもの）
- **課題と痛み**: 日常的に抱えている問題
- **ゴール**: 達成したいこと
- **JTBD**（Jobs to Be Done）: プロダクトを「雇う」理由
- **反論**: 導入を躊躇する理由とその対処法
- **好むチャネル**: 情報収集の手段（SNS、検索、展示会等）

### 3c. 顧客の言葉の収集

ペルソナ定義で最も重要なのは**顧客自身の言葉**:
- 問題をどう表現するか（生の声）
- 製品をどう説明するか（口コミ的表現）
- 使うべき言葉 / 避けるべき言葉

**チェックリスト**:
- [ ] ICP が BtoB / BtoC に応じた粒度で定義されている
- [ ] Primary / Secondary / Negative の 3 層が定義されている
- [ ] 各ペルソナに JTBD が紐づいている
- [ ] 顧客自身の言葉（バーベイタム）が収集されている
- [ ] 反論と対処法がセットで記載されている

## Step 4: ブランドボイスの設定

一貫したコミュニケーションを実現するためのブランドボイスを定義する。
詳細なフレームワークは [brand-voice-guide.md](references/brand-voice-guide.md) を参照。

### 4a. ブランドパーソナリティ

3-5 個の形容詞でブランドの人格を定義する:

```
例: 「プロフェッショナル」「親しみやすい」「革新的」「誠実」「頼もしい」
```

各形容詞に対して、具体的な表現の指針を添える:

| 特性 | するべきこと | してはいけないこと |
|:---|:---|:---|
| プロフェッショナル | 根拠のある主張をする | 過剰な専門用語を使う |
| 親しみやすい | 平易な言葉で説明する | 馴れ馴れしくなる |

### 4b. トーン＆マナー

| 場面 | トーン | 例 |
|:---|:---|:---|
| LP・広告 | 明るく力強い | 「チームの生産性を 3 倍に」 |
| ドキュメント | 正確で簡潔 | 「以下の手順で設定します」 |
| エラーメッセージ | 共感的で具体的 | 「予期せぬ問題が発生しました。お手数ですが...」 |
| SNS | カジュアルだが品位を保つ | 「新機能をリリースしました!」 |

### 4c. NG 表現リスト

| NG 表現 | 理由 | 代替表現 |
|:---|:---|:---|
| 「簡単」「すぐに」 | 期待値の過剰設定 | 「シンプルな操作で」 |
| 「最高」「No.1」 | 根拠なき最上級 | 「業界トップクラスの」 |
| 「ぜひ」の多用 | 押し売り感 | CTA を 1 つに絞る |

### 4d. 用語集（グロッサリー）

プロダクト固有の用語とその定義を統一する:

| 用語 | 定義 | 使い方の例 |
|:---|:---|:---|
| （製品固有の用語） | （統一的な定義） | （文中での使い方） |

**チェックリスト**:
- [ ] ブランドパーソナリティが 3-5 個の形容詞で定義されている
- [ ] 場面別のトーン指針がある
- [ ] NG 表現リストが 5 項目以上ある
- [ ] 用語集が整備されている

## Step 5: 競合分析

市場における自社のポジションを明確にする。
詳細なフレームワークは [competitive-analysis-framework.md](references/competitive-analysis-framework.md) を参照。

### 5a. 競合の特定と分類

| 分類 | 定義 | 例 |
|:---|:---|:---|
| **直接競合** | 同じ問題を同じ方法で解決 | Calendly vs SavvyCal |
| **間接競合** | 同じ問題を別の方法で解決 | Calendly vs Superhuman のスケジュール機能 |
| **代替手段** | 競合する行動パターン | Calendly vs 手動メール調整 |

主要な競合を 3-5 社特定し、以下の観点で分析する。

### 5b. 競合比較マトリクス

| 観点 | 自社 | 競合 A | 競合 B | 競合 C |
|:---|:---|:---|:---|:---|
| 主要機能 | | | | |
| 価格帯 | | | | |
| ターゲット | | | | |
| 強み | | | | |
| 弱み | | | | |

### 5c. ポジショニングマップ

2 軸のマトリクスで市場内のポジションを可視化する:

```
       高機能
         │
    ┌────┼────┐
    │ 競合B│自社 │
    │    │    │
低価格──┼────┼──高価格
    │    │    │
    │ 競合C│競合A│
    └────┼────┘
         │
       シンプル
```

軸の選定基準:
- 顧客が購買判断で最も重視する 2 つの要素を選ぶ
- 自社の差別化が最も際立つ軸を選ぶ

### 5d. スイッチング・ダイナミクス（JTBD 四つの力）

| 力 | 方向 | 質問 |
|:---|:---|:---|
| **Push** | 離脱を促す | 現状の何に不満を感じているか？ |
| **Pull** | 引き寄せる | 自社のどこに魅力を感じるか？ |
| **Habit** | 現状維持 | 何が乗り換えを阻んでいるか？ |
| **Anxiety** | 不安 | 乗り換えに対してどんな不安があるか？ |

**チェックリスト**:
- [ ] 直接・間接・代替の 3 分類で競合を特定している
- [ ] 競合比較マトリクスが完成している
- [ ] ポジショニングマップの軸が顧客視点で選定されている
- [ ] スイッチング・ダイナミクスの 4 つの力を分析している

## Step 6: カスタマージャーニーの設計

顧客の認知から定着までの旅路を設計する。

### 6a. ファネルの 5 段階

| 段階 | 目的 | 主要チャネル | KPI |
|:---|:---|:---|:---|
| **認知** | プロダクトの存在を知ってもらう | SNS広告、コンテンツマーケ、PR | インプレッション、リーチ |
| **検討** | 選択肢として検討してもらう | LP、事例、比較記事、ウェビナー | サイト訪問、資料DL |
| **獲得** | 導入・購入を決定してもらう | トライアル、デモ、営業 | CVR、CAC |
| **活性化** | 初期価値を体験してもらう | オンボーディング、チュートリアル | アクティベーション率 |
| **定着** | 継続利用・推奨してもらう | CS、コミュニティ、機能強化 | リテンション率、NPS |

### 6b. 各段階のコンテンツマッピング

段階ごとに必要なコンテンツとメッセージを整理する:

| 段階 | コンテンツ形式 | キーメッセージ | CTA |
|:---|:---|:---|:---|
| 認知 | ブログ、SNS 投稿 | 課題の啓発 | 「詳しく見る」 |
| 検討 | ホワイトペーパー、事例 | 解決策の提示 | 「資料をダウンロード」 |
| 獲得 | LP、料金ページ | 具体的な価値 | 「無料で始める」 |
| 活性化 | ガイド、ツアー | 最初の成功体験 | 「設定を完了する」 |
| 定着 | Tips メール、新機能告知 | 継続的な価値 | 「新機能を試す」 |

### 6c. タッチポイントの設計

各段階で顧客が接触するポイントを列挙し、体験を設計する:
- **デジタル**: Webサイト、アプリ、メール、SNS
- **対人**: 営業、CS、ウェビナー、展示会
- **コンテンツ**: ブログ、動画、ポッドキャスト、事例

**チェックリスト**:
- [ ] 5 段階のファネルが定義されている
- [ ] 各段階のコンテンツとメッセージが整理されている
- [ ] タッチポイントが漏れなく列挙されている
- [ ] 段階間の遷移条件（何をもって次の段階に進むか）が明確

## Step 7: KPI・目標の設定

マーケティング活動の成果を測定する指標体系を設計する。

### 7a. NORTH STAR メトリクス

プロダクトの成長を最もよく表す **1 つの指標** を定める:

| 要素 | 内容 |
|:---|:---|
| **メトリクス名** | 例: 「週間アクティブプロジェクト数」 |
| **なぜこの指標か** | 顧客価値と事業成長の両方を反映するため |
| **目標値** | 現状: X → 目標: Y（期間: Z か月） |
| **計測方法** | データソースと計測頻度 |

### 7b. ファネル指標

| 段階 | 指標 | 現状値 | 目標値 | 計測ツール |
|:---|:---|:---|:---|:---|
| 認知 | インプレッション / リーチ | | | Google Analytics |
| 検討 | サイト訪問数 / 滞在時間 | | | Google Analytics |
| 獲得 | CVR / CAC / CPA | | | 広告プラットフォーム |
| 活性化 | アクティベーション率 | | | プロダクト分析 |
| 定着 | リテンション率 / LTV / NPS | | | CS ツール |

### 7c. マーケティング ROI

| 指標 | 計算式 | 目標 |
|:---|:---|:---|
| CAC | マーケ費用合計 / 新規顧客数 | LTV の 1/3 以下 |
| LTV | ARPU × 平均継続月数 | CAC の 3 倍以上 |
| LTV/CAC 比率 | LTV / CAC | 3.0 以上 |
| Payback Period | CAC / 月間粗利 | 12 か月以内 |

**チェックリスト**:
- [ ] NORTH STAR メトリクスが 1 つ定められている
- [ ] ファネルの各段階に KPI が紐づいている
- [ ] 現状値と目標値のギャップが把握されている
- [ ] LTV/CAC 比率の目標が設定されている

## Step 8: marketing-context.md の生成とレビュー

Step 1-7 の全情報を統合し、構造化ドキュメントとして出力する。

### 8a. ドキュメント構造

```markdown
# Marketing Context

*最終更新: YYYY-MM-DD*
*次回更新予定: YYYY-MM-DD*

## 1. プロダクト概要
## 2. ターゲットペルソナ
## 3. ブランドボイス
## 4. 競合分析
## 5. カスタマージャーニー
## 6. KPI・目標
## 7. 用語集
## 8. 変更履歴
```

### 8b. 保存と連携

1. `marketing-context.md` をプロジェクトルートまたは `.claude/` に保存する
2. 他のマーケティングスキルが自動的に参照できる場所に配置する
3. ユーザーに保存場所と利用方法を説明する

### 8c. 定期更新サイクル

| タイミング | 更新内容 |
|:---|:---|
| **月次** | KPI 実績値の更新、市場変動の反映 |
| **四半期** | ペルソナの見直し、競合情報の更新 |
| **半期** | ブランドボイスの再評価、ジャーニーの改定 |
| **都度** | 新製品リリース、価格改定、大きな市場変化 |

### 8d. 変更管理

更新時は以下を記録する:
- **変更日**: いつ変更したか
- **変更者**: 誰が変更したか
- **変更箇所**: どのセクションを変更したか
- **変更理由**: なぜ変更したか

**チェックリスト**:
- [ ] 全 7 セクションが漏れなく記載されている
- [ ] ドキュメントの保存場所が明確になっている
- [ ] 他のマーケティングスキルから参照可能な状態になっている
- [ ] 更新サイクルと変更管理ルールが定められている
- [ ] ユーザーに「他のマーケティングスキルがこのコンテキストを自動参照する」旨を説明した

## Examples

### Example 1: BtoB SaaS のコンテキスト新規作成

```
「クラウド勤怠管理SaaSのマーケティングコンテキストを作って」

→ Step 1 でコードベースを自動スキャン（README、LP）
→ Step 2 でプロダクト情報を整理（SaaS、サブスクリプション）
→ Step 3 でペルソナ定義（Primary: 人事部長、Secondary: 総務担当）
→ Step 4 でブランドボイス設定（プロフェッショナル×親しみやすい）
→ Step 5 で競合分析（KING OF TIME、ジョブカン、freee）
→ Step 6 でジャーニー設計（認知: HR メディア → 獲得: 無料トライアル）
→ Step 7 で KPI 設定（NORTH STAR: 月間アクティブ企業数）
→ Step 8 で marketing-context.md を生成
```

### Example 2: BtoC アプリのペルソナ再定義

```
「フィットネスアプリのペルソナを見直して」

→ Step 1 で既存の marketing-context.md を読み込み
→ Step 3 に集中してペルソナを再定義
  Primary: 30代共働き会社員（運動習慣を作りたい）
  Secondary: 50代健康志向シニア（医師に運動を勧められた）
  Negative: アスリート志向（機能が物足りない）
→ Step 8 でペルソナセクションのみ更新
```

### Example 3: スタートアップの初期コンテキスト構築

```
「まだ LP もないけど、マーケの土台を作りたい」

→ Step 1 でコードベースが最小限であることを確認
→ Step 1b の対話形式で情報を段階的に収集
→ Step 2 でワンライナーと USP を言語化（最も重要な初期作業）
→ Step 3 で仮説ベースのペルソナを設定
→ Step 5 で「顧客が今使っている代替手段」を中心に競合分析
→ Step 8 で MVP 版の marketing-context.md を生成
→ 「仮説ベースなので、顧客インタビュー後に更新しましょう」と案内
```

### Example 4: 競合参入に伴うポジショニング更新

```
「大手が類似サービスを出した。ポジショニングを見直して」

→ Step 1 で既存コンテキストを読み込み
→ Step 5 で新競合を追加し、比較マトリクスを更新
→ Step 5c でポジショニングマップを再作成
→ Step 5d でスイッチング・ダイナミクスを再評価
→ Step 2b で USP の差別化ポイントを強化
→ Step 4 でメッセージングの調整を提案
→ Step 8 で関連セクションを更新
```

### Example 5: 新規事業のブランドボイス策定

```
「新しいブランドのトーン＆マナーを決めて」

→ Step 1 でブランドの背景情報を収集
→ Step 3 でターゲットペルソナを確認（ボイスはペルソナに合わせる）
→ Step 4a でブランドパーソナリティを 5 つの形容詞で定義
→ Step 4b で場面別トーンを設計（LP / ドキュメント / SNS / エラー）
→ Step 4c で NG 表現リストを作成
→ Step 4d で用語集を整備
→ Step 8 でブランドボイスセクションを生成
```

### Example 6: KPI 再設計と NORTH STAR メトリクス選定

```
「マーケティング KPI を再設定して。何を追うべき？」

→ Step 1 で既存コンテキストと現在の KPI を確認
→ Step 6 でカスタマージャーニーを再確認（KPI はジャーニーに紐づく）
→ Step 7a で NORTH STAR メトリクスの候補を 3 つ提示
→ 「顧客価値」と「事業成長」の両面から最適な 1 つを選定
→ Step 7b でファネル各段階の KPI を再設計
→ Step 7c で LTV/CAC 比率の目標を設定
→ Step 8 で KPI セクションを更新
```

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| コンテキストが漠然として使えない | 情報の具体性が不足 | Step 2 で「顧客はどんな言葉で検索する？」等の具体的な質問で深掘りする |
| ペルソナが実際の顧客と乖離している | 仮説ベースのまま更新されていない | 顧客インタビューや CS フィードバックで検証し、Step 3 で更新する |
| ブランドボイスが形骸化している | 場面別の具体例が不足 | Step 4b でシーン別の例文を 3 つ以上追加する |
| 競合分析が古い | 市場変化を反映していない | Step 8c の四半期更新サイクルに従い、Step 5 を再実行する |
| USP が競合と差別化できていない | 機能比較に終始している | 「顧客が選ぶ理由」に焦点を当て、体験・サポート・哲学の違いを言語化する |
| ジャーニーの各段階が機能していない | コンテンツと CTA が段階に合っていない | Step 6b でコンテンツマッピングを見直し、段階に適したメッセージに修正する |
| KPI が多すぎて何を見ればいいかわからない | NORTH STAR が未定義 | Step 7a で「1 つだけ追うならどれか」を決める |
| 他のスキルがコンテキストを参照できない | ファイルの保存場所が不適切 | Step 8b で `.claude/marketing-context.md` に配置する |
| コンテキストが長すぎて読まれない | 情報の優先度が不明確 | 各セクションに「要約」行を追加し、詳細は折りたたみにする |
| 部門間でメッセージが一貫しない | ブランドボイスが共有されていない | Step 4 の成果物を全チームに展開し、NG 表現リストを周知する |
| 新メンバーがコンテキストを理解できない | 前提知識が必要な記述 | 用語集（Step 4d）を充実させ、略語は初出時に正式名称を併記する |
| 更新のたびに全体を作り直してしまう | 差分更新の仕組みがない | Step 8d の変更管理ルールを徹底し、変更箇所のみを更新する |
