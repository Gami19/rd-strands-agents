# Skill: Decision Framework（意思決定の構造化・記録・追跡）

> **判断を属人化させるな ── プロセスで再現し、記録で組織知にせよ**

## Instructions

### ワークフロー内の位置

```
story-map / distill / data-arch → [decision-framework] → review → drm
                                         ↓
                                    意思決定の構造化
                                    ・ADR 生成・管理
                                    ・トレードオフ分析
                                    ・DACI/RACI 整理
                                    ・リスク評価
                                    ・フィットネス関数設計
```

### 入力

| 入力 | 説明 | 例 |
|:---|:---|:---|
| 判断対象 | 何を決める必要があるか | 「REST と gRPC のどちらを採用するか」 |
| 制約条件 | 期限、予算、技術的制約 | 「3ヶ月以内に実装、チームに gRPC 経験なし」 |
| ステークホルダー情報 | 関係者とその役割 | 「CTO が最終承認、テックリードが推進」 |
| 既存の判断記録 | 過去の ADR や議事録 | 「ADR-001〜ADR-015 が docs/adr/ に存在」 |

### 出力

| 出力 | 形式 | 説明 |
|:---|:---|:---|
| ADR | Markdown | 7セクション構造の意思決定記録 |
| トレードオフ分析 | Markdown テーブル | 重み付きスコアリングの多軸比較マトリクス |
| DACI / RACI チャート | Markdown テーブル | ステークホルダーの役割マッピング |
| リスク評価 | Markdown テーブル | リスクマトリクス + 緩和策 |
| フィットネス関数仕様 | Markdown | 決定の遵守を自動検証する仕組みの設計書 |

## Step 1: 判断対象の特定とスコープ定義

判断が必要な事項を明確にし、プロセスの骨格を定める。

### 1a. アーキテクチャ的重要性の判定

全ての判断が構造化プロセスを必要とするわけではない。以下の基準で重要性を判定する。

| 基準 | 該当すれば構造化プロセスが必要 |
|:---|:---|
| **構造への影響** | システムの構造・コンポーネント構成を変更する |
| **非機能特性への影響** | 性能・可用性・セキュリティ等に影響を与える |
| **依存関係への影響** | コンポーネント間の依存構造を変更する |
| **外部インターフェースへの影響** | 外部システムとの接点を変更する |
| **技術選択** | 主要な技術・フレームワーク・プラットフォームを選定する |
| **コストへの影響** | 組織が定める閾値を超える投資が発生する |
| **他チームへの影響** | 他チームの作業や成果物に影響を与える |

> **原則**: 1つでも該当すれば、構造化プロセスで判断を記録すべき。

### 1b. スコープの明確化

| 観点 | 確認事項 |
|:---|:---|
| **What** | 具体的に何を決定するか（1つの ADR = 1つの決定） |
| **Why** | なぜ今この決定が必要か（駆動力・トリガー） |
| **When** | いつまでに決定する必要があるか（最終責任時点） |
| **Who** | 誰が決定に関与するか（Step 2 で詳細化） |
| **Scope** | 決定の影響範囲（システム、チーム、組織） |

### 1c. 最終責任時点（Last Responsible Moment）の判断

```
情報が十分に集まったか？
  ├── まだ不十分 → 判断を保留し、情報収集を継続
  │     ↓
  │   判断を遅延させるコストが発生し始めたか？
  │     ├── いいえ → 保留を継続
  │     └── はい → 今が最終責任時点 → 判断に進む
  │
  └── 十分 → 判断に進む
```

**チェックリスト**:
- [ ] 判断対象を具体的に特定した（What / Why / When を明記）
- [ ] アーキテクチャ的重要性の判定基準に照らして構造化の必要性を確認した
- [ ] 影響範囲（Scope）を明確にした
- [ ] 最終責任時点を評価した（早すぎず遅すぎず）
- [ ] 1つの ADR = 1つの決定の原則を確認した

## Step 2: ステークホルダー分析（DACI / RACI チャート生成）

### 2a. DACI モデル

意思決定向けの役割定義。**1つの決定に Driver と Approver は各1名のみ**。

| 役割 | 説明 | 人数制約 |
|:---|:---|:---|
| **D**river | 決定プロセスを推進する責任者。情報を集め、選択肢を整理し、決定を導く | 1名のみ |
| **A**pprover | 最終的な承認権限を持つ人。拒否権を持つ | 1名のみ |
| **C**ontributor | 知見・意見を提供する人。判断材料を提供するが決定権なし | 複数可 |
| **I**nformed | 決定結果を通知される人。決定プロセスには参加しない | 複数可 |

### 2b. RACI モデル

タスク・プロセス向けの役割定義。複数の活動にまたがる場合に使う。

| 役割 | 説明 |
|:---|:---|
| **R**esponsible | 実行する人（複数可） |
| **A**ccountable | 最終責任者（1名のみ） |
| **C**onsulted | 相談される人（双方向コミュニケーション） |
| **I**nformed | 報告される人（一方向コミュニケーション） |

### 2c. DACI vs RACI の選択基準

| 状況 | 推奨モデル |
|:---|:---|
| 単一の意思決定 | **DACI** |
| 複数のタスク・プロセスの役割分担 | **RACI** |
| 決定 + 実装の両方を管理 | **DACI**（決定）+ **RACI**（実装） |

**チェックリスト**:
- [ ] DACI と RACI のどちらが適切か判断した
- [ ] Driver（推進者）を1名に限定した
- [ ] Approver（承認者）を1名に限定した
- [ ] Contributor と Informed を混同していない（双方向 vs 一方向）
- [ ] 自己承認 vs エスカレーションの判断基準を確認した（Step 6 で適用）

## Step 3: 選択肢の列挙と評価軸の設定

### 3a. 選択肢の列挙

> **原則**: 最低 3 つの選択肢を列挙する。2 択は偽りの二項対立に陥りやすい。

| ガイドライン | 説明 |
|:---|:---|
| **最低 3 つ** | "Do nothing" も有効な選択肢として含める |
| **早期排除の禁止** | 直感で排除せず、全候補を評価テーブルに載せる |
| **ハイブリッド案の検討** | 複数候補の利点を組み合わせた折衷案を検討 |
| **段階的導入の検討** | 一括導入 vs 段階的移行も選択肢として含める |

### 3b. 評価軸（ディメンション）の設定

評価軸はビジネスコンテキストに応じて選定する。汎用的な評価軸の候補:

| カテゴリ | 評価軸 | 説明 |
|:---|:---|:---|
| **技術** | 性能 | レスポンスタイム、スループット |
| | スケーラビリティ | 負荷増大への対応力 |
| | 保守性 | 変更の容易さ、可読性 |
| | テスト容易性 | テストの書きやすさ、自動化適性 |
| | セキュリティ | 攻撃対象面、データ保護 |
| **組織** | 学習コスト | チームが習得にかかる時間 |
| | 採用可能性 | スキルを持つ人材の採用しやすさ |
| | チーム自律性 | チームが独立して意思決定・運用できるか |
| **ビジネス** | 初期コスト | 導入時の費用 |
| | 運用コスト | 継続的な費用 |
| | 市場投入速度 | 実装完了までの期間 |
| | リスク | 失敗時の影響度 |

### 3c. 重み付けの設定

ビジネス優先度に基づき、各評価軸に重み（1-5）を割り当てる。

> **原則**: 全軸を同一重みにしない。ビジネス上の優先度の差を明示する。重みの合計は固定しない（各軸が独立して重要度を持つ）。

**チェックリスト**:
- [ ] 3 つ以上の選択肢を列挙した（"Do nothing" を含む）
- [ ] 早期排除をせず、全候補を評価テーブルに載せた
- [ ] 評価軸を 5-8 個に絞った（多すぎると意味が薄まる）
- [ ] ビジネス優先度に基づく重みを設定した
- [ ] ハイブリッド案・段階的導入案を検討した

## Step 4: トレードオフ分析（重み付きマトリクス）

### 4a. マトリクスの構築

[tradeoff-matrix.md](references/tradeoff-matrix.md) のテンプレートを使用して分析する。

基本構造:

| 評価軸 | 重み (W) | 選択肢A (S) | A Score (W*S) | 選択肢B (S) | B Score (W*S) | 選択肢C (S) | C Score (W*S) |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| 性能 | 5 | 4 | 20 | 3 | 15 | 5 | 25 |
| 保守性 | 4 | 3 | 12 | 5 | 20 | 2 | 8 |
| 初期コスト | 3 | 5 | 15 | 3 | 9 | 2 | 6 |
| ... | | | | | | | |
| **合計** | | | **47** | | **44** | | **39** |

- **S (Score)**: 各軸での評価（1-5、5が最高）
- **W (Weight)**: ビジネス優先度に基づく重み（1-5）
- **W*S**: 重み付きスコア

### 4b. 感度分析

最終スコアが僅差（10%以内）の場合、重みの変動による結果の安定性を確認する。

```
最も重みの高い軸の重みを ±1 変動させる
  → 結論が変わるか？
    ├── 変わらない → 結論は堅牢
    └── 変わる → 追加の定性分析が必要（判断材料が不足している可能性）
```

### 4c. 定性的トレードオフの補足

スコアリングで表現しきれない質的な判断を記録する:

- **不可逆性**: この決定を後から覆せるか（覆せないなら慎重に）
- **段階的適用**: 小さく試してから拡大できるか
- **組織的準備度**: チームがこの選択肢を受け入れる準備があるか

**チェックリスト**:
- [ ] 全選択肢を全評価軸でスコアリングした
- [ ] 重み付けがビジネス優先度を正確に反映している
- [ ] 僅差の場合は感度分析を実施した
- [ ] 定性的トレードオフ（不可逆性、段階的適用、組織的準備度）を補足した
- [ ] スコアだけでなく、スコアの根拠を各セルに記録した

## Step 5: リスク評価（リスクストーミング + マトリクス）

### 5a. リスクマトリクスの構築

2 次元（影響度 x 発生確率）でリスクを定量化する。

| | 低確率 (1) | 中確率 (2) | 高確率 (3) |
|:---|:---:|:---:|:---:|
| **高影響 (3)** | 3 (中) | 6 (高) | 9 (高) |
| **中影響 (2)** | 2 (低) | 4 (中) | 6 (高) |
| **低影響 (1)** | 1 (低) | 2 (低) | 3 (中) |

**リスクレベル**:
- **1-2 (低)**: 監視のみ。緩和策は不要
- **3-4 (中)**: 緩和策を策定し、定期的にモニタリング
- **6-9 (高)**: 緩和策を即時策定し、判断への影響を再評価

> **原則**: 影響度を先に評価し、確率を後に評価する。確率が不明な場合は高 (3) を仮定する。

### 5b. リスクストーミング（協調的リスク識別）

チームで実施するリスク識別手法。3つのフェーズで進行する。

| Phase | 内容 | 時間 |
|:---|:---|:---:|
| **1. 個別識別** | 各参加者が独立にリスクを付箋に書き出す（赤=高/黄=中/緑=低） | 5-10 分 |
| **2. コンセンサス** | チームでリスクの認識を共有し、合意レベルを確定 | 10-15 分 |
| **3. 緩和策** | 高リスク（6-9）の項目に緩和策・責任者・期限を策定 | 10-15 分 |

詳細な手順は [risk-assessment-guide.md](references/risk-assessment-guide.md) を参照。

**チェックリスト**:
- [ ] 全選択肢に対してリスクを識別した
- [ ] 影響度を先に評価し、確率を後に評価した
- [ ] 確率不明のリスクは高 (3) を仮定した
- [ ] 高リスク（6-9）の項目に緩和策を策定した
- [ ] リスクストーミングの3フェーズを実施した（チーム判断の場合）

## Step 6: ADR の作成と承認判定

### 6a. ADR の生成

[adr-template.md](references/adr-template.md) のテンプレートを使用して ADR を生成する。

7 セクション構造:

| セクション | 目的 | 記述のポイント |
|:---|:---|:---|
| **Title** | 通番 + 短い説明句 | 具体的かつ曖昧さのないタイトル |
| **Status** | 現在のステータス | Proposed → Accepted → Superseded のライフサイクル管理 |
| **Context** | 駆動力と状況 | 何がこの決定を迫っているか。選択肢の簡潔な列挙 |
| **Decision** | 決定内容と根拠 | 断定的な表現（"We will use..."）。**Why が最も重要** |
| **Consequences** | 影響（正・負両面） | トレードオフを明示。リスクと緩和策も記載 |
| **Compliance** | 遵守の確認方法 | 手動レビュー or 自動フィットネス関数 |
| **Notes** | メタデータ | 著者、承認者、日付、関連 ADR へのリンク |

> **Second Law of Software Architecture**: "Why is more important than how." -- ADR の Decision セクションでは、**何を決めたか**よりも**なぜそう決めたか**を重点的に記述する。

### 6b. ADR のライフサイクル管理

```
[RFC] → [Proposed] → [Accepted] → [Superseded by ADR-NNN]
                  └→ [Rejected]
```

**管理ルール**: ADR は絶対に削除しない（Superseded で履歴保持）。Superseded ADR には後継リンクを付与。RFC には期限を明記。四半期ごとに Accepted ADR を棚卸し。詳細は [adr-template.md](references/adr-template.md) を参照。

### 6c. 自己承認 vs エスカレーションの判断

| 判断基準 | 自己承認可能 | エスカレーション必要 |
|:---|:---|:---|
| **コスト** | 組織の閾値以内 | 閾値超過 |
| **他チームへの影響** | 影響なし | 他チームに影響あり |
| **セキュリティ** | セキュリティ影響なし | セキュリティ影響あり |
| **可逆性** | 容易に覆せる | 覆すのが困難または不可能 |

### 6d. ADR の保管場所

コードリポジトリ内（`docs/adr/`）を強く推奨。Wiki や共有ドライブはバージョン管理の欠如により非推奨。

**チェックリスト**:
- [ ] ADR を 7 セクション構造で作成した
- [ ] Decision セクションで「Why」を重点的に記述した
- [ ] 断定的な表現（"We will use..."）を使用した
- [ ] Consequences にポジティブ・ネガティブ両面を記載した
- [ ] 自己承認 vs エスカレーションの基準を適用した
- [ ] 保管場所を決定し、ADR を配置した

## Step 7: フィットネス関数の設計

決定が実装後も遵守されていることを自動的に検証する仕組みを設計する。

### 7a. フィットネス関数の分類

| 分類軸 | 種類 | 説明 | 例 |
|:---|:---|:---|:---|
| **スコープ** | 原子的（Atomic） | 単一の特性を検証 | 循環依存の検出 |
| | 全体的（Holistic） | 複数の特性を複合的に検証 | セキュリティ制約下での性能 |
| **実行タイミング** | トリガー型（Triggered） | 特定のイベントで実行 | CI/CD パイプライン、PR レビュー |
| | 継続型（Continual） | 常時実行 | 本番モニタリング、SLO 監視 |

### 7b. 典型的なフィットネス関数の例

| 決定の種類 | フィットネス関数 | 実装手段 |
|:---|:---|:---|
| レイヤー分離の決定 | レイヤー間の不正な依存がないか | ArchUnit / Dependency-Cruiser |
| パフォーマンス SLO | レスポンスタイムが閾値以内か | k6 / Gatling + CI/CD |
| セキュリティ方針 | 既知の脆弱性がないか | OWASP ZAP / Trivy + CI/CD |
| API 互換性の決定 | 破壊的変更がないか | OpenAPI diff / Contract testing |
| コスト上限の決定 | クラウドコストが月額閾値以内か | AWS Cost Explorer / Infracost |

**チェックリスト**:
- [ ] 高リスクまたは高影響の ADR にフィットネス関数を設計した
- [ ] フィットネス関数の分類（スコープ × 実行タイミング）を明確にした
- [ ] 合格基準を具体的な数値で定義した
- [ ] 実装手段を決定した（設計段階で止めない）

## Step 8: セルフレビューと出力

### 8a. 意思決定アンチパターンの検出

生成した成果物を以下のアンチパターンに照らして検証する。

| アンチパターン | 症状 | 矯正策 |
|:---|:---|:---|
| **Covering Your Assets** | 決定を避け続ける。分析が無限に続く | 最終責任時点を設定し、期限で決定を強制する |
| **Groundhog Day** | 同じ決定が何度も蒸し返される | ADR を作成し、過去の Context / Decision / Consequences を参照可能にする |
| **Email-Driven Architecture** | 決定がメールに埋もれ、発見できない | ADR をリポジトリに保管。メールは通知のみ |
| **Analysis Paralysis** | 完璧な情報を待ち続けて行動できない | 「70% の情報で判断する」ルールを適用 |
| **Groupthink** | 集団の同調圧力で異論が出ない | DACI で Contributor の意見を個別収集してから議論 |
| **HiPPO** | 最も高い地位の人の意見が無条件で通る | トレードオフマトリクスの定量評価で議論の基盤を作る |

### 8b. 成果物の構成

```markdown

### 1. エグゼクティブサマリー
（決定内容と根拠の要約）

（Step 1 の結果）

（Step 2 の結果）

（Step 3 の結果）

（Step 4 の結果）

（Step 5 の結果）

（Step 6 の結果）

（実装計画、レビュー予定、モニタリング開始日）
```

### 8c. 部分実行ガイド

全ステップの完全実行が不要な場合、以下のショートカットを使用する。

| ユーザーの要求 | 実行ステップ |
|:---|:---|
| 「ADR を書いて」 | Step 1 → Step 6 |
| 「トレードオフを分析して」 | Step 1 → Step 3 → Step 4 |
| 「リスク評価をして」 | Step 1 → Step 5 |
| 「DACI を作って」 | Step 1 → Step 2 |
| 「フィットネス関数を設計して」 | Step 1 → Step 7 |
| 「ADR を整理・棚卸しして」 | Step 6b のライフサイクル管理 |

### Decision Framework Slop 防止チェック（Distributional Convergence 対策）

意思決定フレームワークの均質化は最も危険な Slop の一つである。判断プロセスが汎用テンプレートに収束すると、異なる文脈の決定に同じ構造・同じ分析・同じ重み付けが適用され、「構造化した」という安心感だけで実質的な判断の質が向上しない。

| # | Decision Framework Slop パターン | 検出方法 | 対策 |
|:---|:---|:---|:---|
| DF-1 | **ADR テンプレートの画一的適用** | クラウド選定も認証方式もDB選定も同じ ADR 構造 | ドメイン固有 ADR テンプレート（adr-template-library.md）を使用。判断の種類に応じて評価軸を変える |
| DF-2 | **DACI/RACI の機械的割当** | 組織図から役職でそのまま D/A/C/I を割当 | 意思決定の影響範囲と専門性から逆算。同じ人が D と A を兼ねる危険を検出 |
| DF-3 | **2x2 マトリクスの軸固定** | トレードオフ分析が毎回同じ「コスト × パフォーマンス」軸 | ビジネスの最重要特性から軸を導出。「開発速度 × 保守性」「柔軟性 × 一貫性」等の文脈依存軸 |
| DF-4 | **リスク評価の均一カテゴリ** | 毎回「技術リスク・スケジュールリスク・コストリスク」の 3 分類 | ドメイン固有のリスクカテゴリ（規制リスク・データ品質リスク・ベンダーリスク等）を追加 |
| DF-5 | **教科書的フィットネス関数** | 全判断に同じ KPI テンプレートを適用 | 判断の成功基準を具体的なビジネスメトリクスから定義。自動検証可能な形式に落とし込む |
| DF-6 | **重み付けの均等配分** | 5 つの評価軸に各 20% を配分 | ステークホルダーと対話し、最重要軸を明確に。重み配分の根拠を記録する |
| DF-7 | **選択肢が常に 3 つ** | A案・B案・ハイブリッド案の定型構成 | "Do Nothing" を必ず含める。選択肢の数は判断空間の広さで決まる。2 つでも 5 つでもよい |
| DF-8 | **振り返りの形骸化** | 棚卸しで全 ADR に「有効」とスタンプを押すだけ | ADR ごとに「Context が変化したか」「Decision の仮定は正しかったか」を具体的に検証 |

> **核心原則**: **フレームワークは判断を代行しない、判断の品質を上げるための足場にすぎない** — テンプレートを埋めることが目的ではなく、テンプレートを通じて「見落としていた観点」を発見することが目的。「別の判断に同じ分析結果を流用しても違和感がないか？」→ 違和感がないなら Decision Framework Slop である。

**最終チェックリスト**:
- [ ] 8 ステップ（対象特定 → DACI → 選択肢 → トレードオフ → リスク → ADR → FF → レビュー）を完了した
- [ ] 意思決定アンチパターンの検出を実施した
- [ ] ADR の Decision セクションに Why が十分に記述されている
- [ ] 高リスク項目に緩和策がある
- [ ] フィットネス関数が設計されている（高影響の決定のみ）
- [ ] 成果物がステークホルダーに説明可能な形式になっている
- [ ] Decision Framework Slop チェック（DF-1〜DF-8）に 2 つ以上該当しない
- [ ] 出力がドメイン固有の要素を含み、汎用テンプレートに見えない

## Examples

### Example 1: 技術選定の ADR 生成（REST vs gRPC）

```
「マイクロサービス間の通信プロトコルを REST か gRPC か決めたい。ADR を書いて」

→ Step 1 で判断対象を特定（サービス間通信プロトコルの選定、3 ヶ月以内に決定）
→ Step 6 に直行（ADR 単体の要求）
   Context: 10 サービス間の通信、レイテンシ要件 < 50ms、チームは REST 経験あり
   Decision: "We will use gRPC for internal service-to-service communication"
   Consequences: 正（低レイテンシ、型安全）、負（学習コスト、デバッグ困難）
→ 成果物: ADR-NNN（7 セクション構造）
```

### Example 2: 認証方式のトレードオフ分析

```
「新規 SaaS の認証方式を OAuth 2.0 / SAML / API Key から選びたい。
 トレードオフを分析して」

→ Step 1 で判断対象を特定（B2B SaaS の認証方式選定）
→ Step 3 で 3 候補 + "ハイブリッド（OAuth + API Key）" を列挙
   評価軸: セキュリティ(5), UX(4), 実装コスト(3), 運用負荷(3), エンプラ互換(4)
→ Step 4 でトレードオフマトリクス構築
   OAuth 2.0: 総合 78、SAML: 総合 72、API Key: 総合 51、ハイブリッド: 総合 82
→ Step 4b で感度分析（セキュリティの重みを ±1 → 結論変わらず、堅牢）
→ 成果物: トレードオフ分析マトリクス + 推奨案（ハイブリッド）
```

### Example 3: クラウド移行のリスク評価

```
「オンプレの基幹システムを AWS に移行する計画のリスクを評価して」

→ Step 1 で判断対象を特定（基幹システムの AWS 移行、影響範囲: 全社）
→ Step 5 でリスクストーミング実施
   Phase 1: データ移行の完全性(高影響×中確率=6)、ダウンタイム(高×高=9)、
            セキュリティ境界の変更(高×低=3)、コスト超過(中×中=4)
   Phase 2: チームでコンセンサス形成
   Phase 3: 高リスク(9) のダウンタイムに段階移行戦略を策定
→ 成果物: リスク評価テーブル + 緩和策一覧 + 残存リスクの許容判断
```

### Example 4: 大型案件の DACI チャート生成

```
「5,000 万円の DX 案件で関係者が多い。ステークホルダーを整理して DACI を作って」

→ Step 1 で判断対象を特定（DX 案件の技術選定・アーキテクチャ決定）
→ Step 2 で DACI モデルを選択（単一の大型意思決定）
   Driver: テックリード田中（技術選定を推進）
   Approver: CTO 佐藤（5,000 万円 → 組織閾値超過のためエスカレーション）
   Contributor: インフラチーム、セキュリティチーム、ビジネスオーナー
   Informed: 経営企画、法務
→ Step 2 で RACI も補助的に作成（設計、実装、テスト、デプロイの各フェーズ）
→ 成果物: DACI チャート + RACI マトリクス
```

### Example 5: フィットネス関数の設計（SLO 準拠）

```
「"API レスポンスは p99 で 200ms 以内" という SLO を決めた。
 これを継続的に守る仕組みを設計して」

→ Step 1 で判断対象を特定（SLO の自動検証メカニズム）
→ Step 7 でフィットネス関数を設計
   対象 ADR: ADR-027（API パフォーマンス SLO）
   分類: 全体的 × 継続型 + 全体的 × トリガー型
   FF-1: CI/CD パイプラインで k6 負荷テスト（p99 < 200ms で合格）
   FF-2: 本番で Datadog APM 継続監視（p99 > 200ms で PagerDuty アラート）
   FF-3: 週次レポートで SLO 達成率の可視化（目標 99.9%）
→ 成果物: フィットネス関数設計書 3 件 + CI/CD 組み込み手順
```

### Example 6: ADR ライフサイクルの棚卸し

```
「リポジトリに ADR が 30 本あるが、古い決定がそのまま残っている。
 棚卸しして整理したい」

→ Step 6b のライフサイクル管理に直行
→ 全 30 ADR を読み込み、ステータスを評価:
   Accepted のまま有効: 18 本（変更不要）
   実質的に Superseded: 7 本（後続 ADR との紐付けが欠落 → 修正）
   コンテキストが変化して再評価が必要: 3 本（RFC ステータスに戻す）
   Rejected 相当だが未記録: 2 本（Rejected ステータスに更新）
→ Superseded ADR に後継 ADR へのリンクを追加
→ 成果物: ADR 棚卸しレポート + 修正済み ADR 7 本 + RFC 再開通知 3 本
```

### Example 7: 意思決定アンチパターンの検出と矯正

```
「プロジェクトで技術選定が3ヶ月間決まらない。何が問題か診断して」

→ Step 8a のアンチパターン検出を実施
→ 診断結果:
   ・Covering Your Assets: アーキテクトが責任回避 → 最終責任時点を設定
   ・Analysis Paralysis: PoC が 5 回実施されても結論が出ない → 70% ルール適用
   ・HiPPO: 部長の一言で方針が毎週変わる → DACI チャートで Approver を明確化
→ 矯正策:
   1. Step 1 で最終責任時点を「2 週間後」に設定
   2. Step 2 で DACI チャートを作成し、Decision Maker を 1 名に限定
   3. Step 4 でトレードオフマトリクスを構築し、定量的な根拠を確立
→ 成果物: アンチパターン診断レポート + 矯正アクションプラン
```

## Troubleshooting

| 問題 | 原因 | 解決策 |
|:---|:---|:---|
| 判断を先送りにし続ける（Covering Your Assets） | 決定の責任を負いたくない、失敗への恐怖 | 最終責任時点を明確に設定する。「決めないこと」のコストを可視化する |
| 同じ決定が何度も蒸し返される（Groundhog Day） | 過去の決定が文書化されていない、または発見できない | ADR を作成し、リポジトリの `docs/adr/` に集約する。過去の Context と Decision を参照可能にする |
| 決定がメールに埋もれる（Email-Driven Architecture） | 意思決定の正本がない | ADR を正本とし、メールは通知のみに限定。ADR へのリンクをメールに含める |
| トレードオフ分析が主観的になる | 定性的な「良い/悪い」で評価 | 重み付きスコアリングで客観化する。各スコアの根拠を記録する |
| 全ステークホルダーの合意を求めて前に進めない | DACI が未設定、全員が同等の発言権を持つと認識 | DACI チャートで Driver と Approver を各1名に限定する。Contributor は意見提供のみ |
| ADR が書かれた後に放置される | ライフサイクル管理のプロセスが不在 | 四半期ごとの ADR 棚卸しを実施。Superseded の追跡を運用ルールに組み込む |
| リスク評価で確率を楽観的に見積もる | 正常性バイアス | 「確率が不明なら高 (3) を仮定する」ルールを厳守。影響度を先に評価する |
| フィットネス関数が設計段階で止まる | 実装手段を決めていない | Step 7 で具体的なツール・テストフレームワークまで特定してから完了とする |
| DACI と RACI を混同する | 目的の違いを理解していない | DACI は意思決定向け、RACI はタスク・プロセス向け。Step 2c の選択基準を参照 |
| 選択肢が 2 つしかない | 偽りの二項対立、"Do nothing" の未検討 | 最低 3 つの選択肢を列挙する。"Do nothing" とハイブリッド案を必ず検討 |
| 判断に使った情報が古い | ADR の Context が更新されていない | 定期棚卸しで Context の有効性を再確認。環境変化があれば RFC に戻す |
| 汎用フレームワークをドメイン固有の判断に無理に適用する | decision-framework のスコープ外の使い方 | アーキテクチャスタイル選定は software-architecture、データ基盤設計は data-arch に委譲する |
